# 畫布系統概念篇（固定座標、縮放與滑鼠換算）

所屬章節：[03-畫布系統](./README.md)

## 這篇要解決什麼

這篇先不急著看 `PPTist` 原始碼，而是先把「畫布系統到底在做什麼」講清楚。

畫布系統的核心，其實只有三件事：

1. 先定義一張固定大小的**邏輯畫布**
2. 再把這張邏輯畫布**縮放後顯示到畫面上**
3. 最後把滑鼠操作拿到的**畫面座標**，換算回**邏輯座標**

只要先把這三件事分開理解，後面再讀拖曳、縮放、旋轉、縮圖、選取框等功能時，就不容易全部混在一起。

---

## 先統一這篇的名詞

為了避免混淆，這篇只固定使用下面四個詞：

* **邏輯畫布**：系統內部那張固定尺寸的投影片
* **畫面顯示**：使用者目前在螢幕上看到的結果
* **邏輯座標**：元素資料儲存的位置與尺寸
* **畫面座標**：滑鼠事件、拖曳位移在螢幕上量到的距離

你在其他地方可能會看到「投影片世界」、「資料世界」、「虛擬投影片」、「螢幕座標」這些說法。
這篇為了保持一致，後面全部統一叫做：

* 投影片世界 / 資料世界 / 虛擬投影片 → **邏輯畫布**
* 螢幕座標 / 畫面上的距離 → **畫面座標**

## 30 秒複習入口

* 畫布系統不是直接拿 DOM 的實際尺寸當座標系，而是先定義一張固定大小的**邏輯畫布**
* 預設邏輯寬度是 `1000`，高度是 `1000 * 0.5625 = 562.5`
* 畫面上看到的大或小，只是這張邏輯畫布被整體縮放後的結果
* 滑鼠操作如果要回寫到資料，核心公式就是：

```ts
邏輯座標 = 畫面座標 / canvasScale
```

## 先用最簡單的方式理解

先不要想元件、hooks、store，也先不要想 `PPTist` 原始碼。
先只看這條最基本的流程：

```text
元素資料
  -> 放進固定大小的邏輯畫布
  -> 再把整張邏輯畫布縮放到畫面上
  -> 使用者在畫面上操作
  -> 程式把畫面座標換回邏輯座標
```

你可以把它想成這個模型：

* 系統心中永遠有一張固定大小的投影片
* 這張投影片預設是 `1000 x 562.5`
* 元素資料永遠記在這張投影片裡
* 使用者目前看到的大或小，只是這張投影片被整體縮放後的結果

一句話先記住：

> **資料永遠活在邏輯畫布裡，畫面只是邏輯畫布的縮放結果。**

## 例子 1：資料怎麼看

如果有一個文字框資料是：

```ts
{ left: 100, top: 50, width: 200, height: 80 }
```

先不要把它理解成：

> 現在畫面上看到的 `100px`、`50px`、`200px`、`80px`

而要把它理解成：

> 這個元素在**邏輯畫布**中的位置與尺寸

它表示的是：

* 左上角在邏輯畫布的 `(100, 50)`
* 寬度是 `200`
* 高度是 `80`

所以這筆資料描述的是：

> 元素在邏輯畫布中的幾何資訊

而不是：

> 元素目前在螢幕上看起來幾個像素

也就是說，即使畫面之後被縮小或放大，資料本身仍然不變：

```ts
{ left: 100, top: 50, width: 200, height: 80 }
```

變的是**怎麼顯示**它，不是**怎麼儲存**它。

## 例子 2：畫面怎麼看

假設現在：

```ts
canvasScale = 0.5
```

代表整張邏輯畫布在畫面上被縮成一半。

所以你在螢幕上看到的結果會像是：

* 左邊距看起來像 `50px`
* 上邊距看起來像 `25px`
* 寬看起來像 `100px`
* 高看起來像 `40px`

但資料本身沒有變，仍然是：

```ts
{ left: 100, top: 50, width: 200, height: 80 }
```

這裡要特別分清楚：

* **邏輯座標**：資料真正存的值
* **畫面顯示**：縮放後看起來的大小與位置

對照 demo：

* [Demo 02：畫面怎麼看](./demos/02-screen-view.html)

---

### 為什麼 Demo 02 要有兩層：外層顯示容器 + 內層邏輯畫布

Demo 02 裡有兩層：

* `screenFrame`：外層顯示容器，代表目前在畫面上實際佔多少空間
* `screenViewport`：內層邏輯畫布，代表固定尺寸的投影片本體

這兩個名稱是這篇教學用的概念名，**不是 `PPTist` 原始碼中的正式命名**。

對照到 `PPTist` 的實作，可以先這樣理解：

* `screenFrame` → `viewport-wrapper`
* `screenViewport` → `viewport`

也就是說：

* 外層負責「實際顯示多大、放在哪裡」
* 內層負責「保留固定邏輯座標系」

其中內層的 `screenViewport` 固定維持：

```ts
width = 1000
height = 562.5
```

然後再用：

```ts
screenViewport.style.transform = `scale(${scale})`
```

把整張邏輯畫布縮小或放大。

---

### 為什麼只做 `scale()` 還不夠

這裡最容易混淆的地方是：

> `transform: scale()` 會改變元素**看起來的大小**，但不會改變它在 CSS layout 裡原本的尺寸語意

也就是說，當 `scale = 0.5` 時：

* 視覺上它看起來像 `500 x 281.25`
* 但在 layout 的語意裡，它本質上還是 `1000 x 562.5`

這也是為什麼外層還要額外設定：

```ts
screenFrame.style.width = `${1000 * scale}px`
screenFrame.style.height = `${562.5 * scale}px`
```

因為如果只做 `scale()`，你只是讓它**看起來變小**；
但外層 layout 並不會自動把它當成「真的只有這麼大」。

所以這裡的分工可以記成：

* `screenViewport`：保留固定邏輯尺寸
* `transform: scale()`：負責視覺縮放
* `screenFrame`：承接縮放後真正的顯示尺寸與版面佔位

一句話整理：

```text
scale() 解決的是「看起來多大」，frame 寬高解決的是「在 layout 裡實際佔多少空間」。
```

## 例子 3：滑鼠為什麼要除以 `canvasScale`

假設現在：

```ts
canvasScale = 0.5
```

代表整張邏輯畫布在畫面上被縮成一半。

這時候你在螢幕上量到的距離，其實都是**縮小後的畫面距離**。

例如你用滑鼠點到某個位置，量到它距離左上角是 `50px`：

* 這個 `50` 是**畫面座標**
* 但資料要寫回的是**邏輯座標**

因為畫面已經縮成一半，所以它在邏輯畫布中的對應位置其實是：

```ts
邏輯座標 = 50 / 0.5 = 100
```

也就是說：

* 畫面上看到的是 `50`
* 邏輯畫布裡對應的是 `100`

所以滑鼠事件拿到的是畫面座標，
但元素資料需要的是邏輯座標。

中間一定要做這個換算：

```ts
邏輯座標 = 畫面座標 / canvasScale
```

* [Demo 03：滑鼠為什麼要除以 canvasScale](./demos/03-mouse-scale.html)

## 範例 1：先定義固定的邏輯畫布與元素資料

先不要碰 DOM，先只看資料長什麼樣：

```ts
type ElementBox = {
  left: number
  top: number
  width: number
  height: number
  text: string
}

const viewportSize = 1000
const viewportRatio = 9 / 16

const slide = {
  width: viewportSize,
  height: viewportSize * viewportRatio,
  elements: [
    { left: 100, top: 50, width: 220, height: 80, text: '標題' },
    { left: 360, top: 180, width: 260, height: 120, text: '內容區塊' },
  ] satisfies ElementBox[],
}
```

這段資料的重點是：

* 邏輯畫布固定是 `1000 x 562.5`
* 元素永遠記在這張邏輯畫布裡
* 不管畫面實際顯示多大，資料本身都不變

## 範例 2：把邏輯畫布縮放後顯示到畫面

先記住這一句：

> **邏輯畫布的尺寸不改，改的是它在畫面上看起來多大。**

假設系統內部的邏輯畫布寬度固定是 `1000`，
但現在你想讓它在螢幕上只顯示成 `600px` 寬。

那代表的不是：

> 把邏輯畫布改成 600

而是：

> 仍然保留 `1000` 這套邏輯座標系，只是在畫面上把整張畫布縮小成 `600px`

所以縮放比是：

```ts
const logicalWidth = 1000
const actualWidth = 600
const scale = actualWidth / logicalWidth // 0.6
```

這裡的 `0.6` 意思是：

> 邏輯畫布原本寬 1000，現在視覺上要縮成 600，所以整張畫布要乘上 0.6。

---

### 最小結構

```html
<div class="frame">
  <div class="viewport" id="viewport">
    <div class="box title">標題</div>
    <div class="box content">內容區塊</div>
  </div>
</div>
```

```css
.frame {
  width: 600px;
  height: 337.5px;
  position: relative;
  overflow: hidden;
  border: 1px solid #999;
}

.viewport {
  width: 1000px;
  height: 562.5px;
  position: relative;
  transform: scale(0.6);
  transform-origin: 0 0;
}

.box {
  position: absolute;
  border: 1px solid #333;
  background: #f5f5f5;
}
```

---

### 這兩層各自負責什麼

這裡一定要分成兩層看：

* `.viewport`：固定的邏輯畫布，永遠維持 `1000 x 562.5`
* `.frame`：畫面上的實際顯示區域，現在是 `600 x 337.5`

分工可以直接記成：

* `.viewport` 決定「資料世界多大」
* `.frame` 決定「畫面上看起來多大」

也就是說：

* 你寫元素資料時，仍然是照 `1000 x 562.5` 這套邏輯座標寫
* 真正讓它看起來變小的，是 `.viewport` 的 `scale(0.6)`

---

### 為什麼不能直接把 `.viewport` 設成 `600 x 337.5`

關鍵不是「能不能顯示成 600」，而是：

> **你要不要讓 600 變成資料本身的座標基準。**

這兩種做法看起來畫面都能顯示，但背後的意思完全不同。

---

#### 做法 A：保留固定邏輯畫布，再用 `scale()` 顯示成 600

假設系統內部永遠固定一張：

```ts
viewport = 1000 x 562.5
```

元素資料是：

```ts
{ left: 100, top: 50, width: 200, height: 80 }
```

這表示：

* 元素在邏輯畫布中的位置是 `(100, 50)`
* 寬 `200`
* 高 `80`

如果現在只是把整張畫布縮成 `0.6` 倍顯示到螢幕上，那麼它在畫面上**看起來**會變成：

* 左上角像在 `(60, 30)`
* 寬像 `120`
* 高像 `48`

但要注意：

> **變的是畫面，不是資料。**

也就是說，資料還是：

```ts
{ left: 100, top: 50, width: 200, height: 80 }
```

---

#### 做法 B：直接把 `.viewport` 設成 `600 x 337.5`

如果你直接把 `.viewport` 本身改成 `600 x 337.5`，那問題就在於：

> 這時候元素的座標，到底該用哪一套數字？

原本的資料是：

```ts
{ left: 100, top: 50, width: 200, height: 80 }
```

這組數字本來是建立在：

```ts
viewport = 1000 x 562.5
```

這套邏輯座標系上。

但如果你現在直接把 `.viewport` 變成 `600 x 337.5`，那讀者很容易直覺地認為：

* `left: 100` 就是 600 這個畫面裡的 100
* `width: 200` 就是 600 這個畫面裡的 200

這樣一來，**資料座標就開始跟當前顯示尺寸綁在一起了**。

---

#### 真正麻煩的地方：畫面尺寸一變，資料基準也跟著亂掉

假設今天你用 `600px` 顯示畫布，明天改成 `800px`。

如果你採用「固定邏輯畫布 + scale」：

* 資料永遠還是

  ```ts
  { left: 100, top: 50, width: 200, height: 80 }
  ```
* 只是顯示倍率不同而已

但如果你採用「直接把 `.viewport` 改成目前顯示尺寸」：

* 在 `600px` 顯示時，你可能會把它理解成 `(100, 50)`
* 到了 `800px` 顯示時，這組數字的意義又變了
* 之後拖曳、縮放、滑鼠點擊時，就會一直碰到：

  * 現在拿到的是哪個尺寸下的座標？
  * 資料要不要跟著重算？
  * 這次改成 600，下次改成 800，是不是又要重新換算？

也就是說：

> **只要顯示尺寸一改，資料座標的解讀方式就可能一起變。**

這就是為什麼不能直接讓 `.viewport` 的尺寸跟著畫面跑。

---

#### 固定邏輯畫布的真正好處

固定邏輯畫布，不是因為它比較帥，而是因為它能把兩件事硬切開：

##### 1. 資料永遠只管邏輯世界

例如：

```ts
{ left: 100, top: 50, width: 200, height: 80 }
```

這組資料永遠只代表：

* 在邏輯畫布裡的 `(100, 50)`
* 寬 `200`
* 高 `80`

不管現在畫面顯示成 600、800、1200，這筆資料都不用改。

##### 2. 顯示永遠只管畫面世界

你現在要畫面顯示成多大，是顯示層的事。

例如：

* 顯示成 600 → `scale = 0.6`
* 顯示成 800 → `scale = 0.8`

資料不動，只改顯示倍率。

---

#### 為什麼這對互動很重要

因為拖曳、縮放、滑鼠點擊，最後都要回到同一套座標系。

例如你在畫面上點到：

```ts
screenX = 60
screenY = 30
```

如果現在 `scale = 0.6`，那就換回：

```ts
logicalX = 60 / 0.6 = 100
logicalY = 30 / 0.6 = 50
```

這樣程式永遠知道：

> 不管畫面怎麼縮，最後都回到 `1000 x 562.5` 那張邏輯畫布裡處理。

如果 `.viewport` 本身就跟著顯示尺寸一直改，那你每次互動都要先搞清楚：

> 我現在拿到的，是哪個尺寸下的座標？

整個系統會變得很難穩定。

## 範例 3：用程式產生元素，而不是手寫 CSS

上面的元素位置如果改成根據資料渲染，會更接近實際專案思路：

```ts
const viewport = document.querySelector('#viewport') as HTMLDivElement

slide.elements.forEach(element => {
  const node = document.createElement('div')
  node.className = 'box'
  node.textContent = element.text

  node.style.left = `${element.left}px`
  node.style.top = `${element.top}px`
  node.style.width = `${element.width}px`
  node.style.height = `${element.height}px`

  viewport.appendChild(node)
})
```

這段最重要的觀察是：

* 你直接渲染的是**邏輯座標**
* 元素自己不知道畫面現在是不是縮成 `0.6`
* 變大或變小這件事，是外層邏輯畫布在做，不是元素自己在改資料

## 範例 4：把滑鼠點擊位置換回邏輯座標

假設現在這張投影片在畫面上縮成 `0.6` 倍。
你在螢幕上點到距離左邊 `60px`、距離上方 `30px` 的位置。

要把這個位置換回邏輯畫布中的座標，就要除以縮放比：

```ts
const scale = 0.6

const screenX = 60
const screenY = 30

const logicalX = screenX / scale // 100
const logicalY = screenY / scale // 50
```

這就是畫布系統裡最常出現的核心公式：

```ts
logical = screen / scale
```

---

## 範例 5：新增元素時，資料應該記哪個座標

假設你點擊畫面上的 `(240, 120)`，而目前縮放比是 `0.6`：

```ts
const scale = 0.6

const newElement = {
  left: 240 / scale,   // 400
  top: 120 / scale,    // 200
  width: 200,
  height: 60,
  text: '新文字框',
}
```

注意，新增進資料時，應該存的是：

```ts
left: 400
top: 200
```

而不是畫面上的：

```ts
left: 240
top: 120
```

因為資料對應的是**邏輯畫布**，不是當下縮放過的畫面。

* 資料：永遠使用邏輯座標
* 顯示：整張邏輯畫布一起縮放
* 互動：先拿到畫面座標，再換回邏輯座標

---

## 一張圖先看懂

```text
資料層
  ├─ viewportSize = 1000
  ├─ viewportRatio = 0.5625
  └─ elements: left / top / width / height ...

顯示層
  ├─ wrapper / frame   決定「實際顯示多大、放在哪」
  └─ viewport          保持「邏輯尺寸 1000 x 562.5」，再做 scale(canvasScale)
       └─ elements     一律用邏輯座標渲染

互動層
  └─ 滑鼠事件先拿到畫面座標，再除以 canvasScale 換回邏輯座標
```

---

## 名詞對照

在這篇教學中：

* 最小範例裡的 `.frame`，對應 `PPTist` 的 `viewport-wrapper`
* 最小範例裡的 `.viewport`，對應 `PPTist` 的 `viewport`

另外幾個重要變數：

* `viewportSize`：邏輯畫布寬度
* `viewportRatio`：邏輯畫布高寬比
* `canvasScale`：邏輯畫布映射到畫面時的縮放倍率

補充一個之後在 `PPTist` 實作裡會看到的概念：

* `canvasPercentage`：畫布想占外層容器幾成

也就是說：

* `canvasPercentage` 決定「想顯示多大」
* `canvasScale` 決定「最後實際要縮放多少倍」

這個概念在前面先知道就好，後面速查區會整理完整公式。

---

## 常見混淆點

### 1. `viewportSize` 和 `canvasScale` 不是同一件事

* `viewportSize` 定義的是邏輯畫布有多大
* `canvasScale` 決定的是這張邏輯畫布現在顯示成多大

前者是**資料基準**，後者是**顯示倍率**。

---

### 2. `viewport-wrapper` 和 `viewport` 不是重複包一層

* `viewport-wrapper` 承接實際顯示大小、位置、置中與版面佔位
* `viewport` 保留固定邏輯尺寸，讓元素能繼續使用同一套座標系

外層解決的是顯示與 layout，內層解決的是座標系穩定。

---

### 3. `transform: scale()` 改的是畫面，不是資料

* 元素資料的 `left / top / width / height` 不會因為縮放而改值
* 縮放改變的是「現在看起來多大」
* 它不會改寫元素在邏輯畫布中的定義

---

### 4. 滑鼠拿到的是畫面座標，不是邏輯座標

* 滑鼠事件先量到的是畫面上的距離
* 要寫回資料前，必須先換回邏輯座標

所以才會一直看到：

```ts
邏輯座標 = 畫面座標 / canvasScale
```

---

### 5. `canvasScale` 是顯示倍率，不是資料倍率

* 它不會改寫元素資料
* 它只決定邏輯畫布如何映射到目前畫面

---

## 速查區

### 核心定義

* `viewportSize`：邏輯畫布寬度
* `viewportRatio`：邏輯畫布高寬比
* `canvasPercentage`：畫布可視區想占外層容器多少百分比
* `canvasScale`：邏輯畫布映射到實際畫面的縮放倍率

---

### 常用公式

先不要把下面這些公式看成一堆零散算式。
它們其實只是在回答四個問題：

1. **邏輯畫布有多大？**
2. **這次想讓它在容器裡顯示多大？**
3. **因此要縮放幾倍？**
4. **滑鼠拿到的畫面座標，怎麼換回邏輯座標？**

---

#### 第 1 步：先定義邏輯畫布尺寸

這一步回答的是：

> 系統內部那張固定投影片，本來有多大？

```ts
width = viewportSize
height = viewportSize * viewportRatio
viewportRatio = height / width
```

例如：

```ts
viewportSize = 1000
viewportRatio = 9 / 16
```

那麼邏輯畫布就是：

```ts
width = 1000
height = 1000 * 9 / 16 = 562.5
```

也就是說，系統內部固定有一張：

```ts
1000 x 562.5
```

的邏輯畫布。

---

#### 第 2 步：先算出「這次想顯示多大」

這一步回答的是：

> 外層容器這麼大，而我想讓畫布大約占其中幾成？

```ts
viewportActualWidth = canvasWidth * (canvasPercentage / 100)
viewportActualHeight = canvasHeight * (canvasPercentage / 100)
```

例如外層容器是：

```ts
canvasWidth = 1200
canvasHeight = 800
canvasPercentage = 80
```

那就表示：

```ts
viewportActualWidth = 1200 * 0.8 = 960
viewportActualHeight = 800 * 0.8 = 640
```

注意，這兩個值不是說畫布最後一定會同時變成 `960 x 640`。
它們只是代表：

* 如果這次用寬度當基準，最多想顯示到多寬
* 如果這次用高度當基準，最多想顯示到多高

---

#### 第 3 步：根據容器形狀，決定這次用寬度算還是用高度算

這一步回答的是：

> 這次縮放時，應該讓寬度貼齊，還是讓高度貼齊？

```ts
// 容器較高時，用寬度當基準
canvasScale = viewportActualWidth / viewportSize

// 容器較扁時，用高度當基準
canvasScale = viewportActualHeight / (viewportSize * viewportRatio)
```

可以這樣理解：

* **容器比較高、比較瘦**時，通常寬度會先碰到邊，所以用**寬度**算
* **容器比較扁、比較矮**時，通常高度會先碰到邊，所以用**高度**算

本質上是在做這件事：

> 哪一邊比較先成為限制，就用哪一邊來決定 `canvasScale`

---

#### 第 4 步：算出真正的縮放倍率 `canvasScale`

如果這次是用寬度當基準：

```ts
canvasScale = viewportActualWidth / viewportSize
```

例如：

```ts
viewportActualWidth = 900
viewportSize = 1000
```

那麼：

```ts
canvasScale = 900 / 1000 = 0.9
```

意思就是：

> 這張邏輯寬度 1000 的畫布，現在要縮成 0.9 倍來顯示。

如果這次是用高度當基準：

```ts
canvasScale = viewportActualHeight / (viewportSize * viewportRatio)
```

例如：

```ts
viewportActualHeight = 450
viewportSize * viewportRatio = 562.5
```

那麼：

```ts
canvasScale = 450 / 562.5 = 0.8
```

意思就是：

> 這張邏輯高度 562.5 的畫布，現在要縮成 0.8 倍來顯示。

---

#### 第 5 步：算出畫面上真正看到的大小

這一步回答的是：

> 縮放完之後，畫布在螢幕上實際看起來多大？

```ts
screenWidth = width * canvasScale
screenHeight = height * canvasScale
```

例如邏輯畫布是：

```ts
width = 1000
height = 562.5
canvasScale = 0.8
```

那麼畫面上看到的大小就是：

```ts
screenWidth = 1000 * 0.8 = 800
screenHeight = 562.5 * 0.8 = 450
```

也就是說：

* 邏輯世界還是 `1000 x 562.5`
* 只是畫面上現在看起來變成 `800 x 450`

---

#### 第 6 步：把滑鼠的畫面座標換回邏輯座標

這一步回答的是：

> 使用者點到畫面上的某個位置時，對應到邏輯畫布的哪裡？

```ts
logicalX = screenX / canvasScale
logicalY = screenY / canvasScale
```

例如現在：

```ts
canvasScale = 0.8
screenX = 160
screenY = 80
```

那換回邏輯座標就是：

```ts
logicalX = 160 / 0.8 = 200
logicalY = 80 / 0.8 = 100
```

意思就是：

* 使用者在畫面上點到 `(160, 80)`
* 實際對應的是邏輯畫布裡的 `(200, 100)`

---

### 把整段公式壓成一條流程

如果你只想快速記住，可以記成下面這條：

```text
先定義邏輯畫布大小
-> 再根據容器大小與 canvasPercentage 算出這次想顯示多大
-> 再決定用寬度還是高度當基準算 canvasScale
-> 再得到畫面上真正顯示的大小
-> 最後把滑鼠的畫面座標除以 canvasScale，換回邏輯座標
```

---

### `canvasPercentage` 和 `canvasScale` 的關係

先記住一句最重要的話：

> **`canvasPercentage` 是「想顯示多大」的目標值，`canvasScale` 是「最後真的要縮放幾倍」的結果值。**

兩者不是同一件事。

---

#### 先看 `canvasPercentage`

假設：

```ts
canvasPercentage = 90
```

它的意思不是：

> 畫布一定縮放成 `0.9` 倍

而是：

> **我希望這張投影片顯示時，大約占外層容器的 90% 空間。**

所以 `canvasPercentage` 比較像是一個「顯示目標」或「佔比偏好」：

* 想讓畫布看起來大一點，可以設 95
* 想讓畫布留白多一點，可以設 80

它回答的是：

> **畫布想占容器幾成？**

但它還沒有直接回答：

> **那到底要縮放幾倍？**

這個問題要交給 `canvasScale`。

---

#### 再看 `canvasScale`

`canvasScale` 才是真正用來做縮放的值。

它回答的是：

> **這張固定大小的邏輯畫布，現在要乘上幾倍，才能顯示成我要的大小？**

所以：

```ts
canvasScale = 實際想顯示的尺寸 / 邏輯畫布尺寸
```

這句是對的，但要注意：

> 這裡的「實際想顯示的尺寸」不是直接寫死的，
> 而是先由 `canvasPercentage` 和容器大小一起決定，再拿去算出 `canvasScale`。 

---

#### 為什麼 `canvasPercentage = 90` 不代表 `canvasScale = 0.9`

因為 `90%` 是相對於**容器**說的，
但 `canvasScale` 是相對於**邏輯畫布**說的。

這兩個基準根本不同。

例如：

* 外層容器寬 `1200`
* 邏輯畫布寬 `1000`
* `canvasPercentage = 90`

那麼你先得到的是：

```ts
viewportActualWidth = 1200 * 0.9 = 1080
```

這表示：

> 我希望畫布這次最多顯示成 `1080px` 寬

接著才去算：

```ts
canvasScale = 1080 / 1000 = 1.08
```

你會發現：

* `canvasPercentage = 90`
* 但 `canvasScale = 1.08`

也就是說，畫布不但不是 `0.9` 倍，反而還是**放大**。

原因很簡單：

* `90%` 是拿容器的 `1200` 去算
* `canvasScale` 是拿結果 `1080` 再去除以邏輯畫布的 `1000`

所以它們本來就不會直接一樣。

---

#### 那為什麼還要看容器比例？

因為畫布有固定比例，容器也有自己的比例。

如果你只看寬度，不看高度，可能會發生：

> 寬度放得下，但高度爆掉

反過來也可能：

> 高度放得下，但寬度爆掉

所以程式要先判斷：

> **這次到底是寬度先成為限制，還是高度先成為限制？**

這就是你原文裡這段判斷的意思：

```ts
if (canvasHeight / canvasWidth > viewportRatio) {
  viewportActualWidth = canvasWidth * (canvasPercentage / 100)
  canvasScale = viewportActualWidth / viewportSize
}
else {
  viewportActualHeight = canvasHeight * (canvasPercentage / 100)
  canvasScale = viewportActualHeight / (viewportSize * viewportRatio)
}
```

可以把它翻成白話：

* 如果容器相對比較高，就先用**寬度**決定這次顯示大小
* 如果容器相對比較扁，就先用**高度**決定這次顯示大小

所以 `canvasPercentage` 只是先說：

> 我想占容器大約 90%

但最後到底轉成「多寬」還是「多高」，要看容器形狀和投影片比例。

---

### 關鍵規則

* 元素資料一律使用邏輯座標
* `canvasPercentage` 控制畫布想占容器的比例
* `canvasScale` 控制最後實際縮放倍率
* 顯示縮放交給 `scale()`
* 事件座標回寫資料前，要先除以 `canvasScale`
