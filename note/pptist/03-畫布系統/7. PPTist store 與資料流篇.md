# `PPTist` store 與資料流篇

所屬章節：[03-畫布系統](./README.md)

---

## 這篇要解決什麼

前面六篇已經把畫布系統的幾條主線講清楚了：

```text
固定邏輯畫布
-> viewport-wrapper 決定外框
-> viewport 承載內容並 scale
-> operates 承接操作工具
-> 互動座標換回邏輯座標
-> 元素照邏輯資料渲染
```

但如果你繼續往下讀 `PPTist` 原始碼，會遇到另一個更核心的問題：

> 畫布上看到的 slide、element、選取狀態、縮放倍率、歷史紀錄，到底分別存在誰身上？

這篇就是要把這件事講清楚。

更精準地說，這篇要回答：

1. `slidesStore` 和 `mainStore` 到底怎麼分工？
2. 為什麼元素資料存在 `slidesStore`，但選取狀態存在 `mainStore`？
3. `Canvas/index.vue` 怎麼從 store 拿資料，轉成畫布上的 `elementList`？
4. 使用者拖曳、建立、刪除元素後，資料怎麼寫回 store？
5. `snapshotStore` 在 undo / redo 裡扮演什麼角色？
6. 為什麼 PPTist 不是每個互動都直接改 `currentSlide.elements`？
7. 讀 PPTist 資料流時，應該照什麼順序看？

---

## 先說清楚：這篇主要對照哪些 Source

這篇主要對照下面幾個位置：

```text
src/store/index.ts
src/store/slides.ts
src/store/main.ts
src/store/snapshot.ts
src/store/keyboard.ts
src/store/screen.ts

src/views/Editor/Canvas/index.vue

src/hooks/useCreateElement.ts
src/hooks/useDeleteElement.ts
src/hooks/useSlideHandler.ts
src/hooks/useHistorySnapshot.ts

src/utils/database.ts
```

可以先把它們理解成這樣：

| Source                  | 主要責任                                                        |
| ----------------------- | ----------------------------------------------------------- |
| `store/index.ts`        | 統一匯出所有 store                                                |
| `slides.ts`             | 保存簡報資料本體：slides、theme、slideIndex、viewportSize、viewportRatio |
| `main.ts`               | 保存編輯器 UI / 互動狀態：選取、縮放、建立元素狀態、工具列狀態                          |
| `snapshot.ts`           | 保存 undo / redo 指標，並把 slides 快照放進 IndexedDB                  |
| `keyboard.ts`           | 保存 Ctrl / Shift / Space 等鍵盤狀態                               |
| `screen.ts`             | 保存是否進入放映狀態                                                  |
| `Canvas/index.vue`      | 把 store 資料取出來，轉成畫布顯示與互動入口                                   |
| `useCreateElement.ts`   | 建立元素，寫入 `slidesStore`，更新選取狀態，加入歷史快照                         |
| `useDeleteElement.ts`   | 刪除元素，更新 slide elements，加入歷史快照                               |
| `useSlideHandler.ts`    | 新增、刪除、切換、排序 slide                                           |
| `useHistorySnapshot.ts` | 對 `snapshotStore` 的 add / undo / redo 做節流與防抖                |
| `utils/database.ts`     | 用 Dexie 建立 IndexedDB，保存 snapshots 等資料                       |

`PPTist` 目前的 `src/store` 目錄包含 `index.ts`、`keyboard.ts`、`main.ts`、`screen.ts`、`slides.ts`、`snapshot.ts`；`index.ts` 只是把這些 store 統一匯出，方便其他模組從 `@/store` 取用。([GitHub][1])

# 一、先抓一句話核心

PPTist 的 store 分工可以先用一句話記住：

```text
slidesStore 管「簡報資料」；
mainStore 管「編輯器狀態」；
snapshotStore 管「歷史快照」；
keyboardStore / screenStore 管「全域操作狀態」。
```

這裡最重要的是：

> **不要把所有狀態都當成簡報資料。**

例如：

```text
element.left
element.top
element.width
element.height
```

這些是簡報資料，應該跟著 slide 存。

但下面這些就不是簡報資料：

```text
目前選到哪個元素
目前畫布縮放幾倍
目前是否按著空白鍵
目前是否開啟標尺
目前是否正在建立元素
目前右側工具列在哪個狀態
```

這些是編輯器狀態，應該和真正的 slide 資料分開。

---

# 二、先把 store 分成 5 類

## 先講原理

讀 PPTist store 時，不要一開始就陷進每個欄位。

你可以先把它們分成五類：

```text
1. 簡報資料 store
   -> slidesStore

2. 編輯器互動 store
   -> mainStore

3. 歷史紀錄 store
   -> snapshotStore

4. 鍵盤狀態 store
   -> keyboardStore

5. 放映狀態 store
   -> screenStore
```

這樣分完之後，你再看每個欄位就會清楚很多。

---

## 再對回 Source

`store/index.ts` 目前統一匯出了：

```ts
useMainStore
useSlidesStore
useSnapshotStore
useKeyboardStore
useScreenStore
```

也就是說，PPTist 不是只有一個大 store，而是按責任拆成多個 store。這種拆法的好處是：真正的簡報資料、畫布互動狀態、鍵盤狀態、歷史紀錄不會全部混在同一個物件裡。([GitHub][2])

可以先記成：

```text
slidesStore：
  資料本體

mainStore：
  編輯狀態

snapshotStore：
  undo / redo

keyboardStore：
  鍵盤修飾鍵狀態

screenStore：
  放映狀態
```

---

# 三、`slidesStore` 是簡報資料本體

## 先講原理

`slidesStore` 是整個 PPTist 最接近「檔案內容」的地方。

如果你把 PPTist 想成一個線上 PowerPoint 編輯器，那真正會被保存、匯出、還原的核心資料，大部分都在這一層。

它負責的是：

```text
這份簡報叫什麼？
目前有哪些 slides？
目前在哪一頁？
簡報主題是什麼？
投影片邏輯尺寸是多少？
每一頁有哪些 elements？
```

所以它不是單純給畫布用的 store，而是整份簡報資料的主資料源。

---

## 再對回 Source

`slides.ts` 裡的 `SlidesState` 包含：

```ts
title: string
theme: SlideTheme
slides: Slide[]
slideIndex: number
viewportSize: number
viewportRatio: number
templates: SlideTemplate[]
```

其中最重要的是：

```text
slides：
  所有投影片資料

slideIndex：
  目前正在編輯哪一頁

viewportSize / viewportRatio：
  邏輯畫布基準尺寸

theme：
  全域主題樣式
```

`slidesStore` 的初始狀態裡，`viewportSize` 是 `1000`，`viewportRatio` 是 `0.5625`，這正好接上你前幾篇筆記建立的「固定邏輯畫布」模型：資料世界先固定一張 1000 × 562.5 的投影片，畫面只是縮放後的結果。([GitHub][3]) 

所以 `slidesStore` 可以這樣記：

```text
slidesStore：
  保存簡報資料本體。
  slide / element / theme / viewport 基準都在這裡。
```

---

# 四、`slidesStore.currentSlide` 是目前畫布資料入口

## 先講原理

畫布只需要渲染「目前這一頁」。

所以畫布不會直接拿整個 `slides` 陣列全部渲染，而是先透過：

```text
slideIndex
```

找到目前頁面。

這就是 `currentSlide` 的角色。

它不是新的資料來源，而是：

```text
slides[slideIndex] 的 getter 包裝。
```

---

## 再對回 Source

在 `slides.ts` 的 getters 裡：

```ts
currentSlide(state) {
  return state.slides[state.slideIndex]
}
```

這代表：

```text
currentSlide = slides[slideIndex]
```

所以讀資料流時要注意：

```text
slidesStore.slides：
  全部投影片資料

slidesStore.slideIndex：
  目前頁面索引

slidesStore.currentSlide：
  目前畫布要看的那一頁
```

`currentSlideAnimations` 和 `formatedAnimations` 也都是基於 `currentSlide` 的元素與動畫資料再做整理，這表示 `slidesStore` 不只保存頁面資料，也會提供一些和目前頁面相關的衍生資料。([GitHub][3])

---

# 五、`slidesStore` 的 actions 是資料修改入口

## 先講原理

`slidesStore` 不只是存資料，它也提供修改資料的入口。

常見操作可以分成兩類：

```text
1. 修改 slide
   setSlides
   addSlide
   updateSlide
   deleteSlide
   updateSlideIndex

2. 修改 element
   addElement
   deleteElement
   updateElement
   removeElementProps
```

這裡要先建立一個很重要的觀念：

> **真正會改變簡報內容的操作，最後大多會落到 `slidesStore`。**

例如：

```text
新增元素
刪除元素
拖曳元素
縮放元素
修改背景
新增頁面
刪除頁面
切換頁面
排序頁面
```

這些都是在改簡報資料。

---

## 再對回 Source

`slides.ts` 的 actions 裡，`setSlides()` 會直接替換整份 slides，`addSlide()` 會把新頁面插到目前頁面後方並更新 `slideIndex`，`updateSlide()` 會把目前 slide 或指定 slide 合併新的 props，`deleteSlide()` 會刪除指定 slide 並重新計算新的 `slideIndex`。([GitHub][3])

元素操作則更直接：

```ts
addElement(element)
deleteElement(elementId)
updateElement({ id, props, slideId })
removeElementProps({ id, propName })
```

其中 `addElement()` 會把元素加到目前頁面的 `elements` 後面；`deleteElement()` 會從目前頁面的 `elements` 過濾掉指定 id；`updateElement()` 會根據 id 找到元素並合併新的 props。([GitHub][3])

所以你可以把 `slidesStore` 的資料修改壓成一句：

```text
slide 級資料：
  改 slides / slideIndex

element 級資料：
  改 currentSlide.elements
```

# 六、`mainStore` 是編輯器狀態，不是簡報資料本體

## 先講原理

`mainStore` 很容易被誤解。

因為它裡面有很多和畫布有關的東西，例如：

```text
activeElementIdList
canvasScale
showRuler
creatingElement
toolbarState
```

所以很多人會直覺覺得：

> 這也是簡報資料吧？

但更精準的說法是：

> **`mainStore` 保存的是編輯器目前怎麼操作這份簡報，而不是簡報本身的內容。**

舉例來說：

```text
某個元素的 left / top：
  是簡報資料

目前選到這個元素：
  是編輯器狀態

某頁的 background：
  是簡報資料

目前右側工具列打開哪個 panel：
  是編輯器狀態

某個元素存在於 slide.elements：
  是簡報資料

某個元素暫時被 hidden：
  是編輯器狀態
```

所以 `mainStore` 和 `slidesStore` 的分工是：

```text
slidesStore：
  這份 PPT 是什麼

mainStore：
  使用者現在怎麼編輯它
```

---

## 再對回 Source

`main.ts` 的 `MainState` 包含大量編輯器狀態，例如：

```ts
activeElementIdList
handleElementId
activeGroupElementId
hiddenElementIdList

canvasPercentage
canvasScale
canvasDragged

thumbnailsFocus
editorAreaFocus
disableHotkeys

gridLineSize
showRuler

creatingElement
creatingCustomShape

toolbarState
clipingImageElementId
isScaling
richTextAttrs
selectedTableCells
selectedSlidesIndex
dialogForExport
textFormatPainter
shapeFormatPainter
showSelectPanel
showSearchPanel
...
```

這些欄位大多不是 slide 本身的內容，而是編輯器當下的操作狀態。尤其 `activeElementIdList`、`handleElementId`、`canvasScale`、`creatingElement`、`showRuler`，都和畫布目前怎麼被操作有關。([GitHub][4])

所以 `mainStore` 可以這樣記：

```text
mainStore：
  保存 UI 狀態、畫布狀態、選取狀態、工具狀態。
  它不應該被理解成簡報內容本身。
```

# 七、為什麼選取狀態不放在 element 裡

## 先講原理

這點非常重要。

你可能會想：

```ts
element.selected = true
```

這樣不是很方便嗎？

但 PPTist 沒這樣做。

它把元素資料和選取狀態拆開：

```text
element 資料：
  放在 slidesStore.currentSlide.elements

目前選到哪些元素：
  放在 mainStore.activeElementIdList
```

這樣做的好處是：

```text
1. 簡報資料比較乾淨
2. 匯出時不會混入編輯器暫態
3. 切換頁面、清除焦點、取消選取更容易
4. undo / redo 時可以只還原 slide 資料，不一定要還原 UI 選取狀態
```

所以要先記住：

> **選取狀態是編輯器狀態，不是元素資料。**

---

## 再對回 Source

`mainStore` 裡用 `activeElementIdList` 表示目前選取的元素 id 集合，並用 `handleElementId` 表示正在操作的元素 id。`setActiveElementIdList()` 裡還有一個細節：如果只選中一個元素，就把 `handleElementId` 設成該元素；如果選中多個元素，`handleElementId` 會被清空。([GitHub][4])

`mainStore` 的 getters 也證明它不是自己保存元素資料，而是透過 `useSlidesStore()` 取得目前 slide，再根據 `activeElementIdList` 或 `handleElementId` 從 `currentSlide.elements` 裡找出元素。([GitHub][4])

所以資料流是：

```text
mainStore.activeElementIdList
  只保存 id

slidesStore.currentSlide.elements
  保存元素本體

mainStore.activeElementList getter
  用 id 回 currentSlide.elements 找元素
```

這是一個很典型的設計：

```text
資料本體放一處；
UI 狀態只保存引用關係。
```

# 八、`Canvas/index.vue` 是 store 到畫布的組裝層

## 先講原理

`Canvas/index.vue` 不是資料本體。

它比較像：

```text
把 store 資料取出來，
組裝成畫布渲染與互動所需資料的中介層。
```

它做幾件事：

```text
1. 從 mainStore 取出畫布與互動狀態
2. 從 slidesStore 取出 currentSlide
3. 從 keyboardStore 取出鍵盤狀態
4. 建立本地 elementList
5. 把 elementList 傳給 EditableElement / Operate / MouseSelection / hooks
6. 互動完成後，再透過 hooks 寫回 store
```

所以你讀 `Canvas/index.vue` 時，不要把它看成「資料來源」。

它更像是：

```text
store -> 畫布 DOM / hooks 的資料轉接站
```

---

## 再對回 Source

`Canvas/index.vue` 會 import `storeToRefs`，並從 `@/store` 取出 `useMainStore`、`useSlidesStore`、`useKeyboardStore`。接著它從 `mainStore` 取出 `activeElementIdList`、`activeGroupElementId`、`handleElementId`、`hiddenElementIdList`、`editorAreaFocus`、`gridLineSize`、`showRuler`、`creatingElement`、`creatingCustomShape`、`canvasScale`、`textFormatPainter` 等狀態；從 `slidesStore` 取出 `currentSlide`；從 `keyboardStore` 取出 `ctrlKeyState` 和 `spaceKeyState`。([GitHub][5])

這裡代表一件事：

```text
Canvas/index.vue 不自己保存全域資料。
它是把不同 store 的資料取出來，組合成畫布需要的狀態。
```

---

# 九、`elementList` 是畫布本地工作副本

## 先講原理

這是第 7 篇最重要的點之一。

在 `Canvas/index.vue` 裡，畫布沒有直接拿 `currentSlide.elements` 去做所有互動。

它會建立一個本地的：

```ts
elementList
```

然後用 `watchEffect` 把目前 slide 的 elements 深拷貝進來。

這代表：

```text
slidesStore.currentSlide.elements：
  store 裡的正式資料

Canvas.elementList：
  畫布目前操作用的本地工作副本
```

為什麼要這樣做？

因為拖曳、縮放、旋轉這種互動，過程中會非常頻繁地更新元素狀態。

如果每一個 mousemove 都直接改 store，會讓資料流過度頻繁，也會讓歷史紀錄、吸附線、暫態運算更難控制。

所以比較合理的分工是：

```text
拖曳過程：
  先改 elementList，讓畫面即時動

拖曳結束：
  再把結果寫回 slidesStore
  再加入 history snapshot
```

---

## 再對回 Source

在 `Canvas/index.vue` 裡：

```ts
const elementList = ref<PPTElement[]>([])

const setLocalElementList = () => {
  elementList.value = currentSlide.value
    ? JSON.parse(JSON.stringify(currentSlide.value.elements))
    : []
}

watchEffect(setLocalElementList)
```

這段說明 `elementList` 是從 `currentSlide.elements` 深拷貝出來的本地狀態，而不是直接引用 store 裡的 elements。後面 `elementList` 會被傳給 `EditableElement`、`Operate`、`MouseSelection`，也會被傳給 `useDragElement`、`useScaleElement`、`useRotateElement` 等互動 hooks。([GitHub][5])

這裡可以先記住：

```text
currentSlide.elements：
  store 正式資料

elementList：
  Canvas 本地渲染 / 互動資料
```

# 十、資料從 store 流到畫布的主線

## 先講原理

從 store 到畫布，大致可以整理成這條線：

```text
slidesStore.slides
  -> slidesStore.slideIndex
  -> slidesStore.currentSlide
  -> Canvas/index.vue
  -> elementList
  -> EditableElement
  -> TextElement / ImageElement / ShapeElement / ...
```

也就是：

```text
store 裡的 slide 資料
-> 目前頁面
-> 畫布本地副本
-> 元素包裝層
-> 真正元素元件
```

---

## 再對回 Source

`Canvas/index.vue` template 裡，`EditableElement` 是用 `v-for="(element, index) in elementList"` 渲染，並把 `elementInfo`、`elementIndex`、`isMultiSelect`、`selectElement` 等資料傳進去。這表示畫布上真正被渲染的元素，不是直接從 template 裡讀 `currentSlide.elements`，而是透過 `elementList`。([GitHub][5])

可以把這條畫成：

```text
slidesStore.currentSlide.elements
  ↓ 深拷貝
Canvas.elementList
  ↓ v-for
EditableElement
  ↓ dynamic component
TextElement / ImageElement / ShapeElement / ...
```

這條就是「資料到畫布」的正向資料流。

# 十一、互動後資料怎麼回寫 store

## 先講原理

前面是正向資料流：

```text
store -> Canvas -> element
```

互動後要走反向資料流：

```text
滑鼠操作
-> hook 計算新的 element 狀態
-> 更新 elementList
-> 寫回 slidesStore
-> 加入 history snapshot
```

不同互動的細節不一樣，但大方向是這樣：

```text
操作過程：
  使用 elementList 做即時互動

操作完成：
  用 slidesStore.updateElement / updateSlide / addElement 寫回正式資料
```

---

## 再對回 Source

`Canvas/index.vue` 把 `elementList`、`alignmentLines`、`canvasScale`、`viewportRef` 等資料分發給不同 hooks，例如 `useMouseSelection(elementList, viewportRef)`、`useDragElement(elementList, alignmentLines, canvasScale)`、`useScaleElement(elementList, alignmentLines, canvasScale)`、`useRotateElement(elementList, viewportRef, canvasScale)`、`useMoveShapeKeypoint(elementList, canvasScale)`。你前一篇已經把這些 hooks 的座標責任整理過：有些需要 `viewportRef` 做點座標換算，有些需要 `canvasScale` 做位移換算，有些需要 `elementList` 判斷或修改元素。([GitHub][5]) 

所以互動資料流可以這樣記：

```text
Canvas/index.vue：
  分發資料

hooks：
  處理互動計算

slidesStore：
  接收最後結果
```

---

# 十二、建立元素的資料流

## 先講原理

建立元素是最適合拿來理解資料流的例子。

它大致是：

```text
使用者觸發建立
-> createElement 建立 PPTElement 物件
-> slidesStore.addElement(element)
-> mainStore.setActiveElementIdList([element.id])
-> mainStore.setCreatingElement(null)
-> mainStore.setEditorareaFocus(true)
-> addHistorySnapshot()
```

這裡可以看到兩種狀態同時被更新：

```text
slidesStore：
  加入正式元素資料

mainStore：
  更新編輯器狀態，例如選取新元素、結束建立狀態、讓編輯區聚焦
```

這正好說明：

> **新增元素不是只改資料，也要同步改編輯器狀態。**

---

## 再對回 Source

`useCreateElement.ts` 裡的 `createElement()` 會先呼叫 `slidesStore.addElement(element)`，接著把新元素 id 放進 `mainStore.setActiveElementIdList([element.id])`，如果目前有 `creatingElement` 就把它清掉，接著讓 editor area focus，最後呼叫 `addHistorySnapshot()`。([GitHub][6])

也就是：

```text
slidesStore.addElement：
  改簡報資料

mainStore.setActiveElementIdList：
  改選取狀態

mainStore.setCreatingElement(null)：
  結束建立元素模式

mainStore.setEditorareaFocus(true)：
  回到編輯區焦點

addHistorySnapshot：
  記錄歷史
```

這個例子很適合用來理解 `slidesStore` 和 `mainStore` 的差別：

```text
元素本體歸 slidesStore；
元素被選到這件事歸 mainStore。
```

---

# 十三、刪除元素的資料流

## 先講原理

刪除元素也一樣，不是只把元素從資料裡移掉而已。

它至少要做三件事：

```text
1. 判斷要刪除哪些元素
2. 清空選取狀態
3. 更新 slide.elements
4. 加入歷史快照
```

這裡同樣可以看到：

```text
mainStore：
  處理選取狀態

slidesStore：
  處理真正元素資料

snapshotStore：
  保存歷史
```

---

## 再對回 Source

`useDeleteElement.ts` 裡，`deleteElement()` 會先看 `activeElementIdList` 是否為空；如果有 `activeGroupElementId`，就優先刪掉群組中被獨立操作的元素，否則刪掉所有被選取的元素。接著它會呼叫 `mainStore.setActiveElementIdList([])` 清空選取，再用 `slidesStore.updateSlide({ elements: newElementList })` 更新目前頁面的 elements，最後呼叫 `addHistorySnapshot()`。([GitHub][7])

所以刪除元素的資料流可以畫成：

```text
mainStore.activeElementIdList / activeGroupElementId
  ↓ 決定刪除目標
newElementList
  ↓
mainStore.setActiveElementIdList([])
  ↓
slidesStore.updateSlide({ elements: newElementList })
  ↓
addHistorySnapshot()
```

這裡要注意：

> **刪除目標的判斷依賴 mainStore，但真正刪除資料是改 slidesStore。**

---

# 十四、切換 / 新增 / 刪除 slide 的資料流

## 先講原理

slide 操作和 element 操作很像。

差別是：

```text
element 操作：
  大多改 currentSlide.elements

slide 操作：
  會改 slides 陣列或 slideIndex
```

例如：

```text
新增頁面：
  addSlide + 更新 slideIndex

刪除頁面：
  deleteSlide + 重新計算 slideIndex

切換頁面：
  updateSlideIndex

排序頁面：
  setSlides + updateSlideIndex
```

---

## 再對回 Source

`useSlideHandler.ts` 會從 `mainStore` 和 `slidesStore` 取資料，並提供 `resetSlides`、`updateSlideIndex`、`createSlide`、`createSlideByTemplate`、`deleteSlide`、`sortSlides` 等操作。切換頁面時，如果目前有選取元素，會先清空 `activeElementIdList`，再呼叫 `slidesStore.updateSlideIndex()`；建立空白頁時會建立 `emptySlide`，清空選取，呼叫 `slidesStore.addSlide(emptySlide)`，再加入歷史快照；刪除頁面時，如果刪光全部頁面就 reset，否則呼叫 `slidesStore.deleteSlide()`，並清掉多選頁面狀態後加入歷史快照。([GitHub][8])

這裡你可以抓一個原則：

```text
切換頁面前，通常要清掉元素選取狀態。
因為 activeElementIdList 是目前頁面的 UI 狀態，
切到另一頁後不應該繼續保留。
```

---

# 十五、`snapshotStore` 是歷史紀錄指標，不是直接存在記憶體的大陣列

## 先講原理

undo / redo 系統如果直接把所有歷史都放在記憶體裡，簡報資料一大會很重。

PPTist 的設計是：

```text
snapshotStore：
  保存 snapshotCursor / snapshotLength

IndexedDB：
  保存真正 snapshots
```

所以 `snapshotStore` 本身不是把所有快照資料完整放在 Pinia state 裡。

它更像是：

```text
歷史紀錄控制器
```

而真正的快照資料是存在 `utils/database.ts` 建立的 Dexie database 裡。

---

## 再對回 Source

`snapshot.ts` 的 state 只有：

```ts
snapshotCursor: -1
snapshotLength: 0
```

actions 裡的 `initSnapshotDatabase()`、`addSnapshot()`、`unDo()`、`reDo()` 會透過 `db.snapshots` 讀寫快照。`addSnapshot()` 會把目前 `slidesStore.slideIndex` 和深拷貝後的 `slidesStore.slides` 加進 snapshots；undo / redo 則會從 IndexedDB 取出對應 snapshot，再呼叫 `slidesStore.setSlides(slides)` 和 `slidesStore.updateSlideIndex(slideIndex)` 還原資料，最後清空選取狀態。([GitHub][9])

`utils/database.ts` 裡用 Dexie 建立資料庫，並宣告 `snapshots: '++id'` 和 `writingBoardImgs: 'id'` 兩張表；資料庫名稱包含 `PPTist`、`databaseId` 和當前時間，另外也有刪除過期 / 失效資料庫的邏輯。([GitHub][10])

所以可以這樣記：

```text
snapshotStore：
  管目前歷史指標

Dexie / IndexedDB：
  存真正歷史資料

slidesStore：
  被 undo / redo 還原的目標
```

---

# 十六、`useHistorySnapshot.ts` 是歷史紀錄的操作包裝

## 先講原理

很多地方都需要加入歷史紀錄，例如：

```text
新增元素
刪除元素
新增 slide
刪除 slide
修改元素屬性
```

如果每個地方都直接呼叫 `snapshotStore.addSnapshot()`，會有幾個問題：

```text
1. mousemove 過程可能呼叫太頻繁
2. undo / redo 可能被連續觸發太快
3. 歷史紀錄操作會散在各處
```

所以 PPTist 做了一層 `useHistorySnapshot.ts`。

它不是新的 store，而是：

```text
snapshotStore 的操作包裝器。
```

---

## 再對回 Source

`useHistorySnapshot.ts` 會從 `@/store` 取得 `useSnapshotStore()`，然後把 `addSnapshot()` 包成 debounce，延遲 300ms trailing 執行；`redo()` 和 `undo()` 則用 throttle 包起來，100ms 內只允許 leading 執行。([GitHub][11])

所以這裡可以這樣理解：

```text
snapshotStore：
  真正管理歷史狀態

useHistorySnapshot：
  控制歷史操作的呼叫節奏
```

這對拖曳、縮放這種高頻互動尤其重要，因為你不希望每一個滑鼠移動事件都變成一筆歷史紀錄。

---

# 十七、`keyboardStore` 是跨元件共享的鍵盤暫態

## 先講原理

`keyboardStore` 很小，但很重要。

它保存的是：

```text
Ctrl 是否按下
Shift 是否按下
Space 是否按下
```

這些狀態不是簡報資料，也不是某個單一元件的私有狀態。

因為很多功能都可能需要知道：

```text
使用者是否按著 Ctrl？
是否按著 Shift？
是否按著 Space？
```

例如：

```text
Ctrl + 拖曳可能複製元素
Shift 可能影響多選或等比例操作
Space 可能進入拖曳整張畫布模式
```

所以這類狀態適合放到 store。

---

## 再對回 Source

`keyboard.ts` 裡的 state 包含 `ctrlKeyState`、`shiftKeyState`、`spaceKeyState`，getter `ctrlOrShiftKeyActive` 會回傳 Ctrl 或 Shift 是否任一啟用，actions 則提供 `setCtrlKeyState()`、`setShiftKeyState()`、`setSpaceKeyState()`。([GitHub][12])

`Canvas/index.vue` 也會從 `keyboardStore` 取出 `ctrlKeyState` 和 `spaceKeyState`；其中 `spaceKeyState` 會影響空白區滑鼠按下時到底要框選，還是拖曳整張畫布。([GitHub][5])

所以可以這樣記：

```text
keyboardStore：
  管跨元件需要知道的鍵盤狀態。
  它不是資料本體，而是全域互動暫態。
```

---

# 十八、`screenStore` 是放映模式狀態

## 先講原理

`screenStore` 比較單純。

它只需要回答：

```text
目前是不是進入放映狀態？
```

這種狀態也不是簡報內容。

它是應用目前處在哪個模式。

---

## 再對回 Source

`screen.ts` 裡只有一個 state：

```ts
screening: false
```

以及一個 action：

```ts
setScreening(screening: boolean)
```

所以它可以理解成放映模式的全域狀態開關。([GitHub][13])

# 二十、用「建立文字元素」串一次完整流程

這裡用雙擊建立文字來把整條資料流串起來。

## 先講原理

雙擊建立文字時，PPTist 做的事其實跨了前面幾篇全部主題：

```text
1. 滑鼠事件拿到畫面座標
2. 用 viewportRef 找到投影片左上角
3. 除以 canvasScale 換回邏輯座標
4. 建立 text element
5. 寫進 slidesStore
6. 更新 mainStore 選取狀態
7. 加入 history snapshot
8. Canvas 從 currentSlide 重新同步 elementList
9. EditableElement 渲染新元素
```

---

## 再對回 Source

`Canvas/index.vue` 的 `handleDblClick()` 會先用 `viewportRef.value.getBoundingClientRect()` 取得 viewport 位置，再用 `(e.pageX - viewportRect.x) / canvasScale.value` 和 `(e.pageY - viewportRect.y) / canvasScale.value` 算出邏輯 `left / top`，接著呼叫 `createTextElement()`。這正好接上你前一篇提到的「滑鼠畫面座標 -> 扣 viewport 左上角 -> 除以 canvasScale -> 邏輯座標」流程。([GitHub][5]) 

`createTextElement()` 最後會走 `useCreateElement.ts` 裡的 `createElement()`，而 `createElement()` 會呼叫 `slidesStore.addElement(element)`、`mainStore.setActiveElementIdList([element.id])`，並加入歷史快照。([GitHub][6])

所以完整資料流是：

```text
Canvas.handleDblClick
  ↓
viewportRef + canvasScale 換算座標
  ↓
createTextElement
  ↓
slidesStore.addElement
  ↓
mainStore.setActiveElementIdList
  ↓
addHistorySnapshot
  ↓
snapshotStore.addSnapshot
  ↓
Canvas watchEffect 同步 elementList
  ↓
EditableElement 渲染新文字元素
```

---

# 二十一、這篇最容易誤會的 6 個點

## 1. 不要把 `slidesStore` 和 `mainStore` 混成同一種資料

```text
slidesStore：
  簡報內容

mainStore：
  編輯器狀態
```

元素的 `left / top / width / height` 是內容資料；目前選到哪個元素是編輯器狀態。

---

## 2. 不要以為 `activeElementIdList` 是元素資料的一部分

`activeElementIdList` 只是目前選取的元素 id 集合。

真正的元素資料還是在：

```text
slidesStore.currentSlide.elements
```

---

## 3. 不要以為 `Canvas/index.vue` 是資料來源

`Canvas/index.vue` 是資料組裝層。

它從 store 拿資料，建立本地 `elementList`，再把資料傳給各種元素元件與 hooks。

---

## 4. 不要忽略 `elementList` 是本地副本

`elementList` 不是 `currentSlide.elements` 的直接引用。

它是深拷貝出來的畫布工作副本。

---

## 5. 不要以為所有操作都會立即直接寫 store

拖曳、縮放、旋轉這類高頻互動，常會先改本地 `elementList` 以支撐即時畫面，再在適當時機寫回 store 並建立 history。

---

## 6. 不要把 snapshot 理解成普通陣列存在 Pinia 裡

`snapshotStore` 主要保存 cursor 和 length；真正 snapshots 是透過 Dexie 存在 IndexedDB 的 `snapshots` 表裡。([GitHub][9])

---

# 二十二、建議閱讀 Source 的順序

這篇建議照下面順序讀：

```text
1. src/store/index.ts

2. src/store/slides.ts

```

這個順序的重點是：

```text

```

# 二十三、這篇最後要帶走的結論

PPTist 的 store 與資料流，不是把所有東西都塞進一個大狀態物件。

它真正做的是把資料分成幾種責任：

```text
slidesStore：
  保存簡報資料本體。

mainStore：
  保存目前編輯器怎麼操作這份簡報。

Canvas/index.vue：
  把 store 資料轉成畫布本地渲染與互動資料。

elementList：
  畫布本地工作副本。

hooks：
  處理具體互動，必要時更新 elementList 並回寫 store。

snapshotStore：
  管 undo / redo 指標，真正快照放 IndexedDB。

keyboardStore：
  管 Ctrl / Shift / Space 這種跨元件鍵盤暫態。

screenStore：
  管是否進入放映狀態。
```

一句話總結：

> **PPTist 的資料流核心是：`slidesStore` 保存正式簡報資料，`mainStore` 保存編輯器暫態，`Canvas/index.vue` 產生本地 `elementList` 供畫布渲染與互動，互動結束後再透過 hooks 回寫 `slidesStore` 並加入 `snapshotStore` 歷史。**

如果你能把這條講清楚：

```text
slidesStore.currentSlide.elements
-> Canvas.elementList
-> EditableElement / hooks
-> slidesStore.updateElement / updateSlide
-> snapshotStore.addSnapshot
```

那你讀 PPTist 的新增元素、刪除元素、拖曳元素、縮放元素、切換頁面、undo / redo，就會知道它們其實都在同一套資料流裡運作。
