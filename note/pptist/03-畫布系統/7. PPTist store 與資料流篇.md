# `PPTist` store 與資料流篇（誰負責什麼、資料怎麼流、誰才是 source of truth）

所屬章節：[03-畫布系統](./README.md)

## 這篇要解決什麼

前面的筆記已經把畫布系統的幾何主幹講清楚了：

* `slides.ts` 定義邏輯畫布基準 `viewportSize / viewportRatio`。
* `main.ts` 持有 `canvasScale`。
* `useViewportSize.ts` 根據容器尺寸與比例，計算後呼叫 `mainStore.setCanvasScale(...)`。
* `Canvas/index.vue` 會消費 `canvasScale`：外層 `viewport-wrapper` 用它決定實際占位，內層 `viewport` 用它做 `transform: scale(...)`。
* 互動事件與互動 hooks 又會再用 `canvasScale` 把畫面座標換回邏輯座標。

但只要你真的開始讀原始碼，很快就會碰到另一個更工程化的問題：

> 這些值到底誰管？
> 哪些是資料本體，哪些只是顯示結果，哪些又只是互動過程中的暫時狀態？

這篇就是專門處理這個問題。

---

## 先講一句話核心

畫布系統要穩，關鍵不是把所有值都丟進同一個 store，
而是要分清楚：

> **哪些值是 source of truth，哪些值是 derived state，哪些值只是 interaction state。**

你目前筆記已經足夠確認至少有這種責任切分：

* `slides.ts` 比較像在管理邏輯畫布與元素所依附的資料世界。
* `main.ts` 比較像在管理畫布當前的顯示倍率 `canvasScale`。
* `useViewportSize.ts` 承擔的是「把容器條件換成顯示倍率」的計算責任，而不是元素資料本身。
* `Canvas/index.vue` 與互動 hooks 則屬於消費端：一邊渲染，一邊換算互動。

---

## 先把目前可直接確認的分工整理成一句話

你可以先把目前已確認的畫布鏈路記成這樣：

```text
slides.ts 定義邏輯畫布基準
-> useViewportSize.ts 根據容器算出 canvasScale
-> main.ts 持有 canvasScale
-> Canvas/index.vue 消費 canvasScale 做顯示
-> 互動 hooks 再用 canvasScale 把事件換回邏輯座標
```

這不是推測，而是你目前實作對照篇已經能直接支撐的最短鏈路。

---

## 一、先分清三種狀態

如果你想把畫布原始碼看得穩，最重要的是先把狀態拆成三類。

## A. 資料本體狀態（domain state）

這一類狀態回答的是：

> 投影片在邏輯世界裡到底長什麼樣？

從你目前筆記可直接確認，至少包括：

* `viewportSize`
* `viewportRatio`
* 元素資料的 `left / top / width / height`
* 元素幾何資訊所依附的固定邏輯畫布。

這一類狀態最重要的特徵是：

> **它不應該因為目前畫面放大或縮小，就跟著改值。**

這也呼應你前面筆記一直反覆強調的核心：
資料活在固定邏輯畫布裡，畫面只是顯示結果。

---

## B. 顯示衍生狀態（derived display state）

這一類狀態回答的是：

> 同一份資料，現在在這個容器裡應該怎麼顯示？

目前最明確的例子就是 `canvasScale`。
你已經確認它存在 `main.ts`，並且不是固定值，而是由 `useViewportSize.ts` 根據容器大小、`canvasPercentage`、`viewportSize / viewportRatio` 動態算出來。

這一類狀態的特徵是：

* 它依賴資料本體狀態
* 也依賴容器條件
* 它改變時會影響畫面呈現
* 但它不是元素資料本身。

也就是說：

> `canvasScale` 很重要，但它不是 source of truth；
> 它是「顯示如何成立」的中樞結果值。

---

## C. 互動過程狀態（interaction state）

這一類狀態回答的是：

> 使用者現在正在做什麼？

從你目前筆記裡可以直接確認的互動場景至少包括：

* 雙擊新增文字
* 框選
* 旋轉。

這代表畫布系統一定還有一類短生命週期的狀態，例如：

* 框選起點
* 目前滑鼠位置
* 拖曳中的位移
* 旋轉中的即時計算結果

這些值很重要，但它們通常不是最終資料本體，而是：

> **互動進行中的中間狀態。**

這一類狀態如果跟資料本體混在一起，畫布系統就很容易變亂。

---

## 二、目前可直接確認的 store 角色

下面這一段只寫你目前筆記已經能直接確認的事。

## 1. `slides.ts`：邏輯畫布與元素資料所依附的基準層

你目前已經確認：

```ts
viewportSize: 1000
viewportRatio: 0.5625
```

定義在 `slides.ts`，它們不是 DOM 顯示尺寸，而是在定義邏輯畫布本身。

這代表 `slides.ts` 至少承擔了：

* 邏輯畫布的尺寸基準
* 元素資料解讀的座標世界

所以你可以先把 `slides.ts` 當成：

> **資料世界的基底 store。**

---

## 2. `main.ts`：畫布顯示中樞狀態的承接層

你目前已經確認：

* `canvasScale` 存在 `main.ts`
* `useViewportSize.ts` 會呼叫 `mainStore.setCanvasScale(...)`。

這代表 `main.ts` 至少承擔了：

* 畫布當前顯示倍率的共享狀態
* 讓 `Canvas/index.vue`、互動 hooks 等多方都能共同消費同一個顯示基準

所以你可以把 `main.ts` 理解成：

> **畫布顯示態的共享 store。**

---

## 3. `useViewportSize.ts`：計算邏輯，不是資料源頭

你目前的筆記已經很清楚整理出這段：

```ts
if (canvasHeight / canvasWidth > viewportRatio.value) {
  const viewportActualWidth = canvasWidth * (canvasPercentage.value / 100)
  mainStore.setCanvasScale(viewportActualWidth / viewportSize.value)
}
else {
  const viewportActualHeight = canvasHeight * (canvasPercentage.value / 100)
  mainStore.setCanvasScale(viewportActualHeight / (viewportSize.value * viewportRatio.value))
}
```

這說明 `useViewportSize.ts` 的責任不是存資料，而是把：

* 容器大小
* 顯示目標
* 邏輯畫布比例

換算成顯示倍率。

所以它比較像：

> **derived state 的計算層。**

---

## 4. `Canvas/index.vue`：狀態消費與畫面裝配層

你目前已經確認 `Canvas/index.vue` 同時做兩件事：

* 外層 `viewport-wrapper` 用 `canvasScale` 算畫面占位
* 內層 `viewport` 用 `canvasScale` 做內容縮放。

也就是說，它不是 source of truth 本身，而是：

> **把多個上游狀態組裝成畫面結果的主要消費者。**

這一點很重要，因為很多人讀到 component 很大時，會誤以為它「擁有」所有狀態。
實際上，它常常只是最大的裝配點。

---

## 三、把整條資料流畫成一張圖

你可以先用下面這張圖來理解畫布系統的資料流：

```text
資料本體層
  slides.ts
  - viewportSize
  - viewportRatio
  - element data

計算層
  useViewportSize.ts
  - 讀容器大小
  - 讀顯示目標
  - 讀 viewportSize / viewportRatio
  - 推回 canvasScale

顯示共享層
  main.ts
  - canvasScale

消費層
  Canvas/index.vue
  - wrapper 占位
  - viewport 縮放
  - 內容渲染

互動層
  Canvas/index.vue / useMouseSelection.ts / useRotateElement.ts
  - 使用 canvasScale 把事件座標換回邏輯座標
  - 更新選取、框選、旋轉等結果
```

這張圖的前半段，可以直接由你目前的筆記支持；後半段則是把已確認的互動 hooks 放回資料流位置去理解。

## 四、source of truth 到底應該怎麼判斷

這一段非常重要。

你之後讀任何一段畫布原始碼，都可以先問：

> 這個值是「應該被保存」的，還是「可以被推算」的？

你可以先用這個最實用的判斷法。

### 應該被保存的

* `viewportSize`
* `viewportRatio`
* 元素幾何資料
* 其他真正定義投影片內容的值。

### 比較像推算結果的

* `canvasScale`
* 畫面上的實際寬高
* 當前框選矩形
* 拖曳中偏移量
* 旋轉中的當下角度計算結果

這不是在說第二類不重要，而是在說：

> 它們更像「由條件推得出的狀態」，而不是內容本體本身。

一旦這個界線立住，你在設計 state 時就不容易讓 derived state 反客為主。

## 五、最常見的資料流錯覺

這裡我直接把最容易踩雷的地方講出來。

## 1. 誤以為 `canvasScale` 是資料的一部分

不是。
從你目前筆記來看，`canvasScale` 的責任是把固定邏輯畫布映射到目前容器。它很重要，但它不是元素資料本體。

## 2. 誤以為 component 大，就代表它擁有所有狀態

`Canvas/index.vue` 很核心，但它比較像畫面裝配點，而不是所有狀態的唯一來源。

## 3. 誤以為互動過程值應該直接寫回資料

框選中的暫時矩形、拖曳中的滑鼠位移、旋轉中的中間向量，通常都比較像 interaction state，不應該直接當作永久資料。
這一點雖然你目前筆記沒有逐一列欄位，但從互動 hooks 的性質已經可以合理推出。

---

## 六、讀原始碼時最實用的 5 個問題

之後你只要看到某個 state、computed、hook，可以先這樣問：

### 1. 它定義的是資料本體，還是顯示結果？

如果是邏輯畫布、元素資料，偏向前者；如果是 `canvasScale` 或顯示寬高，偏向後者。

### 2. 它會不會因為容器尺寸改變而改變？

如果會，那它通常不是資料本體，更可能是 derived display state。

### 3. 它會不會因為滑鼠移動每幀都變？

如果會，那它通常是 interaction state。

### 4. 這個值應不應該被保存？

如果答案是「可以每次重新算」，那它更可能不是 source of truth。

### 5. 這段程式在做「寫資料」，還是在做「組畫面」？

這個問題非常有效，因為很多複雜 component 都同時混著兩種責任。

## 七、把這一篇和前幾篇串起來

這一篇和前幾篇的關係可以壓成一句話：

```text
前幾篇解決的是「畫布怎麼成立」，
這一篇解決的是「那些讓畫布成立的值，到底誰在管」。
```

也就是說：

* 第一到五篇偏幾何、顯示、互動原理
* 這一篇開始轉向架構與狀態責任

這一步很重要，因為你之後再讀更深的編輯器原始碼時，
難的常常不再是公式，而是「哪一層該負責哪一種狀態」。

## 最容易混淆的 7 個點

### 1. `slides.ts` 和 `main.ts` 不是同一種 store

前者更接近邏輯畫布與資料世界，後者至少明確承接了 `canvasScale` 這種顯示共享狀態。

### 2. `useViewportSize.ts` 不是資料源頭

它負責計算，不是畫布內容本體的定義者。

### 3. `canvasScale` 很重要，但不是 source of truth

它是顯示鏈路中的中樞倍率。

### 4. component 消費狀態，不等於擁有狀態

`Canvas/index.vue` 是畫面裝配點，不應自動等同於資料源頭。

### 5. interaction state 不等於 element data

互動中間值通常只是短生命週期狀態，不是投影片本體資料。

### 6. store 的責任不是愈集中愈好

真正重要的是責任邊界清楚。

### 7. source of truth 與 derived state 一旦混掉，後面所有邏輯都會開始拖泥帶水

這是可視化編輯器最常見的維護問題之一。

```text
useViewportSize.ts：顯示倍率計算
Canvas/index.vue：畫面組裝與消費

```



### 最重要的分類

```text
資料本體狀態
顯示衍生狀態
互動過程狀態
```

### 一句話記憶

```text
先分清誰在存資料、誰在算顯示、誰只是在互動過程中暫時存在。
```

## 一句話總結

`PPTist` 的畫布資料流，不是把所有值塞進同一個地方，而是：

> **由資料本體層定義邏輯世界，由計算層推導顯示倍率，由共享顯示層承接畫布當前狀態，再由畫面層與互動層共同消費。**
