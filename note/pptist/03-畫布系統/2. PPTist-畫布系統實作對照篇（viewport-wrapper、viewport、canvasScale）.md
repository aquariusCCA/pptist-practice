## 先給一張總圖

先不要急著看程式碼，先把整個系統想成這四個層次：

1. **邏輯畫布**

   * 元素資料真正活著的世界
   * 座標、大小、旋轉、框選都以這一層為準

2. **縮放倍率 `canvasScale`**

   * 負責把邏輯畫布映射到目前螢幕大小

3. **外層 `viewport-wrapper`**

   * 決定這塊可視區域在畫面上占多大、放在哪裡

4. **內層 `viewport`**

   * 真正裝元素內容
   * 用 `transform: scale(...)` 把整張邏輯畫布放大或縮小

如果這四層分得開，後面讀 `PPTist` 原始碼就不容易混在一起。

## 1. 畫布系統真正要解的問題

如果元素的位置直接綁死在真實 DOM 像素上，畫布一縮放，很多事情都會立刻變複雜：

* 不同螢幕尺寸下的版面不一致
* 拖曳、框選、旋轉時，滑鼠位移很難直接對回元素資料
* 主畫布、縮圖、播放頁可能得維護不同座標系統
* 匯出時也不容易保證版面一致

所以 `PPTist` 沒有直接把元素綁在螢幕像素上。
它先建立一個固定的「邏輯世界」，然後讓畫面只是那個世界在不同容器中的顯示結果。這也是我們反覆強調的主軸：元素不是直接跟著真實螢幕尺寸跑，而是先活在固定邏輯畫布中，再透過縮放映射到畫面上。

對應程式位置：

* [slides.ts](../../../PPTist-SourceCode/src/store/slides.ts)
* [Canvas.md](../../../PPTist-SourceCode/doc/Canvas.md)

## 2. 邏輯畫布的來源：`viewportSize` 與 `viewportRatio`

在 [slides.ts](../../../PPTist-SourceCode/src/store/slides.ts) 第 54 行附近，預設值是：

```ts
viewportSize: 1000
viewportRatio: 0.5625
```

這兩個值不是「現在 DOM 顯示多大」，而是在定義**邏輯畫布本身**。

它們的意思是：

* `viewportSize`：邏輯畫布的寬度基準
* `viewportRatio`：邏輯畫布的高寬比

所以邏輯畫布大小就是：

```ts
width = 1000
height = 1000 * 0.5625 = 562.5
```

也就是說，`PPTist` 預設會先建立一個 `1000 x 562.5` 的邏輯座標世界。
元素資料裡的 `left / top / width / height`，本質上都是以這個世界為基準。

可以整理成下面幾個固定關係：

```ts
width = viewportSize
height = viewportSize * viewportRatio

viewportRatio = height / width
height = width * viewportRatio
width = height / viewportRatio
```

這裡最重要的一句話是：

> `viewportSize = 1000` 不代表畫面最多只能顯示到 1000px。
> 它只是邏輯基準，不是實際顯示尺寸。

這點如果分不清，後面很容易把「邏輯寬度」和「畫面寬度」混在一起。

## 3. 實際畫面怎麼決定大小：`viewport-wrapper`

真正顯示在畫面上的尺寸，不是去改 `viewportSize`，而是由 `Canvas` 外層來決定。

在 [Canvas/index.vue](../../../PPTist-SourceCode/src/views/Editor/Canvas/index.vue) 第 19 行附近：

```vue
<div
  class="viewport-wrapper"
  :style="{
    width: viewportStyles.width * canvasScale + 'px',
    height: viewportStyles.height * canvasScale + 'px',
    left: viewportStyles.left + 'px',
    top: viewportStyles.top + 'px',
  }"
>
```

這一層可以先理解成「畫面上的可視區盒子」。

它主要做兩件事：

1. 決定這塊投影片區域在畫面上**占多大**
2. 決定這塊投影片區域在畫面上**放在哪裡**

所以：

* `viewportStyles.width`、`viewportStyles.height` 仍然是邏輯尺寸
* 乘上 `canvasScale` 之後，才變成真正畫在螢幕上的顯示尺寸

例如：

```ts
viewportStyles.width = 1000
canvasScale = 0.8
```

那畫面上看到的寬度就是：

```ts
1000 * 0.8 = 800px
```

所以 `viewport-wrapper` 的工作，不是重新定義邏輯畫布，
而是告訴瀏覽器：

> 這張邏輯投影片，現在在畫面上要占多少空間、擺在哪裡。

## 4. 內容怎麼被放大或縮小：`viewport`

同一個檔案裡，真正裝元素內容的內層是 `viewport`。
在 [Canvas/index.vue](../../../PPTist-SourceCode/src/views/Editor/Canvas/index.vue) 第 60 行附近：

```vue
<div
  class="viewport"
  ref="viewportRef"
  :style="{ transform: `scale(${canvasScale})` }"
>
```

這裡才是整個畫布系統最核心的地方：

* 內層內容仍然維持原本的邏輯座標與邏輯尺寸
* 真正的放大縮小交給 `transform: scale(...)`

也就是說，元素本身不用知道現在畫面放大幾倍、縮小幾倍。
它只要照著邏輯座標去渲染就好。

---

## 4.1 很容易誤會的地方：這樣是不是縮放了兩次？

很多人看到這裡會疑惑：

* 外層 `viewport-wrapper` 的寬高已經乘了 `canvasScale`
* 內層 `viewport` 又做了 `transform: scale(canvasScale)`

那這樣不是重複縮放嗎？

不是。這兩層的職責不同：

### `viewport-wrapper` 做的是「盒子佔位」

它負責的是：

* 可視區域在畫面上實際占多少空間
* 這塊區域在畫面上放在哪裡

也就是說，它是在處理**外層排版與定位**

### `viewport` 做的是「內容縮放」

它負責的是：

* 裡面那張固定邏輯畫布，現在要放大幾倍、縮小幾倍

也就是說，它是在處理**內容如何顯示**

所以這兩者不是「重複縮放」，而是：

* 外層決定盒子看起來有多大
* 內層決定內容怎麼按比例畫進去

這個分工如果分清楚，`wrapper + viewport` 這套設計就會突然變得很合理。

## 5. `canvasScale` 怎麼算出來

`canvasScale` 存在 [main.ts](../../../PPTist-SourceCode/src/store/main.ts) 第 53 行附近的全域 store 中。它不是固定值，而是由 `useViewportSize.ts` 根據容器尺寸動態計。

它的計算邏輯可以拆成四步：

1. 先看外層 `canvas` 容器目前有多大
2. 判斷這次應該用「寬度」還是「高度」當縮放基準
3. 用 `canvasPercentage` 決定投影片想占容器幾成
4. 最後推回 `canvasScale`

這裡一定要分清楚兩個值：

* `canvasPercentage`：控制值，表示「希望投影片約占容器多少比例」
* `canvasScale`：結果值，表示「邏輯畫布最後要縮放多少倍」

所以更精準地說：

> `canvasScale` 不是直接等於 `canvasPercentage`
> 而是先由 `canvasPercentage` 算出可視寬高，再換算成縮放倍率

---

## 5.1 寬度基準與高度基準

你不能直接假設永遠拿寬度來算。
因為有時候容器雖然夠寬，但如果照這個寬度去放，投影片高度可能會超出容器。

所以判斷邏輯是：

* 如果先用容器寬度算，投影片高度也放得下，就以寬度為基準
* 如果先用容器寬度算，投影片高度放不下，就改以高度為基準

對照 [useViewportSize.ts](../../../PPTist-SourceCode/src/views/Editor/Canvas/hooks/useViewportSize.ts)：

```ts
if (canvasHeight / canvasWidth > viewportRatio.value) {
  const viewportActualWidth = canvasWidth * (canvasPercentage.value / 100)
  mainStore.setCanvasScale(viewportActualWidth / viewportSize.value)
}
else {
  const viewportActualHeight = canvasHeight * (canvasPercentage.value / 100)
  mainStore.setCanvasScale(viewportActualHeight / (viewportSize.value * viewportRatio.value))
}
```

---

## 5.2 直接看一個數字例子

只看公式容易抽象，直接帶一次數字會比較清楚。

假設：

```ts
canvasWidth = 1600
canvasHeight = 900

viewportSize = 1000
viewportRatio = 0.5625

canvasPercentage = 90
```

### 情況一：以寬度為基準

如果目前容器比例允許以寬度來算：

```ts
viewportActualWidth = 1600 * 0.9 = 1440
canvasScale = 1440 / 1000 = 1.44
```

意思就是：

* 邏輯畫布寬度原本是 1000
* 現在要顯示成 1440
* 所以要放大 `1.44` 倍

### 情況二：以高度為基準

如果目前容器高度才是限制條件：

```ts
viewportActualHeight = 900 * 0.9 = 810
canvasScale = 810 / 562.5 = 1.44
```

因為邏輯畫布高度是：

```ts
1000 * 0.5625 = 562.5
```

所以這次也是把邏輯高度 `562.5` 映射到實際高度 `810`，得到相同的縮放倍率。

這樣看就會更明白：

> `canvasScale` 本質上就是
> 「實際顯示尺寸 ÷ 邏輯尺寸」

## 6. 元素怎麼渲染：全部都站在邏輯世界裡

元素元件本身大多直接使用資料上的 `left / top / width / height`。

例如文字元素 [TextElement/index.vue](../../../PPTist-SourceCode/src/views/components/element/TextElement/index.vue) 第 4 行附近：

```vue
:style="{
  top: elementInfo.top + 'px',
  left: elementInfo.left + 'px',
  width: elementInfo.width + 'px',
  height: elementInfo.height + 'px',
}"
```

注意這裡沒有乘上 `canvasScale`。

不是它忘了乘，而是因為外層 `viewport` 已經整體做了縮放。
所以元素元件本身只需要專心描述：

* 在邏輯畫布裡我的位置在哪
* 在邏輯畫布裡我的大小是多少

至於畫面現在放大幾倍、縮小幾倍，那是外層系統要處理的事。

這種設計的好處很明顯：

* 元素元件更單純
* 資料模型更穩定
* 主畫布、縮圖、播放頁可以共用同一份元素資料

## 7. 滑鼠操作為什麼一直要除以 `canvasScale`

畫面上的滑鼠事件拿到的是**螢幕座標**。
但元素資料要寫回的是**邏輯座標**。

所以中間一定要做座標轉換。

這個轉換其實分兩步：

1. 先扣掉 `viewport` 在畫面上的偏移
2. 再除以 `canvasScale`，把顯示座標換回邏輯座標

例如在 [Canvas/index.vue](../../../PPTist-SourceCode/src/views/Editor/Canvas/index.vue) 第 221 行附近，雙擊空白新增文字：

```ts
const left = (e.pageX - viewportRect.x) / canvasScale.value
const top = (e.pageY - viewportRect.y) / canvasScale.value
```

這段可以拆開理解成：

```ts
螢幕中的點 - viewport 左上角位置 = 在畫布顯示區內的相對位置
相對位置 / canvasScale = 邏輯畫布中的位置
```

也就是：

```ts
邏輯座標 = (螢幕座標 - 畫布左上角偏移) / canvasScale
```

這就是為什麼你在原始碼裡會一直看到「先減位置、再除縮放」。

## 7.1 框選、旋轉也都是同一件事

在 [useMouseSelection.ts](../../../PPTist-SourceCode/src/views/Editor/Canvas/hooks/useMouseSelection.ts) 第 24 行附近，滑鼠框選也是同樣邏輯：

```ts
const left = (startPageX - viewportRect.x) / canvasScale.value
const top = (startPageY - viewportRect.y) / canvasScale.value
const offsetWidth = (currentPageX - startPageX) / canvasScale.value
const offsetHeight = (currentPageY - startPageY) / canvasScale.value
```

在 [useRotateElement.ts](../../../PPTist-SourceCode/src/views/Editor/Canvas/hooks/useRotateElement.ts) 第 54 行附近，旋轉也會先把滑鼠座標換回邏輯座標：

```ts
const mouseX = (currentPageX - viewportRect.left) / canvasScale.value
const mouseY = (currentPageY - viewportRect.top) / canvasScale.value
```

所以你看到的不是很多套不同公式，核心其實只有一套：

```ts
邏輯座標 = 顯示中的相對座標 / canvasScale
```

而「顯示中的相對座標」通常又是：

```ts
螢幕座標 - viewport 左上角偏移
```

## 8. 為什麼縮圖可以沿用同一套模型

縮圖元件 [ThumbnailSlide/index.vue](../../../PPTist-SourceCode/src/views/components/ThumbnailSlide/index.vue) 跟主畫布本質上是同一個思路。

做法仍然是：

* 外層容器決定縮圖實際顯示多大
* 內層維持固定邏輯尺寸
* 再用 `transform: scale(...)` 做整體縮小

所以縮圖不是另一套資料結構，
而只是同一張邏輯畫布的另一種顯示比例。

這點非常重要，因為它說明了：

> `PPTist` 的核心不是「替不同畫面各做一套座標」
> 而是「所有畫面共用同一個邏輯世界，再各自決定顯示比例」

## 9. 把整個 `PPTist` 畫布系統再收斂成一句話

如果只用一句話來描述這套系統，可以這樣說：

> `PPTist` 不是把元素直接綁在螢幕像素上，
> 而是先維持一張固定邏輯畫布，再把它映射到目前容器中。

再拆得更具體一點，就是：

### 外層：顯示層

負責：

* 可視區域有多大
* 投影片放在畫面的哪裡
* 目前縮放倍率是多少

### 內層：邏輯層

負責：

* 元素真正的座標與尺寸
* 拖曳、框選、旋轉等幾何運算
* 主畫布、縮圖、播放頁共用同一份資料模型

只要把這兩層分開，後面不管你讀拖曳、縮放、框選、旋轉、縮圖還是播放頁，都不容易混掉。

## 一句話抓核心

`PPTist` 的畫布系統，本質上就是：

**固定邏輯畫布 + 顯示時整體縮放 + 事件發生時再換回邏輯座標**
