# `PPTist` 畫布定位篇（`viewportStyles`、置中公式、可視區定位）

所屬章節：[03-畫布系統](./README.md)

## 這篇要解決什麼

前面的筆記已經處理了三件核心事情：

1. 為什麼要先有固定大小的邏輯畫布
2. 為什麼顯示縮放和資料座標要拆開
3. 為什麼互動事件最後都要換回邏輯座標。 

但當你開始對照 `PPTist` 原始碼時，還會卡在另一個實作問題：

> `viewport-wrapper` 到底是怎麼決定
> 「顯示多大」
> 以及
> 「放在哪裡」？

這篇就是專門處理這個缺口。

更具體地說，這篇要回答的是：

* `viewportStyles.width / height / left / top` 各代表什麼
* 這些值為什麼不直接寫死
* 為什麼 `width / height` 還要再乘上 `canvasScale`
* 為什麼畫布能夠在容器中置中
* 這些值裡面，哪些屬於邏輯層，哪些屬於顯示層

---

## 先講一句話核心

`viewportStyles` 本質上不是在定義「投影片資料」，而是在定義：

> 這張固定邏輯畫布，現在在畫面上應該占哪一塊區域、放在什麼位置。

也就是說：

* `viewportSize / viewportRatio` 定義的是 **邏輯畫布本身**
* `canvasScale` 定義的是 **要縮放幾倍**
* `viewportStyles` 定義的是 **這塊顯示區在畫面上如何佈局**

這也符合前面已經建立的兩層分工：
外層 `viewport-wrapper` 負責顯示區大小與位置，內層 `viewport` 負責保留固定邏輯尺寸並做 `scale()`。 

---

## 先備知識

如果下面三件事還不穩，建議先回去看前兩篇：

* 固定邏輯畫布
* `wrapper + viewport` 兩層分工
* `canvasScale` 是顯示倍率，不是資料倍率。 

---

## 先把四個值分清楚

在 `PPTist` 裡，`viewport-wrapper` 會吃到這組 style：

```vue
:style="{
  width: viewportStyles.width * canvasScale + 'px',
  height: viewportStyles.height * canvasScale + 'px',
  left: viewportStyles.left + 'px',
  top: viewportStyles.top + 'px',
}"
```

你可以先把這四個值分成兩組看：

### 第一組：大小

* `viewportStyles.width`
* `viewportStyles.height`

這兩個值回答的是：

> 這張邏輯畫布本來有多大？

注意，這裡的「本來」不是指畫面像素，而是邏輯基準。

在前面的筆記裡，這一套基準已經很明確：

```ts
width = viewportSize
height = viewportSize * viewportRatio
```

預設情況下就是：

```ts
viewportWidth = 1000
viewportHeight = 562.5
```

也就是一張固定的邏輯畫布。

---

### 第二組：位置

* `viewportStyles.left`
* `viewportStyles.top`

這兩個值回答的是：

> 這塊可視區域在外層容器裡，應該擺在哪裡？

它們不是元素資料的 `left / top`，
而是整張投影片顯示區的定位值。

這一點很重要。因為如果把它跟元素資料搞混，後面你看到拖曳、框選、滑鼠偏移時就會亂掉。

---

## 為什麼 `width / height` 還要乘上 `canvasScale`

這裡最容易誤會。

很多人第一次看到會想：

> `viewportStyles.width` 不是已經是寬度了嗎？
> 為什麼還要再乘一次 `canvasScale`？

答案是：

> 因為 `viewportStyles.width / height` 表示的是邏輯尺寸，
> 不是最後畫在螢幕上的顯示尺寸。

前面的筆記已經反覆強調過：

* `viewportSize = 1000` 不代表畫面最多只能顯示到 `1000px`
* 它只是邏輯基準，不是實際顯示尺寸。

所以如果：

```ts
viewportSize = 1000
canvasScale = 0.8
```

那最後畫面上真正占用的寬度才是：

```ts
1000 * 0.8 = 800
```

這也就是前面實作篇已經講到的：

> `viewport-wrapper` 不是重新定義邏輯畫布，
> 而是把這張邏輯投影片映射成現在畫面上要占多少空間。

---

## 把 `viewport-wrapper` 想成「畫布的外框盒子」

要理解 `left / top`，先把 `viewport-wrapper` 想成一個盒子。

這個盒子有兩個責任：

1. 它要知道自己 **看起來多大**
2. 它要知道自己 **在畫面上放哪裡**

這件事其實跟內層 `viewport` 完全不同。

* `viewport-wrapper` 處理的是外層排版與定位
* `viewport` 處理的是內層內容縮放。

也就是說：

* `wrapper` 問的是：**這個盒子放哪裡？**
* `viewport` 問的是：**盒子裡的那張邏輯畫布怎麼縮放？**

這兩層一起看，整套系統才會通。

---

## 為什麼畫布能置中

現在進到這篇最核心的問題。

所謂「置中」，本質上不是一個神祕效果，而只是：

> 先知道容器有多大
> 再知道投影片顯示後有多大
> 最後把剩下的空間平均分到兩邊。

---

## 先定義兩組尺寸

### 1. 容器尺寸

這是外層編輯區目前可用的空間，例如：

```ts
canvasWidth = 1600
canvasHeight = 900
```

### 2. 投影片顯示尺寸

這不是邏輯尺寸，而是縮放後真正顯示在畫面上的尺寸：

```ts
viewportActualWidth = viewportSize * canvasScale
viewportActualHeight = viewportSize * viewportRatio * canvasScale
```

如果：

```ts
viewportSize = 1000
viewportRatio = 0.5625
canvasScale = 0.8
```

那就得到：

```ts
viewportActualWidth = 800
viewportActualHeight = 450
```

---

## 置中公式其實很單純

只要容器比投影片大，剩下的空間就可以平均分配：

```ts
viewportLeft = (canvasWidth - viewportActualWidth) / 2
viewportTop = (canvasHeight - viewportActualHeight) / 2
```

例如：

```ts
canvasWidth = 1600
canvasHeight = 900

viewportActualWidth = 800
viewportActualHeight = 450
```

那就得到：

```ts
viewportLeft = (1600 - 800) / 2 = 400
viewportTop = (900 - 450) / 2 = 225
```

意思就是：

* 投影片左邊留 `400`
* 右邊也留 `400`
* 上面留 `225`
* 下面也留 `225`

這就是畫布置中的本質。

---

## 這裡一定要再分一次：邏輯尺寸 vs 顯示尺寸

很多人會在這裡混掉。

### 邏輯尺寸

```ts
viewportWidth = 1000
viewportHeight = 562.5
```

它定義的是：

> 內部資料世界本來有多大

### 顯示尺寸

```ts
viewportActualWidth = 1000 * 0.8 = 800
viewportActualHeight = 562.5 * 0.8 = 450
```

它定義的是：

> 現在畫面上看起來多大

### 定位值

```ts
viewportLeft = 400
viewportTop = 225
```

它定義的是：

> 這塊顯示區在容器裡放哪裡

把這三層拆開，你之後讀原始碼時就不會把所有 `width / height / left / top` 都看成同一種東西。

---

## 一張圖先看懂

```text
邏輯層
  viewportSize = 1000
  viewportRatio = 0.5625
  ->
  viewportWidth = 1000
  viewportHeight = 562.5

顯示層
  canvasScale = 0.8
  ->
  viewportActualWidth = 800
  viewportActualHeight = 450

定位層
  canvasWidth = 1600
  canvasHeight = 900
  ->
  viewportLeft = (1600 - 800) / 2 = 400
  viewportTop = (900 - 450) / 2 = 225
```

---

## 你可以把 `viewportStyles` 暫時理解成這樣

如果先用教學模型抽象化，`viewportStyles` 可以先理解成：

```ts
const viewportStyles = {
  width: viewportSize,
  height: viewportSize * viewportRatio,
  left: (canvasWidth - viewportSize * canvasScale) / 2,
  top: (canvasHeight - viewportSize * viewportRatio * canvasScale) / 2,
}
```

但這裡要注意一件事：

> 這是一個教學模型，目的是幫你先理解公式關係。
> 真正的 `PPTist` 原始碼可能會把「邏輯尺寸」與「縮放後顯示尺寸」拆在不同 computed / hook 裡處理。

也就是說，這個公式現在可以拿來建立心智模型，
但之後你還是要對照實際原始碼，把「公式位置」補精準。

---

## 為什麼不能把 `left / top` 也乘上 `canvasScale`

這也是一個很容易出錯的點。

`width / height` 乘 `canvasScale`，是因為它們在回答：

> 邏輯畫布縮放後，看起來多大？

但 `left / top` 回答的是：

> 這整塊畫布顯示區在容器裡放哪裡？

它們是**定位結果**，本身已經是顯示層數值，不需要再乘一次縮放。

換句話說：

* `width / height` 是由邏輯尺寸推到顯示尺寸，所以要乘 `canvasScale`
* `left / top` 是顯示定位值，本來就是畫面像素，不應該再縮放

這就是為什麼：

```vue
width: viewportStyles.width * canvasScale
height: viewportStyles.height * canvasScale
left: viewportStyles.left
top: viewportStyles.top
```

這種寫法非常合理。

---

## 為什麼只做 `scale()` 還不夠，跟這篇有什麼關係

這篇跟前面概念篇其實是連起來的。

前面已經說過：

> `transform: scale()` 只會改變「看起來多大」，
> 不會改變 layout 裡實際佔多少空間。

所以才需要：

* 內層 `viewport` 做 `scale()`
* 外層 `viewport-wrapper` 提供真實顯示尺寸與版面定位。 

這也剛好說明了：

> `viewportStyles` 存在的價值，不只是給寬高，
> 更是為了讓外層 wrapper 知道自己在 layout 裡到底該怎麼站。

---

## 最容易混淆的 5 個點

### 1. `viewportStyles.width` 不是畫面寬度本身

它是邏輯尺寸基準，真正畫到畫面上的寬度通常還要再乘上 `canvasScale`。

### 2. `viewportStyles.left` 不是元素的 `left`

它是整張投影片顯示區的偏移，不是某個元素的位置。

### 3. `canvasScale` 不負責置中

`canvasScale` 只回答「縮放幾倍」，不回答「放在哪裡」。
置中要靠 `left / top`。

### 4. `wrapper` 和 `viewport` 不是重複包一層

外層處理 layout 與定位，內層處理內容縮放。

### 5. 畫布置中不是資料層邏輯

它不會改元素資料，不會改邏輯座標，只是在顯示層決定整張投影片的擺放位置。
這也符合你前面一直強調的「資料活在邏輯畫布裡，畫面只是映射結果」。

## 速查區

### 核心角色

```ts
viewportSize   // 邏輯畫布寬度
viewportRatio  // 邏輯畫布高寬比
canvasScale    // 顯示縮放倍率
viewportStyles // wrapper 的邏輯尺寸與定位資料
```

### 核心關係

```ts
viewportWidth = viewportSize
viewportHeight = viewportSize * viewportRatio

viewportActualWidth = viewportWidth * canvasScale
viewportActualHeight = viewportHeight * canvasScale

viewportLeft = (canvasWidth - viewportActualWidth) / 2
viewportTop = (canvasHeight - viewportActualHeight) / 2
```

### 一句話記憶

```text
width / height 解決「看起來多大」，
left / top 解決「放在哪裡」。
```

---

## 一句話總結

`viewportStyles` 不是在描述元素資料，而是在描述：

> 這張固定邏輯畫布，經過縮放後，應該以多大的顯示尺寸、放在容器中的哪個位置。
