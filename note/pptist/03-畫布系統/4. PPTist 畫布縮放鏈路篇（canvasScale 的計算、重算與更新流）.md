# `PPTist` 畫布縮放鏈路篇（`canvasScale` 的計算、重算與更新流）

所屬章節：[03-畫布系統](./README.md)

## 這篇要解決什麼

前面的筆記已經解決了兩件很重要的事：

1. 為什麼系統要先建立一張固定大小的邏輯畫布
2. 為什麼畫面顯示、資料座標、滑鼠座標換算要拆開理解。 

但當你開始真正讀 `PPTist` 原始碼時，還會遇到另一個關鍵問題：

> `canvasScale` 到底是怎麼活起來的？

換句話說，這篇不是只問：

> `canvasScale` 的公式是什麼？

而是要再往前、再往後各多問一步：

* **往前問**：它是由誰算出來的？
* **往後問**：它一旦改變，整個畫布系統哪些地方會跟著更新？

這篇就是專門處理這條鏈路。

---

## 先講一句話核心

`canvasScale` 不是單純的一個數字，而是整個畫布顯示系統的**中樞倍率**。

它站在整條鏈路的中間：

```text
容器尺寸
-> 計算 canvasScale
-> wrapper 決定畫面占位
-> viewport 做內容縮放
-> 滑鼠事件再用它換回邏輯座標
```

也就是說：

* 前面連著 **容器大小與顯示目標**
* 後面連著 **畫布顯示與互動換算**

只要這個倍率改了，整張畫布的顯示方式和互動換算基準都會一起變。

## 先備知識

如果下面三件事還不穩，建議先回去看前一輪筆記：

* 固定邏輯畫布：`viewportSize`、`viewportRatio`
* `wrapper + viewport` 的兩層分工
* 畫面座標與邏輯座標的差異。

## 先把四個角色分清楚

在這篇裡，最容易混掉的是下面四個值：

### 1. `viewportSize`

邏輯畫布的寬度基準。
在你目前整理的實作對照篇裡，預設值是 `1000`。

### 2. `viewportRatio`

邏輯畫布的高寬比。
在你目前筆記裡，預設值是 `0.5625`，所以邏輯畫布高度就是 `1000 * 0.5625 = 562.5`。

### 3. `canvasPercentage`

顯示目標值。
它表示的是：

> 希望投影片大約占外層容器幾成空間。

它不是縮放倍率本身。這點你前面的概念篇已經講得很清楚。

### 4. `canvasScale`

真正的縮放結果值。
它回答的是：

> 這張固定邏輯畫布，最後到底要縮放幾倍，才能顯示成我要的大小？

這四個角色如果分不開，後面整條鏈路會很容易糊掉。

---

## `canvasScale` 在整個系統裡的位置

先直接把你現有筆記裡能確定的原始碼定位整理成一張表：

| 角色     | 位置                   | 職責                                |
| ------ | -------------------- | --------------------------------- |
| 邏輯畫布基準 | `slides.ts`          | 提供 `viewportSize`、`viewportRatio` |
| 縮放結果值  | `main.ts`            | 存放 `canvasScale`                  |
| 計算縮放倍率 | `useViewportSize.ts` | 根據容器尺寸與比例計算 `canvasScale`         |
| 消費縮放倍率 | `Canvas/index.vue`   | `wrapper` 用來占位，`viewport` 用來縮放內容  |

你可以先把這個分工記成一句話：

> `slides.ts` 定義畫布本體，`useViewportSize.ts` 算倍率，`main.ts` 存倍率，`Canvas/index.vue` 用倍率。

## 先看最短鏈路

如果先把整套東西壓到最短，可以先這樣看：

```text
viewportSize / viewportRatio
+ canvas 容器尺寸
+ canvasPercentage
-> useViewportSize.ts 算出 canvasScale
-> mainStore.setCanvasScale(...)
-> Canvas/index.vue 讀取 canvasScale
-> wrapper 寬高改變
-> viewport scale 改變
-> 互動公式的除數改變
```

這條鏈路就是這篇最核心的東西。

## 第 1 段：先有一張固定邏輯畫布

這一段其實不是 `canvasScale` 本身，但沒有它，`canvasScale` 就無從成立。

在 `PPTist` 裡，邏輯畫布不是直接拿 DOM 當基準，而是先有一組固定邏輯尺寸：

```ts
viewportSize: 1000
viewportRatio: 0.5625
```

所以邏輯畫布大小是：

```ts
width = 1000
height = 562.5
```

這代表：

* 元素資料的 `left / top / width / height` 都是建立在這張邏輯畫布上
* `canvasScale` 不是去改寫這張畫布本身
* `canvasScale` 做的是把這張固定畫布映射到目前顯示容器中。

這裡一定要先穩住，因為後面所有「重算」都不是在重建畫布，而是在重算「怎麼顯示它」。

---

## 第 2 段：`canvasScale` 的輸入不是只有一個

很多人第一次看會以為：

> `canvasScale` 不就是某個固定比例嗎？

不是。從你目前整理到的內容看，`canvasScale` 至少同時依賴三類輸入：

### A. 邏輯畫布基準

```ts
viewportSize
viewportRatio
```

### B. 外層容器尺寸

```ts
canvasWidth
canvasHeight
```

### C. 顯示目標

```ts
canvasPercentage
```

也就是說，`canvasScale` 不是單獨存在的數字，而是這三組資訊共同決定的結果。 

---

## 第 3 段：`canvasScale` 怎麼算出來

你目前的實作對照篇已經把公式整理得很完整：

```ts
if (canvasHeight / canvasWidth > viewportRatio.value) {
  const viewportActualWidth = canvasWidth * (canvasPercentage.value / 100)
  mainStore.setCanvasScale(viewportActualWidth / viewportSize.value)
}
else {
  const viewportActualHeight = canvasHeight * (canvasPercentage.value / 100)
  mainStore.setCanvasScale(viewportActualHeight / (viewportSize.value * viewportRatio.value))
}
```

這段程式碼其實回答了三件事。

### 1. 先看容器形狀

程式先比較：

```ts
canvasHeight / canvasWidth
```

和

```ts
viewportRatio
```

本質上是在問：

> 這次應該讓寬度先成為限制，還是讓高度先成為限制？ 

### 2. 再算這次想顯示多大

如果是寬度基準，就先算：

```ts
viewportActualWidth = canvasWidth * (canvasPercentage / 100)
```

如果是高度基準，就先算：

```ts
viewportActualHeight = canvasHeight * (canvasPercentage / 100)
```

也就是先決定：

> 這次想讓投影片在容器中占到多大。

### 3. 最後才換成縮放倍率

如果用寬度當基準：

```ts
canvasScale = viewportActualWidth / viewportSize
```

如果用高度當基準：

```ts
canvasScale = viewportActualHeight / (viewportSize * viewportRatio)
```

所以 `canvasScale` 的本質就是：

> **實際想顯示的尺寸 ÷ 邏輯尺寸**。 

---

## 第 4 段：為什麼 `canvasScale` 不直接等於 `canvasPercentage`

這一點一定要單獨講，因為太容易誤會。

例如：

```ts
canvasPercentage = 90
```

很多人會直覺以為：

```ts
canvasScale = 0.9
```

但這是錯的。

因為：

* `canvasPercentage` 是拿**容器**當基準
* `canvasScale` 是拿**邏輯畫布**當基準

這兩者的分母根本不一樣。

例如：

```ts
canvasWidth = 1200
viewportSize = 1000
canvasPercentage = 90
```

那先得到的是：

```ts
viewportActualWidth = 1200 * 0.9 = 1080
```

接著才是：

```ts
canvasScale = 1080 / 1000 = 1.08
```

也就是說：

* `canvasPercentage = 90`
* 但 `canvasScale = 1.08`

這剛好說明：

> `canvasPercentage` 是顯示目標，`canvasScale` 是換算結果。

---

## 第 5 段：`canvasScale` 為什麼要存在 store，而不是只存在區域變數

這裡是這篇真正進到「鏈路」層的地方。

從你目前整理的內容來看，`canvasScale` 放在 `main.ts` 的全域 store 中，而不是只在某個局部 hook 內部存在。

這代表一件事：

> `canvasScale` 不是某個單一元件的私有資訊，
> 而是整個畫布系統的共享顯示狀態。

它至少同時被三類地方使用：

### 1. 外層顯示區使用

`viewport-wrapper` 會用它來決定畫面上實際占位尺寸：

```vue
width: viewportStyles.width * canvasScale
height: viewportStyles.height * canvasScale
```

也就是 wrapper 會隨著 `canvasScale` 改變而改變。

### 2. 內層內容使用

`viewport` 會直接做：

```vue
transform: scale(canvasScale)
```

也就是整張邏輯畫布的視覺縮放會跟著它改。

### 3. 互動事件使用

新增文字、框選、旋轉等互動，都會拿它把顯示座標換回邏輯座標：

```ts
邏輯座標 = (螢幕座標 - 偏移) / canvasScale
```

所以互動換算基準也跟著它改。

這三類需求同時存在，正是它適合放進 store 的原因。

---

## 第 6 段：`canvasScale` 一旦改變，誰會跟著更新

這一段是本篇最重要的「往後看」。

你可以把 `canvasScale` 的更新後效應分成三層：

### A. 顯示占位層更新

外層 `viewport-wrapper` 的寬高是：

```ts
viewportStyles.width * canvasScale
viewportStyles.height * canvasScale
```

所以倍率一改，wrapper 的畫面占位大小就會改。

### B. 內容縮放層更新

內層 `viewport` 的：

```ts
transform: scale(canvasScale)
```

也會跟著改。
也就是同一張邏輯畫布會被重新放大或縮小。

### C. 互動換算層更新

所有把畫面座標換回邏輯座標的公式，除數都會跟著改：

```ts
logical = relative / canvasScale
```

所以倍率一改，拖曳、框選、旋轉、雙擊新增元素這些互動的換算基準也會一起改。

這三層加在一起，就能解釋為什麼 `canvasScale` 是整個畫布系統的中樞值，而不是單純的樣式數字。

---

## 第 7 段：什麼叫「重算鏈路」

到這裡可以正式把「鏈路」這兩個字講清楚。

所謂 `canvasScale` 的重算鏈路，不只是說：

> 有一個公式把它算出來。

而是要把下面這整條關係看成一個連動系統：

```text
容器可用空間改變
-> useViewportSize.ts 重新判斷寬度/高度基準
-> 重新根據 canvasPercentage 算出顯示目標尺寸
-> 重新推回 canvasScale
-> mainStore.setCanvasScale(...)
-> Canvas/index.vue 重新用新倍率渲染
-> wrapper 佔位變化
-> viewport 視覺縮放變化
-> 互動換算基準同步變化
```

這才叫完整鏈路。 

---

## 第 8 段：那到底什麼時候會重算？

這裡要非常誠實地分成兩層來講。

### 從你目前筆記可以**直接確認**的部分

可以直接確認的是：

* `canvasScale` 不是固定值
* 它由 `useViewportSize.ts` 根據容器尺寸動態計算。

所以至少可以確定：

> **只要容器尺寸的判定條件需要更新，`canvasScale` 就必須重算。**

---

### 從目前資料可以做的**保守推論**

雖然你目前上傳的筆記沒有把所有 watcher、listener、hook 呼叫時機逐行列出來，但從 `useViewportSize.ts` 的命名與職責，以及你目前整理出的鏈路來看，下面幾類場景通常都屬於需要重算的時機：

* 畫布第一次建立時
* 外層容器尺寸改變時
* 會影響可用編輯區尺寸的 UI 變化發生時
* `canvasPercentage` 改變時
* 會改變邏輯畫布比例的設定改變時。 

這一段我刻意寫成「保守推論」，因為它符合目前你筆記所揭示的系統職責，但還不是逐行 source-code 驗證版。

如果你之後要把這篇從 9.5 再推到 9.8，最值得補的就是：

> 把真正的 watch / resize observer / lifecycle hook 名稱補進來。

## 第 9 段：把這件事和第一篇連起來

你第一篇談的是：

* `viewport-wrapper` 怎麼決定在畫面上占多大
* `left / top` 怎麼讓投影片置中
* 什麼是邏輯尺寸，什麼是顯示尺寸。

而這一篇談的是：

* 決定顯示尺寸的那個倍率，是怎麼被算出來、存起來、再分發給各個顯示層與互動層的。

兩篇之間的關係其實很直接：

```text
第一篇：畫布現在放哪裡、看起來多大
第二篇：決定「看起來多大」的倍率，到底怎麼被算出來
```

所以這兩篇不是平行關係，而是前後銜接關係。

## 最容易混淆的 6 個點

### 1. `canvasScale` 不是邏輯畫布尺寸

它不會改 `viewportSize`、不會改元素資料，只是決定這張畫布現在顯示成多大。

### 2. `canvasScale` 不是 `canvasPercentage`

前者是結果值，後者是目標值。

### 3. `canvasScale` 不是只給 CSS 用

它不只影響 `scale()`，也影響外層占位與互動座標換算。 

### 4. 重算 `canvasScale` 不等於重建元素資料

資料仍然活在固定邏輯畫布裡。重算的是顯示倍率，不是資料本身。

### 5. `useViewportSize.ts` 算的是倍率，不是元素位置

它的責任是處理畫布如何映射到容器，不是處理某個元素應該在哪裡。

### 6. `canvasScale` 一改，互動公式也要跟著改

因為畫面座標換回邏輯座標時，除數就是它。

## 速查區

### 一句話抓核心

```text
canvasScale 是「固定邏輯畫布」映射到「目前顯示容器」時的結果倍率。
```

### 公式核心

```ts
// 寬度基準
canvasScale = viewportActualWidth / viewportSize

// 高度基準
canvasScale = viewportActualHeight / (viewportSize * viewportRatio)
```



### 鏈路核心

```text
容器尺寸 -> useViewportSize.ts -> setCanvasScale -> wrapper / viewport / 互動換算
```

 

### 判斷基準

```text
先判斷是寬度先成為限制，還是高度先成為限制，再決定用哪一邊推回 canvasScale。
```

## 一句話總結

`canvasScale` 不是一個孤立的縮放數字，而是：

> **由邏輯畫布基準、容器尺寸與顯示目標共同算出的中樞倍率；它一旦改變，畫布占位、內容縮放與互動座標換算都會一起跟著更新。**
