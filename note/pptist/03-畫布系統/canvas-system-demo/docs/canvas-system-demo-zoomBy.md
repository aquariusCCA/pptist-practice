# `canvas-system-demo` 中 `zoomBy()` 的計算邏輯

這份筆記對應到 `canvas-system-demo` 的 `zoomBy()` 實作，重點在說明：**這個函式如何根據目前縮放倍率，計算下一次的縮放結果。** `zoomBy()` 本身只負責更新倍率，並不處理滑鼠錨點、平移補償或視角校正。

## 核心公式

`zoomBy()` 的核心寫法如下：

```ts
const nextZoom = clamp(
  zoom.value * (direction > 0 ? ZOOM_STEP : 1 / ZOOM_STEP),
  MIN_ZOOM,
  MAX_ZOOM,
)
```

在這個 demo 中，相關常數如下：

* `ZOOM_STEP = 1.12`
* `MIN_ZOOM = 0.4`
* `MAX_ZOOM = 2.4`

其中：

* `zoomBy(1)` 表示放大，新的倍率為 `目前倍率 × 1.12`
* `zoomBy(-1)` 表示縮小，新的倍率為 `目前倍率 × (1 / 1.12)` 

## 為什麼這裡用乘法，而不是加減法

因為 `zoom` 代表的是**縮放倍率**，不是一個固定增減的位移量。既然它描述的是比例關係，更新方式就應該採用乘法，而不是每次加上一個固定數值。

例如目前倍率是 `0.9`：

* 放大一次：`0.9 × 1.12 = 1.008`
* 再縮小一次：`1.008 × (1 / 1.12) = 0.9`

這種做法的好處是，縮放行為會以目前倍率為基準等比變化，因此整體操作會更符合使用者對「放大 / 縮小」的直覺。

## 為什麼縮小要用 `1 / ZOOM_STEP`

這樣設計的目的，是讓放大與縮小形成一組**互為反操作**的倍率變換。

假設放大是乘上 `1.12`，那麼對應的縮小就應該乘上 `1 / 1.12`，而不是任意選一個小於 `1` 的數值。原因在於：

* 放大與縮小的步進能保持對稱
* 來回操作時，數學關係比較一致
* 更符合「倍率變化」而不是「固定差值變化」的模型 

更精確地說，理論上先乘上 `ZOOM_STEP`，再乘上 `1 / ZOOM_STEP`，會回到原本的倍率；如果在實際程式中出現極小誤差，通常是浮點數表示造成的，而不是公式本身有問題。

## `clamp()` 的作用

倍率計算完成後，還會再透過 `clamp()` 限制結果範圍：

```ts
clamp(nextZoom, 0.4, 2.4)
```

這表示最終縮放倍率一定會被限制在 `0.4 ~ 2.4` 之間。

這樣做的目的很直接：

* 避免畫布被縮得太小，導致內容幾乎無法辨識
* 避免畫布被放得太大，超出此 demo 預期的互動範圍 

## 實際計算範例

以下以 `App.vue` 中的預設值 `zoom = 0.9` 為例。

### 範例一：執行一次 `zoomBy(1)`

```ts
0.9 * 1.12 = 1.008
```

因此新的倍率為：

```ts
zoom = 1.008
```

### 範例二：接著執行一次 `zoomBy(-1)`

```ts
1.008 * (1 / 1.12) = 0.9
```

因此倍率又回到：

```ts
zoom = 0.9
```



### 範例三：連續放大 5 次，再連續縮小 5 次

| 步驟 | 動作       |     結果 |
| -- | -------- | -----: |
| 0  | 起始       | 0.9000 |
| 1  | zoom in  | 1.0080 |
| 2  | zoom in  | 1.1290 |
| 3  | zoom in  | 1.2644 |
| 4  | zoom in  | 1.4152 |
| 5  | zoom in  | 1.5850 |
| 6  | zoom out | 1.4152 |
| 7  | zoom out | 1.2644 |
| 8  | zoom out | 1.1290 |
| 9  | zoom out | 1.0080 |
| 10 | zoom out | 0.9000 |

這個例子可以看出，`zoomBy()` 採用的是等比縮放模型，因此倍率變化具有一致性與可預測性。

## `zoomBy()` 的職責邊界

`zoomBy()` 的責任非常單純：**根據目前倍率與縮放方向，計算出新的縮放倍率。**

它**不負責**以下事情：

* 不處理以滑鼠位置為中心的縮放
* 不更新平移位置（pan）
* 不計算游標對應的縮放錨點 

這些邏輯是在 `WorkspaceStage.vue` 的 `handleWheel()` 與 `zoomAroundPoint()` 中處理的。也就是說，`zoomBy()` 可以視為一個**純倍率更新函式**，而不是完整的視角縮放控制函式。

## 總結

`zoomBy()` 的本質可以概括成一句話：

```ts
新的縮放倍率 = 目前倍率 × 縮放步進
```

其中：

* 放大時乘上 `1.12`
* 縮小時乘上 `1 / 1.12`
* 最後再用 `clamp()` 把結果限制在 `0.4 ~ 2.4` 的範圍內 

因此，`zoomBy()` 解決的是「**倍率如何更新**」這件事；至於「**畫面應該以哪個點為中心縮放，縮放後畫面如何補償位移**」，則是其他函式的責任。
