# 常見狀態流案例

這一篇專門把 `PPTist` 中常見的使用者操作串成「狀態流」。

前面幾篇已經分別說明：

- `slides store` 管簡報內容
- `main store` 管編輯器互動狀態
- `snapshot store` 管 undo / redo 歷史版本
- `keyboard store` 管鍵盤環境狀態
- `screen store` 管播放模式

這一篇要回答的是：

> 使用者真的操作畫面時，這些 store 是怎麼串起來的？

---

## 閱讀這篇的方式

每個案例都用同一個格式：

```txt
使用者操作
→ hook / component
→ store action
→ state 改變
→ getter / component 更新
→ 必要時記錄 snapshot
```

讀狀態流時，不要只看「哪個 action 被呼叫」，還要看三件事：

1. **資料來源在哪裡**
2. **誰負責改資料**
3. **改完後誰會重新推導畫面**

---

## 1. 新增空白頁

### 使用者操作

使用者按下新增頁面按鈕，或在縮圖區按下新增頁面的快捷鍵。

### 狀態流

```txt
使用者新增頁面
→ useSlideHandler.createSlide()
→ 建立 emptySlide
→ mainStore.setActiveElementIdList([])
→ slidesStore.addSlide(emptySlide)
→ addHistorySnapshot()
→ snapshotStore.addSnapshot()
→ currentSlide 改變
→ 縮圖區與畫布重新渲染
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `main store` | 清空目前選取的元素 |
| `slides store` | 新頁插入目前頁後方，並更新 `slideIndex` |
| `snapshot store` | 新增一份歷史快照 |

### 原始碼閱讀入口

- `src/hooks/useSlideHandler.ts`
- `src/store/slides.ts`
- `src/hooks/useHistorySnapshot.ts`
- `src/store/snapshot.ts`

### 重要理解

`createSlide()` 不是只呼叫 `slidesStore.addSlide()`。

它還會先清空選取狀態，再加入歷史快照。

所以完整操作不是：

```txt
新增頁面 = slidesStore.addSlide()
```

而是：

```txt
新增頁面 = 清空互動狀態 + 更新內容資料 + 記錄歷史版本
```

## 2. 根據模板新增頁面

### 使用者操作

使用者從模板或頁面資料中新增一頁。

### 狀態流

```txt
使用者選擇模板頁
→ useSlideHandler.createSlideByTemplate(slide)
→ 重新產生 slide id
→ 重新產生 element id
→ 重新對應 groupId
→ mainStore.setActiveElementIdList([])
→ slidesStore.addSlide(newSlide)
→ addHistorySnapshot()
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `main store` | 清空目前選取元素 |
| `slides store` | 插入新頁，頁面與元素 id 會被重建 |
| `snapshot store` | 加入歷史快照 |

### 重要理解

模板頁不能直接塞進 `slides`。

原因是原本模板中的元素 id 可能與既有頁面衝突，因此要先重建：

- `slide.id`
- `element.id`
- `groupId`

這也是為什麼新增模板頁會經過 `createElementIdMap()`。

## 3. 刪除頁面

### 使用者操作

使用者在縮圖區選取頁面後刪除。

### 狀態流

```txt
使用者刪除頁面
→ useSlideHandler.deleteSlide()
→ 判斷是否刪除全部頁面
→ 如果全部刪除：resetSlides()
→ 如果不是全部刪除：slidesStore.deleteSlide(targetSlidesId)
→ mainStore.updateSelectedSlidesIndex([])
→ addHistorySnapshot()
→ 縮圖區與畫布重新渲染
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `slides store` | 移除指定頁面，重新計算 `slideIndex` |
| `main store` | 清空縮圖多選狀態 |
| `snapshot store` | 記錄刪除後的版本 |

### `slidesStore.deleteSlide()` 的特殊處理

刪除頁面時有兩個重點：

1. 如果刪掉的頁面有 `sectionTag`，會嘗試把章節標記轉移給下一頁。
2. 刪除後會把 `slideIndex` 移到合理位置，避免指向不存在的頁面。

### 重要理解

刪除頁面不是單純：

```txt
slides.splice(index, 1)
```

它還會處理：

- 多頁刪除
- 章節標記延續
- 目前頁索引修正
- 縮圖選取狀態清空
- 歷史快照

## 4. 切換目前頁面

### 使用者操作

使用者點擊縮圖，或使用上下方向鍵切換頁面。

### 狀態流

```txt
使用者切換頁面
→ useSlideHandler.updateSlideIndex(command)
→ 如果目前有選取元素，先清空 activeElementIdList
→ slidesStore.updateSlideIndex(newIndex)
→ currentSlide getter 指向新的頁面
→ 畫布、縮圖選中狀態、右側面板更新
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `main store` | 清空目前元素選取狀態 |
| `slides store` | 更新 `slideIndex` |

### 重要理解

頁面切換的核心不是改整份 `slides`，而是改：

```ts
slideIndex
```

`currentSlide` 是根據：

```ts
slides[slideIndex]
```

動態推導出來的。

---

## 5. 選取單一元素

### 使用者操作

使用者點擊畫布上的某個元素。

### 狀態流

```txt
使用者點擊元素
→ useSelectElement.selectElement(id)
→ 檢查是否已經是 handleElement
→ 檢查元素是否被隱藏
→ 檢查元素是否被鎖定
→ mainStore.setActiveElementIdList([id])
→ activeElementIdList 更新
→ handleElementId 同步更新
→ activeElementList / handleElement getter 重新推導
→ 選取框、工具列、屬性面板更新
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `main store` | 更新 `activeElementIdList` 與 `handleElementId` |
| `slides store` | 不改變內容，只提供元素資料給 getter 查詢 |

### 重要理解

`main store` 不保存完整元素。

它只保存：

```ts
activeElementIdList: string[]
handleElementId: string
```

真正的元素資料仍然在：

```ts
slidesStore.currentSlide.elements
```

所以選取元素的本質是：

```txt
main store 存 ID
→ getter 回到 slides store 找元素資料
```

## 6. 全選目前頁面的元素

### 使用者操作

使用者在編輯區按下 `Ctrl + A`。

### 狀態流

```txt
使用者按 Ctrl + A
→ useGlobalHotkey.selectAll()
→ editorAreaFocus 為 true
→ useSelectElement.selectAllElements()
→ 從 currentSlide.elements 過濾出可選元素
→ 排除 locked 元素
→ 排除 hiddenElementIdList 中的元素
→ mainStore.setActiveElementIdList(newActiveElementIdList)
→ 畫布顯示多選狀態
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `main store` | 更新目前選取的元素 ID 集合 |
| `slides store` | 提供目前頁面的元素清單 |

### 重要理解

全選不是選取所有元素，而是選取：

```txt
沒有 lock 且沒有 hidden 的元素
```

這代表 `useSelectElement` 不只是呼叫 store，它也承載了選取規則。

---

## 7. 全選頁面

### 使用者操作

使用者在縮圖區按下 `Ctrl + A`。

### 狀態流

```txt
使用者按 Ctrl + A
→ useGlobalHotkey.selectAll()
→ thumbnailsFocus 為 true
→ useSlideHandler.selectAllSlide()
→ mainStore.setActiveElementIdList([])
→ mainStore.updateSelectedSlidesIndex(allIndexes)
→ 縮圖區顯示多選頁面
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `main store` | 清空元素選取，更新縮圖頁面選取索引 |
| `slides store` | 提供 slides 長度 |

### 重要理解

同樣是 `Ctrl + A`，行為會依焦點不同而改變：

| 焦點 | 結果 |
|---|---|
| `editorAreaFocus` | 全選目前頁面的元素 |
| `thumbnailsFocus` | 全選縮圖區的頁面 |

這是 `main store` 中 `editorAreaFocus` / `thumbnailsFocus` 的價值。

## 8. Tab 切換操作元素

### 使用者操作

使用者在編輯區按下 `Tab`。

### 狀態流

```txt
使用者按 Tab
→ useGlobalHotkey.tabActiveElement()
→ 如果目前頁沒有元素，直接返回
→ 如果目前沒有 handleElementId，選第一個元素
→ 如果已經有 handleElementId，找到下一個元素
→ mainStore.setActiveElementIdList([nextElementId])
→ handleElementId 更新
→ 選取框切到下一個元素
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `main store` | 更新目前操作元素 |
| `slides store` | 提供目前頁元素順序 |

### 重要理解

`Tab` 的選取順序來自：

```ts
currentSlide.elements
```

所以元素在陣列中的順序會影響 Tab 切換順序。

## 9. 方向鍵移動元素

### 使用者操作

使用者選取元素後按方向鍵。

### 狀態流

```txt
使用者按方向鍵
→ useGlobalHotkey.move(key)
→ 如果 activeElementIdList 有資料：useMoveElement.moveElement(key)
→ 根據 key 計算新的 left / top
→ 如果 activeGroupElementId 存在，優先移動群組內被選取成員
→ 否則移動 activeElementIdList 中的元素
→ slidesStore.updateSlide({ elements: newElementList })
→ addHistorySnapshot()
→ 畫布重新渲染元素位置
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `main store` | 提供目前選取元素 ID 與群組內操作 ID |
| `slides store` | 更新目前頁面的 `elements` |
| `snapshot store` | 記錄移動後版本 |

### 重要理解

移動元素時：

- `main store` 決定「誰被移動」
- `slides store` 負責「元素位置真的改變」
- `snapshot store` 負責「這次移動可以 undo」

這是非常典型的跨 store 狀態流。

---

## 10. 方向鍵切換頁面

### 使用者操作

使用者沒有選取元素，但按上下方向鍵。

### 狀態流

```txt
使用者按 UP / DOWN
→ useGlobalHotkey.move(key)
→ activeElementIdList 為空
→ 如果 key 是 UP / DOWN：useSlideHandler.updateSlideIndex(key)
→ slidesStore.updateSlideIndex(newIndex)
→ currentSlide 改變
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `slides store` | 更新目前頁面索引 |
| `main store` | 必要時清空選取狀態 |

### 重要理解

方向鍵有兩種語意：

| 條件 | 行為 |
|---|---|
| 有選取元素 | 移動元素 |
| 沒有選取元素，按上下鍵 | 切換頁面 |

所以不能只看 key，還要看目前互動狀態。

## 11. 刪除元素

### 使用者操作

使用者選取元素後按 `Delete` 或 `Backspace`。

### 狀態流

```txt
使用者按 Delete / Backspace
→ useGlobalHotkey.remove()
→ 如果 activeElementIdList 有資料：useDeleteElement.deleteElement()
→ 判斷是否正在操作群組內元素
→ 產生 newElementList
→ mainStore.setActiveElementIdList([])
→ slidesStore.updateSlide({ elements: newElementList })
→ addHistorySnapshot()
→ 畫布重新渲染
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `main store` | 清空選取元素 |
| `slides store` | 移除目前頁面的元素 |
| `snapshot store` | 記錄刪除後版本 |

### 重要理解

刪除元素後一定要清空選取狀態。

原因是被刪除的元素 id 已經不存在，如果 `activeElementIdList` 還保留舊 id，就會產生「互動狀態指向不存在內容」的問題。

---

## 12. 刪除全部元素

### 使用者操作

使用者觸發「刪除目前頁全部元素」的功能。

### 狀態流

```txt
使用者刪除全部元素
→ useDeleteElement.deleteAllElements()
→ 檢查 currentSlide.elements 是否為空
→ mainStore.setActiveElementIdList([])
→ slidesStore.updateSlide({ elements: [] })
→ addHistorySnapshot()
```

### 重要理解

這個流程和刪除部分元素很像，但不依賴 `activeElementIdList`。

它直接把目前頁的元素陣列清空。

---

## 13. 修改元素樣式或內容

### 使用者操作

使用者在右側面板修改元素樣式，例如字體、顏色、位置、尺寸、邊框、陰影等。

### 常見狀態流

```txt
使用者修改屬性
→ component / hook 收集新的 props
→ slidesStore.updateElement({ id, props })
→ currentSlide.elements 中對應元素被合併更新
→ 畫布與屬性面板重新渲染
→ 必要時 addHistorySnapshot()
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `slides store` | 更新元素內容資料 |
| `main store` | 通常只提供目前操作元素 id |
| `snapshot store` | 若該操作需要 undo，就記錄快照 |

### 重要理解

`updateElement()` 支援三種重要情境：

1. 更新單一元素。
2. 更新多個元素。
3. 指定 `slideId` 跨頁更新元素。

所以它不是單純的「目前頁單元素 setter」。

---

## 14. 移除元素指定屬性

### 使用者操作

使用者清除某個元素的特定屬性，例如取消某種樣式。

### 狀態流

```txt
使用者清除屬性
→ slidesStore.removeElementProps({ id, propName })
→ 找到目前頁對應元素
→ 使用 omit 移除指定欄位
→ 更新 currentSlide.elements
→ 畫面重新渲染
```

### 重要理解

`removeElementProps()` 和 `updateElement()` 不同。

| action | 用途 |
|---|---|
| `updateElement()` | 合併或覆蓋元素屬性 |
| `removeElementProps()` | 從元素物件中移除某些欄位 |

---

## 15. 複製、剪下、快速複製

### 使用者操作

使用者按下：

- `Ctrl + C`
- `Ctrl + X`
- `Ctrl + D`

### 狀態流

```txt
使用者按快捷鍵
→ useGlobalHotkey.copy() / cut() / quickCopy()
→ 判斷 activeElementIdList 是否有資料
→ 如果有選取元素：操作元素
→ 如果縮圖區有焦點：操作頁面
```

### 重要理解

複製、剪下、快速複製是「上下文敏感」操作。

| 條件 | 操作對象 |
|---|---|
| `activeElementIdList.length > 0` | 元素 |
| `thumbnailsFocus === true` | 頁面 |

這代表快捷鍵不是單純對應一個 action，而是要先判斷目前焦點與選取狀態。

## 16. Undo

### 使用者操作

使用者按下 `Ctrl + Z`。

### 狀態流

```txt
使用者按 Ctrl + Z
→ useGlobalHotkey.undo()
→ useHistorySnapshot.undo()
→ snapshotStore.unDo()
→ snapshotCursor - 1
→ 從 IndexedDB 依 id 排序取出快照陣列
→ 取得上一份 snapshot
→ slidesStore.setSlides(snapshot.slides)
→ slidesStore.updateSlideIndex(snapshot.index)
→ snapshotStore.setSnapshotCursor(newCursor)
→ mainStore.setActiveElementIdList([])
→ 畫面回到上一個歷史版本
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `snapshot store` | 游標往前移動 |
| `slides store` | 整份簡報內容回復到快照版本 |
| `main store` | 清空選取元素 |

### 重要理解

Undo 回復的是：

```txt
整份 slides + slideIndex
```

不是回復某一個元素的某一個欄位。

---

## 17. Redo

### 使用者操作

使用者按下 `Ctrl + Y`。

### 狀態流

```txt
使用者按 Ctrl + Y
→ useGlobalHotkey.redo()
→ useHistorySnapshot.redo()
→ snapshotStore.reDo()
→ snapshotCursor + 1
→ 從 IndexedDB 取出下一份 snapshot
→ slidesStore.setSlides(snapshot.slides)
→ slidesStore.updateSlideIndex(snapshot.index)
→ snapshotStore.setSnapshotCursor(newCursor)
→ mainStore.setActiveElementIdList([])
```

### 重要理解

Redo 的邏輯和 Undo 對稱，只是游標方向不同。

但如果使用者在 Undo 之後又做了新的操作，`addSnapshot()` 會刪除游標後面的快照，Redo 分支就不存在了。

---

## 18. Undo 後新增操作

### 使用者操作

使用者先 undo 幾次，然後新增或修改內容。

### 狀態流

```txt
使用者 undo
→ snapshotCursor 往前移
→ 使用者做新操作
→ addHistorySnapshot()
→ snapshotStore.addSnapshot()
→ 刪除 snapshotCursor 後方的快照
→ 新增新的快照
→ snapshotCursor 指向最新快照
```

### 重要理解

這就是一般編輯器的歷史分支規則：

```txt
A → B → C → D
        ↑ undo 到 B
        做新操作 E
結果變成：A → B → E
C、D 被刪掉
```

## 19. 畫布縮放

### 使用者操作

使用者按：

- `Ctrl + -`
- `Ctrl + =`
- `Ctrl + 0`

### 狀態流

```txt
使用者按縮放快捷鍵
→ useGlobalHotkey.scaleCanvas() / resetCanvas()
→ useScaleCanvas.scaleCanvas(command)
→ mainStore.setCanvasPercentage(percentage)

```

### 重置畫布

```txt
使用者按 Ctrl + 0
→ resetCanvas()
```

### 重要理解

畫布縮放屬於互動顯示狀態，所以在 `main store`，不是 `slides store`。

簡報內容沒有變，只是觀看比例變了。

## 20. 開關搜尋面板

### 使用者操作

使用者按 `Ctrl + F`。

### 狀態流

```txt
使用者按 Ctrl + F
→ useGlobalHotkey
→ mainStore.setSearchPanelState(!showSearchPanel)
→ 搜尋面板開啟或關閉
```

### 重要理解

面板開關屬於 UI 互動狀態，所以全部放在 `main store`。

## 21. 使用快捷鍵進入元素插入狀態

### 使用者操作

使用者在編輯區按：

- `T`：插入文字
- `R`：插入矩形
- `O`：插入圓形
- `L`：插入線條

### 狀態流

```txt
使用者按 T / R / O / L
→ useGlobalHotkey
→ 確認 editorAreaFocus 為 true
→ 確認沒有 Ctrl / Shift 修飾鍵
→ mainStore.setCreatingElement(...)
→ 畫布進入繪製 / 插入元素狀態
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `main store` | 設定 `creatingElement` |
| `slides store` | 這一刻還不一定新增元素，真正畫完才會寫入內容 |

### 重要理解

`creatingElement` 表示「正在準備建立元素」，不是「元素已經存在於 slides」。

真正的元素資料要等插入完成後才會進入 `slidesStore.currentSlide.elements`。

---

## 22. 進入播放模式：從目前頁開始

### 使用者操作

使用者按 `Shift + F5`。

### 狀態流

```txt
使用者按 Shift + F5
→ useGlobalHotkey
→ useScreening.enterScreening()
→ enterFullscreen()
→ screenStore.setScreening(true)
→ 進入播放模式
→ keyboardStore.setShiftKeyState(false)
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `screen store` | `screening` 設為 `true` |
| `slides store` | 不改變 `slideIndex`，從目前頁播放 |
| `keyboard store` | 重置 shift 狀態 |

### 重要理解

從目前頁播放不會改變 `slideIndex`。

它只是把目前的頁面帶入播放模式。

---

## 23. 進入播放模式：從第一頁開始

### 使用者操作

使用者按 `F5`。

### 狀態流

```txt
使用者按 F5
→ useGlobalHotkey
→ useScreening.enterScreeningFromStart()
→ slidesStore.updateSlideIndex(0)
→ enterScreening()
→ screenStore.setScreening(true)
→ 進入播放模式
```

### 關鍵狀態變化

| store | 變化 |
|---|---|
| `slides store` | `slideIndex` 設為 `0` |
| `screen store` | `screening` 設為 `true` |

### 重要理解

`F5` 和 `Shift + F5` 的差異在於：

| 快捷鍵 | 行為 |
|---|---|
| `F5` | 從第一頁開始播放 |
| `Shift + F5` | 從目前頁開始播放 |

---

## 24. 退出播放模式

### 使用者操作

使用者退出簡報播放。

### 狀態流

```txt
使用者退出播放
→ useScreening.exitScreening()
→ screenStore.setScreening(false)
→ 如果目前是 fullscreen，呼叫 exitFullscreen()
→ 回到編輯模式
```

### 重要理解

`screen store` 只保存播放模式旗標：

```ts
screening: boolean
```

它不是播放引擎，也不保存播放步驟或動畫進度。

---

## 25. 按鍵狀態更新

### 使用者操作

使用者按下或放開 Ctrl、Shift、Space。

### 狀態流

```txt
keydown
→ useGlobalHotkey.keydownListener
→ keyboardStore.setCtrlKeyState(true)
→ keyboardStore.setShiftKeyState(true)
→ keyboardStore.setSpaceKeyState(true)
```

```txt
keyup / window blur
→ useGlobalHotkey.keyupListener
→ keyboardStore.setCtrlKeyState(false)
→ keyboardStore.setShiftKeyState(false)
→ keyboardStore.setSpaceKeyState(false)
```

### 重要理解

`keyboard store` 不是快捷鍵執行器。

它只記錄鍵盤環境狀態，例如：

```ts
ctrlKeyState
shiftKeyState
spaceKeyState
```

真正決定快捷鍵要做什麼的是 `useGlobalHotkey.ts`。

---

## 常見誤區

### 誤區 1：以為所有操作都直接寫在 store 裡

不是。

store 多數時候只提供狀態與 action。

完整流程通常在：

```txt
hook / component
```

裡面串起來。

---

### 誤區 2：以為 `main store` 會保存元素資料

不是。

`main store` 主要保存互動狀態，例如選取 id。

完整元素資料仍然在 `slides store`。

---

### 誤區 3：以為每次 `slides store` 更新都自動產生快照

不是。

多數情況要透過：

```ts
addHistorySnapshot()
```

明確加入歷史快照。

所以讀程式碼時要找：

```txt
slidesStore.updateXXX()
addHistorySnapshot()
```

這種組合。

---

### 誤區 4：以為 undo / redo 回復的是操作指令

不是。

PPTist 的 undo / redo 回復的是快照版本。

也就是：

```txt
某個時間點的 slides + slideIndex
```

---

## 這篇的核心總結

`PPTist` 的狀態流可以用一句話理解：

```txt
hook 負責串流程，store 負責保存與更新狀態，getter 負責推導畫面需要的資料。
```

五個 store 的分工是：

```txt
slides store：內容真的改在哪裡
main store：使用者目前怎麼操作
snapshot store：能不能回到過去版本
keyboard store：目前鍵盤環境如何
screen store：目前是不是播放模式
```
