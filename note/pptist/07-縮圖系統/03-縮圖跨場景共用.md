# 03-縮圖跨場景共用

返回：[04-縮圖系統](./README.md)

## 這篇在解決什麼問題

前面兩篇已經拆過：

```text
ThumbnailSlide
  負責把一張 slide 依照指定寬度渲染出來

ThumbnailElement
  負責把 slide.elements 分派給對應的 BaseElement
```

如果只看 `src/views/Editor/Thumbnails/index.vue`，很容易以為 `ThumbnailSlide` 只是「編輯器左側縮圖」的內部元件。

但從整個 `PPTist` 來看，它更像是：

> 一個可被多個場景共用的「單頁 slide 縮放 renderer」。

這裡故意不用只說「縮圖 renderer」，是因為它不一定只輸出很小的縮圖。

例如：

* 左側縮圖列表用 `size = 120`。
* 模板預覽用 `size = 180`。
* 連結預覽用 `size = 500`。
* 匯出圖片 / PDF 用 `size = 1600`。
* 手機播放場景會用接近播放器畫面寬度的 `size`。

所以更精準的理解是：

> `ThumbnailSlide` 的核心能力不是「小」，而是「用同一份 slide 資料，以指定寬度輸出一張投影片畫面」。

這篇要回答三個問題：

1. 哪些場景在使用 `ThumbnailSlide`？
2. 這些場景的差異是什麼？
3. 為什麼同一個 renderer 可以支撐預覽、導覽、排序、播放與匯出？

## 一句話看懂跨場景共用

`PPTist` 把「單頁 slide 如何渲染」抽到 `ThumbnailSlide`，把「這張畫面在不同場景中要怎麼被使用」留給外層場景容器。

可以畫成：

```text
共用核心：ThumbnailSlide
  ├─ 輸入 slide
  ├─ 輸入 size
  ├─ 可選 visible
  └─ 輸出一張指定尺寸的 slide 畫面

      ↓ 被不同場景包起來

Editor/Thumbnails/index.vue
  └─ 左側頁面管理列表：切頁、多選、拖曳、section、右鍵

Editor/Thumbnails/Templates.vue
  └─ 模板挑選：分類、預覽、插入單張、插入全部

Editor/Canvas/LinkDialog.vue
  └─ 投影片連結預覽：選擇要連到哪一頁

Screen/SlideThumbnails.vue
  └─ 播放時的全頁縮圖導覽：點擊跳頁並關閉導覽

Screen/BottomThumbnails.vue
  └─ 播放底部橫向縮圖列：橫向捲動、點擊跳頁、自動置中

Screen/PresenterView.vue
  └─ 講者模式縮圖列：輔助跳頁、同步目前頁、搭配備註區

Mobile/MobilePreview.vue
  └─ 手機直向預覽：用接近手機寬度顯示整份簡報

Mobile/MobileThumbnails.vue
  └─ 手機縮圖排序列：橫向列表、點擊切頁、長按拖曳排序

Mobile/MobilePlayer.vue
  └─ 手機播放畫面：用 ThumbnailSlide 渲染目前頁附近的 slide

ExportDialog/ExportImage.vue
  └─ 匯出圖片：用大尺寸渲染後交給 exportImage

ExportDialog/ExportPDF.vue
  └─ 匯出 PDF：用大尺寸渲染後交給 print
```

這張圖要抓住一個重點：

> 場景越多，越能看出 `ThumbnailSlide` 的邊界必須保持單純。

如果 `ThumbnailSlide` 裡面混入左側列表的多選、拖曳、section、右鍵選單，它就很難被播放、手機、匯出場景共用。

## 原始碼對照表

| --------- | ----------------------------------------------- | --------------------: | ------------------------- | ------------------------------ |
| 編輯器左側縮圖列表 | `src/views/Editor/Thumbnails/index.vue`         |                 `120` | `index < slidesLoadLimit` | 頁面管理、切頁、多選、拖曳、section、右鍵、備註    |

| 講者模式縮圖列   | `src/views/Screen/PresenterView.vue`            | `120 / viewportRatio` | `index < slidesLoadLimit` | 講者模式中的縮圖跳頁與目前頁同步               |

| 匯出圖片      | `src/views/Editor/ExportDialog/ExportImage.vue` |                `1600` | 不傳，預設完整渲染                 | 將指定範圍 slide 渲染成圖片輸出來源          |
| 匯出 PDF    | `src/views/Editor/ExportDialog/ExportPDF.vue`   |                `1600` | 不傳，預設完整渲染                 | 將目前頁或全部 slide 渲染後交給列印 / PDF 流程 |

這張表可以看出幾個規律：

1. **長列表預覽場景**通常會搭配 `visible` 和 `useLoadSlides()`。
2. **單頁預覽場景**通常不需要 `visible`。
3. **匯出場景**不能延後渲染，因為輸出時必須拿到完整畫面。
4. `size` 不一定是固定小值，也可以根據高度、手機寬度、匯出解析度動態計算。

## 1. 編輯器左側頁面管理：`Editor/Thumbnails/index.vue`

這是最複雜、也最容易讓人誤會的使用場景。

它的核心使用方式是：

```vue
<ThumbnailSlide
  class="thumbnail"
  :slide="element"
  :size="120"
  :visible="index < slidesLoadLimit"
/>
```

在這裡，`ThumbnailSlide` 只負責畫出單頁縮圖。

外層 `Editor/Thumbnails/index.vue` 才負責：

* 顯示頁碼。
* 顯示 active / selected 狀態。
* 點擊切換目前頁。
* Ctrl / Shift 多選頁面。
* 拖曳排序。
* section 顯示與編輯。
* 右鍵選單。
* 備註旗標。
* 雙擊播放。
* 自動捲動到目前頁。
* 縮圖列表焦點。
* 接入 `useLoadSlides()`。

這個場景說明了 `ThumbnailSlide` 的第一個設計價值：

> 單頁畫面渲染和列表互動被拆開。

如果你要自己做 PPT 編輯器，這個切法很值得學。

不要把「一張 slide 怎麼畫」和「縮圖列表怎麼操作」寫在同一個元件裡。

## 2. 模板挑選：`Editor/Thumbnails/Templates.vue`

模板挑選場景使用：

```vue
<ThumbnailSlide
  class="thumbnail"
  :slide="slide"
  :size="180"
/>
```

這裡有幾個特徵：

* `slide` 不是目前簡報中的 slide，而是模板資料中的 slide。
* `size` 是 `180`，比左側列表大一點。
* 沒有傳 `visible`，所以使用 `ThumbnailSlide` 預設的完整渲染。
* 外層有分類、類型篩選、載入狀態、插入按鈕。
* 點擊「插入模板」時，不是切頁，而是 emit 給外層插入模板。

這裡很能看出跨場景共用的價值。

同樣是 `ThumbnailSlide`：

```text
在左側縮圖列表：代表簡報中的某一頁，可切頁、可排序、可多選
在模板挑選器：代表可插入的模板頁，可預覽、可插入
```

縮圖畫面一樣，但外層語義完全不同。

所以場景差異不應該寫進 `ThumbnailSlide`。

## 3. 投影片連結預覽：`Editor/Canvas/LinkDialog.vue`

在連結設定視窗中，當使用者選擇連結到某一張投影片時，會顯示預覽：

```vue
<ThumbnailSlide
  class="thumbnail"
  :slide="selectedSlide"
  :size="500"
/>
```

這個場景有幾個特徵：

* 只顯示單張 slide。
* 尺寸較大，`size = 500`。
* 不需要 lazy load。
* 目的不是頁面管理，而是幫使用者確認連結目標。

這裡要建立一個重要觀念：

> `ThumbnailSlide` 不一定要出現在列表中，它也可以作為單頁預覽元件使用。

這和左側縮圖列表完全不同。

左側列表關心的是：

```text
我現在要操作哪幾頁？
```

連結預覽關心的是：

```text
我選到的連結目標是不是這張 slide？
```

同樣的 renderer，被不同外層賦予不同產品語義。

## 4. 播放全頁縮圖導覽：`Screen/SlideThumbnails.vue`

播放模式中有一個全頁縮圖導覽場景。

使用方式是：

```vue
<ThumbnailSlide
  :slide="slide"
  :size="150"
  :visible="index < slidesLoadLimit"
/>
```

外層行為是：

* 固定全螢幕覆蓋層。
* 顯示所有 slide 的縮圖。
* 目前頁加上 active 樣式。
* 點擊某張縮圖後跳到該頁。
* 跳頁後關閉縮圖導覽。
* 使用 `useLoadSlides()` 分批渲染。

這個場景和編輯器左側列表有點像，都是多頁列表。

但它少了很多頁面管理能力：

* 沒有拖曳排序。
* 沒有多選。
* 沒有 section 編輯。
* 沒有右鍵選單。
* 沒有新增 / 刪除頁面。

它的核心目標是：

```text
播放時快速跳到某一頁
```

所以同樣是縮圖列表，外層責任可以非常不同。

## 5. 播放底部縮圖列：`Screen/BottomThumbnails.vue`

播放頁底部有一條橫向縮圖列。

使用方式是：

```vue
<ThumbnailSlide
  :slide="slide"
  :size="100 / viewportRatio"
  :visible="index < slidesLoadLimit"
/>
```

這裡的 `size` 很值得注意。

`ThumbnailSlide` 的 `size` 代表寬度。

但底部縮圖列比較像是想固定高度約 `100px`。

因為：

```ts
height = size * viewportRatio
```

如果希望：

```ts
height = 100
```

就要反推：

```ts
size = 100 / viewportRatio
```

所以這個場景展示了 `size` API 的一個重要用法：

> 當場景想固定高度時，可以用 `目標高度 / viewportRatio` 反推要傳入的寬度。

外層還做了幾件事：

* 橫向排列縮圖。
* 將滑鼠滾輪的 `deltaY` 轉成水平捲動。
* 點擊縮圖切到指定頁。
* 監聽 `slideIndex`，讓目前頁縮圖自動捲動到中間。
* 使用 `useLoadSlides()` 分批渲染。

這些都不是 `ThumbnailSlide` 的責任。

它只管輸出縮圖畫面。

## 6. 講者模式縮圖列：`Screen/PresenterView.vue`

講者模式中也使用 `ThumbnailSlide` 顯示底部縮圖列：

```vue
<ThumbnailSlide
  :slide="slide"
  :size="120 / viewportRatio"
  :visible="index < slidesLoadLimit"
/>
```

這和 `BottomThumbnails.vue` 很像，但它出現在講者模式的大 layout 中。

講者模式同時還有：

* 目前播放畫面。
* 備註區。
* 倒數計時。
* 畫筆工具。
* 全螢幕相關控制。
* 觀眾視圖入口。

縮圖列在這裡不是主要畫面，而是輔助導覽工具。

外層同樣負責：

* 橫向捲動。
* 點擊跳頁。
* 目前頁 active 樣式。
* 目前頁自動置中。
* 使用 `useLoadSlides()`。

所以 `PresenterView` 說明了另一個重點：

> `ThumbnailSlide` 可以被放進更大的複合場景中，只負責其中一小塊 slide 預覽。

## 7. 手機直向預覽：`Mobile/MobilePreview.vue`

手機預覽場景使用：

```vue
<ThumbnailSlide
  :slide="slide"
  :size="screenWidth - 20"
  :visible="index < slidesLoadLimit"
/>
```

這裡的縮圖其實已經不像「小縮圖」，更像手機畫面中的整頁預覽。

它的特徵是：

* 垂直排列所有 slide。
* `size` 根據手機容器寬度計算。
* 每張 slide 接近螢幕寬度。
* 使用 `useLoadSlides()` 分批渲染。
* 底部有切換到編輯、播放、AIPPT 的 menu。

這個場景再次說明：

> `ThumbnailSlide` 的使用方式不是固定小尺寸，而是根據場景給定輸出寬度。

在手機直向預覽中，它輸出的其實是「縮放後的完整 slide 預覽」。

## 8. 手機縮圖排序列：`Mobile/MobileThumbnails.vue`

手機端還有一個橫向縮圖列，使用：

```vue
<ThumbnailSlide
  class="thumbnail"
  :slide="element"
  :size="120"
  :visible="index < slidesLoadLimit"
/>
```

外層是 `Draggable`：

```vue
<Draggable
  :modelValue="slides"
  :delayOnTouchOnly="true"
  :delay="800"
  @end="handleDragEnd"
>
```

這個場景和桌面左側縮圖列表很像，但簡化很多。

它負責：

* 橫向顯示頁面縮圖。
* 顯示頁碼 label。
* 點擊切頁。
* 目前頁 active 樣式。
* 長按拖曳排序。
* 拖曳結束後呼叫 `sortSlides(newIndex, oldIndex)`。
* 使用 `useLoadSlides()` 分批渲染。

它沒有桌面端那麼多功能：

* 沒有 Ctrl / Shift 多選。
* 沒有右鍵選單。
* 沒有 section 編輯。
* 沒有備註旗標。

這說明：

> 同樣是頁面排序場景，桌面版和手機版可以共用 `ThumbnailSlide`，但外層互動設計完全不同。

## 9. 手機播放畫面：`Mobile/MobilePlayer.vue`

手機播放場景是跨場景共用中很值得注意的一個案例。

它使用：

```vue
<ThumbnailSlide
  :slide="slide"
  :size="slideSize.width"
/>
```

而且外層有：

```vue
v-if="Math.abs(slideIndex - index) < 2"
```

意思是：

* 只渲染目前頁附近的 slide。
* 每張 slide 的尺寸根據播放器畫面計算。
* 不傳 `visible`，因此使用 `ThumbnailSlide` 預設完整渲染。
* 外層負責轉場 class、觸控切頁、目前頁 / 前後頁狀態。

這裡的 `ThumbnailSlide` 已經不是傳統意義上的「縮圖」。

它在手機播放器中更像：

> 用共用 slide renderer 渲染播放畫面。

這正好證明前面為什麼要把它理解成「可縮放的單頁 slide renderer」，而不是只理解成「小縮圖元件」。

## 10. 匯出圖片：`ExportDialog/ExportImage.vue`

匯出圖片場景使用：

```vue
<ThumbnailSlide
  class="thumbnail"
  v-for="slide in renderSlides"
  :key="slide.id"
  :slide="slide"
  :size="1600"
/>
```

這裡有幾個重點。

### 第一，尺寸很大

`size = 1600`，表示它不是小縮圖，而是用較大的寬度輸出高解析畫面。

### 第二，不傳 `visible`

匯出不能使用 placeholder。

如果某些頁面還沒真正渲染，輸出結果就會錯。

所以匯出場景需要完整渲染。

### 第三，外層負責匯出範圍

`renderSlides` 會根據：

* 全部
* 目前頁
* 自訂範圍

決定要渲染哪些 slide。

### 第四，真正輸出不是 `ThumbnailSlide` 做的

`ThumbnailSlide` 只負責把畫面渲染到 DOM。

真正匯出圖片是外層把容器 ref 交給：

```ts
exportImage(imageThumbnailsRef.value, format, quality, ignoreWebfont)
```

所以匯出圖片可以拆成：

```text
ThumbnailSlide：產生可截圖的 slide DOM
ExportImage.vue：決定範圍、格式、品質、是否忽略線上字體
useExport.exportImage：執行實際圖片輸出
```

這個切分很重要。

## 11. 匯出 PDF：`ExportDialog/ExportPDF.vue`

PDF 匯出也重用 `ThumbnailSlide`。

目前頁：

```vue
<ThumbnailSlide
  class="thumbnail"
  :slide="currentSlide"
  :size="1600"
  v-if="rangeType === 'current'"
/>
```

全部頁：

```vue
<ThumbnailSlide
  class="thumbnail"
  :class="{ 'break-page': (index + 1) % count === 0 }"
  v-for="(slide, index) in slides"
  :key="slide.id"
  :slide="slide"
  :size="1600"
/>
```

這裡的重點是：

* PDF 也使用大尺寸 `1600`。
* 不使用 `visible`，避免 placeholder 進入輸出。
* `rangeType` 支援目前頁或全部頁。
* `count` 控制每頁數量。
* `break-page` class 用來控制列印分頁。
* 最後由 `print(pdfThumbnailsRef.value, pageSize)` 進入列印 / PDF 流程。

這說明 PDF 匯出也不是完全另外做一套 slide renderer。

它仍然是：

```text
先用 ThumbnailSlide 把 slide 渲染成 DOM
再由列印流程把 DOM 輸出成 PDF
```

但 PDF 場景有它自己的外層責任，例如頁面尺寸、邊距、分頁、每頁數量。

這些都不應該塞回 `ThumbnailSlide`。

## 12. 哪些場景適合傳 `visible`

從目前使用方式可以歸納出一個規則。

適合傳 `visible` 的場景通常是：

```text
多頁列表 + 頁數可能很多 + 允許後面的頁面稍晚渲染
```

例如：

* 編輯器左側縮圖列表。
* 播放全頁縮圖導覽。
* 播放底部縮圖列。
* 講者模式縮圖列。
* 手機直向預覽。
* 手機縮圖排序列。

這些場景都會搭配：

```ts
const { slidesLoadLimit } = useLoadSlides()
```

再傳：

```vue
:visible="index < slidesLoadLimit"
```

這種策略的目標是降低初始渲染壓力。

但要記住：

> `visible` 是渲染時機控制，不是資料載入控制。

資料已經在 `slides` 中，`visible` 只決定背景與元素 DOM 是否現在建立。

## 13. 哪些場景不適合傳 `visible`

不適合傳 `visible` 的場景通常是：

```text
需要立即拿到完整畫面，或者頁數少、單頁預覽，不需要分批渲染
```

例如：

* 模板挑選。
* 投影片連結預覽。
* 手機播放畫面。
* 匯出圖片。
* 匯出 PDF。

尤其是匯出場景，不能用 placeholder。

因為輸出流程會讀取 DOM 畫面，如果有些 slide 還停在 placeholder，輸出就會不完整。

所以匯出場景寧可直接完整渲染，並把 `size` 設得很大。

## 14. 各場景真正不同的是什麼

雖然都用 `ThumbnailSlide`，但每個場景差異很大。

差異主要在外層容器，而不是 renderer 本身。

### 差異一：尺寸不同

例如：

```text
120                 編輯器左側列表、手機縮圖列
150                 播放全頁縮圖導覽
180                 模板挑選
500                 投影片連結預覽
100 / viewportRatio 播放底部縮圖列，固定高度推寬度
120 / viewportRatio 講者模式縮圖列，固定高度推寬度
screenWidth - 20    手機直向預覽
slideSize.width     手機播放畫面
1600                匯出圖片 / PDF
```

這表示 `size` 是場景配置，而不是 renderer 內部寫死。

### 差異二：排列方式不同

有些是垂直列表：

* 編輯器左側縮圖列表。
* 手機直向預覽。

有些是橫向列表：

* 播放底部縮圖列。
* 講者模式縮圖列。
* 手機縮圖排序列。

有些是網格或包裹排列：

* 播放全頁縮圖導覽。
* 模板挑選。

有些不是列表：

* 投影片連結預覽。
* 手機播放器目前頁。
* 匯出圖片 / PDF 的輸出 DOM。

### 差異三：互動不同

不同場景的點擊行為不同：

| 場景      | 點擊行為                |
| ------- | ------------------- |
| 編輯器左側列表 | 切頁、多選、右鍵操作、雙擊播放     |
| 模板挑選    | 插入模板                |
| 連結預覽    | 幾乎只是確認畫面            |
| 播放縮圖導覽  | 跳到指定頁並關閉導覽          |
| 底部縮圖列   | 跳到指定頁               |
| 手機縮圖列   | 切頁、長按拖曳排序           |
| 匯出場景    | 不面向使用者直接互動，而是作為輸出來源 |

這些差異都留在場景容器。

### 差異四：是否需要分批渲染

多頁預覽和導覽場景通常使用 `useLoadSlides()`。

匯出和單頁預覽通常不使用。

### 差異五：資料來源可能不同

大多數場景直接使用 `slides store` 的 `slides`。

但模板挑選場景使用的是模板資料中的 slides。

匯出圖片場景則會根據範圍產生 `renderSlides`。

所以 `ThumbnailSlide` 只要求有一個符合 `Slide` 結構的資料，不要求這個 slide 一定來自目前簡報列表。

## 15. 哪些東西是跨場景穩定不變的

雖然場景差異很多，但 `ThumbnailSlide` 本身的核心規則是穩定的。

跨場景不變的是：

1. 都接收一張 `slide`。
2. 都透過 `size` 決定輸出寬度。
3. 高度都由 `size * viewportRatio` 推導。
4. 內層仍建立原始投影片內容區。
5. 仍用 `scale = size / viewportSize` 進行整頁縮放。
6. 背景與元素都交給同一套渲染鏈路。
7. 元素仍由 `ThumbnailElement` 分派到 BaseElement。
8. 元素仍透過 `target="thumbnail"` 接收縮圖場景訊號。

也就是說：

```text
場景容器可以千變萬化
但單頁 slide 的渲染模型保持一致
```

這就是跨場景共用能成立的基礎。

## 16. 這種設計的工程價值

### 價值一：畫面一致性高

同一張 slide 在不同地方出現時，不需要每個場景各寫一套渲染邏輯。

例如：

* 左側列表。
* 模板預覽。
* 播放導覽。
* 手機預覽。
* 匯出圖片。
* 匯出 PDF。

都可以基於同一個 `ThumbnailSlide`。

這樣比較不容易出現：

```text
主畫布看起來正常
左側縮圖跑版
手機預覽又是另一種樣子
匯出圖片又和預覽不同
```

### 價值二：維護成本低

如果某個元素在縮圖場景有問題，通常可以回到：

```text
ThumbnailSlide
ThumbnailElement
對應的 BaseElement
```

去看，而不是每個場景各修一份。

### 價值三：新增場景容易

假設未來要新增：

* 分享前預覽。
* 列印前選頁。
* AI 生成結果預覽。
* 版本比較縮圖。
* 頁面搜尋結果預覽。

理論上都可以優先考慮：

```vue
<ThumbnailSlide :slide="slide" :size="someSize" />
```

再由外層決定點擊、排列、載入、選取等行為。

### 價值四：效能策略可以按場景選擇

`ThumbnailSlide` 只提供 `visible`。

要不要搭配 `useLoadSlides()`，由場景自己決定。

這讓它同時適合：

* 長列表預覽：傳 `visible`，分批渲染。
* 匯出輸出：不傳 `visible`，完整渲染。
* 單頁預覽：不傳 `visible`，直接渲染。

## 17. 這種設計的限制

### 限制一：共用 renderer 不代表完全等於正式播放畫布

`ThumbnailSlide` 是用縮放方式輸出 slide 畫面，並透過 `target="thumbnail"` 通知 BaseElement 目前是縮圖場景。

這意味著它很適合預覽、導覽、匯出 DOM 畫面，但不一定等於主編輯畫布或正式播放引擎的全部行為。

例如：

* 編輯互動不在這裡。
* 元素選取框不在這裡。
* 拖曳 resize 不在這裡。
* 動畫播放細節不一定在這裡。

所以不要把 `ThumbnailSlide` 理解成完整編輯器畫布。

它是共用的 slide 畫面 renderer，不是完整 Canvas 編輯系統。

### 限制二：大量頁面最終仍可能建立很多 DOM

`useLoadSlides()` 只是分批放行，不是完整 virtual list。

當頁數很多、元素很多時，最終仍會有不少 DOM。

所以這套策略是務實折衷，不是極限效能方案。

### 限制三：輸出品質仍受 DOM 渲染限制

匯出圖片 / PDF 重用 `ThumbnailSlide` 能提升一致性，但輸出品質仍可能受：

* 字體載入。
* 圖片跨域。
* browser print 行為。
* DOM 轉圖片工具限制。
* CSS 與列印樣式差異。

影響。

所以共用 renderer 可以減少「多套畫面不一致」的問題，但不能保證所有輸出細節都完美。

## 18. 容易誤解的地方

| 容易誤解                             | 精準理解                                                                |
| -------------------------------- | ------------------------------------------------------------------- |
| `ThumbnailSlide` 只屬於左側縮圖列表       | 不是，它被多個場景共用                                                         |
| `ThumbnailSlide` 只能畫小縮圖          | 不是，它可以用 `size=1600` 做匯出，也可以用播放器寬度做手機播放畫面                            |
| 所有使用場景都要傳 `visible`              | 不是，只有長列表預覽類場景通常需要                                                   |
| 匯出場景也可以 lazy load                | 不適合，匯出需要完整畫面，不能輸出 placeholder                                       |
| 不同場景的縮圖互動應該寫在 `ThumbnailSlide` 裡 | 不應該，互動應留在場景容器                                                       |
| 模板預覽的 slide 一定來自目前簡報             | 不一定，它來自模板資料                                                         |
| `size` 一定是固定數字                   | 不一定，也可以是 `100 / viewportRatio`、`screenWidth - 20`、`slideSize.width` |
| 共用 renderer 就等於完整編輯畫布            | 不是，編輯互動、選取、拖曳、動畫等仍由其他場景或元件處理                                        |
| PDF 匯出是完全獨立 renderer             | 不是，它也先用 `ThumbnailSlide` 渲染 DOM，再交給列印流程                             |

## 19. 讀跨場景共用時的建議順序

建議不要一開始就把所有場景混著看。

可以照這個順序：

### 第一步：先看最簡單的單頁預覽

先看：

```text
Editor/Canvas/LinkDialog.vue
```

它只用：

```vue
<ThumbnailSlide :slide="selectedSlide" :size="500" />
```

這能幫你確認：

> `ThumbnailSlide` 可以不在列表裡，只做單頁預覽。

### 第二步：再看模板挑選

看：

```text
Editor/Thumbnails/Templates.vue
```

它會讓你看到：

* `slide` 可以來自模板資料。
* 外層可以加 hover、分類、插入按鈕。
* `ThumbnailSlide` 不需要知道模板插入邏輯。

### 第三步：再看編輯器左側縮圖列表

看：

```text
Editor/Thumbnails/index.vue
```

這是最複雜的頁面管理場景。

此時你會比較清楚：

```text
複雜的是外層互動，不是 ThumbnailSlide 本身
```

### 第四步：看播放相關場景

依序看：

```text
Screen/SlideThumbnails.vue
Screen/BottomThumbnails.vue
Screen/PresenterView.vue
```

目標是理解：

* 同樣是縮圖導覽，佈局可以是全頁、底部橫向、講者模式內嵌。
* `size` 可以根據高度反推。
* 目前頁可以自動捲動到中間。

### 第五步：看手機相關場景

依序看：

```text
MobilePreview.vue
MobileThumbnails.vue
MobilePlayer.vue
```

目標是理解：

* 手機預覽用接近螢幕寬度的 slide。
* 手機縮圖列支援長按拖曳。
* 手機播放器也能使用 `ThumbnailSlide` 作為 slide 畫面輸出。

### 第六步：最後看匯出場景

看：

```text
ExportImage.vue
ExportPDF.vue
```

目標是理解：

* `ThumbnailSlide` 可以用大尺寸輸出。
* 匯出場景不適合 lazy load。
* renderer 只產生 DOM，真正輸出交給 export / print 流程。

## 20. 這篇最重要的結論

### 結論一：`ThumbnailSlide` 的本質是可縮放的單頁 slide renderer

它不只是小縮圖元件。

它可以被用在：

* 小縮圖。
* 中型預覽。
* 手機整頁預覽。
* 手機播放畫面。
* 匯出大圖。

差別只在於外層給它什麼 `size`，以及要不要傳 `visible`。

### 結論二：場景差異應該留在外層容器

切頁、多選、拖曳、排序、分類、插入、備註、播放、匯出、橫向捲動、自動置中，都不是 `ThumbnailSlide` 的責任。

這些差異應該放在各自場景中。

### 結論三：長列表預覽和輸出場景的策略不同

長列表預覽通常使用：

```vue
:visible="index < slidesLoadLimit"
```

匯出場景則應該完整渲染，不要讓 placeholder 混入輸出。

### 結論四：跨場景共用成立的前提是 renderer 邊界乾淨

`ThumbnailSlide` 只接：

```ts
slide
size
visible?
```

這讓它能被不同場景自由組合。

如果它知道太多外層互動，就會失去共用性。

### 結論五：這是讀大型前端專案時很值得學的抽象方式

PPTist 沒有為每個場景都寫一套 slide preview。

它把共同的「單頁 slide 輸出」抽出來，再讓各場景包上自己的外層行為。

這就是元件復用中很重要的原則：

```text
穩定的視覺輸出抽成共用核心
變動的產品行為留在場景容器
```

## 一句話抓核心

`ThumbnailSlide` 能跨編輯器、模板、播放、手機與匯出場景共用，關鍵不是它處理了所有場景邏輯，而是它只專注於「用同一份 slide 資料，以指定寬度輸出投影片畫面」；至於排列、點擊、排序、載入、播放與匯出，全部交給外層場景容器處理。

把這個設計看懂，就能理解為什麼 PPTist 的縮圖系統不是單一 UI 小功能，而是一個支撐多場景 slide 預覽與輸出的共用渲染基礎。
