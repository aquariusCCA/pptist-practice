# 04-焦點、快捷鍵與多選資料流

返回：[04-縮圖系統](./README.md)

## 這篇在解決什麼問題

左側縮圖列表看起來只是「點一下縮圖切頁」，但在 `PPTist` 裡，它其實和三條狀態線綁在一起：

```text
畫布元素選取
頁面多選
全域快捷鍵路由
```

這也是縮圖列表比一般 preview list 複雜的地方。

如果只看 UI，會以為它只是：

```text
點縮圖 -> 切換 slideIndex
```

但實際上它還要回答幾個問題：

1. 使用者現在是在操作畫布元素，還是在操作整頁投影片？
2. 按 `Ctrl + C` 時，應該複製元素，還是複製頁面？
3. 按 `Delete` 時，應該刪除元素，還是刪除頁面？
4. 按 `Ctrl + A` 時，應該全選元素，還是全選投影片？
5. 使用者多選了幾張 slide 後，右鍵選單應該作用在哪些頁面？
6. 使用者切頁後，舊頁面的元素選取狀態要不要保留？
7. 使用者正在編輯 section 標題時，快捷鍵應不應該生效？

這一篇的核心不是縮圖怎麼畫，而是：

> 縮圖列表如何透過焦點、目前頁、多選集合與元素選取狀態，決定快捷鍵和滑鼠操作到底要作用在哪一層。

## 一句話看懂這套資料流

`PPTist` 不是讓所有快捷鍵都無條件操作同一種資料，而是先判斷目前操作上下文：

```text
如果 activeElementIdList 有值
  -> 優先操作畫布元素

否則如果 thumbnailsFocus 為 true
  -> 操作投影片頁面

否則如果 editorAreaFocus 為 true
  -> 操作編輯區相關能力

如果 disableHotkeys 為 true
  -> 多數快捷鍵暫停，避免和輸入框衝突
```

縮圖列表自己的資料流可以畫成：

```text
使用者點進縮圖列表
  -> setThumbnailsFocus(true)
  -> thumbnailsFocus = true

使用者點擊某張縮圖
  -> 根據 ctrlKeyState / shiftKeyState 分流
  -> 更新 _selectedSlidesIndex 或 slideIndex

頁面操作視圖
  -> selectedSlidesIndex = _selectedSlidesIndex + slideIndex

全域快捷鍵觸發
  -> useGlobalHotkey 判斷 activeElementIdList / editorAreaFocus / thumbnailsFocus
  -> 決定 copy / cut / delete / selectAll / create 等操作走元素層還是頁面層

使用者點出縮圖列表
  -> setThumbnailsFocus(false)
  -> thumbnailsFocus = false
  -> 清空 _selectedSlidesIndex
```

這裡最重要的是：

> `thumbnailsFocus` 不是裝飾性 UI 狀態，而是全域快捷鍵路由到頁面操作的入口條件。

```text

```

## 1. 先分清楚兩種選取狀態

在 `PPTist` 裡，「選取」至少有兩層。

### 第一層：畫布元素選取

由：

```ts
mainStore.activeElementIdList
```

表示。

它代表：

```text
目前選到的是當前 slide 裡的元素
```

例如：

* 文字框
* 圖片
* 形狀
* 線段
* 表格
* 圖表

當 `activeElementIdList.length > 0` 時，快捷鍵通常應該優先操作元素。

例如：

```text
Ctrl + C -> 複製元素
Delete   -> 刪除元素
方向鍵   -> 移動元素
```

### 第二層：投影片頁面選取

由兩個狀態共同構成：

```ts
mainStore.selectedSlidesIndex
slidesStore.slideIndex
```

更精準地說：

* `_selectedSlidesIndex`：額外多選的頁面 index。
* `slideIndex`：目前正在編輯的頁面 index。

在縮圖列表和 `useSlideHandler()` 裡，會把兩者組合成：

```ts
const selectedSlidesIndex = computed(() => [
  ..._selectedSlidesIndex.value,
  slideIndex.value,
])
```

這代表：

```text
額外多選頁面 + 目前頁 = 頁面操作集合
```

這兩層選取一定要分清楚。

否則你會搞不清楚：

```text
Delete 到底刪元素還是刪頁面？
Ctrl + C 到底複製元素還是複製投影片？
Ctrl + A 到底全選元素還是全選頁面？
```

## 2. `thumbnailsFocus` 是頁面層快捷鍵入口

左側縮圖列表根節點有：

```vue
<div
  class="thumbnails"
  @mousedown="() => setThumbnailsFocus(true)"
  v-click-outside="() => setThumbnailsFocus(false)"
>
```

對應函式是：

```ts
const setThumbnailsFocus = (focus: boolean) => {
  if (thumbnailsFocus.value === focus) return
  mainStore.setThumbnailsFocus(focus)

  if (!focus) mainStore.updateSelectedSlidesIndex([])
}
```

這段邏輯有三層意思。

### 第一，點進縮圖列表時，頁面操作取得焦點

當使用者在縮圖區按下滑鼠：

```text
setThumbnailsFocus(true)
```

這表示接下來某些快捷鍵可以進入頁面操作語義。

例如：

* `Ctrl + C` 可以複製頁面。
* `Ctrl + X` 可以剪下頁面。
* `Ctrl + D` 可以複製並貼上頁面。
* `Delete` 可以刪除頁面。
* `Enter` 可以新增頁面。

### 第二，點出縮圖列表時，頁面焦點失效

`v-click-outside` 會呼叫：

```text
setThumbnailsFocus(false)
```

這表示頁面層快捷鍵不應再被縮圖列表接管。

### 第三，失焦時清空額外多選

當 `focus = false` 時：

```ts
mainStore.updateSelectedSlidesIndex([])
```

這代表頁面多選狀態的生命週期和縮圖列表焦點綁在一起。

這樣做可以避免一種危險情境：

```text
使用者多選了幾頁
後來點到畫布編輯元素
但多選頁面狀態仍偷偷保留
接著按 Delete 時語義變得混亂
```

所以失焦時清空多選，是為了讓頁面操作狀態更可控。

## 3. `selectedSlidesIndex` 是操作視圖，不是單一 source of truth

這段很容易寫錯。

原始碼裡有：

```ts
const selectedSlidesIndex = computed(() => [
  ..._selectedSlidesIndex.value,
  slideIndex.value,
])
```

這不代表真正的 store 狀態被合併成一份。

真正的資料來源仍然是兩個：

```text
mainStore.selectedSlidesIndex  -> 額外多選頁面
slidesStore.slideIndex         -> 目前頁
```

這個 computed 只是建立一個方便 UI 和頁面操作使用的「操作視圖」。

可以理解成：

```text
使用者明確多選的頁面 + 目前正在編輯的頁面
```

為什麼要這樣？

因為目前頁在使用者心智裡，本來就應該是有效操作目標。

例如：

* 沒有多選時，按 Delete 應該刪目前頁。
* 沒有多選時，按 Ctrl + C 應該複製目前頁。
* 多選幾頁時，目前頁也應該被納入批次操作語義。

所以這個 computed 的語義是：

> 目前頁永遠被納入頁面操作集合。

但精準說法是：

> `selectedSlidesIndex` 是 `_selectedSlidesIndex + slideIndex` 組合出來的操作視圖，不是唯一資料來源。

## 4. `useSlideHandler()` 也使用同一套頁面操作視圖

這個 computed 不只出現在縮圖列表，也出現在 `useSlideHandler()`。

在 `useSlideHandler()` 中：

```ts
const selectedSlidesIndex = computed(() => [
  ..._selectedSlidesIndex.value,
  slideIndex.value,
])

const selectedSlides = computed(() =>
  slides.value.filter((item, index) => selectedSlidesIndex.value.includes(index))
)

const selectedSlidesId = computed(() =>
  selectedSlides.value.map(item => item.id)
)
```

這代表頁面操作規則，例如：

* 複製頁面
* 剪下頁面
* 刪除頁面
* 全選頁面

都會基於這個「額外多選 + 目前頁」的集合來處理。

這讓 UI 顯示和頁面操作規則保持一致。

也就是說：

```text
縮圖列表看到哪些頁面被納入操作集合
useSlideHandler 就用同一套集合去複製 / 刪除 / 剪下
```

這是這套設計穩定的地方。

## 5. 點擊縮圖時的三種模式

點擊縮圖會進入：

```ts
handleClickSlideThumbnail(e, index)
```

這個函式根據鍵盤狀態分成三種模式：

```text
Ctrl  -> 複選切換
Shift -> 區間選取
其他  -> 單選並切頁
```

這裡用的是 `keyboard store` 中的：

```ts
ctrlKeyState
shiftKeyState
```

而不是直接在 click handler 裡只看 `e.ctrlKey` 或 `e.shiftKey`。

因為整個專案已經有全域 keydown / keyup 在維護修飾鍵狀態。

## 6. 點擊前的兩個保護條件

在真正進入 Ctrl / Shift / 一般切頁前，有兩個重要保護。

### 保護一：正在編輯 section 時不處理縮圖點擊

```ts
if (editingSectionId.value) return
```

使用者正在編輯 section 標題時，滑鼠和鍵盤應該以輸入框為主。

此時不應該因為點到縮圖區而觸發：

* 切頁
* 多選
* 排序
* 頁面操作

這可以避免輸入狀態和頁面操作狀態互相打架。

### 保護二：多選狀態下右鍵點到已選頁，不重置選取

原始碼中有類似邏輯：

```ts
const isMultiSelected = selectedSlidesIndex.value.length > 1

if (isMultiSelected && selectedSlidesIndex.value.includes(index) && e.button !== 0) return
```

這是為了右鍵選單體驗。

如果使用者已經多選了幾張投影片，然後在其中一張上按右鍵，正常預期是：

```text
對這一批已選頁面開右鍵選單
```

而不是：

```text
右鍵瞬間把選取狀態改掉
```

這個細節很像桌面檔案管理器。

## 7. 一般單選：清空多選，切換目前頁

沒有 Ctrl / Shift 時，走一般切頁：

```ts
mainStore.updateSelectedSlidesIndex([])
changeSlideIndex(index)
```

意思是：

1. 清空額外多選頁面。
2. 切換目前頁到點擊的 index。

切頁函式是：

```ts
const changeSlideIndex = (index: number) => {
  mainStore.setActiveElementIdList([])

  if (slideIndex.value === index) return
  slidesStore.updateSlideIndex(index)
}
```

這裡最重要的是第一行：

```ts
mainStore.setActiveElementIdList([])
```

也就是：

> 切頁前先清空畫布元素選取。

原因很簡單。

元素選取狀態是屬於某一張 slide 的。

如果從第 1 頁切到第 2 頁，第 1 頁原本選到的元素不應繼續存在於操作語義中。

否則使用者在第 2 頁按 Delete 時，可能語義上仍然連到第 1 頁的元素，這會非常危險。

所以切頁時清掉 `activeElementIdList` 是必要的一致性維護。

## 8. Ctrl 複選：切換某頁是否在集合裡

當 `ctrlKeyState.value` 為 true 時，進入 Ctrl 複選。

Ctrl 複選的基本語義是：

```text
點未選頁 -> 加入集合
點已選頁 -> 移出集合
```

但目前頁有特殊處理。

如果 Ctrl 點到目前頁：

```ts
if (slideIndex.value === index) {
  if (!isMultiSelected) return

  const newSelectedSlidesIndex = selectedSlidesIndex.value.filter(item => item !== index)
  mainStore.updateSelectedSlidesIndex(newSelectedSlidesIndex)

  changeSlideIndex(selectedSlidesIndex.value[0])
}
```

這段邏輯的目的，是避免「目前頁」被取消後沒有新的操作基準。

因為在這套系統裡，目前頁會被視為頁面操作集合的一部分。

所以如果使用者取消目前頁，系統需要從剩下的選取頁中挑一頁變成新的目前頁。

這不是單純 array toggle，而是同時維護：

```text
多選集合
目前頁 slideIndex
頁面操作基準
```

## 9. Shift 區間選取：建立連續頁面範圍

當 `shiftKeyState.value` 為 true 時，進入 Shift 區間選取。

Shift 選取的語義是：

```text
從目前操作集合的一端，到點擊目標頁，選出連續範圍
```

大致流程是：

```ts
let minIndex = Math.min(...selectedSlidesIndex.value)
let maxIndex = index

if (index < minIndex) {
  maxIndex = Math.max(...selectedSlidesIndex.value)
  minIndex = index
}

const newSelectedSlidesIndex = []
for (let i = minIndex; i <= maxIndex; i++) newSelectedSlidesIndex.push(i)

mainStore.updateSelectedSlidesIndex(newSelectedSlidesIndex)
```

這代表 Shift 不是「多加一頁」，而是建立範圍。

例如：

```text
目前頁是第 3 頁
Shift 點第 8 頁
-> 選取第 3 到第 8 頁
```

如果點到目前頁且沒有多選，則直接 return。

這避免沒有實質變化的操作造成狀態更新。

## 10. 失焦時清空多選，切頁時也會清空多選

縮圖列表有兩個地方會清空額外多選。

### 第一個地方：縮圖列表失焦

```ts
if (!focus) mainStore.updateSelectedSlidesIndex([])
```

也就是使用者點到縮圖外面時，清掉多選頁面。

### 第二個地方：`slideIndex` 變化時

`Thumbnails/index.vue` 會 watch `slideIndex`：

```ts
watch(() => slideIndex.value, () => {
  if (selectedSlidesIndex.value.length) {
    mainStore.updateSelectedSlidesIndex([])
  }

  // 後面接著處理 active thumbnail 自動捲動
}, { immediate: true })
```

這表示當目前頁改變時，也會清空多選。

更精準地說：

* 額外多選集合不會長期跨情境保留。
* 切頁、失焦都會讓多選狀態回到乾淨狀態。

這樣做的目的，是避免使用者在不知情的情況下保留一批頁面操作目標。

## 11. 全域快捷鍵的第一層：先處理部分全域操作

`useGlobalHotkey.ts` 的 keydown handler 不是一開始就要求 `editorAreaFocus` 或 `thumbnailsFocus`。

它會先處理一些更全域的快捷鍵。

例如：

* `Ctrl + P`：打開 PDF 匯出 dialog。
* `Shift + F5`：從目前頁進入播放。
* `F5`：從頭播放。
* `Ctrl + F`：切換搜尋面板。
* `Ctrl + -`：縮小畫布。
* `Ctrl + =`：放大畫布。
* `Ctrl + 0`：重設畫布縮放。

這裡要修正一個容易誤解的點：

> 不是所有快捷鍵都要求 `thumbnailsFocus` 成立。

例如 `F5 / Shift + F5` 在原始碼中是在焦點 gate 之前處理的，因此它們更接近全域播放入口。

後面那些需要判斷元素或頁面語義的快捷鍵，才會進入焦點 gate。

## 12. 全域快捷鍵的第二層：焦點 gate

在處理完前面那批偏全域的快捷鍵後，原始碼會檢查：

```ts
if (!editorAreaFocus.value && !thumbnailsFocus.value) return
```

這行是核心 gate。

意思是：

```text
如果焦點既不在編輯區，也不在縮圖列表，就不處理後續大部分編輯快捷鍵
```

這可以避免使用者在其他 UI 區域輸入或操作時，快捷鍵誤觸編輯行為。

通過這個 gate 後，才會處理像：

* `Ctrl + C`
* `Ctrl + X`
* `Ctrl + D`
* `Ctrl + Z`
* `Ctrl + Y`
* `Ctrl + A`
* `Delete / Backspace`
* 方向鍵
* `PageUp / PageDown`
* `Enter`
* `Tab`

等快捷鍵。

## 13. `disableHotkeys` 是輸入狀態的保護閘

通過焦點 gate 不代表所有快捷鍵都一定會執行。

很多快捷鍵在執行前都會先判斷：

```ts
if (disableHotkeys.value) return
```

這個狀態常用在：

* 編輯文字。
* 編輯 section 標題。
* 正在輸入某些表單內容。

例如 section 標題進入編輯時會呼叫：

```ts
mainStore.setDisableHotkeysState(true)
```

儲存後再恢復：

```ts
mainStore.setDisableHotkeysState(false)
```

這樣可以避免：

```text
使用者在 input 裡按 Ctrl + A
結果不是全選文字，而是全選投影片
```

或：

```text
使用者在 input 裡按 Delete
結果不是刪文字，而是刪頁面
```

所以 `disableHotkeys` 是輸入場景保護快捷鍵系統的重要機制。

## 14. `Ctrl + C / X / D` 如何分流

在 `useGlobalHotkey.ts` 裡，複製、剪下、快速複製都有類似結構。

### 複製

```ts
const copy = () => {
  if (activeElementIdList.value.length) copyElement()
  else if (thumbnailsFocus.value) copySlide()
}
```

語義是：

```text
如果有選到元素 -> 複製元素
否則如果縮圖列表有焦點 -> 複製頁面
```

### 剪下

```ts
const cut = () => {
  if (activeElementIdList.value.length) cutElement()
  else if (thumbnailsFocus.value) cutSlide()
}
```

### 快速複製

```ts
const quickCopy = () => {
  if (activeElementIdList.value.length) quickCopyElement()
  else if (thumbnailsFocus.value) copyAndPasteSlide()
}
```

這裡可以看出一個優先順序：

> 元素選取優先於頁面焦點。

只要 `activeElementIdList` 有值，`Ctrl + C / X / D` 就會走元素層。

只有沒有元素選取，而且縮圖列表有焦點時，才走頁面層。

這能避免使用者選中元素時，誤把整頁投影片複製掉。

## 15. `Delete / Backspace` 如何分流

刪除邏輯是：

```ts
const remove = () => {
  if (activeElementIdList.value.length) deleteElement()
  else if (thumbnailsFocus.value) deleteSlide()
}
```

語義是：

```text
有元素選取 -> 刪元素
沒有元素選取，但縮圖列表有焦點 -> 刪頁面
```

這個分流非常重要。

因為 `Delete` 是危險操作。

如果沒有區分元素層和頁面層，很容易出現：

```text
使用者只是想刪一個文字框
結果整張 slide 被刪掉
```

或相反：

```text
使用者選了縮圖想刪頁
結果沒有反應或刪錯層級
```

所以 `activeElementIdList` 和 `thumbnailsFocus` 在這裡就是操作安全閘。

## 16. `Ctrl + A` 的分流比較特殊

全選邏輯是：

```ts
const selectAll = () => {
  if (editorAreaFocus.value) selectAllElements()
  if (thumbnailsFocus.value) selectAllSlide()
}
```

注意這裡是兩個 `if`，不是 `else if`。

也就是說，設計上它是根據兩個焦點狀態各自判斷。

正常情況下，`editorAreaFocus` 和 `thumbnailsFocus` 應該代表不同操作區域，不應同時為 true。

但從程式碼語義看，它不是「元素優先、頁面其次」的結構，而是：

```text
如果編輯區焦點成立 -> 全選元素
如果縮圖焦點成立 -> 全選頁面
```

這一點要比原本單純說「有縮圖焦點就全選頁面」更精準。

## 17. `Enter` 只在縮圖焦點下新增頁面

建立新頁的邏輯是：

```ts
const create = () => {
  if (!thumbnailsFocus.value) return
  createSlide()
}
```

對應快捷鍵：

```text
Enter -> create()
```

所以 `Enter` 的頁面新增語義有明確條件：

```text
縮圖列表必須有焦點
```

如果焦點不在縮圖列表，`create()` 直接 return。

這樣可以避免使用者在畫布或其他 UI 上按 Enter 時，莫名新增投影片。

## 18. 方向鍵與 PageUp / PageDown 的語義

方向鍵的處理是：

```ts
const move = (key: string) => {
  if (activeElementIdList.value.length) moveElement(key)
  else if (key === KEYS.UP || key === KEYS.DOWN) updateSlideIndex(key)
}
```

意思是：

```text
如果有元素選取 -> 方向鍵移動元素
如果沒有元素選取 -> 只有 Up / Down 會切換頁面
```

要注意：

* 沒有元素選取時，`Left / Right` 不會在這個函式中切頁。
* 頁面上下切換走的是 `Up / Down`。

另外還有：

```ts
const moveSlide = (key: string) => {
  if (key === KEYS.PAGEUP) updateSlideIndex(KEYS.UP)
  else if (key === KEYS.PAGEDOWN) updateSlideIndex(KEYS.DOWN)
}
```

所以：

```text
PageUp   -> 上一頁
PageDown -> 下一頁
```

這些快捷鍵會先經過焦點 gate 和 `disableHotkeys` 判斷。

## 19. `Tab` 是元素層巡覽，不是頁面層操作

`Tab` 會呼叫：

```ts
tabActiveElement()
```

它會在目前 slide 的 elements 裡切換 active element。

如果目前沒有 active element，就選第一個元素。

如果已經有 active element，就選下一個元素，最後一個再回到第一個。

所以 `Tab` 雖然也在全域快捷鍵裡，但它是元素層巡覽，和縮圖列表的頁面多選無關。

這個細節可以幫你理解：

```text
useGlobalHotkey 不是只服務縮圖列表
它同時管元素層、頁面層、播放、搜尋、縮放等快捷鍵
```

## 20. 頁面層快捷鍵整理

只看和縮圖列表 / 頁面操作高度相關的快捷鍵，可以整理成：

| 快捷鍵                  | 條件                               | 行為      |
| -------------------- | -------------------------------- | ------- |
| `Ctrl / Meta + C`    | 有元素選取時                           | 複製元素    |
| `Ctrl / Meta + C`    | 無元素選取，且 `thumbnailsFocus = true` | 複製頁面    |
| `Ctrl / Meta + X`    | 有元素選取時                           | 剪下元素    |
| `Ctrl / Meta + X`    | 無元素選取，且 `thumbnailsFocus = true` | 剪下頁面    |
| `Ctrl / Meta + D`    | 有元素選取時                           | 快速複製元素  |
| `Ctrl / Meta + D`    | 無元素選取，且 `thumbnailsFocus = true` | 複製並貼上頁面 |
| `Ctrl / Meta + A`    | `editorAreaFocus = true`         | 全選元素    |
| `Ctrl / Meta + A`    | `thumbnailsFocus = true`         | 全選頁面    |
| `Delete / Backspace` | 有元素選取時                           | 刪除元素    |
| `Delete / Backspace` | 無元素選取，且 `thumbnailsFocus = true` | 刪除頁面    |
| `Enter`              | `thumbnailsFocus = true`         | 新增頁面    |
| `Up / Down`          | 有元素選取時                           | 移動元素    |
| `Up / Down`          | 無元素選取                            | 前後切換頁面  |
| `PageUp / PageDown`  | 通過焦點 gate                        | 前後切換頁面  |

另外幾個播放 / 全域操作要分開看：

| 快捷鍵                | 精準說法                            |
| ------------------ | ------------------------------- |
| `F5`               | 從頭播放，處理位置在焦點 gate 前             |
| `Shift + F5`       | 從目前頁播放，處理位置在焦點 gate 前           |
| `Ctrl + P`         | 打開 PDF 匯出 dialog，處理位置在焦點 gate 前 |
| `Ctrl + F`         | 切換搜尋面板，處理位置在焦點 gate 前           |
| `Ctrl + - / = / 0` | 畫布縮放相關，處理位置在焦點 gate 前           |

所以不要簡化成「所有快捷鍵都必須縮圖焦點成立」。

更精準的說法是：

> 頁面層編輯操作依賴 `thumbnailsFocus`；部分播放、搜尋、匯出與縮放快捷鍵則在更全域的層級先被處理。

## 21. 為什麼切頁時要清空元素選取

這件事值得再獨立強調。

在 `changeSlideIndex()` 和 `useSlideHandler.updateSlideIndex()` 裡，都能看到切換頁面時會處理 `activeElementIdList`。

這是因為：

```text
元素選取是 slide-local state
頁面切換後，舊頁面的元素選取不應延續到新頁面
```

如果不清空，可能出現幾種問題：

1. 使用者在新頁面按 Delete，卻刪到舊頁面的元素。
2. active element id 指向不存在於目前頁的元素。
3. 快捷鍵分流誤判，以為還在元素操作模式。
4. UI 顯示和實際操作目標不一致。

所以縮圖列表切頁時，雖然主要是在更新 `slideIndex`，但它也必須維護畫布元素選取的一致性。

這是跨層狀態協調的一個典型例子。

## 22. 為什麼失焦時要清空頁面多選

頁面多選是縮圖列表場景的操作狀態。

它不適合在使用者離開縮圖列表後長期保留。

原因是：

```text
多選頁面會影響 copy / cut / delete / selectAll 等頁面操作
```

如果使用者已經離開縮圖列表、回到畫布或其他面板，還保留一批已選頁面，就很容易造成誤操作。

所以失焦時：

```ts
mainStore.updateSelectedSlidesIndex([])
```

是一種安全設計。

它讓頁面批次操作只在縮圖列表焦點生命週期內有效。

## 23. 為什麼這套設計像桌面檔案管理器

縮圖列表的多選行為其實很接近桌面檔案管理器。

對應關係可以這樣看：

| 桌面檔案管理器      | PPTist 縮圖列表   |
| ------------ | ------------- |
| 檔案           | 投影片頁面         |
| 目前選中的檔案      | 目前 slideIndex |
| Ctrl 點選      | 切換某頁是否加入選取集合  |
| Shift 點選     | 選取連續頁面範圍      |
| 右鍵已選項目       | 對整批選取項目開選單    |
| Delete       | 刪除選取檔案 / 頁面   |
| Ctrl + C / X | 複製 / 剪下選取項目   |

這種設計的好處是符合使用者既有操作心智。

但實作上必須小心，因為 PPT 編輯器還多了一層「頁面內元素選取」。

所以它比一般檔案管理器多一個分流問題：

```text
現在選到的是頁面？還是頁面裡的元素？
```

這就是 `activeElementIdList`、`thumbnailsFocus`、`editorAreaFocus` 必須共同判斷的原因。

## 24. 常見操作的完整資料流

### 情境一：點縮圖切到第 5 頁

```text
使用者點第 5 頁縮圖
  -> setThumbnailsFocus(true)
  -> handleClickSlideThumbnail(e, 4)
  -> 沒有 Ctrl / Shift
  -> mainStore.updateSelectedSlidesIndex([])
  -> changeSlideIndex(4)
  -> mainStore.setActiveElementIdList([])
  -> slidesStore.updateSlideIndex(4)
  -> watch(slideIndex) 觸發
  -> 清空多選
  -> 捲動 active thumbnail 到可視範圍
```

### 情境二：縮圖列表焦點下按 Ctrl + C

```text
使用者按 Ctrl + C
  -> useGlobalHotkey keydown
  -> 通過焦點 gate
  -> disableHotkeys 不成立
  -> copy()
  -> 如果 activeElementIdList 有值：copyElement()
  -> 否則如果 thumbnailsFocus 為 true：copySlide()
  -> copySlide() 基於 selectedSlidesIndex 操作視圖複製頁面資料
```

### 情境三：縮圖列表焦點下按 Delete

```text
使用者按 Delete
  -> useGlobalHotkey keydown
  -> 通過焦點 gate
  -> disableHotkeys 不成立
  -> remove()
  -> 如果 activeElementIdList 有值：deleteElement()
  -> 否則如果 thumbnailsFocus 為 true：deleteSlide()
  -> deleteSlide() 根據 selectedSlidesId 刪除頁面
```

### 情境四：Ctrl 多選第 2、4、6 頁後刪除

```text
使用者 Ctrl 點第 2、4、6 頁
  -> handleClickSlideThumbnail()
  -> ctrlKeyState = true
  -> updateSelectedSlidesIndex([...])

頁面操作視圖
  -> _selectedSlidesIndex + slideIndex

使用者按 Delete
  -> remove()
  -> 無 active elements
  -> thumbnailsFocus = true
  -> deleteSlide()
  -> selectedSlidesId 取出要刪除的 slide id
  -> slidesStore.deleteSlide(...)
  -> 清空多選
```

### 情境五：編輯 section 標題時按 Ctrl + A

```text
使用者雙擊 section title
  -> editSection()
  -> disableHotkeys = true
  -> input focus

使用者按 Ctrl + A
  -> useGlobalHotkey keydown
  -> 快捷鍵命中 Ctrl + A
  -> 發現 disableHotkeys = true
  -> return
  -> 不執行 selectAllSlide()
```

這樣使用者就可以在 input 裡正常全選文字，而不是全選投影片。

## 25. 這套資料流的優點

### 優點一：快捷鍵語義清楚

同一組快捷鍵可以根據上下文有不同語義。

例如：

```text
Ctrl + C
  在元素選取時 -> 複製元素
  在縮圖焦點時 -> 複製頁面
```

這比把快捷鍵硬拆成不同按鍵更符合使用者習慣。

### 優點二：頁面多選和目前頁不脫鉤

透過：

```ts
_selectedSlidesIndex + slideIndex
```

目前頁永遠會被納入頁面操作集合。

這讓單頁操作和多頁批次操作可以共用同一套規則。

### 優點三：失焦清空多選，降低誤操作

使用者離開縮圖列表後，多選頁面不會長期保留。

這讓頁面批次操作的生命週期更短、更安全。

### 優點四：元素層和頁面層可以共用快捷鍵

不用為頁面複製、元素複製設計完全不同快捷鍵。

只要焦點和選取狀態判斷清楚，同一組快捷鍵就能自然分流。

## 26. 這套設計的限制與要注意的地方

### 限制一：焦點狀態一定要維護準確

如果 `editorAreaFocus` 或 `thumbnailsFocus` 狀態錯了，快捷鍵就會跑錯層。

例如：

```text
縮圖列表看起來已經失焦，但 thumbnailsFocus 還是 true
-> Delete 可能刪頁面
```

所以焦點狀態是這套系統的核心基礎。

### 限制二：`selectedSlidesIndex` 可能包含重複或目前頁

因為 computed 是直接：

```ts
[..._selectedSlidesIndex, slideIndex]
```

所以讀這段時要理解它是操作視圖。

後續很多地方是透過 `includes(index)` 判斷，而不是把它當嚴格去重的 Set。

如果你自己實作類似系統，可以考慮是否要轉成 Set 來避免重複，但原始碼目前的心智模型是「視圖陣列」。

### 限制三：全域快捷鍵不是全部同一層級

有些快捷鍵在焦點 gate 前處理，有些在 gate 後處理，有些又受 `disableHotkeys` 控制。

所以讀 `useGlobalHotkey.ts` 時，不能只列快捷鍵清單，還要看它的處理位置。

例如：

```text
F5 / Shift + F5 是偏全域播放入口
Delete / Ctrl + C / Enter 是需要焦點與狀態分流的編輯操作
```

### 限制四：頁面操作與元素操作共用快捷鍵，分流邏輯要非常穩

這種設計很好用，但也更依賴狀態正確性。

只要 `activeElementIdList` 沒清掉，快捷鍵就可能被判斷成元素操作。

所以切頁時清空 active elements 很重要。

## 27. 容易誤解的地方

| 容易誤解                                     | 精準理解                                                  |
| ---------------------------------------- | ----------------------------------------------------- |
| `thumbnailsFocus` 只是 CSS focus           | 不是，它是頁面層快捷鍵入口條件                                       |
| `selectedSlidesIndex` 就是 main store 原始狀態 | 不是，這裡是 `_selectedSlidesIndex + slideIndex` 的操作視圖      |
| 目前頁和多選頁是兩套完全分離的操作目標                      | 不是，頁面操作時會把目前頁納入集合                                     |
| 按 Delete 一定刪頁面                           | 不一定，有 active elements 時會先刪元素                          |
| 按 Ctrl + C 一定複製頁面                        | 不一定，有 active elements 時會先複製元素                         |
| `Ctrl + A` 一定只全選頁面                       | 不一定，`editorAreaFocus` 時會全選元素，`thumbnailsFocus` 時會全選頁面 |
| 所有快捷鍵都需要 `thumbnailsFocus`               | 不是，F5、Shift+F5、Ctrl+P、Ctrl+F、縮放類快捷鍵在更全域層處理            |
| Shift 選取是多加一頁                            | 不是，它是建立連續頁面範圍                                         |
| 失焦清空多選是多餘的                               | 不是，它可以避免頁面批次操作狀態殘留                                    |
| `disableHotkeys` 只是不讓快捷鍵動作               | 更精準地說，它是輸入狀態保護，避免 input / section 編輯和全域快捷鍵衝突          |

## 28. 讀這套資料流的建議順序

建議照這個順序讀。

### 第一步：先看 `Thumbnails/index.vue` 的焦點入口

看：

```vue
@mousedown="() => setThumbnailsFocus(true)"
v-click-outside="() => setThumbnailsFocus(false)"
```

再看：

```ts
setThumbnailsFocus()
```

目標是理解縮圖列表焦點如何建立與清除。

### 第二步：看三個 store 狀態

先抓：

```text
slidesStore.slideIndex
mainStore.selectedSlidesIndex
mainStore.activeElementIdList
mainStore.thumbnailsFocus
mainStore.editorAreaFocus
keyboardStore.ctrlKeyState
keyboardStore.shiftKeyState
```

目標是理解狀態來源。

### 第三步：看 `selectedSlidesIndex` computed

理解：

```text
額外多選頁面 + 目前頁 = 頁面操作視圖
```

這是後面 copy / delete / selectAll 的基礎。

### 第四步：看 `handleClickSlideThumbnail()`

依序拆：

```text
editingSectionId return
右鍵多選保護
Ctrl 複選
Shift 區間選取
一般切頁
```

這是縮圖列表滑鼠互動的核心。

### 第五步：看 `changeSlideIndex()`

特別注意：

```ts
mainStore.setActiveElementIdList([])
```

目標是理解切頁如何清掉元素選取。

### 第六步：看 `useGlobalHotkey.ts`

不要只看快捷鍵有哪些，而要看分層：

```text
先處理全域播放 / 搜尋 / 匯出 / 縮放
再檢查 editorAreaFocus / thumbnailsFocus
再檢查 disableHotkeys
最後分流到元素操作或頁面操作
```

### 第七步：看 `useSlideHandler.ts`

理解頁面操作如何使用：

```text
selectedSlides
selectedSlidesId
```

來處理 copy / cut / delete / selectAll。

## 29. 這篇最重要的結論

### 結論一：縮圖列表的難點不是點擊切頁，而是快捷鍵語義分流

同樣一個快捷鍵，在不同狀態下可能操作元素，也可能操作頁面。

這需要 `activeElementIdList`、`editorAreaFocus`、`thumbnailsFocus` 一起判斷。

### 結論二：`thumbnailsFocus` 是頁面操作入口

縮圖列表取得焦點後，頁面層快捷鍵才有操作基礎。

失焦時會清空多選，避免頁面批次操作狀態殘留。

### 結論三：頁面操作集合是 `_selectedSlidesIndex + slideIndex`

目前頁永遠被納入頁面操作集合。

這讓單頁操作和多頁批次操作可以走同一套邏輯。

### 結論四：元素選取優先於頁面操作

像 `Ctrl + C`、`Ctrl + X`、`Ctrl + D`、`Delete` 這些操作，只要有 active elements，就優先操作元素。

沒有元素選取且縮圖列表有焦點時，才操作頁面。

### 結論五：`disableHotkeys` 是輸入狀態保護

編輯 section 標題或其他輸入場景時，需要暫停全域快捷鍵，避免文字輸入被誤判成頁面操作。

## 一句話抓核心

縮圖列表真正複雜的不是渲染縮圖，而是把 `activeElementIdList`、`slideIndex`、`_selectedSlidesIndex`、`thumbnailsFocus`、`editorAreaFocus`、`ctrlKeyState`、`shiftKeyState` 和 `disableHotkeys` 串起來，讓同一組快捷鍵能在「元素層」和「頁面層」之間正確分流。

把這套資料流看懂，你才會真正理解 PPTist 左側縮圖列表為什麼不只是 preview list，而是一個和整個編輯器快捷鍵系統深度整合的頁面操作面板。
