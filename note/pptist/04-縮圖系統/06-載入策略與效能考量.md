# 06-載入策略與效能考量

返回：[04-縮圖系統](./README.md)

## 這篇在解決什麼問題

縮圖看起來只是把投影片縮小，但在 `PPTist` 裡，每一張縮圖本質上仍然會走一套完整的 slide 渲染鏈路：

```text
ThumbnailSlide
  ├─ 渲染 slide background
  └─ v-for slide.elements
        └─ ThumbnailElement
              └─ BaseTextElement / BaseImageElement / BaseShapeElement / ...
```

也就是說，一張縮圖可能包含：

* 背景
* 文字元素
* 圖片元素
* 形狀元素
* 線段元素
* 表格元素
* 圖表元素
* LaTeX 元素
* 影片元素
* 音訊元素

如果一份簡報只有 5 頁，全部直接渲染通常問題不大。

但如果有 50 頁、100 頁，甚至更多頁，每一頁又有多個元素，那左側縮圖列表、播放縮圖導覽、手機預覽這類場景就可能在初始渲染時產生明顯壓力。

所以這篇要回答的是：

1. `PPTist` 如何延後建立縮圖內容 DOM？
2. `useLoadSlides()` 實際做了什麼？
3. 它和 `ThumbnailSlide` 的 `visible` 如何配合？
4. 哪些場景適合使用這個策略？
5. 哪些場景不適合使用這個策略？
6. 為什麼它不是完整虛擬列表？

## 一句話看懂載入策略

`PPTist` 的縮圖載入策略不是 virtual list，也不是 IntersectionObserver，而是：

> 用 `useLoadSlides()` 維護一個逐步增加的 `slidesLoadLimit`，場景容器把 `index < slidesLoadLimit` 傳給 `ThumbnailSlide.visible`，讓前面一批縮圖先真正渲染，後面的縮圖先顯示 placeholder，之後再固定時間分批放行更多頁面。

可以畫成：

```text
useLoadSlides()
  ├─ slidesLoadLimit = 50
  ├─ mounted 後開始 loadSlide()
  ├─ 如果 slides.length > slidesLoadLimit
  │     └─ 600ms 後 slidesLoadLimit += 20
  │         └─ 再次 loadSlide()
  └─ 如果 slides.length <= slidesLoadLimit
        └─ slidesLoadLimit = 9999

      ↓

場景容器
  └─ :visible="index < slidesLoadLimit"

      ↓

ThumbnailSlide
  ├─ visible = true
  │     └─ 建立 .elements、background、ThumbnailElement、BaseElement
  └─ visible = false
        └─ 只顯示 placeholder
```

這套策略的本質是：

```text
延後 heavy content DOM 的建立
```

不是：

```text
只保留可視區 DOM
```

這是理解它的關鍵。

## 原始碼對照表

| 觀念      | 對應檔案                           | 原始碼重點                                | 精準責任                           |
| ------- | ------------------------------ | ------------------------------------ | ------------------------------ |
| 分批放行門檻  | `src/hooks/useLoadSlides.ts`   | `slidesLoadLimit = ref(50)`          | 一開始先允許前 50 頁渲染                 |
| 遞增節奏    | `src/hooks/useLoadSlides.ts`   | `setTimeout(..., 600)`、`+20`         | 每 600ms 多放行 20 頁               |
| 結束狀態    | `src/hooks/useLoadSlides.ts`   | `slidesLoadLimit.value = 9999`       | 當目前頁數已小於等於 limit 時，視為全部放行      |
| 掛載時啟動   | `src/hooks/useLoadSlides.ts`   | `onMounted(loadSlide)`               | 元件掛載後開始分批放行                    |
| 卸載時清理   | `src/hooks/useLoadSlides.ts`   | `onUnmounted(clearTimeout)`          | 避免元件卸載後 timer 繼續執行             |
| 渲染接口    | `ThumbnailSlide/index.vue`     | `visible?: boolean`                  | 決定是否建立背景與元素 DOM                |
| 內容渲染    | `ThumbnailSlide/index.vue`     | `v-if="visible"`                     | `visible=true` 才渲染 `.elements` |
| 占位顯示    | `ThumbnailSlide/index.vue`     | `v-else placeholder`                 | `visible=false` 顯示「加载中 ...」    |
| 場景接線    | 多個縮圖列表場景                       | `:visible="index < slidesLoadLimit"` | 用頁面 index 判斷是否放行渲染             |
| 長列表適用   | Editor / Screen / Mobile 預覽類場景 | 使用 `useLoadSlides()`                 | 降低初始渲染壓力                       |
| 輸出場景不適用 | ExportImage / ExportPDF        | 不傳 `visible`                         | 避免 placeholder 進入輸出結果          |

這張表可以幫你抓住分工：

```text
useLoadSlides     管放行節奏
場景容器          管 index 與 visible 的接線
ThumbnailSlide    管 visible=true / false 時實際渲染什麼
```

## 1. 為什麼縮圖列表需要載入策略

單張縮圖的成本看起來不高，但大量縮圖會累積成本。

例如左側縮圖列表中，每一頁都會渲染：

```vue
<ThumbnailSlide
  :slide="element"
  :size="120"
  :visible="index < slidesLoadLimit"
/>
```

如果沒有 `visible` 控制，而是所有縮圖一次完整渲染，那就代表：

```text
100 張 slide
  -> 100 個 ThumbnailSlide
  -> 每張都建立 background
  -> 每張都遍歷 slide.elements
  -> 每個 element 都建立 ThumbnailElement
  -> 每個 ThumbnailElement 再建立 BaseElement
```

這會帶來幾種成本：

* Vue component 建立成本
* DOM 建立成本
* CSS layout / paint 成本
* 圖片載入與解碼成本
* 表格、圖表、公式等複雜元素渲染成本
* 滾動與拖曳時的畫面壓力

所以 PPTist 沒有一開始就把所有縮圖內容全量渲染，而是先渲染前面一批，後面再慢慢補齊。

## 2. `useLoadSlides()` 的實際程式邏輯

`useLoadSlides.ts` 的程式很短，但設計意圖很明確。

核心狀態是：

```ts
const slidesLoadLimit = ref(50)
```

這代表：

```text
初始只放行 index < 50 的縮圖內容
```

接著定義：

```ts
const loadSlide = () => {
  if (slides.value.length > slidesLoadLimit.value) {
    timer.value = setTimeout(() => {
      slidesLoadLimit.value = slidesLoadLimit.value + 20
      loadSlide()
    }, 600)
  }
  else slidesLoadLimit.value = 9999
}
```

這段邏輯可以翻成：

```text
如果總頁數還大於目前放行數量
  -> 等 600ms
  -> 多放行 20 頁
  -> 再檢查一次

如果總頁數已經小於等於目前放行數量
  -> 直接把 slidesLoadLimit 設成 9999
  -> 表示之後幾乎全部 index 都會通過 visible 判斷
```

啟動時機是：

```ts
onMounted(loadSlide)
```

清理時機是：

```ts
onUnmounted(() => {
  if (timer.value) clearTimeout(timer.value)
})
```

所以 `useLoadSlides()` 的完整責任是：

> 元件掛載後，用固定節奏增加 `slidesLoadLimit`；元件卸載時清掉尚未執行的 timer。

它不處理 DOM，不處理滾動，也不處理縮圖畫面本身。

## 3. `slidesLoadLimit = 50` 的意義

初始值是：

```ts
slidesLoadLimit = 50
```

這代表前 50 張 slide 一開始就會被放行。

如果場景中使用：

```vue
:visible="index < slidesLoadLimit"
```

那麼初始情況下：

```text
index 0 ~ 49 -> visible = true
index 50 以後 -> visible = false
```

這個選擇是一種務實折衷。

它不是只渲染螢幕可視範圍，因為前 50 頁通常已經遠超過首屏可見數量。

但它也不是全部渲染，因為 50 頁以後會延後。

所以它比較像：

```text
先保證前面一大段列表可立即使用
再把後面的重內容慢慢補上
```

這種策略比完整 virtualization 簡單很多，也比較不容易破壞拖曳列表、捲動定位、縮圖高度穩定等行為。

## 4. 每 600ms 增加 20 頁的意義

當 `slides.length > slidesLoadLimit` 時，程式會：

```ts
setTimeout(() => {
  slidesLoadLimit.value = slidesLoadLimit.value + 20
  loadSlide()
}, 600)
```

也就是：

```text
每 600ms 多放行 20 張縮圖內容
```

這是固定時間、固定批量的策略。

它的好處是簡單：

* 不需要計算可視範圍。
* 不需要監聽每個縮圖是否進入 viewport。
* 不需要維護 item 高度與虛擬列表 offset。
* 不需要重構 Draggable 列表。

但它的限制也很明顯：

* 不知道使用者實際捲到哪裡。
* 不會優先載入目前可視區後面的頁面。
* 只按時間慢慢增加。
* 最終仍會全部放行。

所以要精準描述它：

> 這是固定節奏的分批渲染策略，不是根據滾動位置的智慧載入策略。

## 5. 為什麼最後設成 `9999`

當目前總頁數已經小於等於 `slidesLoadLimit` 時，程式會：

```ts
slidesLoadLimit.value = 9999
```

這代表載入流程結束後，不只是停在剛好等於頁數，而是直接設成一個很大的數字。

這樣做的效果是：

```text
大多數情況下，所有 slide index 都會小於 9999
所以後續 visible 判斷都會通過
```

例如：

```ts
index < slidesLoadLimit
```

當 `slidesLoadLimit = 9999` 時，只要簡報頁數不是極端超過 9999，後續新增的頁面也會直接可見。

這是一個簡單粗暴但實用的結束狀態。

不要把 `9999` 理解成精準頁數，它只是代表：

```text
已完成分批放行，之後近似視為全部可渲染
```

## 6. `onUnmounted(clearTimeout)` 的必要性

`useLoadSlides()` 使用 timer 遞迴分批放行。

如果元件已經卸載，但 timer 還沒執行，就可能出現：

* 已卸載元件還在更新 ref。
* 記憶體或計時器殘留。
* 不必要的非同步回呼。

所以它在 `onUnmounted` 中清除 timer：

```ts
if (timer.value) clearTimeout(timer.value)
```

這是基本但重要的清理。

雖然這個 hook 很小，但它仍然有完整的生命週期處理：

```text
mounted -> 開始分批放行
unmounted -> 清理 timer
```

## 7. `visible` 是 `useLoadSlides()` 接回縮圖元件的接口

`useLoadSlides()` 本身不操作 DOM，也不直接控制 `ThumbnailSlide`。

它只回傳：

```ts
return {
  slidesLoadLimit,
}
```

場景容器再把它接到：

```vue
:visible="index < slidesLoadLimit"
```

例如：

```vue
<ThumbnailSlide
  :slide="slide"
  :size="120"
  :visible="index < slidesLoadLimit"
/>
```

這裡的資料流是：

```text
slidesLoadLimit 決定目前放行到哪一頁
index < slidesLoadLimit 得到 true / false
visible 控制 ThumbnailSlide 是否建立內容 DOM
```

這是一個很乾淨的接口設計。

`useLoadSlides()` 不需要知道 `ThumbnailSlide` 內部長什麼樣。

`ThumbnailSlide` 也不需要知道放行節奏是 50、20、600ms。

兩者只透過 `visible` 接起來。

## 8. `ThumbnailSlide.visible` 實際控制什麼

`ThumbnailSlide` 裡面大致是：

```vue
<div class="elements" v-if="visible">
  <div class="background" :style="backgroundStyle"></div>
  <ThumbnailElement
    v-for="(element, index) in slide.elements"
    ...
  />
</div>
<div class="placeholder" v-else>加载中 ...</div>
```

所以 `visible = true` 時：

* `.elements` 會建立。
* background 會建立。
* `ThumbnailElement` 會建立。
* BaseElement 會建立。
* 這張縮圖的主要內容會真正渲染。

`visible = false` 時：

* `.elements` 不建立。
* background 不建立。
* `ThumbnailElement` 不建立。
* BaseElement 不建立。
* 只顯示 placeholder。

這裡要講得精準：

> `visible` 控制的是縮圖內容 DOM 是否建立，不是 slide 資料是否存在。

即使 `visible = false`，`slide` prop 仍然已經傳進來了。

只是暫時不把它展開成背景與元素 DOM。

## 9. placeholder 的作用

當 `visible = false` 時，`ThumbnailSlide` 不會什麼都不顯示，而是顯示：

```text
加载中 ...
```

這個 placeholder 有幾個作用。

### 作用一：維持縮圖外框尺寸

外層 `.thumbnail-slide` 的寬高仍然存在：

```text
width = size
height = size * viewportRatio
```

所以即使內容還沒渲染，列表中的這一項仍然保有正確尺寸。

這可以避免列表高度跳動。

### 作用二：降低使用者困惑

如果後面的縮圖完全空白，使用者可能會以為：

```text
這些頁面壞了
或資料沒有載入
或簡報頁面消失
```

placeholder 至少讓使用者知道：

```text
這裡有一張頁面，只是內容稍後顯示
```

### 作用三：讓拖曳與捲動結構更穩定

因為外框存在，列表項的基本佈局不會因內容延後渲染而完全消失。

這對左側縮圖列表這種支援拖曳排序的場景尤其重要。

## 10. 它不是虛擬列表

這篇最重要的精準點是：

> `useLoadSlides()` 不是 virtual list。

完整虛擬列表通常會做：

```text
只渲染可視區附近的少量 item
離開可視區的 item 會卸載
用 padding / transform 撐出總高度
根據 scrollTop 動態計算要顯示哪些 item
```

但 `useLoadSlides()` 沒有做這些。

它沒有：

* 監聽 scrollTop。
* 計算可視範圍。
* 根據 viewport 決定要渲染哪些 index。
* 卸載已經渲染但離開畫面的縮圖。
* 用虛擬占位高度替代完整列表。

它做的是：

```text
列表項仍然全部存在
只是後面頁面的重內容先不建立
然後按時間逐批建立
最後全部建立
```

所以更準確的名稱是：

```text
分批渲染 / 漸進放行 / 延後內容建立
```

不是：

```text
虛擬列表 / 可視區懶載入 / 智慧 lazy loading
```

## 11. 為什麼不用完整虛擬列表

從工程取捨看，完整虛擬列表雖然效能更極致，但也會帶來複雜度。

縮圖列表不只是普通列表，它可能還要支援：

* 拖曳排序。
* 自動捲動到目前頁。
* active / selected 狀態。
* section title。
* 右鍵選單。
* 備註旗標。
* 不同場景的橫向 / 垂直排列。
* 縮圖高度依 `viewportRatio` 計算。

如果引入完整 virtualization，就要額外處理：

* 拖曳時虛擬 item 和實際 item 的映射。
* section title 在虛擬列表中的高度計算。
* scrollIntoView 與虛擬 offset 的互動。
* 目前頁自動置中。
* 橫向縮圖列的虛擬化。
* placeholder 與真實內容切換的時機。

所以 PPTist 目前這個策略比較像是務實折衷：

> 用低複雜度的方法降低初始渲染壓力，而不是為了極限效能重構整個列表系統。

這對一般簡報頁數來說可能已經足夠。

## 12. 哪些場景適合使用 `useLoadSlides()`

適合的場景通常有三個特徵：

```text
多頁列表
每頁渲染成本不低
允許後面的頁面稍晚顯示內容
```

目前明確適合的場景包括：

### 編輯器左側縮圖列表

檔案：

```text
src/views/Editor/Thumbnails/index.vue
```

特徵：

* 可能有很多頁。
* 每頁都有完整縮圖內容。
* 還要支援拖曳、多選、右鍵、section。
* 初始渲染壓力需要降低。

使用方式：

```vue
:visible="index < slidesLoadLimit"
```

### 播放全頁縮圖導覽

檔案：

```text
src/views/Screen/SlideThumbnails.vue
```

特徵：

* 播放時一次展示很多頁。
* 目的是快速跳頁。
* 後面的頁面可以稍晚補齊。

### 播放底部縮圖列

檔案：

```text
src/views/Screen/BottomThumbnails.vue
```

特徵：

* 橫向縮圖列。
* 可能包含所有頁面。
* 使用者通常先關注目前頁附近，不一定要所有頁面一開始都完整渲染。

### 講者模式縮圖列

檔案：

```text
src/views/Screen/PresenterView.vue
```

特徵：

* 講者模式中有縮圖導覽。
* 也是多頁列表。
* 適合分批渲染。

### 手機直向預覽

檔案：

```text
src/views/Mobile/MobilePreview.vue
```

特徵：

* 手機端垂直排列整份簡報。
* 每張 slide 接近螢幕寬度。
* 若頁數多，直接全量渲染壓力更明顯。

### 手機縮圖排序列

檔案：

```text
src/views/Mobile/MobileThumbnails.vue
```

特徵：

* 手機端橫向縮圖列。
* 支援點擊切頁與長按拖曳排序。
* 適合用分批渲染降低初始成本。

這些場景共同點是：

> 它們主要是預覽 / 導覽 / 排序用的多頁縮圖列表，可以接受後面頁面稍晚幾百毫秒再補齊。

## 13. 哪些場景不適合使用 `useLoadSlides()`

不適合的場景通常有兩類。

### 類型一：輸出場景

例如：

```text
src/views/Editor/ExportDialog/ExportImage.vue
src/views/Editor/ExportDialog/ExportPDF.vue
```

這些場景不適合傳 `visible`，原因很直接：

> 匯出需要完整畫面，不能讓 placeholder 混入輸出結果。

圖片匯出通常會：

```text
用 ThumbnailSlide 渲染大尺寸 DOM
再把 DOM 轉成圖片
```

PDF 匯出通常會：

```text
用 ThumbnailSlide 渲染大尺寸 DOM
再交給 print / PDF 流程
```

如果此時某些頁面還是 placeholder，輸出就會錯。

所以匯出場景通常直接完整渲染，並使用較大的 `size`，例如 `1600`。

### 類型二：單頁預覽或頁數可控場景

例如：

```text
src/views/Editor/Canvas/LinkDialog.vue
src/views/Editor/Thumbnails/Templates.vue
```

連結預覽通常只顯示一張 slide，不需要分批渲染。

模板挑選雖然也可能是列表，但頁數通常由模板資料控制，而且使用者預期直接看到模板內容，所以目前沒有必要接 `useLoadSlides()`。

這裡不要寫成「模板場景永遠不能 lazy load」。

更精準是：

> 目前這些場景沒有接 `useLoadSlides()`；從需求上看，單頁預覽不需要，匯出不適合，模板挑選則取決於產品需求和模板數量。

## 14. `visible` 預設為 true，讓非列表場景更簡單

`ThumbnailSlide` 的 `visible` 有預設值：

```text
visible = true
```

這代表外層如果不傳 `visible`，就會完整渲染內容。

因此匯出、單頁預覽等場景可以直接寫：

```vue
<ThumbnailSlide :slide="slide" :size="1600" />
```

或：

```vue
<ThumbnailSlide :slide="selectedSlide" :size="500" />
```

不需要特別處理載入策略。

這樣設計很好，因為 `visible` 只是可選的效能接口，不是每個場景都必須關心的核心 prop。

## 15. 這套策略的優點

### 優點一：實作非常簡單

`useLoadSlides()` 很短，只需要：

* 一個 `slidesLoadLimit`
* 一個 `timer`
* 一個遞迴 `loadSlide()`
* mounted / unmounted 生命週期

場景容器只要接：

```vue
:visible="index < slidesLoadLimit"
```

就能使用。

### 優點二：不侵入縮圖 renderer

`ThumbnailSlide` 不知道：

* 初始放行 50 頁。
* 每 600ms 多放行 20 頁。
* 什麼時候結束。
* 哪些場景用了這個 hook。

它只知道：

```text
visible=true  -> 渲染內容
visible=false -> 顯示 placeholder
```

這讓渲染元件保持乾淨。

### 優點三：不需要重構列表結構

完整虛擬列表可能需要重構列表 DOM 和滾動邏輯。

但這套策略不需要。

它可以直接加在現有：

* `Draggable` 列表
* 橫向縮圖列
* 垂直縮圖列
* 手機預覽列表

上面。

### 優點四：降低初始渲染壓力

前 50 頁先渲染，後面逐批補齊。

這可以避免「打開縮圖列表時所有頁面同時建立完整 DOM」。

### 優點五：場景可自行決定是否使用

長列表場景可以用。

匯出場景可以不用。

單頁預覽可以不用。

這讓效能策略不是強迫全專案一致，而是按場景選擇。

## 16. 這套策略的限制

### 限制一：最終仍會全量渲染

`slidesLoadLimit` 會一直增加，最後設成 `9999`。

所以它只是延後渲染，不是永遠只保留少量 DOM。

頁數很多時，最終 DOM 數量仍然可能很大。

### 限制二：不根據滾動位置載入

它不知道使用者目前捲到第幾頁。

所以如果使用者快速捲到第 100 頁，而 `slidesLoadLimit` 還沒放行到那裡，可能會先看到 placeholder。

它不會像 IntersectionObserver 那樣優先載入進入 viewport 的元素。

### 限制三：固定批次不一定適合所有設備

目前策略是：

```text
初始 50
每 600ms +20
```

這是固定參數。

但不同設備效能不同：

* 高效能桌機可能可以一次渲染更多。
* 低階手機可能 50 頁仍然太多。
* 圖片很多的簡報和純文字簡報成本差異很大。

目前策略沒有根據設備、頁面元素複雜度或實際 FPS 動態調整。

### 限制四：不是按元素成本調度

`useLoadSlides()` 只看 slide 數量，不看每張 slide 有多少元素。

例如：

```text
第 10 頁有 100 個元素
第 11 頁只有 1 個文字框
```

在這套策略中，它們都只是「一頁」。

沒有根據元素複雜度調整放行速度。

### 限制五：`9999` 是務實常數，不是嚴格上限模型

設定成 `9999` 在一般簡報裡足夠，但它不是嚴格的無限。

如果真的有超過 9999 頁的極端資料，`index < 9999` 仍然會讓後面頁面不通過。

當然，這對一般 PPT 編輯器不是實際問題，但從原始碼精準度來說，`9999` 是一個務實常數。

## 17. 為什麼這套設計仍然值得學

這套策略不是最先進，也不是最強效能方案。

但它很值得學，因為它展現了一種務實工程思路：

```text
不用一開始就上完整 virtual list
先找出最重的部分：縮圖內容 DOM
再用最小改動延後它的建立
```

它的切分也很乾淨：

```text
useLoadSlides
  -> 管節奏

ThumbnailSlide.visible
  -> 管內容是否建立

場景容器
  -> 管 index < slidesLoadLimit 的接線
```

這種設計的優點是：

* 容易讀懂。
* 容易接到不同場景。
* 不會污染 renderer。
* 不需要大幅重構列表。
* 對中等頁數簡報應該已經有幫助。

對你學前端大型專案來說，這比單純知道「要做 lazy load」更重要。

真正值得學的是：

> 效能策略要有清楚接口，不要和渲染細節、場景互動、資料模型全部糾纏在一起。

## 18. 如果要進一步優化，可以怎麼想

這段不是原始碼已經做的事，而是讀完原始碼後可以延伸思考的方向。

### 方向一：IntersectionObserver

可以讓接近 viewport 的縮圖優先渲染。

優點：

* 使用者捲到哪裡，就優先載入哪裡。
* 不必固定按 index 順序放行。

缺點：

* 要處理拖曳、排序、scroll container、橫向列表等情境。
* 複雜度比目前策略高。

### 方向二：真正虛擬列表

只保留可視區附近的 item。

優點：

* 大量頁面時 DOM 數量大幅降低。

缺點：

* 和 Draggable、section title、自動置中、右鍵選單的整合難度高。
* 橫向 / 垂直多場景都要處理。

### 方向三：根據元素複雜度調整批次

例如每次不是放行固定 20 頁，而是根據每頁元素數量估算成本。

優點：

* 複雜頁面可以少放行，簡單頁面可以多放行。

缺點：

* 要定義成本模型。
* 可能增加程式複雜度。

### 方向四：使用 requestIdleCallback

在瀏覽器空閒時逐步渲染後續縮圖。

優點：

* 比固定 timer 更貼近瀏覽器實際負載。

缺點：

* 相容性和排程控制要處理。
* 空閒時機不一定穩定。

### 方向五：圖片快取或縮圖快取

將某些 slide 預先轉成圖片縮圖。

優點：

* 列表滾動會更輕。

缺點：

* 編輯後快取失效管理複雜。
* 縮圖和實際 slide 可能出現同步問題。

這些方向都不是目前 `useLoadSlides()` 的內容，但可以作為你之後自己設計大型互動列表時的思考延伸。

## 19. 容易誤解的地方

| 容易誤解                            | 精準理解                                                          |
| ------------------------------- | ------------------------------------------------------------- |
| `useLoadSlides()` 是虛擬列表         | 不是，它只是分批放行內容渲染                                                |
| `visible=false` 代表 slide 資料還沒載入 | 不是，slide 資料已存在，只是不建立背景與元素 DOM                                 |
| placeholder 代表頁面不存在             | 不是，它代表內容暫時延後渲染                                                |
| `slidesLoadLimit=50` 表示只渲染首屏    | 不精準，它是先放行前 50 頁，不是根據 viewport 計算                              |
| 每 600ms +20 是智慧載入               | 不是，它是固定時間、固定批量的遞增策略                                           |
| 最後設成 9999 表示真正無限                | 不是，它是一個務實的大數值                                                 |
| 匯出場景也應該用 lazy load              | 不適合，匯出需要完整 DOM，不能出現 placeholder                               |
| `useLoadSlides()` 直接控制 DOM      | 不是，它只提供 `slidesLoadLimit`，真正 DOM 控制在 `ThumbnailSlide.visible` |
| 這套策略可以解決所有大量頁面效能問題              | 不行，最終仍會全量渲染，頁數極大時仍可能有壓力                                       |
| 模板列表永遠不適合 lazy load             | 不要這樣寫；目前沒使用，是否需要取決於模板數量與產品需求                                  |

## 20. 讀這篇相關原始碼的建議順序

### 第一步：先看 `ThumbnailSlide.visible`

先看：

```vue
v-if="visible"
v-else placeholder
```

理解 `visible` 真正控制的是：

```text
背景與元素 DOM 是否建立
```

### 第二步：看 `useLoadSlides.ts`

依序看：

```text
slidesLoadLimit = 50
timer
loadSlide()
onMounted(loadSlide)
onUnmounted(clearTimeout)
return { slidesLoadLimit }
```

目標是理解分批節奏。

### 第三步：看場景如何接線

找這類程式：

```vue
:visible="index < slidesLoadLimit"
```

看看哪些場景有使用。

### 第四步：看哪些場景沒有傳 `visible`

例如：

```text
ExportImage.vue
ExportPDF.vue
LinkDialog.vue
Templates.vue
```

思考為什麼它們不需要或不適合延後渲染。

### 第五步：回頭思考取捨

問自己：

```text
這是 lazy rendering 還是 virtual list？
它降低的是初始成本，還是最終 DOM 成本？
它按時間載入，還是按滾動位置載入？
```

這樣才不會把這套策略理解錯。

## 21. 這篇最重要的結論

### 結論一：`useLoadSlides()` 是分批渲染策略

它用 `slidesLoadLimit` 控制目前放行到哪一頁。

初始 50 頁，之後每 600ms 多 20 頁，最後設成 9999。

### 結論二：`visible` 是效能接口

`visible=true` 時，`ThumbnailSlide` 建立背景與元素 DOM。

`visible=false` 時，只顯示 placeholder。

這控制的是 DOM 建立時機，不是資料存在與否。

### 結論三：它不是虛擬列表

它不根據可視區載入，也不卸載離開可視區的 DOM。

它只是延後後面頁面的重內容建立，最後仍會全量放行。

### 結論四：預覽 / 導覽類長列表適合使用

例如：

* 編輯器左側縮圖列表
* 播放縮圖導覽
* 底部縮圖列
* 講者模式縮圖列
* 手機預覽
* 手機縮圖列

這些場景都可以接受後面的頁面稍晚補齊。

### 結論五：匯出場景不適合使用

圖片和 PDF 匯出需要完整畫面。

如果使用 `visible=false` 導致 placeholder 進入 DOM，輸出結果就會錯。

所以匯出場景通常完整渲染，並使用較大的 `size`。

### 結論六：這是一個務實折衷

它沒有追求最極致效能，而是在不重構列表、不引入複雜虛擬化的前提下，降低大量縮圖初始渲染壓力。

## 一句話抓核心

`PPTist` 的縮圖載入策略不是完整 virtualization，而是透過 `useLoadSlides()` 逐步增加 `slidesLoadLimit`，再用 `ThumbnailSlide.visible` 延後背景與元素 DOM 的建立；這讓長縮圖列表可以先快速出現可操作結構，再逐步補齊縮圖內容，而匯出圖片與 PDF 這類需要完整畫面的場景則不使用這個策略。

看懂這一篇，你就能理解：前端效能優化不一定一開始就要上最重的架構，有時候用清楚的接口做低成本分批渲染，就能在複雜度和體感效能之間取得很好的平衡。
