# 01-縮圖渲染元件設計

返回：[04-縮圖系統](./README.md)

## 這篇在解決什麼問題

上一篇總覽已經建立了一個核心觀念：

> `PPTist` 的縮圖系統不是為每個場景重寫一套小型投影片，而是把同一張 `slide` 用共用 renderer 以不同尺寸輸出。

這一篇要進一步拆開真正負責「畫出單頁縮圖」的兩個元件：

* `src/views/components/ThumbnailSlide/index.vue`
* `src/views/components/ThumbnailSlide/ThumbnailElement.vue`

它們分別負責兩件事：

```text
ThumbnailSlide
  負責把一張 slide 放進固定比例的縮圖外框，並用 scale 整頁縮小

ThumbnailElement
  負責根據元素 type，把每個 PPTElement 分派給對應的 BaseElement
```

所以這篇要回答四個問題：

1. `ThumbnailSlide` 怎麼決定縮圖尺寸？
2. 它怎麼把原始投影片縮成縮圖？
3. `visible`、`scale`、`provide` 在這裡各自扮演什麼角色？
4. `ThumbnailElement` 為什麼只是分派器，而不是重新寫一套縮圖元素？

## 先看整體結構

單頁縮圖的渲染鏈路可以畫成：

```text
ThumbnailSlide
  ├─ 接收 slide / size / visible
  ├─ 從 slides store 取得 viewportSize / viewportRatio
  ├─ 根據 size 算出縮圖外框寬高
  ├─ 根據 viewportSize 建立原尺寸內容區
  ├─ 根據 size / viewportSize 算出 scale
  ├─ provide(injectKeySlideScale, scale)
  ├─ 用 useSlideBackgroundStyle() 取得背景樣式
  ├─ visible = true 時渲染背景與元素
  └─ visible = false 時顯示 placeholder

      ↓

ThumbnailElement
  ├─ 接收 elementInfo / elementIndex
  ├─ 外層設定 z-index
  ├─ 根據 elementInfo.type 選擇 BaseXXXElement
  └─ 傳入 elementInfo 與 target="thumbnail"

      ↓

BaseImageElement / BaseTextElement / BaseShapeElement / ...
  └─ 真正渲染圖片、文字、形狀、線段、圖表、表格、公式、影片、音訊
```

這裡最重要的分工是：

> `ThumbnailSlide` 管「頁面尺寸與整體縮放」，`ThumbnailElement` 管「元素類型分派」，BaseElement 管「元素本身如何渲染」。

不要把這三層混在一起。

## 原始碼對照表

| 原始碼位置                                     | 對應觀念     | 作用                                                             |
| ----------------------------------------- | -------- | -------------------------------------------------------------- |
| `defineProps<{ slide, size, visible }>()` | 核心輸入     | 一張 slide、縮圖寬度、是否渲染內容                                           |
| `withDefaults(..., { visible: true })`    | 預設直接渲染   | 不傳 `visible` 時，預設會渲染背景與元素                                      |
| `storeToRefs(useSlidesStore())`           | 投影片比例資料  | 取得 `viewportSize` 和 `viewportRatio`                            |
| 外層 `.thumbnail-slide` style               | 縮圖外框尺寸   | `width = size`，`height = size * viewportRatio`                 |
| 內層 `.elements` style                      | 原尺寸內容區   | `width = viewportSize`，`height = viewportSize * viewportRatio` |
| `transform: scale(${scale})`              | 整頁縮放     | 把原始投影片內容區縮小成縮圖                                                 |
| `scale = size / viewportSize`             | 縮放比例     | 由目標寬度與基準寬度推出縮放倍率                                               |
| `provide(injectKeySlideScale, scale)`     | 往下提供縮放比例 | 讓子孫元件有需要時可以取得目前頁面縮放比例                                          |
| `useSlideBackgroundStyle(background)`     | 背景樣式計算   | 把 slide background 轉成可套用的 CSS style                            |
| `v-if="visible"`                          | 內容渲染開關   | 控制背景與元素 DOM 是否建立                                               |
| `v-else placeholder`                      | 延後渲染占位   | 未渲染內容時維持外框與提示                                                  |
| `ThumbnailElement v-for="slide.elements"` | 元素逐一渲染   | 每個 slide element 都交給 `ThumbnailElement` 分派                     |
| `elementIndex="index + 1"`                | 堆疊順序     | 讓元素 z-index 跟陣列順序保持一致                                          |
| `elementTypeMap`                          | 元素型別對應   | 根據 `ElementTypes` 對應到各 BaseElement                             |
| `target="thumbnail"`                      | 縮圖場景訊號   | 告訴 BaseElement 目前在縮圖場景中渲染                                      |

這張表可以當成讀這兩個檔案時的索引。

## 1. `ThumbnailSlide` 是單頁縮圖 renderer

`ThumbnailSlide` 的 props 非常小：

```ts
const props = withDefaults(defineProps<{
  slide: Slide
  size: number
  visible?: boolean
}>(), {
  visible: true,
})
```

這三個輸入剛好對應三個核心問題：

| prop      | 問題          | 意義               |
| --------- | ----------- | ---------------- |
| `slide`   | 要畫哪一頁？      | 單頁投影片資料          |
| `size`    | 要畫多寬？       | 縮圖外框寬度           |
| `visible` | 目前要不要真的畫內容？ | 控制背景與元素 DOM 是否建立 |

這個設計很乾淨，因為 `ThumbnailSlide` 不需要知道自己在哪個場景中被使用。

它不需要知道：

* 現在是不是編輯器左側縮圖列表。
* 現在是不是模板預覽。
* 現在是不是播放導覽。
* 現在是不是手機預覽。
* 現在是不是匯出圖片。

它只要知道：

```text
給我一張 slide
給我一個寬度 size
告訴我要不要現在渲染內容
```

就能輸出一張縮圖。

## 2. `size` 是縮圖寬度，不是高度，也不是縮放倍率

讀 `ThumbnailSlide` 時，第一個要釐清的就是 `size`。

外層模板是：

```vue
<div
  class="thumbnail-slide"
  :style="{
    width: size + 'px',
    height: size * viewportRatio + 'px',
  }"
>
```

這代表：

```text
縮圖寬度 = size
縮圖高度 = size * viewportRatio
```

所以 `size` 的精準意義是：

> 目標縮圖外框的寬度。

它不是高度，也不是倍率。

例如：

```ts
size = 120
viewportRatio = 0.5625
```

那外框就是：

```ts
width = 120
height = 120 * 0.5625 = 67.5
```

這讓外層場景只需要決定「我要多寬」，高度就能根據投影片比例自動算出來。

這也是為什麼某些場景如果想固定高度，就要反推寬度：

```ts
size = targetHeight / viewportRatio
```

因為 `ThumbnailSlide` 的 API 是吃「寬度」，不是吃「高度」。

## 3. `viewportSize` 與 `viewportRatio` 來自 slides store

`ThumbnailSlide` 自己不硬編投影片比例，而是從 `slides store` 取得：

```ts
const { viewportRatio, viewportSize } = storeToRefs(useSlidesStore())
```

這兩個值分別代表：

| 名稱              | 意義                                 |
| --------------- | ---------------------------------- |
| `viewportSize`  | 投影片基準寬度，常見預設是 `1000`               |
| `viewportRatio` | 投影片高寬比，常見預設是 `0.5625`，也就是 `9 / 16` |

它們共同決定兩件事：

1. 原始投影片內容區的尺寸。
2. 縮圖外框的比例。

外框尺寸是：

```ts
width = size
height = size * viewportRatio
```

內層原始內容區尺寸是：

```ts
width = viewportSize
height = viewportSize * viewportRatio
```

所以整套縮圖渲染不是任意縮小，而是始終跟著投影片本身的比例走。

## 4. 內層不是小尺寸排版，而是原尺寸排版後整頁縮放

這是 `ThumbnailSlide` 最重要的設計點。

內層 `.elements` 的 style 是：

```vue
<div
  class="elements"
  :style="{
    width: viewportSize + 'px',
    height: viewportSize * viewportRatio + 'px',
    transform: `scale(${scale})`,
  }"
  v-if="visible"
>
```

這代表它不是直接建立一個 120px 寬的小畫布，然後重新計算每個元素的小尺寸。

它做的是：

```text
第一步：建立原始投影片尺寸的內容區
第二步：讓背景與元素照原本投影片座標系排版
第三步：用 transform: scale(...) 把整個內容區縮小
```

這個做法非常重要。

如果縮圖重新計算每個元素的小座標，會有很多問題：

* 每個元素都要自己處理縮放。
* 文字、線條、圖表、表格可能出現比例不一致。
* 主畫布與縮圖容易產生位置誤差。
* 新增元素型別時，也要補縮圖專用邏輯。

但用整頁縮放的方式，就可以讓縮圖最大程度沿用主畫布的座標系。

所以核心觀念是：

> 縮圖不是重新排版，而是把原始投影片視覺整頁縮小。

## 5. `scale` 的來源非常直接

`scale` 的計算是：

```ts
const scale = computed(() => props.size / viewportSize.value)
```

意思是：

```text
目標寬度 / 原始投影片基準寬度 = 縮放倍率
```

例如：

```ts
viewportSize = 1000
size = 120
```

那麼：

```ts
scale = 120 / 1000 = 0.12
```

把這個比例套到內層 `.elements`：

```ts
transform: scale(0.12)
```

就能把 1000px 寬的內容區縮成 120px 寬。

這裡要注意：

> `scale` 是由寬度推導出來的，不是由高度推導出來的。

因為高度本來就是根據同一個 `viewportRatio` 算出來，所以只要寬度比例正確，高度比例自然也會正確。

## 6. `transform-origin: 0 0` 保證從左上角縮放

樣式中有：

```scss
.elements {
  transform-origin: 0 0;
}
```

這個細節很重要。

如果沒有指定 `transform-origin`，瀏覽器預設通常會以元素中心點作為縮放原點。這會導致縮小後的內容區位置偏移。

`transform-origin: 0 0` 的意思是：

```text
從左上角開始縮放
```

這樣內層投影片內容區縮小後，才能穩定貼齊外層縮圖盒子的左上角。

所以 `scale` 解決的是「縮多小」，`transform-origin` 解決的是「從哪裡開始縮」。

## 7. 背景也在同一個縮放內容區裡

`ThumbnailSlide` 不只渲染元素，也渲染背景。

script 中有：

```ts
const background = computed(() => props.slide.background)
const { backgroundStyle } = useSlideBackgroundStyle(background)
```

template 中有：

```vue
<div class="background" :style="backgroundStyle"></div>
```

而且背景是在 `.elements` 裡面：

```vue
<div class="elements" ... v-if="visible">
  <div class="background" :style="backgroundStyle"></div>
  <ThumbnailElement ... />
</div>
```

這代表背景會和元素一起處在原始投影片內容區中，再一起被 `scale` 縮小。

所以縮圖上的畫面理論上包含：

* 投影片背景
* 投影片元素
* 元素之間的相對位置
* 元素與背景的相對關係

這也是為什麼縮圖比較像「縮小版投影片」，而不是「元素清單預覽」。

## 8. `visible` 控制的是內容 DOM，不是 slide 資料

`visible` 是這個元件中最容易被誤解的 prop。

模板中是這樣寫的：

```vue
<div class="elements" ... v-if="visible">
  <div class="background" :style="backgroundStyle"></div>
  <ThumbnailElement ... />
</div>
<div class="placeholder" v-else>加载中 ...</div>
```

所以當 `visible = true` 時：

* `.elements` 會被建立。
* 背景會被建立。
* 每個 `ThumbnailElement` 會被建立。
* BaseElement 也會跟著被渲染。

當 `visible = false` 時：

* `.elements` 不會被建立。
* 背景不會被建立。
* 元素 DOM 不會被建立。
* 只顯示 placeholder。

但要特別注意：

> `visible = false` 不代表 `slide` 資料不存在，也不代表資料還沒載入。

`slide` prop 仍然傳進來了，只是暫時不把背景與元素轉成 DOM。

所以更精準的說法是：

```text
visible 控制渲染時機，不控制資料存在。
```

這個設計主要是為了效能。

例如左側縮圖列表可能有很多頁，每頁又包含文字、圖片、圖表、表格。如果一次把所有頁面的縮圖內容都渲染出來，初始成本會很高。

因此外層可以用 `useLoadSlides()` 控制：

```vue
<ThumbnailSlide
  :slide="element"
  :size="120"
  :visible="index < slidesLoadLimit"
/>
```

先渲染前面一批，後面的頁面先保留 placeholder。

## 9. `visible` 預設是 `true`

這一點也很重要。

`ThumbnailSlide` 使用：

```ts
withDefaults(defineProps<...>(), {
  visible: true,
})
```

代表如果外層沒有傳 `visible`，它會直接渲染完整內容。

這讓很多場景可以直接使用：

```vue
<ThumbnailSlide :slide="slide" :size="500" />
```

不用每次都處理載入狀態。

也就是說：

* 預覽型長列表：通常會傳 `visible`，配合分批渲染。
* 單頁預覽或匯出場景：通常可以不傳 `visible`，直接完整渲染。

這讓 `visible` 成為一個可選的效能接口，而不是所有場景都必須使用的核心資料接口。

## 10. `provide(injectKeySlideScale, scale)` 的意義

`ThumbnailSlide` 還有一行很關鍵：

```ts
provide(injectKeySlideScale, scale)
```

這表示 `scale` 不只是給 `.elements` 的 `transform` 使用，也被提供給子孫元件。

可以這樣理解：

```text
ThumbnailSlide 知道目前整張 slide 被縮小了多少
如果內部某些元素或工具函式需要知道這個比例
可以透過 injectKeySlideScale 取得
```

但這裡也要避免過度推論。

`provide` 本身只做一件事：

> 把 `scale` 這個 computed value 提供給子孫元件。

它不代表每個子元件一定會使用這個比例，也不代表縮圖中所有細節都靠 inject 的 scale 計算。

更精準的說法是：

```text
ThumbnailSlide 將頁面縮放比例提供給後代元件，讓需要感知縮放比例的元素可以自行 inject 使用。
```

這是一個擴充點，不是強制所有元素都要依賴它。

## 11. `ThumbnailElement` 是元素分派器

`ThumbnailSlide` 裡面會逐一渲染元素：

```vue
<ThumbnailElement
  v-for="(element, index) in slide.elements"
  :key="element.id"
  :elementInfo="element"
  :elementIndex="index + 1"
/>
```

這裡真正處理單一元素的是 `ThumbnailElement`。

但 `ThumbnailElement` 自己也不是實際的文字、圖片、形狀 renderer。

它的角色是：

> 根據 `elementInfo.type`，選擇要交給哪個 BaseElement 元件渲染。

它的 props 是：

```ts
const props = defineProps<{
  elementInfo: PPTElement
  elementIndex: number
}>()
```

其中：

* `elementInfo`：單一投影片元素資料。
* `elementIndex`：元素在這張 slide 裡的順序。

## 12. `elementTypeMap` 是縮圖元素重用的核心

`ThumbnailElement` 中的核心邏輯是：

```ts
const currentElementComponent = computed<unknown>(() => {
  const elementTypeMap = {
    [ElementTypes.IMAGE]: BaseImageElement,
    [ElementTypes.TEXT]: BaseTextElement,
    [ElementTypes.SHAPE]: BaseShapeElement,
    [ElementTypes.LINE]: BaseLineElement,
    [ElementTypes.CHART]: BaseChartElement,
    [ElementTypes.TABLE]: BaseTableElement,
    [ElementTypes.LATEX]: BaseLatexElement,
    [ElementTypes.VIDEO]: BaseVideoElement,
    [ElementTypes.AUDIO]: BaseAudioElement,
  }

  return elementTypeMap[props.elementInfo.type] || null
})
```

這段的重點不是技術上用了 `computed`，而是架構上做了這件事：

```text
縮圖元素沒有重寫一套 ThumbnailText、ThumbnailImage、ThumbnailShape
而是直接重用 BaseTextElement、BaseImageElement、BaseShapeElement...
```

這樣的好處是：

* 元素 schema 不需要為縮圖再定義一次。
* 元素的主要渲染邏輯可以重用。
* 新增元素型別時，可以集中思考 BaseElement 與各場景分派。
* 主畫布與縮圖比較容易維持視覺一致性。

所以 `ThumbnailElement` 的真正價值是「分派與復用」，不是「自己繪製元素」。

## 13. `target="thumbnail"` 是縮圖場景訊號

`ThumbnailElement` 的 template 是：

```vue
<component
  :is="currentElementComponent"
  :elementInfo="elementInfo"
  target="thumbnail"
></component>
```

這行很重要。

它代表當 BaseElement 被用在縮圖中時，會收到：

```ts
target = 'thumbnail'
```

這個值可以被理解為：

> 目前這個元素不是在主編輯畫布中渲染，而是在縮圖場景中渲染。

BaseElement 如果需要，可以根據 target 做不同處理。

常見可能包括：

* 關閉或弱化互動。
* 關閉游標或點擊行為。
* 加上 thumbnail 專用 class。
* 修正縮小後的樣式細節。
* 避免顯示只屬於編輯場景的控制 UI。

但這裡要講精準：

> `target="thumbnail"` 是統一傳入的場景訊號，不代表每一個 BaseElement 都一定有 thumbnail 專用邏輯。

有些元素會消費這個訊號，有些元素可能只是接收到但沒有額外處理。

所以不要把它理解成萬能開關，而要理解成：

```text
ThumbnailElement 統一標記目前場景
BaseElement 自己決定是否根據場景做差異化渲染
```

## 14. `elementIndex` 與 `z-index` 維持堆疊順序

`ThumbnailElement` 外層是：

```vue
<div
  class="base-element"
  :class="`base-element-${elementInfo.id}`"
  :style="{
    zIndex: elementIndex,
  }"
>
```

`elementIndex` 是從 `ThumbnailSlide` 傳進來的：

```vue
:elementIndex="index + 1"
```

這代表元素的堆疊順序大致跟 `slide.elements` 陣列順序一致。

例如：

```text
slide.elements[0] -> z-index: 1
slide.elements[1] -> z-index: 2
slide.elements[2] -> z-index: 3
```

這個細節很重要。

如果縮圖沒有處理堆疊順序，就可能出現：

* 主畫布中在上面的元素，縮圖卻跑到下面。
* 被遮住的元素，在縮圖中反而露出來。
* 文字、圖片、形狀的遮擋關係不一致。

所以 `z-index` 的重點是讓縮圖盡量維持與投影片本身一致的視覺堆疊。

## 15. `class="base-element-${elementInfo.id}"` 的作用

`ThumbnailElement` 還會給外層加上：

```vue
:class="`base-element-${elementInfo.id}`"
```

這個 class 讓每個元素在 DOM 中有和元素 id 對應的 class 名稱。

在讀這裡時，可以先把它理解成：

> 讓縮圖中的每個元素 DOM 仍然保留可識別的元素 id class。

但不要過度推論它一定在縮圖系統中被大量操作。這裡比較重要的仍然是：

* 外層 wrapper 承載 z-index。
* 動態 component 承載真正元素渲染。
* element id class 保留元素識別性。

## 16. 為什麼縮圖可以不重寫元素元件

綜合前面的內容，可以看出 PPTist 的縮圖元素渲染不是這樣：

```text
ThumbnailTextElement
ThumbnailImageElement
ThumbnailShapeElement
ThumbnailLineElement
...
```

而是這樣：

```text
ThumbnailElement
  ├─ TEXT  -> BaseTextElement
  ├─ IMAGE -> BaseImageElement
  ├─ SHAPE -> BaseShapeElement
  ├─ LINE  -> BaseLineElement
  └─ ...
```

這種設計的前提是：

1. 縮圖使用同一份 `PPTElement` 資料。
2. 元素位置與尺寸仍然基於原始投影片座標系。
3. 外層 `ThumbnailSlide` 已經負責整頁縮放。
4. BaseElement 可以透過 `target` 知道目前渲染場景。

因此元素層不需要重新計算一套 thumbnail 版座標。

這是縮圖系統乾淨的地方：

> 頁面層負責縮放整張 slide，元素層負責照原本資料渲染自己。

## 17. 這種設計的好處

### 好處一：視覺一致性高

因為縮圖不是重新排版，而是原始投影片整頁縮放，所以縮圖和主畫布比較容易保持一致。

尤其對這些元素很重要：

* 文字位置
* 圖片裁切
* 形狀位置
* 線段角度
* 表格大小
* 圖表比例
* 背景樣式

### 好處二：新增場景容易

只要有一張 `slide`，並決定一個 `size`，就可以使用：

```vue
<ThumbnailSlide :slide="slide" :size="120" />
```

如果是長列表，再補上：

```vue
:visible="index < slidesLoadLimit"
```

這讓 `ThumbnailSlide` 很容易被編輯器、播放、手機、匯出等場景共用。

### 好處三：元素渲染邏輯可重用

`ThumbnailElement` 分派到 BaseElement，而不是自己重寫元素，能降低維護成本。

如果某個元素在縮圖場景有問題，通常可以回到該 BaseElement 看它如何處理 `target="thumbnail"`，而不是在多個縮圖場景各修一份。

### 好處四：效能策略可以外掛

`ThumbnailSlide` 只暴露 `visible`，不直接知道 `useLoadSlides()`。

這代表：

```text
ThumbnailSlide 提供延後渲染接口
useLoadSlides 決定分批放行節奏
場景容器負責把兩者接起來
```

這個邊界很乾淨。

## 18. 這種設計的限制

這套設計雖然乾淨，但不是沒有代價。

### 限制一：縮圖仍然渲染真實元素結構

因為它重用 BaseElement，所以每張縮圖仍可能建立不少 DOM。

如果一份簡報有很多頁，而且每頁元素很多，成本仍然不低。

這也是為什麼需要 `visible` 和 `useLoadSlides()` 做分批渲染。

### 限制二：不是完整虛擬列表

`visible` 可以延後內容 DOM 建立，但它不是虛擬列表。

它不會根據滾動位置只保留可視區 DOM，也不會在離開視窗後卸載已渲染的縮圖。

所以它是低成本的分批渲染策略，不是終極效能方案。

### 限制三：BaseElement 仍需自己處理場景差異

`target="thumbnail"` 只是把場景訊號傳下去。

如果某個 BaseElement 在縮圖中需要特殊樣式或互動控制，仍然要由該 BaseElement 自己處理。

所以縮圖共用不是完全免費的，它依賴元素元件本身有良好的場景適配能力。

## 19. 容易誤解的地方

| 容易誤解                                                    | 精準理解                                     |
| ------------------------------------------------------- | ---------------------------------------- |
| `ThumbnailSlide` 是左側縮圖列表                                | 不是，它是單頁縮圖 renderer；左側列表只是使用它的場景之一        |
| `size` 是縮圖大小                                            | 更精準地說，`size` 是縮圖寬度                       |
| `scale` 是外部傳入的倍率                                        | 不是，`scale = size / viewportSize`，由元件內部計算 |
| 縮圖會重新計算每個元素座標                                           | 不會，元素仍在原始投影片內容區中排版，再由外層整頁縮放              |
| `visible=false` 代表資料不存在                                 | 不是，資料仍然存在，只是不建立背景與元素 DOM                 |
| `provide(scale)` 代表所有元素都一定依賴 scale                      | 不一定，只是把 scale 提供給需要的子孫元件                 |
| `ThumbnailElement` 會自己畫文字、圖片、形狀                         | 不會，它只是根據 type 分派到 BaseElement            |
| `target="thumbnail"` 代表所有 BaseElement 都一定有 thumbnail 分支 | 不一定，它是場景訊號，是否使用取決於各 BaseElement          |
| `z-index` 是隨便設的                                         | 不是，它用 `elementIndex` 維持元素堆疊順序            |

## 20. 讀這兩個元件時的建議順序

建議照這個順序讀：

1. 先看 `ThumbnailSlide` 的 props

   * `slide`
   * `size`
   * `visible`

2. 再看外層 `.thumbnail-slide` 的尺寸

   * `width = size`
   * `height = size * viewportRatio`

3. 再看內層 `.elements` 的尺寸與縮放

   * `width = viewportSize`
   * `height = viewportSize * viewportRatio`
   * `transform = scale(...)`
   * `transform-origin = 0 0`

4. 再看 `scale` 的計算

   * `scale = size / viewportSize`

5. 再看背景渲染

   * `slide.background`
   * `useSlideBackgroundStyle()`
   * `.background`

6. 再看 `visible`

   * `v-if="visible"`
   * `placeholder`
   * 預設 `visible = true`

7. 再看 `ThumbnailElement`

   * `elementInfo`
   * `elementIndex`
   * `elementTypeMap`
   * dynamic component
   * `target="thumbnail"`
   * `z-index`

照這個順序讀，會比較不容易被細節打散。

## 21. 這篇最重要的四個結論

### 結論一：`ThumbnailSlide` 是頁面縮放容器

它的核心責任不是管理列表互動，而是把一張 slide 放進縮圖外框中，並用原始投影片內容區加上 `scale` 輸出縮小畫面。

### 結論二：縮圖不是重新排版

背景與元素仍然在 `viewportSize × viewportSize * viewportRatio` 的原始內容區中渲染，再整頁縮小。

這讓縮圖與主投影片更容易保持視覺一致。

### 結論三：`ThumbnailElement` 是分派器

它根據 `elementInfo.type` 選擇 BaseElement，自己不實作文字、圖片、形狀、表格等元素渲染。

### 結論四：`target="thumbnail"` 是場景訊號

它讓 BaseElement 知道目前在縮圖場景中，但不代表所有 BaseElement 都一定有特殊降級邏輯。

## 一句話抓核心

`ThumbnailSlide` 負責用原始投影片尺寸建立內容區並整頁縮放；`ThumbnailElement` 負責把每個 `PPTElement` 分派回既有 BaseElement，並用 `target="thumbnail"` 標記目前是縮圖場景。

把這兩層分清楚，就能看懂 PPTist 為什麼可以用同一套投影片資料，同時支援左側縮圖、模板預覽、播放導覽、手機預覽與匯出場景。
