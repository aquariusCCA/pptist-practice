# 00-縮圖系統總覽

返回：[04-縮圖系統](./README.md)

## 這篇先建立什麼觀念

如果只看左側縮圖列表，很容易把 `PPTist` 的縮圖系統理解成：

* 用 `v-for` 跑每張投影片
* 外面包一個小盒子
* 點一下切換目前頁

但這只看到「編輯器左側列表」這個使用場景，沒有看到縮圖系統真正的設計重點。

在 `PPTist` 裡，縮圖比較適合理解成：

> 一套可被多個場景共用的「單頁投影片縮放渲染器」。

也就是說，`ThumbnailSlide` 不是只服務左側縮圖列表，它還能被模板預覽、播放導覽、手機預覽、匯出圖片、匯出 PDF 等場景共用。

這篇總覽先回答三個問題：

1. 縮圖資料從哪裡來？
2. 單頁縮圖是怎麼被畫出來的？
3. 為什麼同一套縮圖元件可以跨場景共用？

## 一句話看懂縮圖系統

`PPTist` 的縮圖系統不是重新做一套小型投影片，而是：

> 用同一份 `slide` 資料，透過 `ThumbnailSlide` 建立原始投影片尺寸的內容區，再用 `scale` 整頁縮小；頁面中的元素則交給 `ThumbnailElement` 分派到既有的 BaseElement 元件，並傳入 `target="thumbnail"` 表示目前處於縮圖場景。

整條鏈路可以先畫成：

```text
slides store
  ├─ slides：投影片資料來源
  ├─ viewportSize：投影片基準寬度
  └─ viewportRatio：投影片高寬比

      ↓

ThumbnailSlide
  ├─ 接收 slide / size / visible
  ├─ 計算縮圖外框寬高
  ├─ 用 viewportSize 建立原尺寸內容區
  ├─ 用 scale = size / viewportSize 整頁縮小
  ├─ 渲染背景
  └─ v-for 渲染 slide.elements

      ↓

ThumbnailElement
  ├─ 接收單一 PPTElement
  ├─ 根據 elementInfo.type 選擇 BaseXXXElement
  ├─ 用 elementIndex 對應 z-index
  └─ 傳入 target="thumbnail"

      ↓

BaseTextElement / BaseImageElement / BaseShapeElement / ...
  └─ 實際負責元素內容的渲染；是否針對 thumbnail 做特殊處理，取決於各元素元件本身

      ↓

Editor / Screen / Mobile / Export 等場景容器
  └─ 決定尺寸、排列、互動、分批載入、切頁、匯出等外層行為
```

這張圖要抓住一個重點：

> 縮圖的核心是 `ThumbnailSlide`，但縮圖列表的互動不是 `ThumbnailSlide` 的責任。

`ThumbnailSlide` 只管「怎麼把一張 slide 畫成某個尺寸」；至於點擊切頁、拖曳排序、多選、右鍵選單、區段管理，都是外層場景容器的事情。

## 原始碼對照表

| 觀念            | 對應檔案                                                            | 主要責任                                                             |
| ------------- | --------------------------------------------------------------- | ---------------------------------------------------------------- |
| 投影片資料來源       | `src/store/slides.ts`                                           | 提供 `slides`、`viewportSize`、`viewportRatio` 等縮圖渲染會用到的基礎資料         |
| 單頁縮圖 renderer | `src/views/components/ThumbnailSlide/index.vue`                 | 接收一張 `slide`，根據 `size` 和投影片比例建立縮圖外框，並把原尺寸內容區縮放成縮圖                |
| 元素分派器         | `src/views/components/ThumbnailSlide/ThumbnailElement.vue`      | 根據 `elementInfo.type` 選擇對應的 BaseElement，並傳入 `target="thumbnail"` |
| 編輯器左側縮圖列表     | `src/views/Editor/Thumbnails/index.vue`                         | 管理縮圖列表、切頁、多選、拖曳排序、section、右鍵選單、備註旗標等互動                           |
| 分批載入策略        | `src/hooks/useLoadSlides.ts`                                    | 控制哪些頁面的縮圖內容先真正渲染，降低大量頁面時的初始渲染壓力                                  |
| 匯出圖片 / PDF    | `src/views/Editor/ExportDialog/ExportImage.vue`、`ExportPDF.vue` | 用較大的 `size` 重用 `ThumbnailSlide`，把投影片輸出成圖片或 PDF 需要的畫面             |

這張表的用途是幫你把「渲染核心」和「使用場景」分開。讀原始碼時，最怕一開始就陷入 `Editor/Thumbnails/index.vue` 的互動細節，反而看不清楚縮圖 renderer 本身。

## 主要職責分工

### 1. `slides store`：縮圖資料的來源

檔案：`src/store/slides.ts`

縮圖系統不自己維護一份 thumbnail 專用資料，而是直接使用 `slides store` 裡的投影片資料。

和縮圖渲染最相關的是：

```ts
slides: Slide[]
viewportSize: number
viewportRatio: number
```

可以這樣理解：

* `slides`：每一頁投影片本體，包含背景、元素、備註、section 等資料。
* `viewportSize`：投影片基準寬度，預設是 `1000`。
* `viewportRatio`：投影片高寬比，預設是 `0.5625`，也就是 `9 / 16`。

這代表縮圖不需要另外同步一份資料。只要 store 裡的 slide 改變，縮圖用到的資料來源也會跟著改變。

這種設計的好處是：

* 畫布編輯後，縮圖可以基於同一份 slide 資料更新。
* 新增、刪除、排序頁面時，不需要維護另一份 thumbnail list data。
* 編輯器、播放、手機、匯出可以共用同一份投影片資料。

但也要注意：

> `slides store` 只提供資料與部分頁面狀態，並不知道縮圖要怎麼排版、怎麼捲動、怎麼被點擊。

這些都是場景容器的責任。

### 2. `ThumbnailSlide`：單頁縮圖 renderer

檔案：`src/views/components/ThumbnailSlide/index.vue`

`ThumbnailSlide` 是縮圖系統的核心元件。它的 props 很小：

```ts
{
  slide: Slide
  size: number
  visible?: boolean
}
```

這三個 props 分別代表：

* `slide`：這張縮圖要畫哪一頁。
* `size`：縮圖的目標寬度。
* `visible`：現在是否真的渲染背景與元素內容。

這裡最重要的是 `size`。

`size` 不是高度，也不是縮放倍率，而是「縮圖外框的寬度」。高度會由 `viewportRatio` 推出：

```ts
height = size * viewportRatio
```

內部內容區則不是直接用小尺寸排版，而是先建立原始投影片基準尺寸：

```ts
width = viewportSize
height = viewportSize * viewportRatio
```

然後再用：

```ts
scale = size / viewportSize
```

把整張投影片縮小。

所以它的渲染模型是：

```text
先用原始投影片尺寸排好內容
再用 transform: scale(...) 整頁縮小
```

這就是為什麼縮圖上的元素位置可以和正式投影片保持一致：它不是重新計算每個元素的小座標，而是沿用原本座標系，再整體縮放。

另外，`ThumbnailSlide` 裡也會處理背景：

```ts
const background = computed(() => props.slide.background)
const { backgroundStyle } = useSlideBackgroundStyle(background)
```

所以縮圖不是只畫元素，而是把投影片背景與元素一起放進同一個縮放內容區。

### 3. `visible`：延後渲染內容，不是延後資料

`ThumbnailSlide` 的 `visible` 很容易被誤解。

它不是控制 slide 資料有沒有載入，也不是控制這頁是否存在，而是控制「這一頁的背景與元素 DOM 是否先被建立」。

當 `visible = true`：

```vue
<div class="elements" v-if="visible">
  <div class="background" :style="backgroundStyle"></div>
  <ThumbnailElement ... />
</div>
```

會真正渲染背景與元素。

當 `visible = false`：

```vue
<div class="placeholder" v-else>加载中 ...</div>
```

只顯示 placeholder。

所以更精準地說：

> `visible` 是縮圖內容的渲染時機控制，不是資料存在與否的控制。

這個接口讓外層場景可以配合 `useLoadSlides()` 做分批渲染。例如左側縮圖列表會先讓前面一批頁面真正渲染，後面的頁面先顯示 placeholder，以降低初始渲染壓力。

### 4. `ThumbnailElement`：元素分派器，不是元素 renderer

檔案：`src/views/components/ThumbnailSlide/ThumbnailElement.vue`

`ThumbnailElement` 的責任很單純：

1. 接收單一 `PPTElement`。
2. 根據 `elementInfo.type` 找到對應的基礎元素元件。
3. 用動態 component 把它渲染出來。
4. 傳入 `target="thumbnail"`。
5. 用 `elementIndex` 設定外層 `z-index`。

它大致可以理解成這樣：

```ts
IMAGE  -> BaseImageElement
TEXT   -> BaseTextElement
SHAPE  -> BaseShapeElement
LINE   -> BaseLineElement
CHART  -> BaseChartElement
TABLE  -> BaseTableElement
LATEX  -> BaseLatexElement
VIDEO  -> BaseVideoElement
AUDIO  -> BaseAudioElement
```

這裡有一個很重要的邊界：

> `ThumbnailElement` 不自己畫圖片、文字、形狀、表格或圖表，它只是分派到既有的 BaseElement。

這也是 PPTist 縮圖系統可以維持一致性的原因之一。縮圖不是另外定義一份 thumbnail element schema，而是重用投影片原本的元素型別與渲染元件。

### 5. `target="thumbnail"`：縮圖場景入口，不代表每個元素都一定有特殊處理

`ThumbnailElement` 會在動態 component 上傳入：

```vue
<component
  :is="currentElementComponent"
  :elementInfo="elementInfo"
  target="thumbnail"
/>
```

這是縮圖系統很重要的設計點。

它代表：

* 目前這個元素不是在主畫布中渲染。
* 它是在縮圖場景中渲染。
* BaseElement 如果需要，可以根據 `target` 做不同處理。

但這裡要講精準一點：

> `target="thumbnail"` 是統一傳入的場景訊號，不代表每一種 BaseElement 都一定有寫 thumbnail 專用邏輯。

有些元素可能會根據 thumbnail 模式關閉互動、修正樣式、處理 pointer event；有些元素則可能只是接收了這個 prop，但沒有額外分支。

所以更好的理解是：

```text
ThumbnailElement 統一提供 thumbnail 場景訊號
BaseElement 自己決定是否消費這個訊號
```

這比說「每個元素都知道怎麼降級」更精準。

### 6. 場景容器：決定縮圖怎麼被使用

`ThumbnailSlide` 只負責畫一張 slide。真正讓縮圖變成「列表」、「模板預覽」、「播放導覽」、「匯出來源」的是外層場景容器。

常見使用場景可以分成四類。

#### 編輯器頁面管理

代表檔案：

* `src/views/Editor/Thumbnails/index.vue`

這是最複雜的縮圖場景。它不只顯示縮圖，還負責：

* 顯示頁碼
* 切換目前頁
* 多選頁面
* 拖曳排序
* section 顯示與編輯
* 右鍵選單
* 備註旗標
* 自動捲動到目前頁
* 配合快捷鍵焦點狀態
* 接入 `useLoadSlides()` 分批渲染

在這裡，`ThumbnailSlide` 只是視覺核心，真正的頁面管理行為在 `Editor/Thumbnails/index.vue` 和相關 hooks 裡。

#### 模板與連結預覽

代表檔案：

* `src/views/Editor/Thumbnails/Templates.vue`
* `src/views/Editor/Canvas/LinkDialog.vue`

這類場景也使用 `ThumbnailSlide`，但目的不是頁面管理。

例如：

* Templates 用來預覽模板頁，重點是挑選與插入模板。
* LinkDialog 用來預覽要連結到哪一張投影片，重點是單頁辨識。

這說明縮圖不一定是列表，也可以是單頁預覽。

#### 播放與手機端導覽

代表檔案：

* `src/views/Screen/SlideThumbnails.vue`
* `src/views/Screen/BottomThumbnails.vue`
* `src/views/Screen/PresenterView.vue`
* `src/views/Mobile/MobilePreview.vue`
* `src/views/Mobile/MobileThumbnails.vue`
* `src/views/Mobile/MobilePlayer.vue`

這些場景會根據使用情境調整外層排列方式：

* 垂直列表
* 橫向縮圖列
* 全頁縮圖導覽
* 手機直向預覽
* 手機縮圖排序

但單頁縮圖本身仍然可以交給 `ThumbnailSlide`。

#### 匯出與輸出

代表檔案：

* `src/views/Editor/ExportDialog/ExportImage.vue`
* `src/views/Editor/ExportDialog/ExportPDF.vue`

匯出場景很值得注意。

它不是另外寫一套完全獨立的輸出 renderer，而是把 `ThumbnailSlide` 用在更大的尺寸上，例如輸出圖片時可能用較大的 `size` 來取得高解析畫面。

這代表 `ThumbnailSlide` 不只是「小縮圖元件」，而是一個可以輸出不同尺寸投影片畫面的共用 renderer。

## 為什麼 `ThumbnailSlide` 可以跨場景共用

原因不是它功能很多，而是它邊界很乾淨。

它只要求三個輸入：

```ts
slide: Slide
size: number
visible?: boolean
```

這讓它不需要知道：

* 現在是不是在編輯器左側。
* 現在是不是在播放頁。
* 現在是不是在手機端。
* 現在是不是在匯出圖片。
* 點擊後要不要切頁。
* 這張縮圖能不能拖曳排序。
* 這張縮圖是否被選取。

這些差異全部留給外層場景容器處理。

可以整理成一句話：

> `ThumbnailSlide` 負責「畫出一張 slide」，場景容器負責「這張縮圖在這個場景中要怎麼被使用」。

這種切法讓縮圖 renderer 可以穩定復用，也讓編輯器左側列表可以承擔更複雜的頁面管理互動，而不污染單頁縮圖渲染元件。

## 縮圖系統最值得學的四個觀念

### 1. 縮圖不是重新排版，而是原畫布整體縮放

`ThumbnailSlide` 不會用縮圖尺寸重新計算每個元素的位置。

它的做法是：

1. 先用 `viewportSize` 和 `viewportRatio` 建立原始投影片內容區。
2. 讓背景與元素仍然使用原本的投影片座標系。
3. 最後用 `transform: scale(...)` 把整頁縮小。

這樣能最大限度維持縮圖和正式投影片的視覺一致性。

### 2. `size` 是寬度，不是高度，也不是倍率

這是讀 `ThumbnailSlide` 時最容易混淆的點。

`size` 代表縮圖外框寬度。

高度由：

```ts
size * viewportRatio
```

推出。

縮放倍率由：

```ts
size / viewportSize
```

推出。

所以如果：

```ts
viewportSize = 1000
viewportRatio = 0.5625
size = 120
```

那麼：

```ts
縮圖寬度 = 120
縮圖高度 = 120 * 0.5625 = 67.5
縮放倍率 = 120 / 1000 = 0.12
```

理解這個關係後，就能看懂為什麼某些場景想固定高度時，會反推 `size = height / viewportRatio`。

### 3. `target="thumbnail"` 是場景訊號，不是萬能開關

`target="thumbnail"` 很重要，但不要過度解讀。

它的本質是：

```text
ThumbnailElement 告訴 BaseElement：你現在是在縮圖場景中渲染
```

但是否真的關閉 pointer event、是否修正樣式、是否簡化互動，要看各個 BaseElement 有沒有使用這個訊號。

這樣理解會比「每個元素都一定會進入縮圖降級模式」更貼近原始碼。

### 4. `visible` 是效能接口，不是資料接口

`visible` 不決定 slide 是否存在，也不決定資料是否同步。

它只決定：

* 現在要不要建立背景與元素 DOM。
* 還是先顯示 placeholder。

這讓 `useLoadSlides()` 可以用很低的成本做到分批渲染：

```text
先讓前面一批頁面真的渲染
後面的頁面先保留 placeholder
再逐步放行更多頁面
```

但它不是完整虛擬列表，因為外層列表項仍然存在，最後也會逐步全量渲染。

## 容易誤解的地方

| 容易誤解                                         | 更精準的理解                                             |
| -------------------------------------------- | -------------------------------------------------- |
| 縮圖系統就是左側縮圖列表                                 | 左側列表只是其中一個場景；核心是共用的 `ThumbnailSlide` 單頁 renderer   |
| `ThumbnailSlide` 負責切頁、排序、多選                  | 不是。這些是 `Editor/Thumbnails/index.vue` 和相關 hooks 的責任 |
| `size` 是縮圖大小                                 | 更精準地說，`size` 是縮圖寬度；高度與倍率都由它推導出來                    |
| 縮圖會重新計算每個元素的小尺寸座標                            | 不會。它用原始投影片座標系排版，再整頁 `scale`                        |
| `target="thumbnail"` 代表每個元素都有 thumbnail 特殊邏輯 | 不一定。它是統一傳入的場景訊號，是否消費取決於各 BaseElement               |
| `visible=false` 代表資料還沒載入                     | 不是。資料仍在，只是背景與元素 DOM 暫時不建立                          |
| `useLoadSlides` 是虛擬列表                        | 不是。它是分批放行縮圖內容渲染的低成本策略                              |
| 匯出圖片是另一套 renderer                            | 不是完全另一套。匯出場景也會重用 `ThumbnailSlide`，只是尺寸和輸出流程不同      |

## 建議閱讀順序

如果要真的讀懂縮圖系統，建議不要一開始就硬啃 `Editor/Thumbnails/index.vue`。

比較好的順序是：

1. `src/store/slides.ts`

   * 先看 `slides`、`viewportSize`、`viewportRatio`。
   * 目標是知道縮圖的資料和比例從哪裡來。

2. `src/views/components/ThumbnailSlide/index.vue`

   * 看 `slide / size / visible` 三個 props。
   * 看外框尺寸、內層原尺寸、`scale`、背景與 placeholder。
   * 目標是理解「單頁縮圖怎麼畫」。

3. `src/views/components/ThumbnailSlide/ThumbnailElement.vue`

   * 看 `elementTypeMap`。
   * 看動態 component。
   * 看 `target="thumbnail"`。
   * 看 `zIndex: elementIndex`。
   * 目標是理解「頁面元素怎麼接回既有 BaseElement」。

4. `src/hooks/useLoadSlides.ts`

   * 看 `slidesLoadLimit`。
   * 看初始放行數量、定時增加數量、最後設成 `9999`。
   * 目標是理解「縮圖內容如何分批渲染」。

5. `src/views/Editor/Thumbnails/index.vue`

   * 最後再看左側縮圖列表。
   * 這時再理解切頁、多選、拖曳、section、右鍵、焦點與快捷鍵，會清楚很多。

6. 其他使用場景

   * `Templates.vue`
   * `LinkDialog.vue`
   * `Screen/SlideThumbnails.vue`
   * `Screen/BottomThumbnails.vue`
   * `PresenterView.vue`
   * `MobilePreview.vue`
   * `MobileThumbnails.vue`
   * `ExportImage.vue`
   * `ExportPDF.vue`

讀完前五個，再看這些跨場景使用點，你會更容易看出：哪些是 `ThumbnailSlide` 的穩定能力，哪些是場景容器自己的外層行為。

## 讀這章時要一直盯住的問題

1. `size` 到底代表什麼？
2. `viewportSize` 和 `viewportRatio` 如何共同決定縮圖尺寸與縮放比例？
3. `ThumbnailSlide` 為什麼不用重新排版，而是可以整頁縮放？
4. `ThumbnailElement` 為什麼只是分派器，而不是每種元素都重寫一份縮圖版？
5. `target="thumbnail"` 是什麼層級的訊號？哪些事情不能過度推論？
6. `visible` 控制的是 DOM 建立時機，還是 slide 資料本身？
7. 為什麼左側縮圖列表會比 `ThumbnailSlide` 複雜很多？
8. 為什麼匯出場景也能重用 `ThumbnailSlide`？

## 這章的核心結論

`PPTist` 的縮圖系統可以拆成三層：

```text
資料層：slides store
渲染層：ThumbnailSlide + ThumbnailElement + BaseElement
場景層：Editor / Screen / Mobile / Export 等容器
```

資料層提供同一份投影片資料；渲染層把一張 slide 用原始座標系畫出來並整頁縮放；場景層決定這張縮圖要被放在哪裡、顯示多大、是否分批渲染，以及要綁定哪些互動。

真正值得學的不是「怎麼把圖片變小」，而是這種責任切分：

> 單頁縮圖 renderer 保持純粹，複雜互動留給場景容器，效能策略抽成 hook，元素渲染則重用既有 BaseElement。

把這個骨架看懂，後面再讀左側縮圖列表的多選、排序、section、快捷鍵與載入策略，就不會迷失在細節裡。
