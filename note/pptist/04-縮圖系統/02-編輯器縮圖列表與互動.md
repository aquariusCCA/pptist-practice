# 02-編輯器縮圖列表與互動

返回：[04-縮圖系統](./README.md)

## 這篇在解決什麼問題

前兩篇已經拆過縮圖渲染核心：

```text
ThumbnailSlide
  負責把一張 slide 以指定寬度縮放渲染

ThumbnailElement
  負責把 slide.elements 裡的每個元素分派給對應的 BaseElement
```

但在編輯器左側，事情不是「把縮圖畫出來」就結束。

`src/views/Editor/Thumbnails/index.vue` 真正扮演的是：

> 編輯器中的頁面管理面板。

它把 `ThumbnailSlide` 放進一個可操作的列表中，並接上：

* 新增頁面
* 插入模板
* 切換目前頁
* 多選頁面
* 拖曳排序
* section 顯示與編輯
* 右鍵選單
* 備註旗標
* 自動捲動到目前頁
* 縮圖列表焦點狀態
* 配合全域快捷鍵的頁面操作
* 配合 `useLoadSlides()` 做分批渲染

所以這篇要建立的核心觀念是：

> `ThumbnailSlide` 是縮圖 renderer，`Editor/Thumbnails/index.vue` 是縮圖列表場景容器，也是頁面管理互動的入口。

這兩者不能混在一起看。

## 一句話看懂這個檔案

`Editor/Thumbnails/index.vue` 做的事情可以濃縮成：

```text
從 slides store 取得頁面列表
用 Draggable 建立可排序清單
每一頁交給 ThumbnailSlide 渲染縮圖
再把點擊、多選、拖曳、section、右鍵、焦點、快捷鍵相關狀態接回 store 與 hooks
```

它不是重新實作縮圖畫面，而是把縮圖 renderer 接進編輯器的頁面管理工作流。

## 原始碼對照表

| 區塊                    | 對應原始碼概念                                                               | 主要責任                              |
| --------------------- | --------------------------------------------------------------------- | --------------------------------- |
| 根節點 `.thumbnails`     | `@mousedown`、`v-click-outside`                                        | 控制縮圖列表是否取得焦點                      |
| `.add-slide`          | `createSlide()`、`Templates`                                           | 新增空白頁或從模板新增頁面                     |
| `Draggable`           | `:modelValue="slides"`、`@end="handleDragEnd"`                         | 顯示並拖曳排序投影片列表                      |
| `thumbnail-container` | 每一頁的外層容器                                                              | 承載 section title 與 thumbnail item |
| `section-title`       | `sectionTag`、`editingSectionId`                                       | 顯示、編輯、右鍵操作 section                |
| `thumbnail-item`      | `active`、`selected`、`mousedown`、`dblclick`                            | 顯示頁面狀態並處理切頁、多選與播放                 |
| `ThumbnailSlide`      | `:slide="element"`、`:size="120"`、`:visible="index < slidesLoadLimit"` | 單頁縮圖渲染                            |
| `note-flag`           | `element.notes`                                                       | 顯示備註數量並開啟備註面板                     |
| `page-number`         | `slideIndex + 1 / slides.length`                                      | 顯示目前頁與總頁數                         |
| `slides store`        | `slides`、`slideIndex`、`currentSlide`                                  | 頁面資料與目前頁狀態                        |
| `main store`          | `_selectedSlidesIndex`、`thumbnailsFocus`                              | 多選頁面與縮圖焦點                         |
| `keyboard store`      | `ctrlKeyState`、`shiftKeyState`                                        | 判斷點擊時的多選模式                        |
| `useSlideHandler()`   | `createSlide`、`copySlide`、`deleteSlide`、`sortSlides` 等                | 頁面操作規則                            |
| `useSectionHandler()` | `createSection`、`removeSection`、`updateSectionTitle` 等                | section 操作規則                      |
| `useLoadSlides()`     | `slidesLoadLimit`                                                     | 分批放行縮圖內容渲染                        |

這張表可以幫你先把這個檔案拆成三層：

```text
畫面結構：template 中的列表、縮圖、section、右鍵入口
狀態來源：slides store / main store / keyboard store
操作規則：useSlideHandler / useSectionHandler / useLoadSlides / useScreening
```

## 1. 這個檔案不是縮圖 renderer，而是場景容器

`Editor/Thumbnails/index.vue` 很容易讓人誤會成「縮圖元件」。

但真正的縮圖畫面不是它畫的。

單頁縮圖是：

```vue
<ThumbnailSlide
  class="thumbnail"
  :slide="element"
  :size="120"
  :visible="index < slidesLoadLimit"
/>
```

這代表：

* `ThumbnailSlide` 負責畫出一張 slide。
* `Editor/Thumbnails/index.vue` 負責把很多張 slide 放進列表。
* 列表中的互動由 `Editor/Thumbnails/index.vue` 和 hooks 處理。

所以它比較像：

```text
縮圖列表容器 + 頁面管理控制器
```

而不是：

```text
縮圖渲染元件
```

這個邊界很重要。

如果以後要自己設計類似 PowerPoint 的左側頁面列表，也應該盡量讓單頁縮圖 renderer 保持純粹，把列表互動留在場景容器中。

## 2. 模板骨架：從外到內看

這個檔案的 template 可以先抓成四層：

```text
.thumbnails
  ├─ .add-slide
  │   ├─ 新增空白頁按鈕
  │   └─ Templates 模板選擇 Popover
  │
  ├─ Draggable.thumbnail-list
  │   └─ thumbnail-container
  │       ├─ section-title
  │       └─ thumbnail-item
  │           ├─ label
  │           ├─ ThumbnailSlide
  │           └─ note-flag
  │
  └─ page-number
```

這個結構說明了三件事：

1. 最外層 `.thumbnails` 是整個縮圖面板。
2. `Draggable` 才是真正承載投影片列表的容器。
3. 單頁內容中，`ThumbnailSlide` 只負責縮圖畫面，頁碼、section、備註旗標都在外層加上去。

所以這裡的縮圖項目不是只有畫面，它還附帶頁面管理資訊。

## 3. 根節點負責縮圖列表焦點

根節點大致是：

```vue
<div
  class="thumbnails"
  @mousedown="() => setThumbnailsFocus(true)"
  v-click-outside="() => setThumbnailsFocus(false)"
>
```

這裡不是普通的 UI 事件而已。

它在管理一個很重要的狀態：

```ts
thumbnailsFocus
```

可以這樣理解：

```text
滑鼠點進縮圖列表 -> 縮圖列表取得焦點
滑鼠點到外面     -> 縮圖列表失去焦點
```

這個焦點會影響全域快捷鍵的語義。

例如同樣是：

* `Ctrl + C`
* `Delete`
* `Ctrl + A`
* `Enter`

如果焦點在畫布，可能是操作元素；如果焦點在縮圖列表，則可能是操作頁面。

所以 `thumbnailsFocus` 不是裝飾狀態，而是頁面層快捷鍵能不能生效的重要條件。

## 4. 新增列：空白新增與模板新增

上方 `.add-slide` 提供兩種新增入口。

第一種是直接新增空白頁：

```text
點擊添加幻燈片 -> createSlide()
```

這個方法來自 `useSlideHandler()`。

第二種是打開模板選擇：

```text
Popover
  └─ Templates
      ├─ select     -> createSlideByTemplate(slide)
      └─ selectAll  -> insertAllTemplates(...)
```

這說明左側縮圖列表不是單純「看頁面」，它也是新增頁面的入口。

特別是模板插入不是單一邏輯：

* 選單張模板：走 `createSlideByTemplate()`。
* 選整組模板：走 `insertAllTemplates()`。

而 `insertAllTemplates()` 內部還會判斷目前簡報是否只有初始空白頁。如果只有一張空白初始頁，會更偏向直接替換整份 slides；否則才把模板加到現有簡報中。

這代表新增頁面不只是 UI 行為，背後其實有工作流判斷。

## 5. `Draggable`：列表排序入口，不是資料規則本身

列表容器使用 `vuedraggable`：

```vue
<Draggable
  class="thumbnail-list"
  ref="thumbnailsRef"
  :modelValue="slides"
  :animation="200"
  :scroll="true"
  :scrollSensitivity="50"
  :disabled="editingSectionId"
  @end="handleDragEnd"
  itemKey="id"
>
```

這裡要注意幾個點。

### `modelValue="slides"`

列表資料來自 `slides store`。

也就是說：

```text
縮圖列表不是自己維護一份 pages
它直接讀 slides store 的 slides
```

這讓畫布、縮圖、播放、匯出都可以基於同一份頁面資料。

### `itemKey="id"`

每張 slide 用 `id` 作為拖曳列表的 key。

這可以避免只用 index 造成 DOM 重用錯亂，尤其排序時更重要。

### `:disabled="editingSectionId"`

當正在編輯 section 標題時，拖曳會被停用。

這個設計合理，因為使用者正在輸入文字時，不應同時觸發排序操作。

### `@end="handleDragEnd"`

拖曳結束時並不直接在 template 裡改資料，而是進入：

```ts
handleDragEnd(eventData)
```

再交給：

```ts
sortSlides(newIndex, oldIndex)
```

真正的排序規則放在 `useSlideHandler()`。

所以要分清楚：

```text
Draggable 負責收集拖曳結果
useSlideHandler.sortSlides 負責資料重排規則
slides store 負責資料落地
```

這是一個很好的責任切分。

## 6. 單頁縮圖項目的結構

每一頁大致是：

```text
thumbnail-container
  ├─ section-title
  └─ thumbnail-item
      ├─ label
      ├─ ThumbnailSlide
      └─ note-flag
```

### `section-title`

如果這一頁有 `sectionTag`，或者整份簡報有 section 且目前是第一頁，就會顯示 section 標題。

這表示 section 是依附在頁面列表上的。

也可以這樣理解：

> section 不是獨立於頁面列表外的導航樹，而是掛在某張 slide 上，用來表示某段頁面的起點。

### `thumbnail-item`

這是使用者真正點擊的區塊。

它負責：

* 顯示 active 狀態。
* 顯示 selected 狀態。
* 接收 mousedown 切頁或多選。
* 接收 dblclick 進入播放。
* 接上單張縮圖右鍵選單。

### `label`

顯示頁碼。

它不是資料邏輯，只是把 index 轉成使用者看的頁碼，例如 `01`、`02`。

### `ThumbnailSlide`

真正畫出這一頁的縮圖。

這裡固定傳入：

```ts
size = 120
```

並用：

```ts
visible = index < slidesLoadLimit
```

控制是否真正渲染縮圖內容。

### `note-flag`

如果該頁有備註，就顯示備註數量。

點擊後呼叫：

```ts
openNotesPanel()
```

這代表縮圖列表不只是頁面縮圖，也承載頁面層附加資訊。

## 7. `active` 與 `selected` 是兩種不同狀態

縮圖項目上有兩種 class：

```vue
:class="{
  active: slideIndex === index,
  selected: selectedSlidesIndex.includes(index),
}"
```

這兩個狀態不能混在一起。

### `active`

`active` 代表：

```text
目前正在編輯的頁面
```

它由 `slidesStore.slideIndex` 決定。

正常情況下，目前頁只有一頁。

### `selected`

`selected` 代表：

```text
目前被納入頁面批次操作集合的頁面
```

它由：

```ts
selectedSlidesIndex = [..._selectedSlidesIndex, slideIndex]
```

推導而來。

這裡要講精準：

> `_selectedSlidesIndex` 是額外多選集合，`slideIndex` 是目前頁。畫面上用 computed 把兩者合併成頁面操作視圖。

它不是把資料真的合併成唯一 source of truth，而是建立一個方便 UI 與操作判斷使用的 selected view。

## 8. 這個元件依賴三個 store

這個檔案主要接三個 store。

### `slides store`

取用：

```ts
slides
slideIndex
currentSlide
```

用途：

| 狀態             | 用途                         |
| -------------- | -------------------------- |
| `slides`       | 縮圖列表資料來源                   |
| `slideIndex`   | 判斷目前頁、顯示頁碼、切頁              |
| `currentSlide` | 判斷右鍵選單狀態，例如目前頁是否已有 section |

### `main store`

取用：

```ts
selectedSlidesIndex: _selectedSlidesIndex
thumbnailsFocus
```

用途：

| 狀態                     | 用途                   |
| ---------------------- | -------------------- |
| `_selectedSlidesIndex` | 額外多選頁面集合             |
| `thumbnailsFocus`      | 判斷縮圖列表是否取得焦點，影響快捷鍵語義 |

### `keyboard store`

取用：

```ts
ctrlKeyState
shiftKeyState
```

用途：

| 狀態              | 用途                   |
| --------------- | -------------------- |
| `ctrlKeyState`  | 點擊縮圖時進入 Ctrl 複選邏輯    |
| `shiftKeyState` | 點擊縮圖時進入 Shift 區間選取邏輯 |

這三個 store 的分工很清楚：

```text
slides store   管頁面資料與目前頁
main store     管 UI 操作狀態與焦點
keyboard store 管鍵盤修飾鍵狀態
```

## 9. `selectedSlidesIndex` 為什麼要補上 `slideIndex`

原始碼中有：

```ts
const selectedSlidesIndex = computed(() => [
  ..._selectedSlidesIndex.value,
  slideIndex.value,
])
```

這行很重要，但也容易過度解讀。

它的意思不是「把 store 裡的選取狀態真的改掉」。

它是在建立一個 UI 與操作用的選取視圖：

```text
額外多選頁面 + 目前頁 = 頁面操作集合
```

為什麼要這樣做？

因為使用者通常會認為：

> 即使我沒有 Ctrl 多選，目前正在看的那一頁也應該是有效操作目標。

例如：

* 只選目前頁時，按 Delete 應該刪目前頁。
* 只選目前頁時，按 Ctrl + C 應該複製目前頁。
* Ctrl 多選幾頁後，目前頁也應該包含在批次操作語義裡。

所以這個 computed 把「目前頁」和「額外選取頁」合成一個操作集合。

但精準地說：

```text
真正的資料狀態仍然分散在：
  mainStore.selectedSlidesIndex
  slidesStore.slideIndex
computed 只是把它們組合成列表場景需要的視圖
```

這樣理解會比直接說「selectedSlidesIndex 是唯一 source of truth」更接近原始碼。

## 10. 點擊縮圖的主流程

點擊縮圖會進入：

```ts
handleClickSlideThumbnail(e, index)
```

這個函式處理三種模式：

1. Ctrl 複選
2. Shift 區間選取
3. 一般單選切頁

在進入這三種模式前，還有兩個前置判斷。

### 前置判斷一：正在編輯 section 時不處理縮圖點擊

```ts
if (editingSectionId.value) return
```

原因是：

```text
使用者正在改 section 標題時，不應因為點擊或輸入流程誤觸切頁、多選、拖曳等操作
```

### 前置判斷二：多選且右鍵點在已選頁時，不改變選取狀態

原始碼會判斷：

```text
如果目前已經是多選
而且點到的頁在選取集合內
且不是滑鼠左鍵
就直接 return
```

這個細節通常是為了右鍵選單體驗。

使用者多選了幾頁後，在其中一頁上按右鍵，預期是對這一批選取頁開選單，而不是右鍵瞬間改掉選取集合。

這個設計很符合桌面檔案管理器的操作心智。

## 11. 一般單選：清掉多選並切頁

沒有 Ctrl / Shift 時，走一般切頁：

```ts
mainStore.updateSelectedSlidesIndex([])
changeSlideIndex(index)
```

意思是：

1. 清空額外多選頁面。
2. 切換目前頁到點擊的 index。

`changeSlideIndex()` 內部會先做：

```ts
mainStore.setActiveElementIdList([])
```

再更新：

```ts
slidesStore.updateSlideIndex(index)
```

這裡很重要。

切頁時清掉 active elements 是合理的，因為元素選取屬於某一張 slide 內部。

當頁面切走後，上一頁的元素選取狀態不應繼續影響新頁。

所以縮圖列表雖然在操作頁面，但它也會維護畫布元素選取的一致性。

## 12. Ctrl 複選：切換某頁是否在選取集合中

當 `ctrlKeyState.value` 成立時，進入 Ctrl 複選。

Ctrl 複選的語義是：

```text
點到未選頁 -> 加入選取集合
點到已選頁 -> 從選取集合移除
```

但目前頁有特殊處理。

如果點到的是目前頁：

* 如果目前沒有多選，直接 return。
* 如果目前有多選，會把目前頁移出集合，並把目前頁切到其他已選頁中的第一個。

這段邏輯的目的，是避免目前頁被取消後，頁面操作基準消失。

因為在這套設計裡：

```text
目前頁永遠應該是頁面操作集合中的基準之一
```

所以取消目前頁時，需要找另一個已選頁變成目前頁。

這是比單純 array toggle 更細的地方。

## 13. Shift 區間選取：建立連續頁面範圍

當 `shiftKeyState.value` 成立時，進入 Shift 選取。

Shift 的語義不是「多加一頁」，而是：

```text
從目前選取集合的邊界到點擊頁，建立連續範圍
```

流程大致是：

1. 取目前選取集合中的最小 index。
2. 把點擊頁當作另一個邊界。
3. 如果點擊頁比目前最小 index 還小，就反過來用目前最大 index 當另一端。
4. 用迴圈建立 `minIndex ~ maxIndex` 的連續陣列。
5. 寫回 `mainStore.updateSelectedSlidesIndex()`。

這個設計讓縮圖列表操作比較接近桌面檔案管理器。

例如目前在第 3 頁，Shift 點第 8 頁，就會選第 3 到第 8 頁。

## 14. 切頁後自動捲動到目前縮圖

這個元件會 watch：

```ts
slideIndex
```

當目前頁改變時，會做兩件事。

### 第一件事：清掉多選狀態

如果目前有選取集合，就清空：

```ts
mainStore.updateSelectedSlidesIndex([])
```

這代表「頁面切換」會讓額外多選狀態失效。

### 第二件事：確保 active thumbnail 在可視範圍

它會等 DOM 更新後：

1. 找到 `.thumbnail-item.active`。
2. 用 `isElementInViewport()` 判斷它是否在縮圖列表可視範圍內。
3. 如果不在，就用 `scrollIntoView({ behavior: 'smooth' })` 捲動過去。

這裡需要 `nextTick()`，因為 `slideIndex` 改變後，DOM 上的 `active` class 要等 Vue 更新完成才能正確取得。

而 `setTimeout(..., 100)` 則是讓捲動動作在 DOM 狀態穩定後再執行。

這個設計讓使用者透過鍵盤、播放控制或其他方式切頁時，左側縮圖列表也能自動跟上目前頁。

## 15. `useLoadSlides` 如何接進左側列表

這個檔案中使用：

```ts
const { slidesLoadLimit } = useLoadSlides()
```

再傳給每張縮圖：

```vue
:visible="index < slidesLoadLimit"
```

這代表左側列表採用分批渲染策略：

```text
index 小於 slidesLoadLimit 的頁面 -> 真正渲染背景與元素
index 大於等於 slidesLoadLimit 的頁面 -> 先顯示 placeholder
```

這一點要和 `ThumbnailSlide` 的責任分清楚。

```text
useLoadSlides    決定目前放行到第幾頁
Thumbnails/index 把 visible 傳給 ThumbnailSlide
ThumbnailSlide   根據 visible 決定要渲染內容還是 placeholder
```

所以 `useLoadSlides` 不直接操作 DOM，也不直接知道縮圖長什麼樣。

它只是提供一個「目前允許渲染到哪一頁」的數值。

## 16. 拖曳排序：事件在列表，規則在 hook

拖曳排序結束後呼叫：

```ts
handleDragEnd(eventData)
```

邏輯很薄：

```text
如果 newIndex / oldIndex 無效，或兩者一樣，就 return
否則呼叫 sortSlides(newIndex, oldIndex)
```

這是很好的設計。

因為拖曳排序不只是把 array item 換位置，還可能牽涉到：

* 目前頁要跟著移動。
* sectionTag 要保持語義正確。
* 歷史紀錄要不要記錄。
* 刪除、新增、排序的規則要保持一致。

所以 `Thumbnails/index.vue` 不應該自己處理所有資料規則。

它只負責：

```text
接收使用者拖曳事件
取得 newIndex / oldIndex
把排序需求交給 useSlideHandler.sortSlides()
```

真正的頁面重排規則放在 `useSlideHandler()` 中，這讓 UI 層比較乾淨。

## 17. section 標題：頁面列表中的章節入口

縮圖列表每個 `thumbnail-container` 上方可能有 `section-title`。

顯示條件是：

```text
這一頁有 sectionTag
或整份簡報有 section 且目前是第一頁
```

這透露出 PPTist 的 section 設計：

> section 是掛在某張 slide 上的頁面層資料，用來表示一段投影片的起點。

section title 支援：

* 顯示 section 標題。
* 沒有標題時顯示預設文字。
* 雙擊進入編輯。
* blur 或 Enter 儲存。
* 右鍵刪除、刪除 section 內頁面、刪除全部 section、重新命名。

這代表左側縮圖列表同時也是章節管理入口。

## 18. 編輯 section 時要停用快捷鍵

當使用者雙擊 section title 進入編輯時，會做：

```ts
mainStore.setDisableHotkeysState(true)
editingSectionId.value = id || 'default'
nextTick(() => input.focus())
```

這個設計很重要。

使用者正在 input 裡輸入 section 名稱時：

* 按 Delete 不應刪除頁面。
* 按 Ctrl + A 不應全選頁面。
* 按 Enter 應該儲存 section 標題，而不是新增頁面。

所以 section 編輯期間要暫時停用全域快捷鍵。

儲存時再：

```text
更新 section title
清空 editingSectionId
恢復 hotkeys
```

這是一個典型的「文字輸入狀態要接管鍵盤」的設計。

## 19. 右鍵選單：頁面管理能力的 UI 清單

這個檔案有三類右鍵選單。

### 縮圖列表空白處右鍵

提供：

* 貼上
* 全選
* 新建頁面
* 從頭放映

這些偏向「整個縮圖列表」的操作。

### 單張縮圖右鍵

提供：

* 剪切
* 複製
* 貼上
* 全選
* 新建頁面
* 複製頁面
* 刪除頁面
* 增加 section
* 從當前頁放映

這些偏向「目前頁或目前選取頁」的操作。

特別注意：

```text
增加 section 會根據 currentSlide.sectionTag 判斷是否禁用
```

這代表右鍵選單不是靜態選單，也會根據目前頁狀態調整。

### section 標題右鍵

提供：

* 刪除 section
* 刪除 section 和投影片
* 刪除所有 section
* 重新命名 section

這些偏向章節管理。

所以右鍵選單其實把「頁面管理能力」集中暴露在縮圖列表上。

## 20. 備註旗標：頁面附加資訊入口

縮圖右上角可能顯示：

```text
note-flag
```

條件是：

```text
element.notes 存在且長度大於 0
```

顯示內容是備註數量。

點擊後呼叫：

```ts
openNotesPanel()
```

並設定：

```ts
mainStore.setNotesPanelState(true)
```

這說明縮圖列表不只是視覺導航，它也承載頁面層 metadata。

使用者可以從縮圖直接知道：

```text
這一頁有沒有備註
這一頁有幾條備註
能不能快速打開備註面板
```

這是縮圖列表作為「頁面工作台」的一部分。

## 21. 雙擊縮圖：從目前頁開始播放

`thumbnail-item` 上有：

```vue
@dblclick="enterScreening()"
```

這代表雙擊縮圖會進入播放。

這裡呼叫的是 `useScreening()` 提供的播放入口。

它和右鍵選單中的「從當前放映」語義接近，都是基於目前頁進入播放模式。

所以縮圖列表也兼具播放入口，不只是編輯入口。

## 22. `page-number` 顯示目前頁與總頁數

底部有：

```text
幻燈片 slideIndex + 1 / slides.length
```

這是很小的 UI，但它提醒你：

* `slideIndex` 是從 0 開始的程式索引。
* 使用者看到的頁碼是從 1 開始。
* 總頁數直接來自 `slides.length`。

這種地方容易小，但在頁面管理 UI 裡很常見。

## 23. 這個檔案和 hooks 的邊界

這篇最值得學的不是每個選單項，而是邊界切分。

### `Thumbnails/index.vue` 負責什麼？

它負責場景組裝：

* 顯示列表。
* 放入 `ThumbnailSlide`。
* 綁定點擊、拖曳、右鍵、雙擊。
* 管理 section title 的輸入狀態。
* 把 store 狀態轉成 UI class。
* 把 hook 方法接到事件上。

### `ThumbnailSlide` 負責什麼？

只負責：

```text
把一張 slide 畫成縮圖
```

不負責切頁、多選、排序、section、右鍵。

### `useSlideHandler()` 負責什麼？

負責頁面操作規則：

* 新增頁面
* 複製頁面
* 剪下頁面
* 貼上頁面
* 刪除頁面
* 全選頁面
* 排序頁面
* 重置簡報

### `useSectionHandler()` 負責什麼？

負責 section 操作規則：

* 建立 section
* 刪除 section
* 刪除所有 section
* 刪除 section 內頁面
* 更新 section 標題

### `useLoadSlides()` 負責什麼？

只負責提供：

```ts
slidesLoadLimit
```

也就是分批渲染的放行門檻。

### `useScreening()` 負責什麼？

提供播放模式入口，例如從頭播放或從目前頁播放。

這樣切分後，`Thumbnails/index.vue` 雖然長，但它不是把所有邏輯都硬塞在一起。

它更像一個場景協調者。

## 24. 為什麼這個場景比其他縮圖場景複雜

和其他縮圖場景相比，編輯器左側縮圖列表最複雜。

例如：

* `Templates.vue` 重點是選模板。
* `LinkDialog.vue` 重點是預覽連結目標。
* `MobilePreview.vue` 重點是手機預覽。
* `ExportImage.vue` 重點是輸出畫面。

但 `Editor/Thumbnails/index.vue` 要同時處理：

1. 頁面縮圖顯示。
2. 頁面新增。
3. 模板插入。
4. 目前頁切換。
5. Ctrl / Shift 多選。
6. 拖曳排序。
7. section 顯示與編輯。
8. 右鍵選單。
9. 備註入口。
10. 播放入口。
11. 自動捲動。
12. 快捷鍵焦點。
13. 分批載入。

所以它不只是 preview list，而是完整的 page management panel。

## 25. 讀這個檔案時的建議順序

建議不要從 script 最上面一路硬讀到最後。

比較好的順序是：

### 第一步：先看 template 骨架

先看：

```text
.thumbnails
.add-slide
Draggable
thumbnail-container
section-title
thumbnail-item
ThumbnailSlide
note-flag
page-number
```

目標是知道畫面分成哪些區塊。

### 第二步：看 store 來源

再看：

```text
slides / slideIndex / currentSlide
_selectedSlidesIndex / thumbnailsFocus
ctrlKeyState / shiftKeyState
```

目標是知道畫面狀態從哪裡來。

### 第三步：看 selectedSlidesIndex computed

理解：

```text
額外多選集合 + 目前頁 = 頁面操作視圖
```

這是後面多選與右鍵操作的基礎。

### 第四步：看 handleClickSlideThumbnail

依序拆：

```text
前置 return
Ctrl 複選
Shift 區間選取
一般切頁
```

這是這個檔案最重要的互動邏輯。

### 第五步：看 watch(slideIndex)

理解切頁後如何：

* 清掉多選。
* 捲動到 active thumbnail。

### 第六步：看 handleDragEnd

只要知道它把排序交給 `sortSlides()`。

真正排序規則到 `useSlideHandler.ts` 再看。

### 第七步：看 section 與右鍵選單

最後再看：

* `editSection()`
* `saveSection()`
* `contextmenusSection`
* `contextmenusThumbnails`
* `contextmenusThumbnailItem`

這些是頁面管理功能的補完整。

## 26. 容易誤解的地方

| 容易誤解                                | 精準理解                                                        |
| ----------------------------------- | ----------------------------------------------------------- |
| `Thumbnails/index.vue` 是縮圖渲染元件      | 不是。它是編輯器左側縮圖列表場景容器                                          |
| 單頁縮圖是這個檔案自己畫的                       | 不是。單頁縮圖交給 `ThumbnailSlide`                                  |
| `slides` 是列表自己維護的資料                 | 不是。它來自 `slides store`                                       |
| `selectedSlidesIndex` 就是 store 原始狀態 | 不是。這裡的 computed 是 `_selectedSlidesIndex + slideIndex` 的操作視圖 |
| `active` 和 `selected` 是同一件事         | 不是。`active` 是目前頁；`selected` 是頁面操作集合                         |
| 拖曳排序直接在這個檔案改 slides                 | 不是。這裡只接事件，真正排序交給 `sortSlides()`                             |
| `useLoadSlides` 是虛擬列表               | 不是。它只是控制哪些縮圖內容先渲染                                           |
| section 是獨立樹狀結構                     | 不是。從這裡看，它是掛在 slide 上的 `sectionTag`                          |
| 右鍵選單只是 UI 裝飾                        | 不是。它暴露頁面管理與 section 管理能力                                    |
| `thumbnailsFocus` 只是 CSS 焦點         | 不是。它會影響全域快捷鍵要操作頁面還是元素                                       |

## 27. 這篇最重要的結論

### 結論一：左側縮圖列表是頁面管理面板

它不只是顯示縮圖，而是整個編輯器中操作頁面、section、播放、備註與模板插入的重要入口。

### 結論二：縮圖渲染和列表互動被拆開

`ThumbnailSlide` 專心渲染一張 slide。

`Thumbnails/index.vue` 專心處理列表場景的互動。

這個邊界很值得學。

### 結論三：多選狀態是 `_selectedSlidesIndex + slideIndex` 的操作視圖

目前頁永遠被納入頁面操作語義中，但真正狀態仍分散在 main store 與 slides store。

### 結論四：拖曳、section、刪除等複雜規則沒有全部塞在 UI 裡

這個檔案把事件交給：

* `useSlideHandler()`
* `useSectionHandler()`
* `useLoadSlides()`
* `useScreening()`

所以它雖然長，但本質上是場景協調者。

### 結論五：縮圖列表的難點不是畫縮圖，而是狀態協調

真正複雜的是把下面這些狀態接在一起：

```text
slides
slideIndex
currentSlide
_selectedSlidesIndex
thumbnailsFocus
ctrlKeyState
shiftKeyState
editingSectionId
slidesLoadLimit
```

然後讓點擊、拖曳、右鍵、快捷鍵、section 編輯都不互相打架。

## 一句話抓核心

`Editor/Thumbnails/index.vue` 的核心不是「渲染縮圖」，而是把 `ThumbnailSlide` 放進可操作的頁面管理列表中，並協調 slides store、main store、keyboard store 與各種 hooks，完成切頁、多選、排序、section、右鍵選單、備註、播放與分批載入等編輯器工作流。

讀懂這個檔案，你就會知道：真正的 PPT 編輯器左側縮圖欄，不是 preview list，而是一個濃縮版的頁面管理系統。
