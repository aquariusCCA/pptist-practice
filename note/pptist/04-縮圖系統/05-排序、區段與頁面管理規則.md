# 05-排序、區段與頁面管理規則

返回：[04-縮圖系統](./README.md)

## 這篇在解決什麼問題

左側縮圖列表最容易被低估的地方，不是「拖曳排序」本身，而是拖曳、刪除、新增、複製、剪下、貼上、section 這些頁面操作背後都需要維持資料規則。

如果只看 UI，很容易把它想成：

```text
新增頁面 -> slides.push()
刪除頁面 -> slides.splice()
拖曳排序 -> array 移動位置
section -> 顯示一個標題
```

但在 `PPTist` 裡，實際規則比這個更完整。

它至少要處理：

* 新頁面要插在哪裡。
* 新增後目前頁要不要切到新頁。
* 複製模板頁時元素 id 和 group id 要不要重建。
* 刪除全部頁面時能不能讓簡報變成空陣列。
* 刪除或移動有 `sectionTag` 的頁面時，section 起點要如何延續。
* 拖曳排序後目前頁要不要跟著新位置更新。
* section 刪除時，是只刪 section 標記，還是連 section 內頁面一起刪。
* UI 層要不要直接改資料，還是把規則交給 hook / store。

所以這篇要建立的核心觀念是：

> 縮圖列表只是頁面管理操作的入口；真正的頁面規則主要分散在 `useSlideHandler.ts`、`useSectionHandler.ts` 和 `slides store` 中。

## 一句話看懂這篇

`Editor/Thumbnails/index.vue` 負責接收使用者事件；`useSlideHandler()` 負責新增、複製、剪下、刪除、全選、排序等頁面操作；`useSectionHandler()` 負責 section 建立、刪除、重新命名與刪除 section 內頁面；`slides store` 負責真正改變 `slides` 與 `slideIndex`。

可以畫成：

```text
使用者操作縮圖列表
  ├─ 點新增
  ├─ 右鍵複製 / 刪除
  ├─ 拖曳排序
  ├─ 建立 section
  ├─ 重新命名 section
  └─ 刪除 section 內頁面

      ↓

Editor/Thumbnails/index.vue
  └─ 只負責事件入口與 UI 狀態

      ↓

useSlideHandler.ts
  ├─ createSlide()
  ├─ createSlideByTemplate()
  ├─ copySlide()
  ├─ pasteSlide()
  ├─ copyAndPasteSlide()
  ├─ cutSlide()
  ├─ deleteSlide()
  ├─ selectAllSlide()
  └─ sortSlides()

useSectionHandler.ts
  ├─ createSection()
  ├─ removeSection()
  ├─ removeAllSection()
  ├─ removeSectionSlides()
  └─ updateSectionTitle()

      ↓

slides store
  ├─ addSlide()
  ├─ deleteSlide()
  ├─ setSlides()
  ├─ updateSlide()
  ├─ removeSlideProps()
  └─ updateSlideIndex()
```

這張圖要抓住一個重點：

> `Thumbnails/index.vue` 不應該承擔所有頁面資料規則；它應該把操作交給更靠近資料規則的 hook / store。

## 原始碼對照表

| 規則 / 功能        | 主要位置                                  | 精準責任                                                                   |
| -------------- | ------------------------------------- | ---------------------------------------------------------------------- |
| 拖曳結束事件         | `Editor/Thumbnails/index.vue`         | 取得 `newIndex` / `oldIndex`，再呼叫 `sortSlides()`                          |
| 新增空白頁          | `useSlideHandler.ts`                  | 建立空白 slide、清空元素選取、呼叫 `slidesStore.addSlide()`、記錄歷史快照                   |
| 根據模板新增頁        | `useSlideHandler.ts`                  | 重建元素 id / group id，產生新 slide id，插入目前頁後方                                |
| 插入頁面落地         | `slides store` 的 `addSlide()`         | 移除待插入 slide 的 `sectionTag`，插到目前頁後方，並把 `slideIndex` 切到新增位置              |
| 複製頁面           | `useSlideHandler.ts`                  | 把頁面操作集合中的 slides 加密成剪貼簿資料                                              |
| 貼上頁面           | `useSlideHandler.ts`                  | 讀取剪貼簿，交給貼上流程並限制為 slide 貼上                                              |
| 剪下頁面           | `useSlideHandler.ts`                  | 先快取要刪除的 slide id，再複製，最後刪除                                              |
| 刪除頁面           | `useSlideHandler.ts` + `slides store` | 如果刪除全部頁面則 reset；否則交給 store 刪除並更新目前頁                                    |
| 刪除時 section 延續 | `slides store` 的 `deleteSlide()`      | 被刪頁若有 `sectionTag`，嘗試把 section 交給下一頁                                   |
| 全選頁面           | `useSlideHandler.ts`                  | 清空元素選取，將所有 slide index 寫入頁面多選集合                                        |
| 排序頁面           | `useSlideHandler.ts`                  | 深拷貝 slides，處理 sectionTag 延續與第一頁 section，再 setSlides / updateSlideIndex |
| 建立 section     | `useSectionHandler.ts`                | 在目前 slide 上寫入新的 `sectionTag.id`                                        |
| 刪除 section     | `useSectionHandler.ts`                | 移除某張 slide 上的 `sectionTag`                                             |
| 刪除全部 section   | `useSectionHandler.ts`                | 移除所有 slide 上的 `sectionTag`                                             |
| 刪除 section 內頁面 | `useSectionHandler.ts`                | 從 section 起點一路收集到下一個 section 前一頁，交給 `deleteSlide(ids)`                 |
| 更新 section 標題  | `useSectionHandler.ts`                | 更新指定 section 的 title；若是 `default`，會在第一頁建立 sectionTag                   |

這張表可以幫你快速定位：

```text
事件在哪裡發生？
規則在哪裡判斷？
資料在哪裡落地？
```

## 1. 縮圖列表只是事件入口

在 `Editor/Thumbnails/index.vue` 裡，拖曳排序最後會進入：

```ts
handleDragEnd(eventData)
```

這個函式本身很薄，主要做：

```text
取出 newIndex / oldIndex
如果 index 無效或沒有變化，就 return
否則呼叫 sortSlides(newIndex, oldIndex)
```

這代表縮圖列表不直接自己改 `slides`。

它只是把使用者拖曳結果轉成「我要把 oldIndex 的頁面移到 newIndex」這個意圖。

真正的資料規則交給：

```ts
useSlideHandler().sortSlides()
```

這個邊界很重要。

因為排序頁面不是單純 array move，還要考慮：

* 被移動的頁面是否帶有 `sectionTag`。
* 原本 section 是否需要延續到下一頁。
* 移到第一頁時第一頁 section 如何處理。
* 排序後目前頁是否跟到新位置。

如果這些規則全部塞在 `Thumbnails/index.vue`，UI 元件會變得很難維護。

## 2. `useSlideHandler()` 是頁面操作規則中心

`useSlideHandler()` 負責大部分頁面級操作。

它一開始會取用：

```text
main store:
  _selectedSlidesIndex
  activeElementIdList

slides store:
  currentSlide
  slides
  theme
  slideIndex
```

然後建立頁面操作視圖：

```text
selectedSlidesIndex = _selectedSlidesIndex + slideIndex
selectedSlides = slides 中 index 落在 selectedSlidesIndex 的 slide
selectedSlidesId = selectedSlides 的 id 集合
```

這一點和前一篇焦點 / 多選資料流有關。

頁面操作不是只操作 `_selectedSlidesIndex`，而是操作：

```text
額外多選頁面 + 目前頁
```

所以像複製、剪下、刪除，都會以這個集合為基礎。

## 3. `resetSlides()`：簡報不能變成空陣列

`resetSlides()` 會建立一張空白 slide：

```text
id: 新產生
元素: 空陣列
背景: 使用目前 theme 的 backgroundColor
```

然後做：

```text
slideIndex = 0
activeElementIdList = []
slides = [emptySlide]
```

它的核心意義是：

> 簡報至少要保留一張頁面。

這個不變式很重要。

如果刪除全部頁面後讓 `slides = []`，很多地方都會出問題：

* `currentSlide` 會取不到。
* 畫布不知道要顯示什麼。
* 縮圖列表沒有目前頁。
* 新增、貼上、匯出等流程都要處理空簡報特例。

所以當使用者刪掉所有頁面時，PPTist 不是真的讓簡報變空，而是重置成一張空白頁。

## 4. `createSlide()`：新增空白頁插在目前頁後方

新增空白頁的流程大致是：

```text
建立 emptySlide
清空 activeElementIdList
呼叫 slidesStore.addSlide(emptySlide)
記錄 history snapshot
```

這裡有兩個重點。

### 第一，新頁不是加在最後，而是加在目前頁後面

真正插入位置由 `slidesStore.addSlide()` 決定。

它會用：

```text
addIndex = slideIndex + 1
```

也就是插在目前頁後面。

這符合 PPT 編輯器常見行為：

```text
我正在第 3 頁工作
按新增頁面
新頁應該出現在第 3 頁後面
```

而不是永遠 append 到最後。

### 第二，新增後目前頁會切到新頁

`slidesStore.addSlide()` 插入後會把：

```text
slideIndex = addIndex
```

所以新增空白頁後，使用者會直接進入新頁。

這是工作流設計，不只是資料插入。

## 5. `slidesStore.addSlide()` 會移除待插入頁面的 `sectionTag`

這是一個容易漏掉但很重要的細節。

`addSlide()` 接收一張或多張 slide 時，會先處理每張待插入 slide：

```text
如果 slide 上有 sectionTag，就刪掉
```

然後才插入目前頁後方。

這代表：

> 新增 / 插入 slide 時，不會直接把外來 slide 的 sectionTag 原封不動帶進目前簡報。

這個設計有道理。

因為 `sectionTag` 代表的是目前簡報中的章節起點。

如果從模板或其他來源插入一張 slide，直接保留它原本的 `sectionTag`，可能造成目前簡報章節結構錯亂。

所以插入頁面時先去掉 section 標記，再讓使用者或頁面管理邏輯重新建立 section，會比較安全。

## 6. `createSlideByTemplate()`：模板新增需要重建 id

根據模板建立新頁時，不只是把模板 slide 直接插入。

它會做幾件事：

```text
根據模板元素建立 groupIdMap / elIdMap
重寫每個 element.id
如果元素有 groupId，也重寫 groupId
產生新的 slide.id
清空 activeElementIdList
呼叫 slidesStore.addSlide(newSlide)
記錄 history snapshot
```

這裡的核心是：

> 模板頁插入後，必須成為目前簡報中的新頁，而不是和模板資料共用同一批 id。

如果不重建 id，會有風險：

* 元素 id 和原模板資料衝突。
* 群組 id 對不上。
* 後續選取、複製、刪除、動畫或關聯資料可能混亂。

所以模板插入的重點不是「複製一張 slide」而已，而是要重新建立它在目前簡報中的身份。

## 7. `copySlide()`：複製的是頁面操作集合

複製頁面時，`copySlide()` 不是只複製目前頁。

它會使用：

```text
selectedSlides
```

也就是：

```text
額外多選頁面 + 目前頁
```

然後把資料包成：

```text
type: 'slides'
data: selectedSlides
```

再加密後寫入剪貼簿。

這裡有兩個重點。

### 第一，頁面剪貼簿和元素剪貼簿有語義區分

資料裡會標記：

```text
type = slides
```

這讓貼上流程能知道現在貼的是頁面，不是文字或元素。

### 第二，複製成功後會重新讓縮圖列表取得焦點

複製完成後會呼叫：

```text
setThumbnailsFocus(true)
```

這可以讓後續貼上、剪下、刪除等頁面層操作仍然留在縮圖列表語境中。

## 8. `pasteSlide()`：讀剪貼簿後交給專門貼上流程

貼上頁面時，`pasteSlide()` 會：

```text
讀取剪貼簿文字
交給 pasteTextClipboardData(text, { onlySlide: true })
```

這裡重點是 `onlySlide: true`。

它表示這次貼上流程只接受 slide 類資料。

所以頁面貼上不是在 `pasteSlide()` 裡直接解析所有資料細節，而是把剪貼簿資料交給更通用的貼上處理 hook。

這樣可以避免把剪貼簿解析、元素貼上、頁面貼上的全部邏輯塞在 `useSlideHandler()` 裡。

## 9. `copyAndPasteSlide()`：快速複製目前頁

快速複製頁面時，`copyAndPasteSlide()` 會深拷貝目前頁，再交給：

```text
addSlidesFromData([slide])
```

這代表它不是透過系統剪貼簿完成，而是直接把目前頁資料複製後走新增資料流程。

這通常對應快捷鍵或右鍵中的「複製頁面」行為。

這裡可以先記住責任邊界：

```text
copyAndPasteSlide() 決定要複製目前頁
addSlidesFromData() 負責把 slide data 加進目前簡報
```

不要在這篇過度推論 `addSlidesFromData()` 的所有細節，因為那是另一個 hook 的責任。

## 10. `deleteSlide()`：刪除全部時要 reset

`useSlideHandler.deleteSlide()` 的核心規則是：

```text
如果要刪除的頁數等於全部頁數
  -> resetSlides()
否則
  -> slidesStore.deleteSlide(targetSlidesId)

清空 selectedSlidesIndex
記錄 history snapshot
```

這裡最重要的是：

> 刪除全部頁面時，不讓簡報變成空，而是重置成一張空白頁。

這就是前面說的頁面層不變式。

另外，刪除後會清空多選頁面。

因為被選取的那些頁面可能已經不存在，繼續保留 index 會造成狀態錯亂。

## 11. `slidesStore.deleteSlide()`：真正刪資料並修正目前頁

如果不是刪除全部頁面，就會進入 `slidesStore.deleteSlide()`。

它會做幾件事：

1. 把傳入的 id 統一成陣列。
2. 深拷貝目前 slides。
3. 逐一找到要刪除的 slide index。
4. 如果被刪除頁有 `sectionTag`，嘗試把 section 交給下一頁。
5. 從陣列中移除該 slide。
6. 根據被刪除頁的位置計算新的 `slideIndex`。
7. 更新 store 中的 `slides` 和 `slideIndex`。

這裡有兩個關鍵規則。

### 規則一：刪除有 sectionTag 的頁面時，section 會嘗試延續給下一頁

如果被刪掉的頁面有 `sectionTag`：

```text
查看下一頁
如果下一頁存在，且下一頁沒有 sectionTag
  -> 把被刪頁的 sectionTag 交給下一頁
```

這表示 section 的語義是：

> 某個 section 的起點應該跟著該段內容延續，而不是因為刪掉開頭頁就直接消失。

例如：

```text
第 3 頁是「產品介紹」section 起點
第 4、5 頁屬於產品介紹
刪掉第 3 頁後
如果第 4 頁沒有 sectionTag
就把「產品介紹」交給第 4 頁
```

這樣 section 結構比較不容易因刪頁而斷掉。

### 規則二：刪除後目前頁會移到合理位置

刪除後會用被刪頁中的最小 index 作為新的目前頁候選。

如果這個 index 超過剩餘 slides 的最大 index，就調整成最後一頁。

也就是說：

```text
刪除中間頁 -> 目前頁大致落在原位置的下一頁
刪除最後頁 -> 目前頁退到新的最後一頁
```

這是很常見的編輯器行為。

## 12. `cutSlide()`：為什麼要先快取 id

剪下頁面看起來像：

```text
copySlide()
deleteSlide()
```

但原始碼中特別先做：

```text
targetSlidesId = [...selectedSlidesId]
```

原因是：

> 複製操作可能造成多選狀態變化，所以要先把要刪除的 slide id 快取起來。

這個細節很重要。

如果不快取，流程可能變成：

```text
copySlide() 之後 selectedSlides 變了
再 deleteSlide() 時刪除目標不準
```

所以剪下頁面不是簡單「複製再刪除」，而是：

```text
先鎖定刪除目標
再複製
最後用鎖定好的 id 刪除
```

這是很好的資料一致性保護。

## 13. `selectAllSlide()`：全選頁面前會清空元素選取

全選頁面會建立：

```text
[0, 1, 2, ..., slides.length - 1]
```

然後：

```text
清空 activeElementIdList
更新 selectedSlidesIndex
```

這裡的重點是：

> 全選頁面和全選元素是不同層級的操作。

當使用者在縮圖列表中全選頁面時，畫布元素選取應該被清掉。

否則後續快捷鍵可能不知道應該操作元素還是頁面。

小細節：因為頁面操作視圖會再加上 `slideIndex`，所以實作上可能出現目前頁 index 被重複包含的情況；但很多判斷使用 `includes(index)`，所以語義上仍是「全部頁面被納入操作集合」。讀原始碼時要把它理解成操作視圖，而不是嚴格去重的 Set。

## 14. `sortSlides()`：排序不只是 array move

拖曳排序真正進入：

```text
sortSlides(newIndex, oldIndex)
```

這個函式比單純 splice 複雜，因為它要處理 `sectionTag`。

大致流程是：

```text
如果 oldIndex === newIndex，直接 return
深拷貝 slides
取得 movingSlide
如果 movingSlide 有 sectionTag，先處理原位置 section 延續
如果 newIndex === 0，處理第一頁 section 移交
從 oldIndex 移除 slide
插入 newIndex
setSlides(_slides)
updateSlideIndex(newIndex)
```

這裡有三個最重要的規則。

## 15. 排序規則一：先複製 slides 再重排

`sortSlides()` 不是直接在 store 的 `slides` 上原地操作，而是先深拷貝：

```text
_slides = deep clone slides
```

然後在 `_slides` 上處理 section 與排序。

最後才：

```text
slidesStore.setSlides(_slides)
```

這種寫法的好處是：

* 操作過程比較集中。
* 最後一次性提交結果。
* 比較容易在排序前先處理 section 移交規則。

## 16. 排序規則二：移動 section 起點頁時，原 section 要留給下一頁

如果被移動的頁面本身有 `sectionTag`：

```text
取出 movingSlide.sectionTag
刪掉 movingSlide.sectionTag
查看 oldIndex + 1 的下一頁
如果下一頁存在且沒有 sectionTag
  -> 把原 sectionTag 交給下一頁
```

這表示：

> 當一個 section 的開頭頁被拖走時，原本那一段內容仍然應該保留 section 起點。

例如：

```text
第 3 頁有 sectionTag = A
第 4、5 頁原本屬於 A
使用者把第 3 頁拖到第 10 頁
```

如果直接把第 3 頁帶著 section A 移走，那第 4、5 頁就會失去 section 起點。

所以原始碼會先把 A 從第 3 頁拿掉，交給第 4 頁。

這讓 section 的語義更像：

```text
section 屬於一段頁面，而不只是某張被拖走的 slide
```

## 17. 排序規則三：移到第一頁時，要處理第一頁原本的 section

如果 `newIndex === 0`，也就是某張頁面被拖到第一頁，原始碼會檢查原本第一頁是否有 `sectionTag`。

如果有：

```text
移除原第一頁的 sectionTag
把它交給 movingSlide
```

這代表：

> 第一頁如果原本是某個 section 起點，而新的頁面被插到最前面，section 起點需要跟著新的第一頁調整。

這個設計是為了維持第一頁 section 顯示與 section 起點語義。

要注意：這段是在真正 splice 移動前處理的，所以讀程式時要特別小心時間點。

## 18. 排序後目前頁會更新到新位置

排序完成後會做：

```text
slidesStore.updateSlideIndex(newIndex)
```

這代表拖曳哪一頁，排序完成後目前頁會跟到它的新位置。

這符合使用者預期：

```text
我拖的是這張頁
拖完之後，我仍然在操作這張頁
```

不是停留在原本 index 對應的新頁面。

## 19. 關於排序與 history snapshot 的精準說法

新增、刪除、建立 section、更新 section title 等操作中，可以看到明確的 history snapshot 呼叫。

但就 `sortSlides()` 這個函式本身來看，它主要做：

```text
setSlides(_slides)
updateSlideIndex(newIndex)
```

沒有在函式內直接看到 history snapshot。

所以筆記中不要寫成：

```text
sortSlides 一定會記錄歷史紀錄
```

比較精準的說法是：

> `sortSlides()` 本身負責資料重排與目前頁更新；是否有額外歷史記錄，要再看外層或相關歷史機制，不要在這篇過度推論。

這種精準度很重要，因為我們是在做原始碼筆記，不是憑設計常識補劇情。

## 20. `sectionTag` 的本質：section 起點標記

從 `useSlideHandler()`、`useSectionHandler()` 和 `slides store` 的處理可以看出，`sectionTag` 不是一份獨立的 section tree。

它是掛在 slide 上的標記：

```text
這張 slide 是某個 section 的起點
```

一張有 `sectionTag` 的 slide，大致表示：

```text
從這張 slide 開始，到下一張有 sectionTag 的 slide 之前，都屬於同一個 section
```

所以 section 操作和頁面順序高度耦合。

這也是為什麼刪除、排序都要處理 `sectionTag`。

如果 section 是獨立資料結構，也許排序時只要更新 section 的 slide id；但 PPTist 目前的設計是 section 掛在 slide 上，所以頁面移動本身就會影響 section 語義。

## 21. `createSection()`：在目前頁建立 section 起點

建立 section 時，`useSectionHandler.createSection()` 會在目前 slide 上更新：

```text
sectionTag: { id: nanoid(6) }
```

然後記錄 history snapshot。

這表示：

> 建立 section 的本質，就是把目前頁標記成一個 section 起點。

它一開始可以沒有 title。

標題可以之後透過重新命名補上。

## 22. `removeSection()`：只移除 section 標記，不刪頁面

刪除 section 時，`removeSection(sectionId)` 會：

```text
找到 sectionTag.id 等於 sectionId 的 slide
移除該 slide 的 sectionTag 屬性
記錄 history snapshot
```

這裡要分清楚：

> removeSection 是刪 section 標記，不是刪 slide。

也就是說，頁面仍然存在，只是不再作為 section 起點。

這和「刪除 section 及其投影片」是不同操作。

## 23. `removeAllSection()`：移除全部 sectionTag

刪除全部 section 時，會遍歷所有 slide：

```text
如果 slide 有 sectionTag，就刪掉
```

然後：

```text
slidesStore.setSlides(_slides)
記錄 history snapshot
```

這個操作只清掉章節結構，不改變頁面順序與頁面內容。

所以它的語義是：

```text
保留所有投影片，但移除所有章節分組
```

## 24. `removeSectionSlides()`：刪除某個 section 範圍內的頁面

這個函式比較重要。

它的流程是：

```text
如果有 sectionId
  -> 找到該 section 起點 index
如果沒有 sectionId
  -> startIndex = 0

從 startIndex 開始往後掃 slides
  -> 把每張 slide.id 加入 ids
  -> 如果不是 startIndex 且遇到下一個 sectionTag，就停止

最後呼叫 deleteSlide(ids)
```

這表示 section 範圍是根據 slide 順序推導出來的。

例如：

```text
第 1 頁：section A
第 2 頁
第 3 頁
第 4 頁：section B
第 5 頁
```

如果刪除 section A 內頁面，就會刪：

```text
第 1、2、3 頁
```

遇到第 4 頁的 section B 就停止。

這也再次證明：

> sectionTag 是區段起點，section 範圍由「當前 section 起點到下一個 section 起點前」決定。

## 25. `updateSectionTitle()`：default section 的特殊處理

更新 section 標題時有一個特殊情況：

```text
sectionId === 'default'
```

這通常對應一種 UI 狀態：簡報已有 section 概念，但第一頁本身還沒有明確 sectionTag，需要為第一頁建立預設 section。

這時會在第一頁建立：

```text
sectionTag: {
  id: 新產生,
  title
}
```

如果不是 `default`，就會根據 `sectionId` 找到對應 slide，更新它既有 `sectionTag` 的 title。

所以這裡要注意：

> 重新命名 section 不只是改 title，也可能在 default 情況下替第一頁建立 sectionTag。

## 26. 編輯 section 標題時的快捷鍵保護

雖然這篇重點是排序與頁面管理，但 section 編輯和快捷鍵也有交會。

在 `Thumbnails/index.vue` 中，雙擊 section title 進入編輯時，會設定：

```text
disableHotkeys = true
editingSectionId = section id 或 default
focus input
```

儲存後再：

```text
updateSectionTitle(...)
editingSectionId = ''
disableHotkeys = false
```

這是為了避免使用者在輸入 section 名稱時，按鍵被全域快捷鍵接走。

例如：

* 按 Delete 應刪文字，不應刪頁面。
* 按 Ctrl + A 應全選 input 文字，不應全選投影片。
* 按 Enter 應儲存標題，不應新增 slide。

所以 section 管理不是只有資料規則，也涉及輸入焦點與快捷鍵保護。

## 27. 頁面層不變式整理

讀完這幾個檔案後，可以整理出幾條頁面層不變式。

### 不變式一：簡報至少保留一頁

刪除全部頁面時，會 `resetSlides()` 成一張空白頁。

不要讓 `slides` 變成空陣列。

### 不變式二：新增頁面插在目前頁後方

`slidesStore.addSlide()` 使用：

```text
addIndex = slideIndex + 1
```

新增後 `slideIndex` 也會切到新頁。

### 不變式三：插入 slide 時不保留外來 sectionTag

`addSlide()` 會刪掉待插入 slide 的 `sectionTag`。

避免模板或外部 slide 把章節結構污染進目前簡報。

### 不變式四：目前頁永遠是頁面操作集合的一部分

頁面操作集合來自：

```text
_selectedSlidesIndex + slideIndex
```

所以單頁操作和批次操作可以共用一套邏輯。

### 不變式五：刪除頁面後要修正目前頁

刪除中間頁後目前頁落在合理位置；刪除最後頁後目前頁退到新的最後頁。

### 不變式六：sectionTag 代表 section 起點

section 不是獨立樹，而是掛在 slide 上。

section 範圍由 slide 順序和下一個 sectionTag 決定。

### 不變式七：刪除或移動 section 起點時，section 要嘗試延續

刪除有 `sectionTag` 的頁面時，section 會嘗試交給下一頁。

移動有 `sectionTag` 的頁面時，原 section 也會嘗試交給原位置下一頁。

### 不變式八：拖曳排序後目前頁跟到新位置

`sortSlides()` 完成後會更新：

```text
slideIndex = newIndex
```

讓使用者仍然停留在被拖曳的頁面。

## 28. 這套設計的優點

### 優點一：UI 層不直接承擔資料規則

`Thumbnails/index.vue` 雖然是頁面管理面板，但它主要是事件入口。

新增、刪除、排序、section 操作都交給 hook / store。

這讓 UI 檔案比較像「場景協調者」，而不是所有規則的集中地。

### 優點二：頁面操作和快捷鍵操作可以共用同一套 hook

無論是：

* 點右鍵選單刪除頁面
* 按 Delete 刪除頁面
* 按 Ctrl + D 複製頁面
* 點新增按鈕新增頁面

最後都可以回到 `useSlideHandler()` 的同一批方法。

這比每個入口各寫一份邏輯更穩定。

### 優點三：section 規則被顯式維護

刪除和排序時都有處理 `sectionTag`，不是單純把頁面移來移去。

這讓 section 不容易因頁面變動而完全錯亂。

### 優點四：刪除全部頁面有保護

`resetSlides()` 避免簡報進入空狀態。

這讓很多後續邏輯不用到處判斷 `slides.length === 0`。

## 29. 這套設計的限制與注意點

### 限制一：sectionTag 掛在 slide 上，排序 / 刪除都要小心

這種設計簡單直觀，但也代表 section 和頁面順序高度耦合。

只要移動或刪除頁面，就可能影響 section 範圍。

所以所有會改變 slide 順序或數量的操作，都要考慮 `sectionTag`。

### 限制二：多選 index 和刪除 / 排序容易互相影響

多選集合是 index-based。

刪除或排序後，index 會改變。

所以刪除後清空多選是必要的。

如果未來要做更複雜的跨操作保留多選，可能要改成 id-based 或加入映射規則。

### 限制三：`sortSlides()` 本身沒有明確 history snapshot

從目前這個函式本身來看，它只做資料重排和目前頁更新。

如果需求是讓拖曳排序也能被 undo / redo，需要確認歷史機制是否在其他地方處理。

筆記中不要自行假設它一定已經記錄歷史。

### 限制四：section 移交規則是務實處理，不是完整章節模型

目前規則能處理常見情況：

* 刪掉 section 起點時交給下一頁。
* 移走 section 起點時交給原下一頁。
* 移到第一頁時處理第一頁 section。

但它仍然是基於 `sectionTag` 的局部規則，不是完整 section tree 或 range model。

如果未來 section 功能變得更複雜，例如巢狀 section、折疊、跨區拖曳規則，就可能需要更明確的 section 資料結構。

## 30. 容易誤解的地方

| 容易誤解                              | 精準理解                                        |
| --------------------------------- | ------------------------------------------- |
| 縮圖列表直接改 `slides` 排序               | 不是，縮圖列表只接拖曳事件，真正排序交給 `sortSlides()`         |
| 新增頁面是 append 到最後                  | 不是，`addSlide()` 插在目前頁後方                     |
| 新增後仍停在原頁                          | 不是，`slideIndex` 會切到新增頁                      |
| 插入模板頁會保留模板 sectionTag             | 不會，`addSlide()` 會移除待插入 slide 的 `sectionTag` |
| 刪除全部頁面後 slides 會變成空陣列             | 不會，會 reset 成一張空白頁                           |
| deleteSlide 只刪資料，不處理 section      | 不精準，store 刪除時會嘗試把被刪頁的 sectionTag 交給下一頁      |
| sortSlides 只是 splice 移動           | 不精準，它還處理移動 section 起點與移到第一頁的 section 規則     |
| section 是獨立資料表                    | 從目前程式看，section 是掛在 slide 上的 `sectionTag`    |
| removeSection 會刪掉頁面               | 不會，它只移除 sectionTag                          |
| removeSectionSlides 只刪 section 標題 | 不是，它會刪除該 section 範圍內的 slide                 |
| 更新 section title 一定只改既有 section   | 不一定，`default` 情況會在第一頁建立 sectionTag          |
| sortSlides 一定有 history snapshot   | 不要這樣寫；就函式本身看不到直接記錄 history 的動作              |

## 32. 這篇最重要的結論

### 結論一：頁面管理不是單純改陣列

新增、刪除、排序都會牽涉目前頁、元素選取、多選集合、sectionTag 與 history。

所以不能只用 `push / splice` 的角度理解。

### 結論二：`useSlideHandler()` 是頁面操作規則中心

複製、貼上、剪下、刪除、新增、排序、全選都集中在這裡。

縮圖列表、快捷鍵、右鍵選單都可以共用這些操作。

### 結論三：`slides store` 負責真正資料落地

例如 `addSlide()`、`deleteSlide()`、`setSlides()`、`updateSlideIndex()`。

其中 `addSlide()` 和 `deleteSlide()` 也包含重要的 sectionTag 處理。

### 結論四：`sectionTag` 是 section 起點，不是獨立 section tree

section 範圍由 slide 順序推導。

因此刪除、排序時才需要把 sectionTag 交給下一頁或新第一頁。

### 結論五：簡報至少保留一頁是關鍵不變式

刪除所有頁面時走 `resetSlides()`，避免 `slides` 變成空陣列。

這讓整個編輯器可以維持穩定的頁面基礎。

## 一句話抓核心

`PPTist` 的縮圖列表不只是拿來拖曳排序，它背後是一套頁面管理規則：UI 層接收操作，`useSlideHandler()` 處理新增、複製、剪下、刪除、排序與全選，`useSectionHandler()` 處理 section，`slides store` 負責資料落地；其中最關鍵的不變式是「簡報至少一頁存在」以及「`sectionTag` 代表 section 起點，刪除與排序時必須維持 section 延續」。

把這套規則看懂，你才會真正理解左側縮圖列表為什麼是完整的頁面管理系統，而不是單純的 preview + drag list。
