# 原始碼追蹤路線

這一篇是 `PPTist` 的 **store 與狀態流查找索引**。

前面幾篇是按主題整理：

- store 總覽
- slides store
- main store
- snapshot store
- keyboard / screen
- 常見狀態流案例
- store 與 hooks 關係
- store 與畫面渲染關係

這一篇則是反過來：

> 當你想追某個功能時，應該從哪個檔案開始看？接著往哪裡追？

---

## 使用方式

遇到問題時，先找到最接近的問題類型，照著「閱讀順序」追。

建議每次都用這個模板：

```txt
使用者操作入口
→ 對應 hook
→ 相關 store state
→ 相關 store action
→ 是否有 snapshot
→ 哪些畫面會重新渲染
```

## 1. 我想知道整個 store 架構

### 閱讀順序

```txt
src/store/index.ts
→ src/store/slides.ts
→ src/store/main.ts
→ src/store/snapshot.ts
→ src/store/keyboard.ts
→ src/store/screen.ts
```

### 你要看什麼

| 檔案 | 重點 |
|---|---|
| `index.ts` | 匯出哪些 store |
| `slides.ts` | 簡報內容怎麼保存與更新 |
| `main.ts` | 編輯器互動狀態怎麼保存 |
| `snapshot.ts` | undo / redo 怎麼保存快照 |
| `keyboard.ts` | 鍵盤狀態怎麼記錄 |
| `screen.ts` | 播放模式怎麼標記 |

### 不要一開始就看什麼

不要一開始就跳進大型 component。

先知道 store 的責任邊界，再看 component 會比較清楚。


### 重點問題

1. 新頁是在哪裡建立的？
2. 新頁插入到哪個位置？
3. 新增後 `slideIndex` 如何變化？
4. 有沒有清空元素選取？
5. 有沒有加入歷史快照？

### 關鍵結論

新增頁面是：

```txt
useSlideHandler.createSlide()
→ mainStore.setActiveElementIdList([])
→ slidesStore.addSlide(emptySlide)
→ addHistorySnapshot()
```

## 20. 我想知道為什麼某個快捷鍵沒有作用

### 檢查順序

```txt
1. useGlobalHotkey 是否有監聽這個 key？
2. disableHotkeys 是否為 true？
3. editorAreaFocus / thumbnailsFocus 是否符合條件？
4. activeElementIdList 是否符合條件？
5. ctrlKey / metaKey / shiftKey / altKey 是否符合條件？
6. 對應 hook 是否 early return？
```

### 常見原因

| 原因 | 說明 |
|---|---|
| `disableHotkeys = true` | 正在輸入文字或某些場景禁用快捷鍵 |
| 焦點不對 | 快捷鍵只在編輯區或縮圖區生效 |
| 沒有選取元素 | 某些操作需要 `activeElementIdList` |
| 元素被 lock / hidden | 選取流程會擋掉 |
| hook early return | 條件不滿足時直接返回 |

---

## 21. 我想知道為什麼畫面沒有更新

### 檢查順序

```txt
1. 目標 state 是否真的改變？
2. 改的是正確 store 嗎？
3. getter 是否依賴這個 state？
4. component 是否讀的是 getter 還是舊資料？
5. id 是否對得上 currentSlide.elements？
```

### 常見場景

| 問題 | 優先檢查 |
|---|---|
| 元素不見 | `slidesStore.currentSlide.elements` |
| 選取框不見 | `mainStore.activeElementIdList` |
| 右側面板不對 | `mainStore.handleElement` |
| 縮圖不對 | `slidesStore.slides`、`slideIndex` |
| undo 後怪怪的 | `snapshotStore.snapshotCursor`、`slidesStore.slides` |

## 不建議的閱讀方式

### 不建議 1：從大型 UI component 直接開始看

大型 component 通常會同時處理：

- template
- style
- props
- emits
- computed
- store
- hook
- DOM event

一開始直接讀會很混亂。

更好的方式是先抓狀態流。

---

### 不建議 2：只看 store，不看 hook

store 只能告訴你有哪些 state 和 action。

但它不能完整回答：

```txt
什麼時候呼叫？
呼叫前做了什麼判斷？
呼叫後有沒有 snapshot？
```

所以一定要搭配 hook。

---

### 不建議 3：每一行都平均用力讀

讀原始碼要先抓主幹。

建議先抓：

```txt
使用者操作
→ hook
→ store
→ snapshot
→ UI
```

之後再回來補細節。

---

## 最實用的追蹤策略

遇到一個功能，先問五個問題：

```txt
1. 這是內容資料變化，還是 UI 互動狀態變化？
2. 內容資料是否在 slides store？
3. 互動狀態是否在 main store？
4. 這個操作是否需要 snapshot？
5. 是哪個 hook 把流程串起來？
```

只要能回答這五題，通常就能定位要看的原始碼。

---

## 這篇的核心總結

讀 `PPTist` 原始碼時，不要用「檔案順序」讀，要用「功能路線」讀。

最佳心法是：

```txt
先找到操作入口，再追 hook，最後回到 store。
```

也就是：

```txt
我要追功能
→ 先找 hook
→ 再看 store state / action
→ 再看 getter / UI
→ 最後看 snapshot 是否接上
```
