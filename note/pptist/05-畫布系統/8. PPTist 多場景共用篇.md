# `PPTist` 多場景共用篇

所屬章節：[03-畫布系統](./README.md)

---

## 這篇要解決什麼

前面七篇已經建立了幾個核心觀念：

```text
固定邏輯畫布
-> canvasScale 顯示縮放
-> viewport-wrapper / viewport 分層
-> 互動座標換算
-> store 與 elementList 資料流
```

其中最重要的一句是：

> **元素資料永遠活在固定邏輯畫布裡，畫面只是不同場景下的顯示結果。**

這篇要把這個觀念再往外推一層：

```text
同一份 slide / element 資料
可以被不同場景用不同 scale、不同 wrapper、不同互動能力重新渲染。
```

這篇要回答：

1. PPTist 的「多場景共用」到底共用什麼？
2. 為什麼不是所有場景都共用 `Canvas/index.vue`？
3. Editor 主畫布、左側縮圖、放映畫面、放映底部縮圖各自怎麼渲染 slide？
4. `EditableElement`、`ThumbnailElement`、`ScreenElement` 差在哪裡？
5. 為什麼縮圖和放映可以共用同一份 `slide.elements`？
6. `viewportSize / viewportRatio / scale` 在多場景裡扮演什麼角色？
7. 如果你之後要自己設計類似 PPT / Excel / Word 編輯器，這個架構可以學到什麼？

---

## 先說清楚：這篇主要對照哪些 Source

這篇主要對照下面幾個位置：

```text
src/views/Editor/Canvas/index.vue
src/views/Editor/Canvas/EditableElement.vue

src/views/Editor/Thumbnails/index.vue

src/views/components/ThumbnailSlide/index.vue
src/views/components/ThumbnailSlide/ThumbnailElement.vue
src/views/components/element/*

src/views/Screen/hooks/useSlideSize.ts
src/views/Screen/ScreenSlideList.vue
src/views/Screen/ScreenSlide.vue
src/views/Screen/ScreenElement.vue
src/views/Screen/BottomThumbnails.vue

src/views/Mobile/index.vue
```

可以先把它們理解成這樣：

| Source                        | 主要角色                      |
| ----------------------------- | ------------------------- |
| `Editor/Canvas/index.vue`     | 主編輯畫布場景                   |
| `EditableElement.vue`         | 編輯場景的元素包裝器                |
| `Editor/Thumbnails/index.vue` | 左側縮圖列表                    |
| `ThumbnailSlide/index.vue`    | 單張縮圖渲染器                   |
| `ThumbnailElement.vue`        | 縮圖場景的元素包裝器                |
| `components/element/*`        | 多場景共用的元素基礎渲染元件            |
| `Screen/BaseView.vue`         | 放映場景主視圖                   |
| `useSlideSize.ts`             | 放映時根據容器計算 slide 顯示尺寸      |
| `ScreenSlideList.vue`         | 放映場景的 slide 列表與轉場容器       |
| `ScreenSlide.vue`             | 放映場景的單張 slide 渲染器         |
| `ScreenElement.vue`           | 放映場景的元素包裝器                |
| `BottomThumbnails.vue`        | 放映時底部縮圖列                  |
| `Mobile/index.vue`            | Mobile 編輯 / 預覽 / 播放模式切換入口 |

PPTist 的 `src/views` 目錄本身就能看出多場景設計：它分成 `Editor`、`Screen`、`Mobile` 與共用 `components` 等區塊；而共用元素元件放在 `src/views/components/element` 底下，包含文字、圖片、形狀、線條、圖表、表格、影片、音訊、LaTeX 等類型。([GitHub][1])

# 一、先抓一句話核心

PPTist 的多場景共用核心是：

```text
共用的不是同一個畫布元件，
而是同一份 slide / element 資料模型、
同一套邏輯尺寸基準、
以及一批可被不同場景包裝的基礎元素渲染元件。
```

也就是說：

```text
Editor 主畫布
Editor 左側縮圖
Screen 放映畫面
Screen 底部縮圖
Mobile 預覽 / 播放
```

它們不一定共用同一個 Vue component。

但它們共用的是：

```text
slide 資料
element 資料
viewportSize
viewportRatio
元素類型系統
部分基礎元素渲染元件
```

差別在於：

```text
每個場景都有自己的 wrapper
每個場景都有自己的 scale 計算方式
每個場景有不同的互動能力
每個場景有不同的元素包裝器
```

一句話記：

> **PPTist 是「資料模型共用，場景渲染分開」。**

---

# 二、多場景共用不是共用 `Canvas/index.vue`

## 先講原理

很多人第一次想到「多場景共用」時，會以為應該要做一個萬用元件：

```text
SlideRenderer.vue
```

然後所有地方都用它：

```text
Editor 主畫布用它
縮圖用它
放映用它
Mobile 用它
匯出用它
```

但實際上，這種做法不一定好。

因為不同場景的需求差很多。

---

### Editor 主畫布要的是「可編輯」

主畫布需要：

```text
元素選取
拖曳
縮放
旋轉
對齊線
操作框
多選框
框選
滑鼠座標換算
右鍵選單
建立元素
歷史紀錄
```

所以它一定很重。

---

### 縮圖要的是「輕量預覽」

縮圖只需要：

```text
看起來像投影片
可以被點選
可以用來切換頁面
通常不需要完整互動
```

如果縮圖也載入主畫布那套操作框、拖曳、旋轉、對齊線，就太重了。

---

### 放映要的是「播放語意」

放映場景需要：

```text
轉場
動畫
連結跳轉
全螢幕
簡報工具
倒數計時
底部縮圖
雷射筆 / 畫筆類工具
```

它不是拿來編輯元素，而是拿來播放。

---

所以 PPTist 的設計不是：

```text
所有場景共用同一個 Canvas/index.vue
```

而是：

```text
共用 slide / element 資料，
但每個場景用自己的渲染入口。
```

這樣才符合不同場景的責任差異。

---

## 再對回 Source

在 Editor 主畫布中，`Canvas/index.vue` 負責的是完整編輯器畫布。它有 `.canvas` 根節點，接住 `wheel`、`mousedown`、`dblclick`、右鍵選單、click outside 等事件；內部還包含 `ElementCreateSelection`、`ShapeCreateCanvas`、`viewport-wrapper`、`operates`、`viewport`、`MouseSelection`、`EditableElement`、`Ruler`、`Modal` 等結構。([GitHub][2])

但左側縮圖不是使用 `Canvas/index.vue`，而是使用 `ThumbnailSlide`；放映畫面也不是使用 `Canvas/index.vue`，而是走 `ScreenSlideList -> ScreenSlide -> ScreenElement`。這說明 PPTist 的多場景共用，不是把同一個 Canvas 元件硬塞到所有場景，而是讓不同場景各自建立自己的渲染入口。([GitHub][3])

---

# 三、共同契約一：所有場景都吃同一份 `slide` / `element` 資料

## 先講原理

多場景能共用的前提是：

```text
slide 資料本身不能綁死在某一個 DOM 尺寸裡。
```

如果元素資料是這樣設計：

```ts
{
  left: 100,
  top: 50,
  width: 200,
  height: 80
}
```

而且這組資料永遠表示：

```text
在固定邏輯畫布中的位置與尺寸
```

那麼它就可以被不同場景重用。

例如同一個元素資料：

```ts
{
  left: 100,
  top: 50,
  width: 200,
  height: 80
}
```

在不同場景下可以這樣呈現：

```text
Editor 主畫布：
  依 canvasScale 顯示，並可拖曳、縮放、旋轉

左側縮圖：
  依 thumbnail scale 顯示，只用來預覽

放映畫面：
  依 screen scale 顯示，並套用動畫與連結

Mobile 預覽：
  依 mobile 場景尺寸顯示
```

資料沒變。

變的是：

```text
場景 wrapper
scale
互動能力
元素包裝器
```

---

## 再對回 Source

Editor 主畫布會從 `slidesStore.currentSlide` 取得目前頁面，再把 `currentSlide.elements` 深拷貝成本地 `elementList`，最後用 `EditableElement` 渲染。這表示 Editor 使用的資料來源仍然是 slide 裡的 elements，只是為了高頻互動建立本地工作副本。([GitHub][2])

縮圖場景的 `ThumbnailSlide` 直接接收 `slide` prop，並對 `slide.elements` 做 `v-for`，把每個 element 交給 `ThumbnailElement`。([GitHub][3])

放映場景的 `ScreenSlide` 也接收 `slide` prop，並對 `slide.elements` 做 `v-for`，把每個 element 交給 `ScreenElement`。([GitHub][4])

所以三個場景的資料主線其實都是：

```text
slide.elements
  -> 場景自己的 element wrapper
  -> 場景自己的元素渲染
```

# 四、共同契約二：所有場景都從「邏輯尺寸」推導自己的 `scale`

## 先講原理

前面幾篇已經講過，PPTist 的邏輯畫布基準是：

```text
viewportSize = 1000
viewportRatio = 0.5625
```

也就是：

```text
邏輯寬度 = viewportSize
邏輯高度 = viewportSize * viewportRatio
```

這件事對多場景共用非常重要。

因為只要所有場景都認同：

```text
slide / element 資料活在同一個邏輯畫布裡
```

那每個場景就只要根據自己的容器算出一個 scale，就能把同一份資料畫出來。

可以抽象成：

```text
顯示寬度 / viewportSize = scale
```

或：

```text
邏輯資料 × scale = 畫面結果
```

不同場景只是「顯示寬度」來源不同。

---

## 場景 scale 對照

| 場景         | 顯示寬度來源                   | scale 來源                    |
| ---------- | ------------------------ | --------------------------- |
| Editor 主畫布 | 編輯區容器 + canvasPercentage | `canvasScale`               |
| 左側縮圖       | 縮圖指定寬度 `size`            | `size / viewportSize`       |
| 放映畫面       | 放映容器計算出的 `slideWidth`    | `slideWidth / viewportSize` |
| 放映底部縮圖     | 底部縮圖指定寬度                 | `size / viewportSize`       |
| Mobile     | Mobile 場景容器              | 場景自行計算                      |

---

## 再對回 Source：Editor 主畫布

Editor 主畫布裡，`viewport-wrapper` 的寬高是：

```ts
width = viewportStyles.width * canvasScale
height = viewportStyles.height * canvasScale
```

而真正承載內容的 `viewport` 則使用：

```ts
transform: scale(canvasScale)
```

也就是 Editor 場景使用的是 `mainStore.canvasScale`，這個值由前面幾篇講過的畫布縮放鏈路計算出來。([GitHub][2])

---

## 再對回 Source：縮圖場景

`ThumbnailSlide` 會從 `slidesStore` 取出：

```ts
viewportRatio
viewportSize
```

然後根據傳進來的 `size` 算出：

```ts
scale = props.size / viewportSize.value
```

外層縮圖寬高是：

```text
width = size
height = size * viewportRatio
```

內層 `.elements` 還是使用邏輯寬高：

```text
width = viewportSize
height = viewportSize * viewportRatio
```

然後用：

```ts
transform: scale(scale)
```

把邏輯內容縮成縮圖大小。`ThumbnailSlide` 也會 `provide(injectKeySlideScale, scale)`，讓子層元素知道目前縮放比例。([GitHub][3])

這其實就是縮圖版的畫布縮放系統。

---

## 再對回 Source：放映場景

放映場景會先用 `useSlideSize.ts` 根據放映容器尺寸與 `viewportRatio` 算出 `slideWidth` / `slideHeight`。這個 hook 會監聽視窗 resize，並根據容器比例決定是寬受限還是高受限。([GitHub][5])

接著 `ScreenSlideList.vue` 會根據：

```ts
scale = props.slideWidth / viewportSize.value
```

算出放映時的 scale，並把 `scale` 傳給 `ScreenSlide`。它也會 `provide(injectKeySlideScale, scale)`，讓子層元素可以取得播放場景的縮放比例。([GitHub][6])

`ScreenSlide.vue` 本身則仍然用邏輯寬高建立 slide：

```text
width = viewportSize
height = viewportSize * viewportRatio
```

再用：

```ts
transform: scale(scale)
```

把它縮放到放映畫面中。([GitHub][4])

所以 Editor、Thumbnail、Screen 三個場景雖然元件不同，但底層邏輯一致：

```text
固定邏輯尺寸
-> 場景自己算 scale
-> transform: scale(...)
-> 顯示到該場景容器
```

# 五、場景一：Editor 主畫布是「重互動場景」

## 先講原理

Editor 主畫布是最複雜的場景。

它不是只把 slide 畫出來而已，而是要支援完整編輯能力。

所以它需要：

```text
畫布縮放
畫布拖曳
元素選取
元素拖曳
元素縮放
元素旋轉
框選
新增元素
對齊線
操作框
多選框
右鍵選單
鍵盤狀態
歷史紀錄
```

因此 Editor 主畫布不適合拿去給縮圖或放映直接共用。

它太重，而且責任太多。

---

## 再對回 Source

`Canvas/index.vue` 會從 `mainStore` 取出大量編輯器狀態，例如：

```text
activeElementIdList
activeGroupElementId
handleElementId
hiddenElementIdList
gridLineSize
showRuler
creatingElement
creatingCustomShape
canvasScale
textFormatPainter
```

同時也從 `slidesStore` 取出 `currentSlide`，從 `keyboardStore` 取出 `ctrlKeyState`、`spaceKeyState`。([GitHub][2])

它還會建立本地 `elementList`，並把 `elementList`、`canvasScale`、`viewportRef`、`alignmentLines` 等資料傳給：

```text
useMouseSelection
useDragElement
useScaleElement
useRotateElement
useMoveShapeKeypoint
useDragLineElement
```

這代表 Editor 主畫布是「資料 + 操作 + 互動 hooks」的組裝中心。([GitHub][2])

所以可以這樣記：

```text
Editor Canvas：
  不是單純 slide renderer。
  它是完整的編輯場景。
```

---

# 六、場景二：左側縮圖是「輕量預覽 + 頁面導航場景」

## 先講原理

左側縮圖的任務和主畫布完全不同。

它主要負責：

```text
顯示每一頁的縮小預覽
點擊切換頁面
顯示目前頁面 active 狀態
支援多選 / 排序等頁面層級操作
```

它不需要：

```text
元素拖曳
元素縮放
元素旋轉
操作框
對齊線
滑鼠座標換算
```

所以它應該用更輕量的渲染方式。

---

## 再對回 Source

`Editor/Thumbnails/index.vue` 會對 `slides` 做列表渲染，每個縮圖 item 裡放一個：

```vue
<ThumbnailSlide :slide="element" :size="120" />
```

點擊縮圖時會切換 `slideIndex`，而切換前如果有選取元素，會先清空 `activeElementIdList`。([GitHub][7])

`ThumbnailSlide` 則負責把單張 slide 渲染成縮圖。它接收 `slide` 和 `size`，根據 `viewportSize` 算出縮圖 scale，再渲染背景與 `ThumbnailElement` 列表。([GitHub][3])

所以左側縮圖的資料流是：

```text
slidesStore.slides
  -> Editor/Thumbnails
  -> ThumbnailSlide
  -> ThumbnailElement
  -> Base*Element
```

它和主畫布共用的是：

```text
slide 資料
element 資料
viewportSize / viewportRatio
元素基礎渲染
```

但不共用：

```text
Canvas/index.vue
EditableElement
operates
拖曳 / 縮放 / 旋轉 hooks
```

---

# 七、場景三：放映畫面是「播放語意場景」

## 先講原理

放映場景的任務不是編輯，而是播放。

所以它關心的是：

```text
目前播放到哪一頁
目前動畫到哪一步
頁面轉場
元素動畫
連結跳轉
全螢幕控制
底部縮圖
簡報工具
```

它不需要：

```text
元素操作框
拖曳元素
縮放元素
旋轉元素
編輯器右鍵選單
```

因此放映場景也不會直接拿 Editor 的 `Canvas/index.vue` 來用。

它需要自己的播放型 renderer。

---

## 再對回 Source

`Screen/BaseView.vue` 是放映場景的主視圖，它會組裝 `ScreenSlideList`、`SlideThumbnails`、`WritingBoardTool`、`CountdownTimer`、上一頁 / 下一頁控制、工具列與 `BottomThumbnails` 等放映工具。它從 `slidesStore` 取得 `slides` 和 `slideIndex`，並透過 `useSlideSize` 等 hook 處理放映場景尺寸與播放操作。([GitHub][8])

`ScreenSlideList.vue` 會根據 `slidesWithTurningMode` 渲染多張 slide，並根據 `slideIndex` 加上 current / before / after / last / next 等 class，用來支援不同轉場模式。它會把 `slideWidth`、`slideHeight`、`scale` 等資料傳給 `ScreenSlide`。([GitHub][6])

`ScreenSlide.vue` 則負責渲染單張放映 slide：它用 `viewportSize` 和 `viewportRatio` 建立邏輯大小，再用 `transform: scale(scale)` 呈現，並把 `slide.elements` 交給 `ScreenElement`。([GitHub][4])

所以放映資料流可以畫成：

```text
slidesStore.slides
  -> Screen/BaseView
  -> ScreenSlideList
  -> ScreenSlide
  -> ScreenElement
  -> Base*Element / ScreenVideoElement / ScreenAudioElement
```

---

# 八、場景四：放映底部縮圖再次共用 `ThumbnailSlide`

## 先講原理

這裡很值得注意。

左側縮圖和放映底部縮圖，其實都屬於：

```text
縮圖預覽場景
```

差別只是它們出現在不同位置：

```text
Editor 左側：
  用來編輯時切換頁面

Screen 底部：
  用來放映時快速切換頁面
```

所以它們可以共用 `ThumbnailSlide`。

這說明 PPTist 的共用不是粗暴地共用所有東西，而是：

```text
相同責任的場景才抽共用。
不同責任的場景各自實作。
```

---

## 再對回 Source

`Screen/BottomThumbnails.vue` 會對 `slides` 做列表渲染，並且每一項也使用：

```vue
<ThumbnailSlide :slide="slide" />
```

點擊底部縮圖時，會透過播放 hook 切換到指定頁。([GitHub][9])

所以這裡的共用關係是：

```text
Editor 左側縮圖
  -> ThumbnailSlide

Screen 底部縮圖
  -> ThumbnailSlide
```

這代表 `ThumbnailSlide` 的責任很清楚：

```text
給一張 slide 和一個 size，
把它畫成縮圖。
```

它不管自己是在 Editor 裡，還是在 Screen 裡。

---

# 九、場景五：Mobile 是另一組場景入口

## 先講原理

Mobile 不只是把桌面版縮小。

它通常會有自己的模式：

```text
MobileEditor
MobilePreview
MobilePlayer
```

也就是：

```text
手機編輯
手機預覽
手機播放
```

這些模式也會消費同一份簡報資料，但 UI 與互動方式會和桌面不同。

---

## 再對回 Source

`src/views/Mobile` 目錄底下包含 `MobileEditor`、`MobilePlayer.vue`、`MobilePreview.vue`、`MobileThumbnails.vue` 等檔案。`Mobile/index.vue` 會用 `mode` 控制目前要顯示 `editor`、`player` 還是 `preview`，並用動態 component 切換。([GitHub][10])

這代表 Mobile 也是一種場景拆分：

```text
同一份簡報資料
-> MobileEditor
-> MobilePreview
-> MobilePlayer
```

但 Mobile 的細節可以另開一篇，不建議在這篇塞太多。

---

# 十、元素渲染共用：`EditableElement`、`ThumbnailElement`、`ScreenElement` 的差異

## 先講原理

多場景共用裡最重要的一層是元素渲染。

同一個 element 資料，在不同場景會被不同 wrapper 包起來。

可以先分成三種：

```text
Editor：
  EditableElement

Thumbnail：
  ThumbnailElement

Screen：
  ScreenElement
```

它們不是同一個元件。

因為三個場景對元素的需求不同。

---

## 1. `EditableElement`：編輯場景元素包裝器

`EditableElement` 是主畫布用的。

它需要處理：

```text
選取
右鍵選單
編輯狀態
元素容器樣式
元素事件
可編輯元素元件切換
```

在 source 裡，`EditableElement.vue` 會根據 `elementInfo.type` 對應到不同編輯版元素元件，例如 `ImageElement`、`TextElement`、`ShapeElement`、`LineElement`、`ChartElement`、`TableElement`、`LatexElement`、`VideoElement`、`AudioElement`。([GitHub][11])

所以它可以這樣記：

```text
EditableElement：
  編輯場景的元素外殼。
  重點是可操作、可選取、可編輯。
```

---

## 2. `ThumbnailElement`：縮圖場景元素包裝器

`ThumbnailElement` 是縮圖用的。

它不需要完整編輯能力。

它主要做：

```text
接收 elementInfo
根據 type 選擇 Base*Element
傳入 target="thumbnail"
用 base-element 包住
```

在 source 裡，`ThumbnailElement.vue` 會根據 element type 對應到 `BaseImageElement`、`BaseTextElement`、`BaseShapeElement`、`BaseLineElement`、`BaseChartElement`、`BaseTableElement`、`BaseLatexElement`、`BaseVideoElement`、`BaseAudioElement`。([GitHub][12])

所以它可以這樣記：

```text
ThumbnailElement：
  縮圖場景的元素外殼。
  重點是輕量預覽，不做完整編輯。
```

---

## 3. `ScreenElement`：放映場景元素包裝器

`ScreenElement` 是放映用的。

它除了渲染元素外，還要處理：

```text
動畫等待狀態
元素動畫顯示 / 隱藏
連結跳轉
web link
slide link
放映時影音元件
```

在 source 裡，`ScreenElement.vue` 會根據 element type 對應到基礎元素元件，影音則使用 `ScreenVideoElement`、`ScreenAudioElement`；它也會使用 `formatedAnimations` 判斷元素是否需要等待動畫才顯示，並處理 web link 或 slide link 點擊。([GitHub][13])

所以它可以這樣記：

```text
ScreenElement：
  放映場景的元素外殼。
  重點是播放語意、動畫、連結與放映互動。
```

---

# 十一、真正共用的是 Base 元素，而不是場景外殼

## 先講原理

這是多場景共用裡最精準的理解。

PPTist 不是讓所有場景都用同一個 `Element.vue`。

而是拆成：

```text
場景外殼：
  EditableElement
  ThumbnailElement
  ScreenElement

基礎渲染：
  BaseTextElement
  BaseImageElement
  BaseShapeElement
  BaseLineElement
  BaseChartElement
  BaseTableElement
  BaseLatexElement
  ...
```

也就是：

```text
外殼處理場景差異。
Base 元件處理元素怎麼畫。
```

這樣設計很合理。

因為：

```text
文字元素怎麼畫：
  多場景大致相同

文字元素能不能被編輯：
  只有 Editor 需要

文字元素要不要套動畫：
  Screen 需要

文字元素是不是縮圖預覽：
  Thumbnail 需要
```

所以最好的拆法不是把所有東西塞在同一個元件，而是：

```text
共用渲染核心；
場景外殼分開。
```

---

## 對照圖

```text
同一個 elementInfo
        │
        ├── Editor
        │     └── EditableElement
        │           └── TextElement / ImageElement / ShapeElement ...
        │
        ├── Thumbnail
        │     └── ThumbnailElement
        │           └── BaseTextElement / BaseImageElement / BaseShapeElement ...
        │
        └── Screen
              └── ScreenElement
                    └── BaseTextElement / BaseImageElement / ScreenVideoElement ...
```

這裡最重要的是：

> **場景 wrapper 不共用，但元素基礎渲染能力可以共用。**

---

# 十二、背景也可以被不同場景重用

## 先講原理

除了元素，slide background 也是多場景共用的重要部分。

一張投影片的背景可能是：

```text
純色
漸層
圖片
主題背景
```

不管它出現在主畫布、縮圖、放映，都應該看起來一致。

所以背景資料也應該存在 slide 裡，而不是綁死在某個場景 DOM。

---

## 再對回 Source

`ThumbnailSlide` 會使用 `useSlideBackgroundStyle(slide.value.background)` 計算縮圖背景樣式。([GitHub][3])

`ScreenSlide.vue` 也會使用 `useSlideBackgroundStyle(props.slide.background)` 計算放映場景背景樣式。([GitHub][4])

這代表 background 的資料來源是 slide，而不是某個場景獨有。

所以可以這樣記：

```text
slide.background：
  是資料

useSlideBackgroundStyle：
  是把背景資料轉成 CSS 顯示樣式的工具

ThumbnailSlide / ScreenSlide：
  都可以消費同一份 background 資料
```

---

# 十三、為什麼不做一個萬用 `SlideRenderer`

## 先講原理

看到這裡，你可能會想：

> 既然 Editor、Thumbnail、Screen 都是在畫 slide，為什麼不抽一個通用的 `SlideRenderer`？

這個問題很好。

答案是：

```text
可以抽，但不能抽錯層。
```

如果你把所有場景都硬抽成一個大 renderer，會出現幾個問題。

---

## 問題一：Editor 的互動太重

Editor 需要：

```text
selection
drag
resize
rotate
alignment
contextmenu
history
keyboard
```

縮圖和放映不需要。

如果都塞進通用 renderer，縮圖也會被迫背一堆不需要的邏輯。

---

## 問題二：Screen 有播放語意

Screen 需要：

```text
transition
animation
link
fullscreen tools
bottom thumbnails
writing board
timer
```

這些也不是 Editor 和 Thumbnail 的核心責任。

---

## 問題三：Thumbnail 追求輕量

Thumbnail 通常會大量出現。

一份簡報可能有很多頁，左側縮圖一次要渲染很多張。

所以它需要：

```text
簡單
輕量
可延遲渲染
只看不編輯
```

`Editor/Thumbnails/index.vue` 裡也有 `slidesLoadLimit` 這類機制，用來控制縮圖載入數量；這表示縮圖場景確實有自己的性能考量。([GitHub][7])

---

## 所以正確抽象是什麼？

不是抽成：

```text
一個超大的 UniversalSlideRenderer
```

而是抽成：

```text
共用資料模型
共用 viewportSize / viewportRatio
共用 Base 元素渲染
共用 background style 工具
共用 scale 注入機制
```

然後場景層分開：

```text
EditorCanvas
ThumbnailSlide
ScreenSlide
MobilePreview
```

這樣才是比較乾淨的設計。

---

# 十四、多場景共用完整資料流圖

可以把 PPTist 的多場景資料流整理成這張圖：

        │

        │     ThumbnailSlide
        │       ↓
        │     ThumbnailElement
        │       ↓

        │

# 十五、用同一個元素資料看不同場景

假設有一個文字元素：

```ts
{
  id: 'text-1',
  type: 'text',
  left: 100,
  top: 60,
  width: 300,
  height: 80,
  content: 'Hello PPTist'
}
```

在不同場景會變成：

---

## 1. Editor 主畫布

```text
elementInfo
  -> EditableElement
  -> TextElement
```

它可以被：

```text
選取
拖曳
縮放
旋轉
右鍵操作
編輯文字
```

---

## 2. 左側縮圖

```text
elementInfo
  -> ThumbnailElement
  -> BaseTextElement
```

它只需要：

```text
縮小顯示
保持位置與比例
作為頁面預覽的一部分
```

---

## 3. Screen 放映

```text
elementInfo
  -> ScreenElement
  -> BaseTextElement
```

它可能需要：

```text
等待動畫
依動畫狀態顯示 / 隱藏
點擊 link
跟著 slide scale 播放
```

---

所以同一份 element 資料，在不同場景下不是「被改成不同資料」，而是：

```text
同一份資料
-> 不同場景 wrapper
-> 不同互動語意
-> 不同顯示 scale
```

這就是多場景共用的核心。

# 十六、這篇最容易誤會的 6 個點

## 1. 不要以為多場景共用就是共用同一個 Vue 元件

PPTist 共用的是：

```text
資料模型
邏輯尺寸
元素類型
基礎渲染能力
```

不是所有場景都共用 `Canvas/index.vue`。

---

## 2. 不要把 `EditableElement` 拿去當縮圖元素

`EditableElement` 是編輯場景用的，帶有編輯語意。

縮圖場景應該用更輕量的 `ThumbnailElement`。

---

## 3. 不要把 `ScreenElement` 當成普通元素渲染器

`ScreenElement` 不是單純畫元素，它還要處理動畫等待、link、放映影音等播放語意。([GitHub][13])

---

## 4. 不要忽略 `scale` 是每個場景自己算的

Editor 有 `canvasScale`。

Thumbnail 有：

```ts
scale = size / viewportSize
```

Screen 有：

```ts
scale = slideWidth / viewportSize
```

它們都是把固定邏輯畫布映射到各自場景。

---

## 5. 不要以為縮圖只是截圖

PPTist 的縮圖不是簡單把主畫布截圖後貼上去。

它有自己的 `ThumbnailSlide` 和 `ThumbnailElement` 渲染流程，會直接吃 slide / element 資料重新畫出縮圖。([GitHub][3])

---

## 6. 不要把場景狀態寫回 slide 資料

例如：

```text
目前是否被選取
目前是否正在拖曳
目前播放到哪個動畫步驟
目前縮圖是否 visible
目前 Mobile 是 preview 還是 player
```

這些都不是 slide / element 本體資料。

它們應該留在各自場景狀態中，而不是污染簡報資料。

---

# 十八、這篇最後要帶走的結論

PPTist 的多場景共用，不是把所有畫面都塞進同一個大元件。

它真正的設計是：

```text
1. 用 slidesStore 保存共同簡報資料。

2. 用 viewportSize / viewportRatio 建立共同邏輯畫布基準。

3. 每個場景自己根據容器算 scale。

4. Editor 用 Canvas/index.vue + EditableElement 支援完整編輯。

5. Thumbnail 用 ThumbnailSlide + ThumbnailElement 做輕量預覽。

6. Screen 用 ScreenSlideList + ScreenSlide + ScreenElement 做播放語意。

7. BottomThumbnails 再次共用 ThumbnailSlide。

8. Mobile 用自己的 editor / preview / player 場景入口。

9. 基礎元素渲染能力放在 components/element 底下供多場景使用。
```

一句話總結：

> **PPTist 的多場景共用核心是：資料模型共用，渲染場景分離；邏輯尺寸共用，scale 各自計算；元素內容共用，場景外殼各自負責互動語意。**

如果你能把這條主線抓住：

```text
slide.elements
  -> Editor: EditableElement
  -> Thumbnail: ThumbnailElement
  -> Screen: ScreenElement
```

你就會看懂 PPTist 為什麼可以用同一份簡報資料，同時支撐：

```text
編輯
縮圖
放映
放映縮圖
Mobile 預覽 / 播放
匯出 / 列印
```

而不會讓所有場景互相污染。
