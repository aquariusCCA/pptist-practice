# `PPTist` 畫布分層結構篇

所屬章節：[03-畫布系統](./README.md)

---

## 這篇要解決什麼

前面幾篇已經把畫布系統的幾條主線講清楚了：

```text
固定邏輯畫布
-> 根據容器算出 canvasScale
-> viewport-wrapper 建立縮放後外框
-> viewport 用 transform: scale(canvasScale) 顯示內容
-> 互動時再把畫面座標換回邏輯座標
```

但當你真的打開 `Canvas/index.vue` 時，還會看到一堆元件與 DOM 層：

```text
canvas
ElementCreateSelection
ShapeCreateCanvas
viewport-wrapper
operates
AlignmentLine
MultiSelectOperate
Operate
ViewportBackground
viewport
MouseSelection
EditableElement
drag-mask
Ruler
Modal
```

這篇要處理的就是：

> **PPTist 的畫布不是只有一層 `viewport`，而是被拆成「工作區、投影片外框、操作工具層、內容層、外部輔助層」幾個責任層。**

這篇要回答：

1. `doc/Canvas.md` 的概念分層，怎麼對回目前 source？
2. `Canvas/index.vue` 裡真正的 DOM / 組件分層長什麼樣？
3. 為什麼 `operates` 和 `viewport` 是兄弟層？
4. 為什麼 `Ruler` 不放在 `viewport-wrapper` 裡？
5. 為什麼 `ElementCreateSelection`、`ShapeCreateCanvas` 不屬於正式投影片內容？
6. 讀 `Canvas/index.vue` 時，應該用什麼順序看才不會混？

---

# 一、先抓一句話核心

PPTist 的畫布分層可以先用一句話記住：

```text
viewport-wrapper 負責投影片外框；
viewport 負責真正內容；
operates 負責操作工具；
Ruler、drag-mask、Modal 則是畫布工作區外圍的輔助層。
```

這裡最重要的是：

> **不要把所有出現在 Canvas/index.vue 裡的東西，都粗略叫成「畫布內容」。**

因為它們其實分屬不同責任：

```text
有些東西是投影片資料本身。
有些東西是操作資料用的工具。
有些東西只是幫助使用者互動的外部 UI。
有些東西是在建立元素過程中暫時出現的選區。
```

如果這些不分清楚，後面看 `EditableElement`、`Operate`、`AlignmentLine`、`MouseSelection`、`ViewportBackground` 時，就很容易混成一團。

---

# 二、先看作者文件的概念分層

## 先講原理

在 `doc/Canvas.md` 裡，作者把編輯器裡的「畫布」拆成兩大塊：

```text
畫布
├── 可視區域
│    ├── 可編輯元素
│    └── 鼠標選框
│
└── 畫布工具
     ├── 參考線
     ├── 標尺
     ├── 元素操作節點層
     ├── 吸附對齊線
     └── 可視區域背景
```

這是概念模型，不是目前 Vue template 的逐字結構。`doc/Canvas.md` 同時也定義元素資料的基本形狀，例如 `left / top / width / height`，並說明元素是相對於畫布左上角描述位置與尺寸；可視區域預設以 `1000 × 562.5` 為基準比例，實際顯示時再透過縮放比呈現。

所以從概念上看，PPTist 至少有兩種東西：

```text
1. 可視區域裡的內容
2. 圍繞可視區域工作的畫布工具
```

---

## 再對回 Source

目前 `Canvas/index.vue` 不是直接把這兩塊命名成：

```text
VisibleArea
CanvasTools
```

而是落成下面這種結構：

```text
canvas
├── ElementCreateSelection
├── ShapeCreateCanvas
├── viewport-wrapper
│    ├── operates
│    │    ├── AlignmentLine
│    │    ├── MultiSelectOperate
│    │    ├── Operate
│    │    └── ViewportBackground
│    │
│    └── viewport
│         ├── MouseSelection
│         └── EditableElement
│
├── drag-mask
├── Ruler
└── Modal
```

這個結構可以直接從 `Canvas/index.vue` 的 template 對出來：`viewport-wrapper` 用 `viewportStyles.width * canvasScale`、`viewportStyles.height * canvasScale`、`viewportStyles.left`、`viewportStyles.top` 建立投影片外框；它裡面先放 `operates`，再放 `viewport`；`Ruler` 則放在 `viewport-wrapper` 外面，但接收 `viewportStyles` 與 `elementList`。

---

# 三、第一層：`canvas` 是整個畫布工作區

## 先講原理

`canvas` 這一層不是投影片本身。

它比較像：

```text
整個編輯器中間那塊灰色工作區。
```

這一層負責接住全域畫布事件，例如：

```text
滾輪縮放 / 翻頁
滑鼠按下空白區
雙擊建立文字
右鍵選單
點擊外部失焦
```

所以它不是「投影片內容層」，而是：

> **畫布互動入口層。**

---

## 再對回 Source

在 `Canvas/index.vue` 裡，最外層是：

```vue
<div
  class="canvas"
  ref="canvasRef"
  @wheel="$event => handleMousewheelCanvas($event)"
  @mousedown="$event => handleClickBlankArea($event)"
  @dblclick="$event => handleDblClick($event)"
  v-contextmenu="contextmenus"
  v-click-outside="removeEditorAreaFocus"
>
```

這代表 `canvas` 是事件入口，不是單純的背景 DOM。它接住滾輪、滑鼠按下、雙擊、右鍵選單與點擊外部等事件。

CSS 上，`.canvas` 設定了：

```scss
height: 100%;
user-select: none;
overflow: hidden;
background-color: $lightGray;
position: relative;
```

這也說明它是整個畫布工作區：有灰色背景、裁切溢出內容，並且用 `position: relative` 讓內部的絕對定位層可以以它作為定位參考。

所以第一層可以這樣記：

```text
canvas：
  整個畫布工作區。
  負責事件入口、灰色背景、裁切、以及內部絕對定位參考。
```

---

# 四、第二層：`viewport-wrapper` 是縮放後的投影片外框

## 先講原理

`viewport-wrapper` 是理解 PPTist 分層的關鍵。

它不是內容本身，而是：

```text
縮放後的投影片外框。
```

它回答的是：

```text
這張投影片現在在畫面上看起來多大？
它的左上角現在放在哪裡？
```

所以 `viewport-wrapper` 站在「顯示世界」。

它處理的是：

```text
畫面上的投影片矩形
```

而不是：

```text
投影片內部元素資料
```

---

## 再對回 Source

在 `Canvas/index.vue` 裡：

```vue
<div
  class="viewport-wrapper"
  :style="{
    width: viewportStyles.width * canvasScale + 'px',
    height: viewportStyles.height * canvasScale + 'px',
    left: viewportStyles.left + 'px',
    top: viewportStyles.top + 'px',
  }"
>
```

這一段非常重要，因為它直接說明：

```text
wrapper 的寬高 = 邏輯寬高 × canvasScale
wrapper 的 left / top = 目前投影片外框在工作區中的位置
```

也就是：

```text
viewport-wrapper 管的是「縮放後外框」。
```

這和前面〈畫布定位篇〉的主線一致：`viewportStyles.width / height` 本身偏向邏輯寬高，真正的顯示寬高是在 wrapper 這層乘上 `canvasScale` 之後才成立。`useViewportSize.ts` 裡的 `viewportStyles` 也可以對上這點：它回傳 `width: viewportSize`、`height: viewportSize * viewportRatio`，而 `left / top` 來自 `viewportLeft / viewportTop`。([GitHub][3])

所以這層可以這樣記：

```text
viewport-wrapper：
  投影片在畫面上的外框。
  負責寬高、位置、陰影，以及讓操作層和內容層共享同一個投影片外框。
```

---

# 五、第三層：`operates` 是操作工具層

## 先講原理

`operates` 不是投影片內容。

它放的是：

```text
為了編輯投影片內容而存在的工具。
```

例如：

```text
吸附對齊線
單元素操作框
多選操作框
可視區域背景
```

這些東西看起來都貼著投影片，但它們不是資料內容本身。

你可以把它想成：

```text
投影片上方的一層操作輔助膜。
```

它和 `viewport` 放在同一個 `viewport-wrapper` 裡，是因為它們都要對齊同一張投影片外框。

---

## 再對回 Source

在 `Canvas/index.vue` 裡，`operates` 裡放了：

```vue
<div class="operates">
  <AlignmentLine ... />
  <MultiSelectOperate ... />
  <Operate ... />
  <ViewportBackground />
</div>
```

也就是說，目前 source 把 `AlignmentLine`、`MultiSelectOperate`、`Operate`、`ViewportBackground` 都歸在 `operates` 這一層，而不是放進真正承載元素的 `viewport` 裡。

這裡有一個重要細節：

> **`operates` 沒有整層套 `transform: scale(canvasScale)`，但裡面的工具需要時會自己消費 `canvasScale`。**

例如 `AlignmentLine.vue` 裡，對齊線會接收 `canvasScale`，然後把邏輯座標與長度乘上 `canvasScale`，轉成畫面上的位置與尺寸：

```ts
left = props.axis.x * props.canvasScale + 'px'
top = props.axis.y * props.canvasScale + 'px'
height = props.length * props.canvasScale + 'px'
width = props.length * props.canvasScale + 'px'
```

這說明 `AlignmentLine` 雖然是操作工具，但它仍然需要知道畫布縮放倍率，才能把邏輯世界的對齊線畫到正確的畫面位置。([GitHub][4])

所以 `operates` 可以這樣記：

```text
operates：
  操作工具層。
  不是真正的投影片內容。
  它和 viewport 共享 wrapper 外框，但工具元件需要時會自行使用 canvasScale 做座標換算。
```

---

# 六、第四層：`ViewportBackground` 是背景工具，不是普通元素

## 先講原理

`ViewportBackground` 很容易被誤會。

因為它叫背景，看起來像投影片內容的一部分。

但在目前 source 裡，它不是放在 `viewport` 裡，而是放在 `operates` 裡。

這代表它更像是：

```text
可視區域背景工具
```

而不是：

```text
一般 EditableElement
```

這樣設計的好處是，背景可以跟操作工具一起對齊投影片外框，但不會混進元素列表裡。

---

## 再對回 Source

`ViewportBackground.vue` 會根據目前 slide 的 background 算出 `backgroundStyle`，也會根據 `gridLineSize` 決定是否渲染 `GridLines`。它自己的 `.viewport-background` 設定了 `width: 100%`、`height: 100%`、`position: absolute`，表示它是鋪在投影片外框上的背景層。

所以這層可以這樣記：

```text
ViewportBackground：
  負責目前投影片背景與格線。
  它不是元素資料，不會出現在 elementList 裡。
  它是圍繞可視區域工作的背景工具層。
```

---

# 七、第五層：`viewport` 是真正的邏輯內容層

## 先講原理

`viewport` 才是真正承載投影片內容的那一層。

它代表的是：

```text
固定邏輯畫布內容層。
```

也就是：

```text
MouseSelection
EditableElement
```

都放在這裡。

前面幾篇已經講過：

```text
元素資料活在固定邏輯畫布裡；
畫面只是把這張邏輯畫布 scale 後顯示。
```

而 `viewport` 就是這個固定邏輯畫布的 DOM 落點。

---

## 再對回 Source

在 `Canvas/index.vue` 裡：

```vue
<div
  class="viewport"
  ref="viewportRef"
  :style="{ transform: `scale(${canvasScale})` }"
>
  <MouseSelection ... />
  <EditableElement ... />
</div>
```

這裡有三個重點：

```text
1. viewport 有 ref="viewportRef"
2. viewport 整層做 transform: scale(canvasScale)
3. MouseSelection 和 EditableElement 都在 viewport 裡
```

這說明 `viewport` 是內容層，而且是互動座標反推時的基準層。

CSS 裡也可以看到：

```scss
.viewport {
  position: absolute;
  top: 0;
  left: 0;
  transform-origin: 0 0;
}
```

`transform-origin: 0 0` 很重要，代表這一層是以左上角為原點縮放，這和「元素座標相對於投影片左上角」的模型一致。

所以這層可以這樣記：

```text
viewport：
  真正的邏輯內容層。
  元素、框選結果都在這裡。
  它整層使用 transform: scale(canvasScale) 顯示到畫面上。
```

---

# 八、第六層：`EditableElement` 才是真正的元素渲染入口

## 先講原理

`EditableElement` 不是單一元素類型。

它比較像：

```text
畫布中所有可編輯元素的統一包裝器。
```

真正的文字、圖片、形狀、線條、表格、圖表等元件，是在 `EditableElement` 裡根據元素類型動態選出來的。

所以可以這樣分：

```text
viewport：
  承載整張投影片內容

EditableElement：
  承載單一元素的可編輯外殼

TextElement / ImageElement / ShapeElement / LineElement ...：
  真正負責畫出元素內容
```

---

## 再對回 Source

在 `Canvas/index.vue` 裡，`EditableElement` 是用 `v-for="(element, index) in elementList"` 渲染的，並把 `elementInfo`、`elementIndex`、`selectElement` 等資料傳進去。

而在 `EditableElement.vue` 裡，它會用 `<component :is="currentElementComponent">` 動態渲染元素元件；`currentElementComponent` 會根據 `ElementTypes.IMAGE`、`TEXT`、`SHAPE`、`LINE`、`CHART`、`TABLE`、`LATEX`、`VIDEO`、`AUDIO` 對應到不同元件。([GitHub][6])

所以這層可以這樣記：

```text
EditableElement：
  單一元素的可編輯包裝層。
  它不決定畫布縮放。
  它負責根據 elementInfo.type 選擇真正的元素元件。
```

---

# 九、`MouseSelection` 為什麼放在 `viewport` 裡

## 先講原理

`MouseSelection` 是滑鼠框選範圍。

它看起來像一個操作工具，但它被放在 `viewport` 裡，這點很值得注意。

原因是：

```text
框選範圍最後要和 elementList 裡的元素座標比較。
```

而元素資料是在邏輯畫布座標裡。

所以 `MouseSelection` 更適合和 `EditableElement` 一起站在 `viewport` 這個邏輯內容層裡。

---

## 再對回 Source

在 `Canvas/index.vue` 裡，`MouseSelection` 確實放在 `viewport` 裡，並接收 `top / left / width / height / quadrant` 等資料；後面才是 `EditableElement` 的列表。

所以可以這樣記：

```text
MouseSelection：
  雖然是互動輔助視覺，
  但它描述的是投影片內部的選取範圍，
  所以放在 viewport 這個邏輯內容層裡。
```

---

# 十、`Ruler` 為什麼不在 `viewport-wrapper` 裡

## 先講原理

標尺 `Ruler` 很容易讓人誤判。

直覺上會覺得：

```text
標尺是畫布工具，所以應該放在 wrapper 裡吧？
```

但目前 source 不是這樣。

`Ruler` 放在 `viewport-wrapper` 外面。

這代表它不是投影片外框裡的一部分，而是：

```text
圍繞投影片外框工作的外部輔助工具。
```

它需要知道投影片在哪裡、多大，但不需要被放進投影片外框裡。

---

## 再對回 Source

在 `Canvas/index.vue` 裡，`Ruler` 位於 `viewport-wrapper` 後面：

```vue
<Ruler :viewportStyles="viewportStyles" :elementList="elementList" v-if="showRuler" />
```

它接收 `viewportStyles` 和 `elementList`，但不是 `viewport-wrapper` 的子節點。

所以這層可以這樣記：

```text
Ruler：
  外部同步工具。
  不在 wrapper 裡。
  它靠 viewportStyles 追蹤投影片外框的位置與尺寸。
```

這和 `doc/Canvas.md` 的概念模型不衝突，因為 `doc/Canvas.md` 說的是「標尺屬於畫布工具」，不是說它一定要在 `viewport-wrapper` 這個 DOM 裡。

---

# 十一、`ElementCreateSelection` 和 `ShapeCreateCanvas` 是建立前的暫時互動層

## 先講原理

`ElementCreateSelection` 和 `ShapeCreateCanvas` 不應該被看成正式投影片內容。

它們比較像：

```text
使用者正在建立元素時，暫時出現在畫布工作區裡的互動層。
```

例如：

```text
拖曳建立文字框
拖曳建立圖片區域
繪製自訂 shape
```

在元素真正建立完成以前，它還不是 `elementList` 裡的一個正式元素。

---

## 再對回 Source

在 `Canvas/index.vue` 裡，這兩個元件放在 `viewport-wrapper` 前面：

```vue
<ElementCreateSelection
  v-if="creatingElement"
  @created="data => insertElementFromCreateSelection(data)"
/>

<ShapeCreateCanvas
  v-if="creatingCustomShape"
  @created="data => insertCustomShape(data)"
/>
```

它們都透過 `v-if` 受建立狀態控制，完成後才透過事件把資料交給後續插入流程。

所以可以這樣記：

```text
ElementCreateSelection / ShapeCreateCanvas：
  建立元素過程中的暫時互動層。
  它們不是投影片正式內容。
  產生資料後，才會透過 createElement / insert 流程變成真正元素。
```

---

# 十二、`drag-mask` 是拖曳整張畫布時的遮罩層

## 先講原理

`drag-mask` 也不是投影片內容。

它的角色比較像：

```text
按住空白鍵拖曳整張畫布時，蓋在畫布工作區上的互動遮罩。
```

前面〈互動座標換算篇〉已經講過：

```text
拖元素：改資料世界，要除以 canvasScale。
拖畫布：改顯示世界，不除以 canvasScale。
```

`drag-mask` 就是偏向拖曳整張畫布時的顯示 / 互動輔助層。

---

## 再對回 Source

在 `Canvas/index.vue` 裡：

```vue
<div class="drag-mask" v-if="spaceKeyState"></div>
```

而 `handleClickBlankArea()` 裡，如果不是空白鍵狀態就更新框選；如果是空白鍵狀態，就呼叫 `dragViewport(e)` 拖曳整張畫布。

`dragViewport()` 本身在 `useViewportSize.ts` 裡直接更新 `viewportLeft` 和 `viewportTop`，使用的是畫面位移，不會除以 `canvasScale`。([GitHub][3])

所以可以這樣記：

```text
drag-mask：
  拖曳整張畫布時的遮罩層。
  它服務的是畫布工作區互動，不是投影片內容資料。
```

---

# 十三、`Modal` 是浮層 UI，不屬於畫布幾何分層

## 先講原理

`Modal` 在 `Canvas/index.vue` 裡，但不要把它算進畫布幾何層。

它是 UI 浮層，例如連結設定對話框。

它會出現在畫布元件裡，是因為它和畫布操作有關，但它不參與：

```text
邏輯畫布
canvasScale
viewportStyles
元素座標
投影片外框
```

所以它不是畫布幾何系統的一部分。

---

## 再對回 Source

`Canvas/index.vue` 裡的 `Modal` 包著 `LinkDialog`，透過 `linkDialogVisible` 控制顯示。它出現在 template 裡，但位置在 `Ruler` 後面，並不是 `viewport-wrapper` 或 `viewport` 的子層。

```text
Modal：
  和畫布操作相關的浮層 UI。
  不參與投影片內容分層。
  不參與座標縮放模型。
```

# 十五、這篇最重要的分層圖

可以把整個 `Canvas/index.vue` 壓成下面這張圖：

```text
【Canvas 工作區層】
.canvas / canvasRef
  負責事件入口、背景、裁切、定位參考
  │
  ├──【建立前暫存互動層】
  │     ElementCreateSelection
  │     ShapeCreateCanvas
  │
  ├──【投影片外框層】
  │     viewport-wrapper
  │       width  = viewportStyles.width  * canvasScale
  │       height = viewportStyles.height * canvasScale
  │       left   = viewportStyles.left
  │       top    = viewportStyles.top
  │
  │     ├──【操作工具層】
  │     │     operates
  │     │       AlignmentLine
  │     │       MultiSelectOperate
  │     │       Operate
  │     │       ViewportBackground
  │     │
  │     └──【邏輯內容層】
  │           viewport / viewportRef
  │             transform: scale(canvasScale)
  │
  │           ├── MouseSelection
  │           └── EditableElement[]
  │                 └── TextElement / ImageElement / ShapeElement / ...
  │
  ├──【拖曳遮罩層】
  │     drag-mask
  │
  ├──【外部輔助工具層】
  │     Ruler
  │
  └──【浮層 UI】
        Modal / LinkDialog
```

這張圖要記住的不是 DOM 名字而已，而是責任：

```text
canvas 管工作區。
wrapper 管投影片外框。
operates 管操作工具。
viewport 管真正內容。
Ruler 在外面同步投影片外框。
ElementCreateSelection / ShapeCreateCanvas 是建立前暫存層。
drag-mask 是拖曳整張畫布的遮罩。
Modal 是浮層 UI。
```

# 十六、最容易誤會的 5 個點

## 1. 不要把 `viewport-wrapper` 當成內容層

`viewport-wrapper` 是外框，不是內容。

真正內容在：

```text
viewport
  └── EditableElement
```

---

## 2. 不要把 `operates` 當成元素資料層

`operates` 裡的東西是工具：

```text
AlignmentLine
MultiSelectOperate
Operate
ViewportBackground
```

它們服務編輯行為，不是投影片資料本身。

---

## 3. 不要以為 `Ruler` 一定要在 wrapper 裡

`Ruler` 是畫布工具，但目前實作放在 wrapper 外面。

它靠 `viewportStyles` 同步投影片位置，而不是靠父子 DOM 關係。

---

## 4. 不要把 `MouseSelection` 和 `ElementCreateSelection` 混在一起

這兩個名字都跟 selection 有關，但責任不同：

```text
MouseSelection：
  框選既有元素。

ElementCreateSelection：
  拖曳建立新元素時的暫時選區。
```

前者在 `viewport` 裡，後者在 `viewport-wrapper` 外面。

---

## 5. 不要把 template 順序完全等同於視覺堆疊順序

DOM 順序是閱讀 source 的入口，但真正視覺堆疊還會受到：

```text
position
z-index
transform
子元件自己的 CSS
```

影響。

所以讀分層時，重點不是只看「誰寫在前面」，而是看：

```text
它屬於哪個責任層？
它用哪一套座標？
它是不是正式投影片內容？
它需不需要 canvasScale？
```

# 十八、這篇最後要帶走的結論

PPTist 的畫布分層，不是單純的 HTML 包一包。

它真正做的是把這幾件事拆開：

```text
1. 整個工作區在哪裡接事件？
   -> canvas

2. 投影片外框現在在畫面上多大、放哪裡？
   -> viewport-wrapper

3. 哪些東西是編輯工具？
   -> operates

4. 哪些東西是真正投影片內容？
   -> viewport + EditableElement

5. 哪些工具需要跟投影片同步，但不該放進投影片裡？
   -> Ruler

6. 哪些東西只是建立元素過程中的暫時互動？
   -> ElementCreateSelection / ShapeCreateCanvas

7. 哪些東西只是畫布操作輔助或浮層？
   -> drag-mask / Modal
```

一句話總結：

> **PPTist 的畫布分層，本質上是在把「投影片內容」、「操作工具」、「顯示外框」、「工作區互動」拆成不同責任層，讓縮放、定位、選取、操作框、標尺和元素渲染可以各自穩定工作。**

這樣看 `Canvas/index.vue` 時，就不會只看到一堆元件，而是能看出：

```text
canvas 是工作區。
viewport-wrapper 是外框。
operates 是工具。
viewport 是內容。
EditableElement 是元素入口。
Ruler 是外部同步工具。
```

這就是第 6 篇要建立的核心心智模型。
