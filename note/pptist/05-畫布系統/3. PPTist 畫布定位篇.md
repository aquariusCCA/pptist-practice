# `PPTist` 畫布定位篇

所屬章節：[03-畫布系統](./README.md)

---

## 這篇要解決什麼

前兩篇已經把三件最核心的事講清楚了：

1. 為什麼要先固定一張邏輯畫布
2. 為什麼顯示縮放和資料座標要拆開
3. 為什麼互動事件最後要把畫面座標換回邏輯座標

但當你真的去對照 `PPTist` 原始碼時，還會卡在另一個更具體的問題：

> 這張固定邏輯畫布，現在在容器裡到底要顯示多大？
> 又到底要放在哪裡？

這篇就是專門把這件事講清楚。

更精準地說，這篇要回答的是：

* `viewportStyles.width / height / left / top` 各自代表什麼
* 為什麼 `width / height` 還要再乘上 `canvasScale`
* 為什麼畫布能夠在容器中置中
* 為什麼縮放時不只改 `canvasScale`，還要一起修正 `left / top`
* 為什麼拖曳整張畫布時，改的是 `viewportLeft / viewportTop`，不是元素座標

這一版會刻意遵守一個順序：

> **先講原理，再舉例 Source**

也就是先把「它為什麼要這樣設計」講清楚，再去看 `useViewportSize.ts`、`Canvas/index.vue`、`main.ts`、`slides.ts` 這些檔案怎麼落地。

## 閱讀提醒：這篇有 3 種東西要刻意分開

後面你會一直看到三種量，如果不先分開，`viewportStyles` 很容易越看越混：

### 1. 邏輯尺寸

這是資料世界裡的基準尺寸。

例如：

```ts
viewportWidth = 1000
viewportHeight = 562.5
```

它回答的是：

> **這張投影片在邏輯世界裡本來有多大？**

---

### 2. 顯示尺寸

這是畫面上真正看起來多大的尺寸。

例如：

```ts
viewportActualWidth = viewportWidth * canvasScale
viewportActualHeight = viewportHeight * canvasScale
```

它回答的是：

> **這張投影片現在在螢幕上看起來多大？**

---

### 3. 顯示位置

這是整張投影片外框在容器中的位置。

例如：

```ts
viewportLeft
viewportTop
```

它回答的是：

> **這張投影片外框左上角，現在落在容器中的哪裡？**

只要你後面一直把這三層分開，`viewportStyles`、`canvasScale`、`viewport-wrapper` 的責任就會很清楚。

# 一、先抓主問題：畫布定位系統到底在解什麼

## 先講原理

`PPTist` 的畫布定位系統，本質上不是在解「元素怎麼畫」，而是在解：

> **一張固定邏輯尺寸的投影片，現在要怎麼放進目前這個編輯容器裡。**

這裡至少有三個問題要同時成立：

1. 投影片要保持固定比例
2. 投影片要知道現在顯示成多大
3. 投影片要知道自己左上角現在放在哪裡

如果這三件事沒有拆清楚，後面就會開始混掉：

* 元素座標到底是資料值還是畫面值？
* `width / height` 到底是在說邏輯尺寸還是顯示尺寸？
* `left / top` 到底是在說元素位置還是整張畫布外框位置？

所以畫布定位系統真正做的事，不是單純「把畫布顯示出來」，而是把下面這條鏈接起來：

```text
固定邏輯畫布
-> 根據容器條件決定這次要顯示多大
-> 算出縮放倍率
-> 再決定外框要放在哪裡
```

---

## 再對回 Source

`doc/Canvas.md` 先講清楚：可視區域預設以 **寬 1000、高 562.5** 為基準比例。也就是說，一個 `{ width: 1000, height: 562.5, left: 0, top: 0 }` 的元素，會剛好鋪滿整個可視區域。這其實已經把整個畫布模型的源頭講出來了。

```ts
width = 1000
height = 562.5
```

接著 `slides.ts` 把這件事落地成 store 狀態：

```ts
viewportSize: 1000
viewportRatio: 0.5625
```

也就是：

```ts
viewportWidth = viewportSize
viewportHeight = viewportSize * viewportRatio
```

換句話說，畫布定位系統不是從 DOM 現在多寬開始算，而是從：

> **先有一張固定邏輯畫布**

這件事開始算。

---

# 二、先把三組角色釘死：邏輯尺寸、顯示尺寸、顯示位置

## 先講原理

這一節是整篇最重要的基礎。

你可以直接把畫布定位系統拆成三組量：

### 第一組：邏輯尺寸

```ts
viewportWidth = viewportSize
viewportHeight = viewportSize * viewportRatio
```

這組值的角色是：

> **描述資料世界裡，投影片本來的基準大小。**

---

### 第二組：顯示尺寸

```ts
viewportActualWidth = viewportWidth * canvasScale
viewportActualHeight = viewportHeight * canvasScale
```

這組值的角色是：

> **描述投影片現在在螢幕上真正看起來多大。**

---

### 第三組：顯示位置

```ts
viewportLeft
viewportTop
```

這組值的角色是：

> **描述投影片外框左上角，現在在容器中的位置。**

---

## 再對回 Source

在 `main.ts` 裡，與畫布顯示最有關的兩個值是：

```ts
canvasPercentage: 90
canvasScale: 1
canvasDragged: false
```

這裡要非常小心：

* `canvasPercentage` 不是最終縮放倍率
* `canvasScale` 才是最後真的拿去做 `scale()` 的值

更精準地說：

* `canvasPercentage`：這次希望畫布大概占容器多少比例
* `canvasScale`：根據容器大小與百分比算出來的實際縮放倍率

而 `useScaleCanvas.ts` 也直接把這個設計寫出來了：

```ts
// 设置画布缩放比例
// 但不是直接设置该值，而是通过设置画布可视区域百分比来动态计算
```

所以 `PPTist` 的思路不是：

> 我要直接把 `canvasScale` 改成某個值

而是：

> 我要先調整「畫布想占容器多少比例」，再由系統重算真正的 `canvasScale`

這也是為什麼你在講定位系統時，不能把 `canvasPercentage` 和 `canvasScale` 混成同一件事。

---

# 三、`viewportStyles` 到底是什麼

## 先講原理

這一節最容易卡住的地方，不是 `viewportStyles` 這個名字本身，而是很多人一看到它，就會下意識以為：

> **它應該就是「畫面上最後那個完整矩形」吧？**

也就是會直覺把它理解成：

* 最後顯示有多大
* 最後顯示放在哪裡
* 四個值都是同一層意思

但更貼近 `PPTist` 原始碼的說法其實不是這樣。

更精準地說：

> **`viewportStyles` 不是完整的「最終顯示矩形」。**
> **它提供的是：邏輯寬高 + 目前定位。**

這句話一定要先記住，因為它是後面整段理解的核心。

也就是說：

* `width / height`：仍然站在邏輯尺寸那一側
* `left / top`：已經站在顯示位置那一側

所以這四個值雖然被包在同一個物件裡，**但它們不是四個完全同質的值**。

---

## 為什麼很多人會誤會

因為看到下面這種東西時，很容易把它腦補成：

```ts
viewportStyles = {
  width,
  height,
  left,
  top,
}
```

然後直覺覺得：

> 這不就是一個矩形嗎？

這個直覺其實不奇怪。

因為從「名字」和「型態」上看，它真的很像一個完整的矩形描述。
但在 `PPTist` 這裡，更精準的理解應該是：

> **它像是在描述一張投影片外框，但裡面混合了兩種層級的資訊。**

也就是：

### 第一種：邏輯尺寸

```ts
width
height
```

它們描述的是：

> **這張投影片在資料世界裡，本來有多大。**

---

### 第二種：顯示位置

```ts
left
top
```

它們描述的是：

> **這張投影片外框左上角，目前在容器中的哪裡。**

所以如果你把 `viewportStyles` 整個直接理解成「已經縮放好的畫面矩形」，後面就很容易混掉。

---

## 先用一句話把它釘死

你可以先把 `viewportStyles` 記成：

> **它不是最後畫到畫面上的完整矩形，而是「邏輯寬高 + 當前定位」這組中介資訊。**

這樣你後面再看到 `viewport-wrapper` 的 style 綁定時，就不會誤以為 `viewportStyles.width` 本身已經是最後畫面寬度。

---

## 先用一個小例子看懂

假設目前邏輯畫布基準是：

```ts
viewportSize = 1000
viewportRatio = 0.5625
```

那邏輯尺寸就是：

```ts
width = 1000
height = 1000 * 0.5625 = 562.5
```

再假設目前畫布外框的位置是：

```ts
left = 80
top = 195
```

那這時候的 `viewportStyles` 可以先理解成：

```ts
viewportStyles = {
  width: 1000,
  height: 562.5,
  left: 80,
  top: 195,
}
```

看到這裡時，你一定要刻意停一下，提醒自己：

這四個值不是都在回答同一個問題。

### `width = 1000`

回答的是：

> 這張投影片在邏輯世界裡本來多寬。

### `height = 562.5`

回答的是：

> 這張投影片在邏輯世界裡本來多高。

### `left = 80`

回答的是：

> 這張投影片外框左上角，現在距離容器左邊 `80px`。

### `top = 195`

回答的是：

> 這張投影片外框左上角，現在距離容器上邊 `195px`。

所以它不是單純在說：

> 畫面上有一個寬 `1000`、高 `562.5`、放在 `(80, 195)` 的最終矩形

因為真正畫到畫面上的寬高，後面還要再考慮 `canvasScale`。

---

## 為什麼它還不是「最終顯示矩形」

關鍵就在這裡。

假設現在：

```ts
canvasScale = 0.8
```

那畫面上真正看起來的顯示尺寸會是：

```ts
actualWidth = 1000 * 0.8 = 800
actualHeight = 562.5 * 0.8 = 450
```

也就是說，畫面上最後真正占空間的，不是：

```ts
1000 × 562.5
```

而是：

```ts
800 × 450
```

但 `left = 80`、`top = 195` 仍然可以直接成立，因為它們本來就是在描述：

> **這張投影片外框左上角在容器中的位置。**

所以這時候你就可以很清楚看出來：

* `width / height` 還在邏輯層
* `left / top` 已經在顯示定位層

這也就是為什麼 `viewportStyles` 不能直接被講成「最終顯示矩形」。

---

## 你可以把 `viewportStyles` 想成什麼

如果要用最好懂的方式去記，你可以把 `viewportStyles` 想成：

> **給 `viewport-wrapper` 用的一組基礎描述資料。**

這組資料會告訴外層：

1. 這張投影片的邏輯基準尺寸是多少
2. 這張投影片外框目前應該放在哪裡

但它還沒有把「縮放後真正顯示多大」這件事做完。

那一步，是後面 `viewport-wrapper` 再處理的。

所以這裡最好的語言不是：

> `viewportStyles` 就是畫面最後的投影片矩形

而是：

> **`viewportStyles` 是畫布外框的基礎描述；真正的顯示寬高，要再搭配 `canvasScale` 才會成立。**

---

## 再對回 Source

在 `useViewportSize.ts` 裡，`viewportStyles` 是這樣定義的：

```ts
const viewportStyles = computed(() => ({
  width: viewportSize.value,
  height: viewportSize.value * viewportRatio.value,
  left: viewportLeft.value,
  top: viewportTop.value,
}))
```

這段可以直接拆成兩半看。

### 前兩個值

```ts
width: viewportSize.value
height: viewportSize.value * viewportRatio.value
```

這兩行很清楚地告訴你：

> **這裡的 `width / height` 還是邏輯尺寸。**

因為它們直接來自：

* `viewportSize`
* `viewportRatio`

也就是固定邏輯畫布的基準。

---

### 後兩個值

```ts
left: viewportLeft.value
top: viewportTop.value
```

這兩行則是在告訴你：

> **這裡的 `left / top` 是畫面上的定位值。**

因為它們不是從邏輯尺寸推來的，而是來自 `viewportLeft / viewportTop`，而這兩個值在 `useViewportSize.ts` 裡本來就在處理：

* 初始化置中位置
* 縮放後修正位置
* 拖曳後更新位置

所以它們描述的不是元素資料，而是整張畫布外框目前在畫面上的位置。

---

## 再接回 `Canvas/index.vue` 就會更清楚

當你再往下看 `Canvas/index.vue` 的這段：

```vue
:style="{
  width: viewportStyles.width * canvasScale + 'px',
  height: viewportStyles.height * canvasScale + 'px',
  left: viewportStyles.left + 'px',
  top: viewportStyles.top + 'px',
}"
```

你就會突然看懂它為什麼要這樣寫。

因為這段其實是在做兩件不同的事：

### 第一件事：把邏輯尺寸轉成顯示尺寸

```ts
width: viewportStyles.width * canvasScale
height: viewportStyles.height * canvasScale
```

也就是：

> **先拿 `viewportStyles` 裡的邏輯寬高，再乘上 `canvasScale`，變成真正畫面上的外框大小。**

---

### 第二件事：直接使用當前定位

```ts
left: viewportStyles.left
top: viewportStyles.top
```

也就是：

> **直接把目前的外框位置用上去。**

所以更貼近 source 的說法應該是：

> **`viewportStyles` 提供的是邏輯寬高與當前定位；真正的顯示寬高，是到 `viewport-wrapper` 那一層才乘上 `canvasScale` 後成立。**

這一句其實就是第三節最重要的結論。

---

## 這一節最值得記住的一句話

你可以把 `viewportStyles` 直接記成：

> **它不是最終顯示矩形，而是「邏輯寬高 + 當前定位」這組中介資訊；真正的顯示尺寸，要再交給 `viewport-wrapper` 搭配 `canvasScale` 去完成。**

只要這句話記住，後面第四節「為什麼 `width / height` 要乘上 `canvasScale`，但 `left / top` 不用」就會自然很多。

# 四、為什麼 `width / height` 要乘上 `canvasScale`，但 `left / top` 不用

## 先講原理

這一節最容易讓人卡住的地方是：

> 為什麼同一組 `viewportStyles` 裡面的四個值，只有 `width / height` 要乘上 `canvasScale`，但 `left / top` 卻不用？

很多人第一次看到時，會直覺覺得：

> 既然整張畫布都在縮放，那不是四個值都應該一起乘上去嗎？

這個直覺很合理。

但真正的關鍵是：

> **這四個值雖然被放在同一個物件裡，卻不是在回答同一種問題。**

更精準地說：

* `width / height` 在回答：**邏輯畫布本來有多大**
* `left / top` 在回答：**這張畫布外框現在放在哪裡**

所以你不能因為它們都叫 `viewportStyles.xxx`，就把它們當成同一層的量。

---

## 先把兩種問題分開

### 第一種問題：畫布現在看起來多大？

這是在問：

> **固定邏輯畫布經過縮放後，畫面上真正顯示成多大？**

這種問題當然要考慮 `canvasScale`。

例如邏輯寬度是 `1000`，縮放倍率是 `0.8`，那畫面上看起來的寬度就會是：

```ts
1000 * 0.8 = 800
```

所以：

* `width / height` 要乘上 `canvasScale`

因為它們是在從：

```text
邏輯尺寸
```

推到：

```text
顯示尺寸
```

---

### 第二種問題：畫布現在放在哪裡？

這是在問：

> **這張投影片外框左上角，目前在容器中的哪個位置？**

這種問題說的本來就是畫面上的位置值。

例如：

```ts
left = 80
top = 195
```

它的意思是：

* 外框左上角距離容器左邊 `80px`
* 外框左上角距離容器上邊 `195px`

這本來就已經是顯示層的結果，不是邏輯尺寸，所以不需要再乘一次 `canvasScale`。

所以：

* `left / top` 不要再乘上 `canvasScale`

---

## 先用一個小例子看懂

假設目前：

```ts
viewportStyles = {
  width: 1000,
  height: 562.5,
  left: 80,
  top: 195,
}

canvasScale = 0.8
```

這時候要分成兩組看。

### 第一組：`width / height`

```ts
width = 1000
height = 562.5
```

這代表的是：

> 這張投影片的邏輯基準尺寸是 `1000 × 562.5`

但畫面上真正看起來不會還是 `1000 × 562.5`，因為它還要經過縮放：

```ts
actualWidth = 1000 * 0.8 = 800
actualHeight = 562.5 * 0.8 = 450
```

所以畫面上最後真正占空間的尺寸其實是：

```ts
800 × 450
```

這也就是為什麼 `width / height` 一定要乘上 `canvasScale`。

---

### 第二組：`left / top`

```ts
left = 80
top = 195
```

這代表的是：

> 這張投影片外框左上角，現在就在 `(80, 195)`

它不是邏輯尺寸，也不是還沒算完的中間值，而是已經可以直接拿來定位的畫面座標。

所以這時候如果你再把它乘上 `canvasScale`：

```ts
left = 80 * 0.8 = 64
top = 195 * 0.8 = 156
```

那就會出現一個問題：

> **畫布外框的位置被你又縮了一次。**

這會讓整張投影片跑到錯的地方去。

所以 `left / top` 不只是「不用乘」，而是：

> **不能亂乘。**

---

## 為什麼 `left / top` 不能跟著一起縮

因為 `left / top` 和 `width / height` 的角色本來就不同。

### `width / height`

是在說：

> 這張畫布本來有多大，縮放後會顯示成多大。

### `left / top`

是在說：

> 這張畫布外框左上角，在容器中的哪裡。

如果你把 `left / top` 也跟著縮，就等於把「位置」當成「尺寸」處理。

但位置不是尺寸。

更白話地說：

* 畫布有沒有變大，是尺寸問題
* 畫布放在哪裡，是定位問題

這兩件事不能混在一起算。

---

## 你可以把這段想成兩步

如果用最直白的方式去記，`viewport-wrapper` 在做的其實是兩步：

### 第一步：先決定畫面上這個外框應該多大

這一步用的是：

```ts
width * canvasScale
height * canvasScale
```

也就是：

> **把邏輯尺寸轉成顯示尺寸。**

---

### 第二步：再決定這個外框應該放在哪裡

這一步用的是：

```ts
left
top
```

也就是：

> **直接使用目前已經算好的定位值。**

所以更好的理解不是：

> 四個值一起構成一個矩形，然後全部一起縮放

而是：

> **尺寸要做縮放，位置直接套用。**

---

## 再對回 Source

在 `Canvas/index.vue` 裡，`viewport-wrapper` 的 style 是這樣綁的：

```vue

  width: viewportStyles.width * canvasScale + 'px',
  height: viewportStyles.height * canvasScale + 'px',
  left: viewportStyles.left + 'px',
  top: viewportStyles.top + 'px',
}"
```

這段其實已經把責任切得非常清楚。

### 前兩行

```ts

height: viewportStyles.height * canvasScale
```

這兩行代表：

> **先拿邏輯寬高，再乘上縮放倍率，得到畫面上真正顯示的外框尺寸。**

---

### 後兩行

```ts
left: viewportStyles.left
top: viewportStyles.top
```

這兩行代表：

> **直接使用目前算好的外框位置。**

因為 `left / top` 已經是在 `useViewportSize.ts` 裡處理過的定位結果。

---

## 再接回 `useViewportSize.ts` 會更穩

如果你再回頭看 `useViewportSize.ts`，就會發現：

* `width / height` 直接來自邏輯基準：

```ts
width: viewportSize.value
height: viewportSize.value * viewportRatio.value
```

* `left / top` 直接來自定位狀態：

```ts
left: viewportLeft.value
top: viewportTop.value
```

這也正好說明了：

> **`viewportStyles` 本身就已經把「邏輯尺寸」和「畫面定位」放在一起了。**

所以到了 `Canvas/index.vue`，才會出現：

* 尺寸要再乘上 `canvasScale`
* 位置直接用

這完全不是特例，而是前面資料分工自然推下來的結果。

---

## 如果把 `left / top` 也乘上去，畫面會怎麼壞掉

我們還是用剛才那組數字：

```ts
width = 1000
height = 562.5
left = 80
top = 195
canvasScale = 0.8
```

正確做法是：

```ts
displayWidth = 1000 * 0.8 = 800
displayHeight = 562.5 * 0.8 = 450
left = 80
top = 195
```

也就是：

> 畫布外框大小變成 `800 × 450`，左上角仍然放在 `(80, 195)`。

但如果你錯誤地把 `left / top` 也一起乘上去：

```ts
left = 80 * 0.8 = 64
top = 195 * 0.8 = 156
```

那就變成：

> 畫布不只縮小，連位置也被你往左上拉了一次。

這樣整張畫布就不是縮放後放在原本該在的位置，而是整個矩形被你額外偏移了。

所以這裡一定要記住：

> **`left / top` 不是還沒換算的邏輯值，而是已經算好的定位結果。**

---

## 這一節最值得記住的一句話

你可以把這段直接記成：

> **`width / height` 要乘上 `canvasScale`，因為它們在把邏輯尺寸轉成顯示尺寸；`left / top` 不用乘，因為它們本來就是畫面上的定位值。**

只要這句話記住，這一節就不會再混掉。

# 五、為什麼畫布能夠置中

## 先講原理

這一節最容易讓人誤會的地方是：

> 畫布會置中，是不是因為 `canvasScale` 幫我做了置中？

答案不是。

更精準地說：

> **`canvasScale` 只回答「畫布現在要縮放幾倍」，不回答「畫布要放在哪裡」。**

所以畫布之所以能夠置中，不是因為 `scale()` 本身有置中能力，而是因為系統另外算出了：

* 這次畫布在畫面上要顯示多大
* 這次畫布外框左上角要放在哪裡

也就是說，所謂「置中」，本質上不是一個特效，而只是：

> **先知道容器有多大，再知道畫布顯示後有多大，最後把剩下的空白平均分到兩邊。**

所以這一節最核心的公式，其實只有兩句：

```ts
left = (容器寬度 - 畫布顯示寬度) / 2
top = (容器高度 - 畫布顯示高度) / 2
```

但真正的原始碼不會直接只寫這兩句，因為在那之前，它還要先解另一個問題：

> **這次應該用寬度來決定畫布大小，還是用高度來決定畫布大小？**

這就是 `useViewportSize.ts` 裡那個 `if / else` 真正在解的事。

---

## 為什麼要先判斷「用寬度算」還是「用高度算」

因為投影片比例是固定的，但外層容器比例不是固定的。

例如：

* 有時容器比較高、比較瘦
* 有時容器比較扁、比較寬

如果你每次都只拿容器寬度去縮放，有些情況下高度會爆掉；
如果你每次都只拿容器高度去縮放，有些情況下寬度又會留太多空白。

所以系統一定要先判斷：

> **這次到底是寬度先成為限制，還是高度先成為限制。**

如果容器相對比較高，代表高度比較寬裕，這時就比較適合先用寬度來決定畫布顯示大小；
如果容器相對比較扁，代表高度比較容易先不夠，這時就要改成用高度來決定畫布顯示大小。

所以「置中」真正的完整流程其實是：

```text
先判斷這次是寬度受限還是高度受限
-> 算出畫布這次在畫面上應該顯示多大
-> 回推出 canvasScale
-> 再用剩餘空白 ÷ 2 算出 left / top
```

---

## 先用一個寬度分支的例子看懂

假設現在外層容器是：

```ts
canvasWidth = 1600
canvasHeight = 1200
viewportRatio = 0.5625
canvasPercentage = 90
```

這裡你可以先看容器比例：

```ts
canvasHeight / canvasWidth = 1200 / 1600 = 0.75
```

而投影片比例是：

```ts
viewportRatio = 0.5625
```

因為：

```ts
0.75 > 0.5625
```

表示：

> **容器相對於投影片來說比較高。**

換句話說，這次高度比較寬裕，寬度比較容易先成為限制。

所以這時候系統就會走：

> **先用寬度當基準**

來決定畫布顯示大小。

---

### 第一步：先算這次畫布在畫面上想顯示多寬

```ts
viewportActualWidth = canvasWidth * (canvasPercentage / 100)
```

代入數字：

```ts
viewportActualWidth = 1600 * 0.9 = 1440
```

意思是：

> 這次畫布在畫面上，不想撐滿整個容器寬度，而是先顯示成容器寬度的 90%。

---

### 第二步：再用比例反推出這次實際顯示高度

因為投影片比例固定，所以高度不能亂設，而是要跟著寬度一起推：

```ts
viewportActualHeight = viewportActualWidth * viewportRatio
```

代入數字：

```ts
viewportActualHeight = 1440 * 0.5625 = 810
```

所以這次畫布在畫面上實際顯示成：

```ts
1440 × 810
```

---

### 第三步：再回推出 `canvasScale`

這一步是在回答：

> 既然這次畫布要顯示成 `1440` 寬，那固定邏輯寬度 `1000` 應該縮放成幾倍？

公式就是：

```ts
canvasScale = 實際顯示寬度 / 邏輯寬度
```

代入數字：

```ts
canvasScale = 1440 / 1000 = 1.44
```

所以這次真正的縮放倍率是：

```ts
canvasScale = 1.44
```

---

### 第四步：最後再算 `left / top`

現在畫布的顯示尺寸已經知道了：

```ts
viewportActualWidth = 1440
viewportActualHeight = 810
```

接下來就只剩一件事：

> **把它放到容器正中央。**

這裡就直接用：

```ts
left = (容器寬度 - 畫布顯示寬度) / 2
top = (容器高度 - 畫布顯示高度) / 2
```

代入數字：

```ts
left = (1600 - 1440) / 2 = 80
top = (1200 - 810) / 2 = 195
```

也就是：

```ts
viewportLeft = 80
viewportTop = 195
```

這代表：

* 左邊留 `80`
* 右邊也留 `80`
* 上面留 `195`
* 下面也留 `195`

所以畫布就會剛好置中。

---

## 你可以把「置中」想成一張外框被塞進容器裡

如果用最直白的圖像去想，可以把它想成這樣：

```text
canvas（外層容器）
┌──────────────────────────────────────┐
│                                      │
│         195px 空白                   │
│      ┌────────────────────────┐      │
│ 80px │      畫布外框          │ 80px │
│ 空白 │      1440 × 810        │ 空白 │
│      └────────────────────────┘      │
│         195px 空白                   │
│                                      │
└──────────────────────────────────────┘
```

所謂置中，說穿了就是：

> **剩餘空白平均分到左右與上下。**

沒有更神祕的事。

---

## 高度分支其實只是反過來做同一件事

如果容器比較扁，代表高度先成為限制，這時原始碼就會改成：

1. 先用容器高度與 `canvasPercentage` 算出 `viewportActualHeight`
2. 再除以比例，反推出 `viewportActualWidth`
3. 再用這組顯示尺寸去算 `canvasScale`
4. 最後同樣用剩餘空白 ÷ 2 算出 `left / top`

也就是說，高度分支和寬度分支本質上不是兩套不同邏輯，而只是：

> **先決定這次是由哪一邊當作縮放基準。**

後面的目標完全一樣：

> **求出顯示尺寸，再把外框置中。**

---

## 再對回 Source

在 `useViewportSize.ts` 的 `initViewportPosition()` 裡，核心邏輯就是：

```ts
if (canvasHeight / canvasWidth > viewportRatio.value) {
  const viewportActualWidth = canvasWidth * (canvasPercentage.value / 100)
  mainStore.setCanvasScale(viewportActualWidth / viewportSize.value)
  viewportLeft.value = (canvasWidth - viewportActualWidth) / 2
  viewportTop.value = (canvasHeight - viewportActualWidth * viewportRatio.value) / 2
}
else {
  const viewportActualHeight = canvasHeight * (canvasPercentage.value / 100)
  mainStore.setCanvasScale(viewportActualHeight / (viewportSize.value * viewportRatio.value))
  viewportLeft.value = (canvasWidth - viewportActualHeight / viewportRatio.value) / 2
  viewportTop.value = (canvasHeight - viewportActualHeight) / 2
}
```

你可以直接把這段翻成白話：

1. 先比較容器比例和投影片比例
2. 決定這次用寬度還是高度當基準
3. 算出畫布這次在畫面上真正要顯示多大
4. 回推出 `canvasScale`
5. 最後把剩餘空白各分一半，得到 `viewportLeft / viewportTop`

所以更貼近原始碼的說法不是：

> 畫布會置中，是因為 `canvasScale` 幫忙置中

而是：

> **畫布會置中，是因為 `useViewportSize.ts` 先算出顯示尺寸，再用剩餘空白 ÷ 2 算出外框位置。**

---

## 這一節還有一個很重要的提醒

這裡講的「置中」，比較精準地說是：

> **初始化時，或重算位置時，系統如何把畫布外框放到容器中央。**

它不是在說畫布永遠都待在中間。

因為後面如果使用者拖動畫布，`viewportLeft / viewportTop` 就會改掉；
之後縮放時，也會以目前畫布自己的中心去修正位置，而不是每次都強迫回到最初的置中位置。

所以這一節要理解的是：

> **畫布最初為什麼能被正確置中。**

而不是：

> **畫布之後永遠不會離開中心。**

---

## 這一節最值得記住的一句話

你可以把這段直接記成：

> **畫布能置中，不是因為 `canvasScale` 自帶置中能力，而是因為系統先算出顯示尺寸，再把容器剩餘空白平均分到左右與上下，得到 `left / top`。**

只要這句話記住，第五節就不會再模糊。

# 六、為什麼縮放時不只改 `canvasScale`，還要修正 `left / top`

## 先講原理

這一節最容易卡住的地方，不是公式本身，而是你要先搞懂：

> **縮放時，系統真正想固定的，不是 `left / top`，而是「畫布的中心點」。**

很多人一開始會直覺地以為：

> 縮放不就是把 `canvasScale` 改大或改小嗎？

但如果你只改倍率，不去修正畫布外框的位置，畫布看起來就不會是「以原本中心往四周一起放大或縮小」，而會像是只往某一側長大，或只往某一側縮回去。

所以更精準的說法應該是：

> **縮放不只是改「畫布有多大」，還要同時維持「縮放前後中心點不要跑掉」。**

這也是為什麼 `PPTist` 在縮放時，不只會重算 `canvasScale`，還會一起修正：

* `viewportLeft`
* `viewportTop`

因為這兩個值代表的不是元素座標，而是：

> **整張畫布外框左上角，目前在容器中的位置。**

---

## 先把 `left / top` 想成「外框左上角」

你可以先把畫布外框想成一個矩形：

```text
(left, top)
   ↓
   ┌──────────────────────┐
   │      畫布外框         │
   │                      │
   │                      │
   └──────────────────────┘
```

這裡的：

* `viewportLeft`：外框左上角距離容器左邊多少
* `viewportTop`：外框左上角距離容器上邊多少

所以這一節在修的，不是元素資料，也不是邏輯座標，而是整張投影片外框在畫面上的位置。

---

## 為什麼不能只改 `canvasScale`

先看一個最簡單的水平例子。

假設原本畫布外框是這樣：

```ts
oldLeft = 100
oldWidth = 800
```

那原本中心點就是：

```ts
oldCenterX = 100 + 800 / 2 = 500
```

現在如果你把畫布放大，新的寬度變成：

```ts
newWidth = 1000
```

但你完全不改 `left`，也就是：

```ts
newLeft = 100
```

那新的中心點就會變成：

```ts
newCenterX = 100 + 1000 / 2 = 600
```

你會發現：

* 原本中心點是 `500`
* 新中心點變成 `600`

也就是說：

> **畫布的中心往右跑掉了。**

這代表畫布不是以原本中心往左右兩邊一起長大，而是看起來比較像「往右邊擴張」。

這就是為什麼不能只改 `canvasScale`。

---

## 系統真正想維持的是什麼

系統真正想維持的是這件事：

```ts
newCenterX = oldCenterX
newCenterY = oldCenterY
```

也就是：

> **縮放前後，畫布中心不要變。**

如果要讓水平中心不變，就一定要滿足：

```ts
newLeft + newWidth / 2 = oldLeft + oldWidth / 2
```

把這個式子整理一下：

```ts
newLeft = oldLeft - (newWidth - oldWidth) / 2
```

這一行非常重要。

它的意思是：

> **新畫布比舊畫布多出多少寬度，就要把左上角往左補回一半。**

垂直方向完全一樣：

```ts
newTop = oldTop - (newHeight - oldHeight) / 2
```

也就是：

> **新畫布比舊畫布多出多少高度，就要把左上角往上補回一半。**

所以這裡不是什麼特別複雜的技巧，它其實只是在做一件很單純的事：

> **讓新舊畫布的中心點維持不變。**

---

## 用一個完整數字例子看懂

下面直接用 `PPTist` 這段邏輯會遇到的數字來看。

假設現在外層容器是：

```ts
canvasWidth = 1600
canvasHeight = 1200
viewportRatio = 0.5625
```

原本畫布顯示百分比是：

```ts
oldValue = 90
```

現在想把它放大成：

```ts
newValue = 100
```

---

### 第一步：先算舊畫布在畫面上實際顯示多大

```ts
oldViewportActualWidth = 1600 * 0.9 = 1440
oldViewportActualHeight = 1440 * 0.5625 = 810
```

這裡的 `1440 × 810`，是畫布原本在畫面上真正看起來的大小。

---

### 第二步：再算新畫布在畫面上實際顯示多大

```ts
newViewportActualWidth = 1600 * 1 = 1600
newViewportActualHeight = 1600 * 0.5625 = 900
```

所以現在畫布從：

```text
1440 × 810
```

變成：

```text
1600 × 900
```

也就是：

* 寬度多了 `160`
* 高度多了 `90`

---

### 第三步：假設原本畫布是置中的，先看原本位置

原本如果是置中的，那它的 `left / top` 會是：

```ts
oldLeft = (1600 - 1440) / 2 = 80
oldTop = (1200 - 810) / 2 = 195
```

也就是說，原本畫布外框左上角在：

```ts
(80, 195)
```

---

### 第四步：算出新舊尺寸差

```ts
deltaWidth = 1600 - 1440 = 160
deltaHeight = 900 - 810 = 90
```

這代表：

* 畫布總共變寬了 `160`
* 畫布總共變高了 `90`

如果要讓它仍然以原本中心往四周展開，那就要把這些差值平均分配到左右、上下兩邊。

所以：

* 左邊補 `80`
* 右邊也補 `80`
* 上面補 `45`
* 下面也補 `45`

---

### 第五步：更新 `left / top`

所以新的位置就不是維持原本的 `(80, 195)`，而是要改成：

```ts
newLeft = 80 - 160 / 2 = 0
newTop = 195 - 90 / 2 = 150
```

也就是：

```ts
newLeft = 0
newTop = 150
```

---

### 第六步：驗證中心點有沒有變

先看舊中心：

```ts
oldCenterX = 80 + 1440 / 2 = 800
oldCenterY = 195 + 810 / 2 = 600
```

再看新中心：

```ts
newCenterX = 0 + 1600 / 2 = 800
newCenterY = 150 + 900 / 2 = 600
```

你會發現：

```ts
oldCenterX === newCenterX
oldCenterY === newCenterY
```

也就是說：

> **畫布變大了，但中心點完全沒變。**

這就是這段邏輯真正想做到的事。

---

## 為什麼是「扣一半」，不是扣全部

因為新舊尺寸差不是只長在一邊，而是要平均分到兩邊。

例如寬度從 `1440` 變成 `1600`，總共多了 `160`。
如果要讓中心不變，這 `160` 不可能全部長到右邊，而應該是：

* 左邊多 `80`
* 右邊也多 `80`

所以左上角自然就要往左移 `80`：

```ts
left -= 160 / 2
```

高度也完全一樣：

```ts
top -= 90 / 2
```

所以「扣一半」不是公式技巧，而是因為：

> **畫布要以原本中心往四周一起放大或縮小。**

---

## 再對回 Source

在 `useViewportSize.ts` 的 `setViewportPosition(newValue, oldValue)` 裡，寬度分支的關鍵程式碼是：

```ts
const newViewportActualWidth = canvasWidth * (newValue / 100)
const oldViewportActualWidth = canvasWidth * (oldValue / 100)
const newViewportActualHeight = newViewportActualWidth * viewportRatio.value
const oldViewportActualHeight = oldViewportActualWidth * viewportRatio.value

mainStore.setCanvasScale(newViewportActualWidth / viewportSize.value)
viewportLeft.value = viewportLeft.value - (newViewportActualWidth - oldViewportActualWidth) / 2
viewportTop.value = viewportTop.value - (newViewportActualHeight - oldViewportActualHeight) / 2
```

你可以直接把它翻成白話：

1. 先算新舊畫布在畫面上各自有多大
2. 再用新的顯示寬度回推出新的 `canvasScale`
3. 最後把左上角往左、往上各補回一半的尺寸差
4. 讓縮放前後的畫布中心維持不變

所以這段不是單純在「修位置」，而是在做：

> **固定中心點的縮放。**

高度分支也是同樣邏輯，只是改成先從高度當基準去算。

---

## 這段還有一個更深一點的意思

這段程式不是每次縮放都重新算一次「容器正中央」，而是：

> **以目前畫布自己的位置為基準，維持它當下的中心。**

這很重要，因為如果使用者先把畫布拖到別的地方，再縮放，系統也不會突然把畫布送回初始置中的位置；它會沿著目前這個位置的中心去放大或縮小。

也就是說，它保留的是：

> **目前視覺上的中心感**

而不是：

> **永遠回到容器初始中心**

這樣整體互動會自然很多。

---

## 這一節最值得記住的一句話

你可以把這段邏輯直接記成：

> **縮放時，`canvasScale` 決定畫布變多大；`left / top` 決定畫布左上角在哪裡；為了讓中心點不變，左上角必須扣回新舊尺寸差的一半。**

這樣你之後再看到這段程式，就不會只看到公式，而會知道它其實在做什麼。

---

# 七、拖曳整張畫布時，改的不是元素座標，而是外框位置

## 先講原理

這一節最容易和「拖元素」混在一起。

所以一開始要先把兩件事硬切開：

### 拖元素時

你改的是元素自己的資料，例如元素的：

* `left`
* `top`
* `width`
* `height`

也就是說，你改的是：

> **元素在邏輯畫布裡的位置與尺寸。**

---

### 拖整張畫布時

你改的不是元素資料，而是：

* `viewportLeft`
* `viewportTop`

也就是說，你改的是：

> **整張投影片外框在畫面上的位置。**

這兩種拖曳看起來都像「東西在移動」，但本質上完全不是同一件事。

所以這一節最重要的一句話就是：

> **拖畫布時，改的是畫布外框的位置，不是元素的邏輯座標。**

---

## 先把 `viewportLeft / viewportTop` 想成什麼

你可以先把畫面想成這樣：

```text
canvas（外層容器）
┌──────────────────────────────────────┐
│                                      │
│   viewport-wrapper                   │
│   (left = viewportLeft,              │
│    top  = viewportTop)               │
│   ┌──────────────────────────────┐   │
│   │ viewport                     │   │
│   │ element / background / ...   │   │
│   └──────────────────────────────┘   │
│                                      │
└──────────────────────────────────────┘
```

所以：

* `viewportLeft`：畫布外框左上角，距離容器左邊多少
* `viewportTop`：畫布外框左上角，距離容器上邊多少

只要這兩個值變了，整張畫布看起來就會一起移動。

注意這裡移動的是：

> **整張投影片外框**

不是：

> **投影片裡某個元素的資料位置**

---

## 為什麼拖畫布時，不需要除以 `canvasScale`

這一點非常關鍵。

前面講過，當你在拖元素、框選元素、建立元素時，滑鼠事件先拿到的是畫面座標；如果最後要寫回元素資料，就必須把畫面位移換回邏輯位移，所以常常會看到：

```ts
邏輯位移 = 畫面位移 / canvasScale
```

但拖整張畫布時不是這樣。

因為拖畫布時，系統不是在改元素資料，而是在改：

```ts
viewportLeft
viewportTop
```

而這兩個值本來就是：

> **畫面上的定位值**

既然目標就是畫面位置，那當然直接用滑鼠在畫面上移動多少，就讓畫布在畫面上跟著移多少。

所以拖畫布時的邏輯其實是：

```text
滑鼠在畫面上移多少
-> 畫布外框就在畫面上移多少
```

而不是：

```text
滑鼠在畫面上移多少
-> 先換回邏輯位移
-> 再改元素資料
```

這就是為什麼拖畫布時，不需要除以 `canvasScale`。

---

## 先看觸發入口在哪裡

在 `Canvas/index.vue` 裡，點擊空白區時會走這段：

```ts
const handleClickBlankArea = (e: MouseEvent) => {
  if (activeElementIdList.value.length) mainStore.setActiveElementIdList([])

  if (!spaceKeyState.value) updateMouseSelection(e)
  else dragViewport(e)
}
```

你可以直接把它翻成白話：

* 沒按 `Space`：這次是框選模式
* 按了 `Space`：這次改成拖曳整張畫布

所以這裡的前提不是「拖到某個元素」，而是：

> **按住 `Space`，然後拖曳畫布空白區。**

---

## 真正被改的是什麼

真正的拖曳邏輯在 `useViewportSize.ts` 的 `dragViewport()`：

```ts
const dragViewport = (e: MouseEvent) => {
  let isMouseDown = true

  const startPageX = e.pageX
  const startPageY = e.pageY

  const originLeft = viewportLeft.value
  const originTop = viewportTop.value

  document.onmousemove = e => {
    if (!isMouseDown) return

    const currentPageX = e.pageX
    const currentPageY = e.pageY

    viewportLeft.value = originLeft + (currentPageX - startPageX)
    viewportTop.value = originTop + (currentPageY - startPageY)
  }
}
```

這段如果先不看語法，只看邏輯，其實很單純：

1. 先記住滑鼠按下當下的位置
2. 再記住畫布外框當下的位置
3. 之後滑鼠每移動一次，就把「滑鼠移動量」直接加到畫布外框位置上

所以這段不是在改元素資料，而是在做：

> **畫布外框跟著滑鼠一起平移。**

---

## 把每個變數翻成白話

### `startPageX / startPageY`

表示：

> **滑鼠按下那一瞬間，在頁面上的位置。**

例如：

```ts
startPageX = 500
startPageY = 300
```

---

### `originLeft / originTop`

表示：

> **滑鼠按下那一瞬間，畫布外框原本的位置。**

例如：

```ts
originLeft = 80
originTop = 195
```

---

### `currentPageX - startPageX`

表示：

> **滑鼠相對於起點，水平移動了多少。**

如果目前滑鼠移到：

```ts
currentPageX = 620
```

那水平位移就是：

```ts
deltaX = 620 - 500 = 120
```

也就是：

> 滑鼠往右拖了 `120px`

垂直方向同理。

---

## 用一個完整數字例子看懂

假設滑鼠按下那一刻：

```ts
startPageX = 500
startPageY = 300
originLeft = 80
originTop = 195
```

這代表：

* 滑鼠起點在 `(500, 300)`
* 當時畫布外框左上角在 `(80, 195)`

現在如果你拖到：

```ts
currentPageX = 620
currentPageY = 360
```

那滑鼠相對起點的位移就是：

```ts
deltaX = 620 - 500 = 120
deltaY = 360 - 300 = 60
```

這代表：

* 滑鼠往右拖了 `120`
* 滑鼠往下拖了 `60`

那畫布新位置就直接變成：

```ts
newLeft = 80 + 120 = 200
newTop = 195 + 60 = 255
```

也就是：

```ts
viewportLeft.value = 200
viewportTop.value = 255
```

你可以直接把這件事想成：

> **滑鼠拖了多少，畫布外框就跟著平移多少。**

這就是整段邏輯最核心的意思。

---

## 為什麼是 `originLeft + deltaX`，不是一直累加當前值

這個細節也值得講清楚。

程式不是寫成：

```ts
viewportLeft.value += 某個移動量
```

而是：

```ts
viewportLeft.value = originLeft + (currentPageX - startPageX)
```

這樣寫的好處是：

> **每一次滑鼠移動時，都是直接用「起點 + 總位移」重新算一次當前位置。**

這樣比一直累加更穩，因為你不用擔心事件頻率不同、或前一次值誤差累積造成位置飄掉。

你可以把它想成：

```text
不是每次在舊位置上再推一小步
而是每次都重新回答：
「相對於起點，我總共拖了多少？」
```

所以這種寫法會更穩定。

---

## 這段和第六節其實是連起來的

第六節講的是：

> 縮放時要維持中心感，所以要修 `left / top`

而第七節講的是：

> 拖畫布時，改的本來就是 `left / top`

所以你把兩節串起來看，就會更清楚：

* `viewportLeft / viewportTop` 是整張畫布外框的位置
* 縮放時會修它們，因為中心要維持不變
* 拖曳時也會改它們，因為整張畫布要跟著滑鼠平移

換句話說：

> **`viewportLeft / viewportTop` 是整套畫布顯示位置控制的核心。**

---

## `mouseup` 時還做了什麼

在這段拖曳流程裡，滑鼠放開時還會做一件事：

```ts
mainStore.setCanvasDragged(true)
```

這代表系統會記住：

> **這張畫布已經被使用者拖曳過。**

這個狀態不是拿來改元素，而是讓整套畫布系統知道：

* 目前畫布位置不再只是初始化置中的結果
* 之後如果要重置畫布，系統就知道它曾經被平移過

所以 `canvasDragged` 比較像是一個「互動狀態記號」。

---

## 這一節最值得記住的一句話

你可以直接把拖畫布這段程式記成：

> **拖曳整張畫布時，系統先記住滑鼠起點和畫布外框原點；之後滑鼠每移動多少，畫布外框就直接跟著平移多少。**

所以這段程式真正做的不是：

> 改元素邏輯座標

而是在做：

> **改整張畫布外框的畫面位置。**

---

# 八、目前 `Canvas/index.vue` 的分層，怎麼講才更貼 source

## 先講原理

這一節最容易被講模糊。

更貼近原始碼的講法不是：

> 畫布就是一層 `viewport`

而是：

> **`viewport-wrapper` 是顯示外框**
> **`viewport` 是邏輯內容層**

再進一步說：

* `viewport-wrapper`：負責縮放後外框的佔位與定位
* `operates`：承接大部分操作層
* `viewport`：承接真正內容層，並做 `scale()`
* `Ruler`：不在 wrapper 裡，但靠 `viewportStyles` 跟畫布同步

這樣講，會比把所有東西都粗略說成「畫布的一部分」更精準。

---

## 再對回 Source

在 `Canvas/index.vue` 裡，畫布主體是這樣：

```vue
<div
  class="viewport-wrapper"
  :style="{
    width: viewportStyles.width * canvasScale + 'px',
    height: viewportStyles.height * canvasScale + 'px',
    left: viewportStyles.left + 'px',
    top: viewportStyles.top + 'px',
  }"
>
  <div class="operates">
    <AlignmentLine ... />
    <MultiSelectOperate ... />
    <Operate ... />
    <ViewportBackground />
  </div>

  <div class="viewport" :style="{ transform: `scale(${canvasScale})` }">
    <MouseSelection ... />
    <EditableElement ... />
  </div>
</div>

<Ruler :viewportStyles="viewportStyles" :elementList="elementList" v-if="showRuler" />
```

這段很值得注意三件事：

### 1. `operates` 和 `viewport` 都在 `viewport-wrapper` 裡

代表它們共享同一個投影片外框基準。

### 2. `viewport` 才是內容層，並且直接做 `scale()`

所以元素一直是在固定邏輯座標裡渲染，只是整張內容層被縮放。

### 3. `Ruler` 不在 wrapper 裡

但它透過 `viewportStyles` 取得可視區域的位置與大小，再和畫布同步。

因此，比較精準的講法是：

> **`operates`、`viewport` 共享同一個顯示外框；`Ruler` 在外面，但靠 `viewportStyles` 同步同一張畫布。**

這會比「所有畫布相關圖層都在 wrapper 裡」更貼近目前 source。

---

# 一句話總結

`PPTist` 的畫布定位系統，本質上是在做這件事：

> **先用 `slides.ts` 定義固定邏輯畫布，再由 `useViewportSize.ts` 根據容器大小、比例與百分比，算出 `canvasScale` 與 `left / top`；最後交給 `Canvas/index.vue` 用 `viewport-wrapper` 顯示外框、用 `viewport` 縮放內容。**
