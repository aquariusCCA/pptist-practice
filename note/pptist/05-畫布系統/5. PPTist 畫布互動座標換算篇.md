# `PPTist` 畫布互動座標換算篇

所屬章節：[03-畫布系統](./README.md)

---

## 這篇要解決什麼

前面幾篇已經把畫布系統的「顯示鏈路」講清楚了：

```text
固定邏輯畫布
-> 根據容器與顯示目標算出 canvasScale
-> viewport-wrapper 建立縮放後外框
-> viewport 用 transform: scale(canvasScale) 顯示內容
```

但真正做編輯器時，還有一條更重要的「反向鏈路」：

```text
滑鼠事件拿到畫面座標
-> 找到 viewport 目前在畫面上的位置
-> 扣掉 viewport 左上角
-> 除以 canvasScale
-> 變回邏輯畫布座標
-> 寫回元素資料或拿去做互動判斷
```

這篇要專門回答：

1. 為什麼滑鼠事件不能直接寫入元素資料
2. `pageX / pageY`、`clientX / clientY`、`getBoundingClientRect()` 的座標系差在哪裡
3. 為什麼 `viewportRef.getBoundingClientRect()` 是互動換算入口
4. 為什麼大部分元素互動都要除以 `canvasScale`
5. 為什麼拖曳整張畫布不用除以 `canvasScale`
6. 雙擊新增文字、拖曳建立元素、框選、拖曳元素、縮放元素、旋轉元素、調整 shape keypoint、拖曳線條端點分別怎麼對回 source

這一篇的核心不是背公式，而是建立一個判斷能力：

```text
這次操作是在改「資料世界」？
還是在改「顯示世界」？
```

只要能回答這個問題，就能判斷這次互動到底要不要除以 `canvasScale`。

# 一、先抓一句話核心

這篇最重要的核心是：

```text
使用者在畫面上操作的是「縮放後的投影片」；
但 PPTist 真正要修改的是「固定尺寸的邏輯投影片資料」。
```

所以互動座標換算要做的事就是：

```text
把滑鼠在畫面上的位置或距離，
換回 PPTist 資料真正使用的邏輯座標或邏輯距離。
```

---

## 先用一個最簡單的例子理解

假設 PPTist 的邏輯畫布是：

```text
1000 × 562.5
```

但現在畫面被放大 2 倍顯示：

```ts
canvasScale = 2
```

這代表：

```text
邏輯世界的 100px
顯示到畫面上會變成 200px
```

所以顯示方向是：

```text
邏輯距離 × canvasScale = 畫面距離
```

反過來，使用者拖曳時拿到的是畫面距離。

如果使用者在畫面上拖了 `200px`，那它回到邏輯世界其實只有：

```ts
200 / 2 = 100
```

所以互動方向是：

```text
畫面距離 ÷ canvasScale = 邏輯距離
```

這就是為什麼大部分元素互動都會看到：

```ts
moveX = (currentPageX - startPageX) / canvasScale
moveY = (currentPageY - startPageY) / canvasScale
```

---

## 但是點座標還多一個問題

如果只是拖曳距離，例如：

```text
從 A 點拖到 B 點，總共移動 200px
```

那只要除以 `canvasScale` 就好。

但如果是「滑鼠點在投影片上的哪裡」，就不能只除以 `canvasScale`。

因為滑鼠事件拿到的：

```ts
pageX
pageY
```

是整個頁面上的座標，不是投影片內部座標。

所以系統要先知道：

```text
目前 viewport 這張投影片的左上角在畫面哪裡？
```

這就是：

```ts
viewportRef.value.getBoundingClientRect()
```

的作用。

它會告訴你 viewport 在畫面上的位置，例如：

```ts
viewportRect.x
viewportRect.y
```

所以點座標要先做第一步：

```text
滑鼠畫面座標 - viewport 左上角位置
```

得到：

```text
滑鼠相對於投影片左上角的畫面座標
```

再做第二步：

```text
除以 canvasScale
```

得到邏輯畫布座標。

所以點座標公式是：

```ts
logicX = (pageX - viewportRect.x) / canvasScale
logicY = (pageY - viewportRect.y) / canvasScale
```

---

## 所以要先分清楚兩種換算

### 1. 點座標換算

用在：

```text
滑鼠點在投影片上的哪裡？
```

例如：

```text
雙擊新增文字
拖曳建立元素的 left / top
旋轉時計算滑鼠點
```

公式是：

```ts
logicX = (pageX - viewportRect.x) / canvasScale
logicY = (pageY - viewportRect.y) / canvasScale
```

因為它要處理兩件事：

```text
1. 先扣掉 viewport 左上角
2. 再除以 canvasScale
```

---

### 2. 位移量換算

用在：

```text
滑鼠從起點到終點，總共拖了多少？
```

例如：

```text
拖曳元素
縮放元素
拖曳線條端點
調整 shape keypoint
```

公式是：

```ts
logicMoveX = (currentPageX - startPageX) / canvasScale
logicMoveY = (currentPageY - startPageY) / canvasScale
```

因為它本來就是兩個滑鼠點之間的距離，所以不需要再扣 `viewportRect`。

---

## 最後還要記住一個例外：拖曳整張畫布

不是所有拖曳都要除以 `canvasScale`。

如果拖的是元素，改的是：

```ts
element.left
element.top
```

這是邏輯資料，所以要除以 `canvasScale`。

但如果拖的是整張畫布，改的是：

```ts
viewportLeft
viewportTop
```

這是畫面上的顯示位置，不是元素邏輯資料，所以不用除以 `canvasScale`。

可以先用這句話記：

```text
拖元素：改資料世界，要除以 canvasScale。
拖畫布：改顯示世界，不除以 canvasScale。
```

---

## 這章真正要先記住的版本

```text
PPTist 顯示畫布時：
邏輯資料 × canvasScale = 畫面結果

PPTist 處理互動時：
畫面操作 ÷ canvasScale = 邏輯資料

如果是滑鼠點：
先扣 viewport 左上角，再除以 canvasScale。

如果是拖曳距離：
直接把畫面距離除以 canvasScale。

如果是拖曳整張畫布：
改的是畫面位置，不除以 canvasScale。
```

# 二、先把座標換算中的幾個角色分清楚

互動座標換算最容易混，是因為 source 裡會同時看到：

```text
pageX / pageY
clientX / clientY
getBoundingClientRect()
viewportRect.x / y
viewportRect.left / top
canvasScale
```

但它們不是同一類東西。

可以先用一句話分開：

```text
pageX / pageY、clientX / clientY：
  是滑鼠事件給你的「滑鼠在哪裡」。

getBoundingClientRect()：
  是 DOM 告訴你「viewport 這個元素目前在哪裡」。

canvasScale：
  是把「畫面距離」換回「邏輯距離」的倍率。
```

所以這章不是要背 API，而是要搞懂下面這條換算鏈：

```text
滑鼠在頁面上的位置
-> 扣掉 viewport 在畫面上的位置
-> 得到滑鼠相對於投影片左上角的位置
-> 除以 canvasScale
-> 得到邏輯畫布座標
```

---

## 1. `pageX / pageY`：滑鼠在整份頁面中的位置

`pageX / pageY` 是滑鼠事件給你的座標。

它回答的是：

```text
滑鼠現在在整份 document 裡的哪個位置？
```

例如：

```ts
e.pageX
e.pageY
```

你可以先把它想成：

```text
滑鼠目前在整個網頁座標系中的位置。
```

但是要注意，它還不是投影片內部的位置。

例如使用者在投影片中間雙擊，事件拿到的可能是：

```ts
pageX = 800
pageY = 420
```

這只能代表：

```text
滑鼠在整份頁面中的位置是 (800, 420)
```

它還不能直接代表：

```text
這個點在投影片裡的邏輯座標是 (800, 420)
```

因為投影片本身可能不是從頁面的 `(0, 0)` 開始顯示。

---

## 2. `getBoundingClientRect()`：viewport 目前在畫面上的位置

PPTist 互動裡常看到：

```ts
const viewportRect = viewportRef.value.getBoundingClientRect()
```

這行的意思是：

```text
取得 viewport 這個 DOM 元素目前在畫面上的矩形資訊。
```

其中最重要的是：

```ts
viewportRect.x
viewportRect.y
```

或：

```ts
viewportRect.left
viewportRect.top
```

它們回答的是：

```text
viewport 左上角目前在畫面上的哪裡？
```

你可以把它想成：

```text
投影片左上角目前在畫面上的位置。
```

例如：

```ts
viewportRect.x = 300
viewportRect.y = 120
```

意思是：

```text
這張投影片的左上角，目前在畫面座標 (300, 120) 的位置。
```

---

## 3. 為什麼要用 `pageX - viewportRect.x`

現在假設使用者點了一下，滑鼠事件給你：

```ts
pageX = 800
pageY = 420
```

而 viewport 的左上角是：

```ts
viewportRect.x = 300
viewportRect.y = 120
```

那麼：

```ts
pageX - viewportRect.x = 800 - 300 = 500
pageY - viewportRect.y = 420 - 120 = 300
```

這代表：

```text
滑鼠點擊的位置，
距離投影片左上角的畫面距離是：

x = 500
y = 300
```

所以第一階段換算是：

```text
整個頁面中的滑鼠位置
-> 扣掉投影片左上角位置
-> 變成滑鼠相對於投影片左上角的位置
```

也就是：

```ts
relativeX = pageX - viewportRect.x
relativeY = pageY - viewportRect.y
```

但這時候的 `relativeX / relativeY` 還是**畫面座標**。

它只是代表：

```text
滑鼠在縮放後的投影片上，看起來離左上角多少 px。
```

還不是 PPTist 元素資料要用的邏輯座標。

---

## 4. 為什麼還要除以 `canvasScale`

假設目前：

```ts
canvasScale = 2
```

代表畫布被放大 2 倍顯示。

這時候畫面上看到的：

```ts
relativeX = 500
relativeY = 300
```

回到邏輯畫布，其實是：

```ts
logicX = 500 / 2 = 250
logicY = 300 / 2 = 150
```

所以完整公式才會是：

```ts
logicX = (pageX - viewportRect.x) / canvasScale
logicY = (pageY - viewportRect.y) / canvasScale
```

這個公式其實分成兩步：

```text
第一步：
pageX - viewportRect.x
把「整頁座標」變成「相對於投影片左上角的畫面座標」

第二步：
/ canvasScale
把「畫面座標」變成「邏輯畫布座標」
```

所以不要把整個公式硬背成一坨。

它真正的意思是：

```text
先找到滑鼠在投影片畫面上的相對位置，
再把這個畫面位置縮回邏輯畫布位置。
```

---

## 5. `clientX / clientY` 是另一種滑鼠座標

`clientX / clientY` 也是滑鼠事件座標。

但它回答的是：

```text
滑鼠在瀏覽器可視區中的位置。
```

而不是整份 document 中的位置。

所以：

```text
pageX / pageY：
  相對於整份 document

clientX / clientY：
  相對於瀏覽器 viewport
```

如果頁面沒有捲動，它們看起來可能很接近。

但如果 document 有 scroll，兩者就會不同。

---

## 6. `getBoundingClientRect()` 回傳的是 viewport 座標

這裡是最容易混的地方。

`getBoundingClientRect()` 回傳的：

```ts
rect.x
rect.y
rect.left
rect.top
```

是相對於瀏覽器 viewport 的位置。

也就是說，它和 `clientX / clientY` 比較同一套。

所以理論上最嚴謹的寫法通常會是：

```ts
logicX = (clientX - viewportRect.left) / canvasScale
logicY = (clientY - viewportRect.top) / canvasScale
```

或者如果要用 `pageX / pageY`，就要把 rect 也轉成 page 座標：

```ts
logicX = (pageX - (viewportRect.left + window.scrollX)) / canvasScale
logicY = (pageY - (viewportRect.top + window.scrollY)) / canvasScale
```

---

## 7. 那為什麼 PPTist source 會寫 `pageX - viewportRect.x`

PPTist source 裡多處會看到類似：

```ts
const left = (e.pageX - viewportRect.x) / canvasScale.value
const top = (e.pageY - viewportRect.y) / canvasScale.value
```

嚴格來說：

```text
pageX 是 document 座標
viewportRect.x 是 viewport 座標
```

兩者不是完全同一套。

但在 PPTist 這種全頁式編輯器布局裡，整份 document 通常不會有明顯 scroll，所以：

```text
pageX 約等於 clientX
pageY 約等於 clientY
```

因此 source 這樣寫在目前場景下通常可以正常運作。

讀 source 時可以這樣理解：

```text
PPTist 用 pageX - viewportRect.x，
實務上是在取得滑鼠相對於 viewport 左上角的畫面位置。
```

不要被這個細節卡住，但你要知道它背後嚴謹的座標系其實不是完全一樣。

---

## 8. `viewportRect.x / y` 和 `viewportRect.left / top` 怎麼看

在 `DOMRect` 裡：

```ts
viewportRect.x
viewportRect.y
viewportRect.left
viewportRect.top
```

在一般情況下可以這樣看：

```text
x    ≈ left
y    ≈ top
```

PPTist source 有些地方用：

```ts
viewportRect.x
viewportRect.y
```

有些地方用：

```ts
viewportRect.left
viewportRect.top
```

讀 source 時不用把它們理解成兩套不同座標系。

它們都在扮演同一個角色：

```text
viewport 左上角目前在畫面中的位置。
```

所以看到：

```ts
e.pageX - viewportRect.x
```

或：

```ts
currentPageX - viewportRect.left
```

都可以先理解成：

```text
滑鼠位置 - 投影片左上角位置
```

---

## 9. 這章真正要記住的版本

```text
pageX / pageY：
  滑鼠在整份 document 裡的位置。

clientX / clientY：
  滑鼠在瀏覽器可視區裡的位置。

getBoundingClientRect()：
  DOM 元素在瀏覽器可視區裡的位置與尺寸。

viewportRect.x / y、left / top：
  viewport 左上角目前在畫面中的位置。

canvasScale：
  不是座標系，而是畫面距離與邏輯距離之間的換算倍率。
```

PPTist 的互動座標換算可以簡化成：

```text
第一步：
滑鼠位置 - viewport 左上角位置
= 滑鼠相對於投影片左上角的畫面位置

第二步：
相對畫面位置 / canvasScale
= 滑鼠在邏輯畫布中的位置
```

公式就是：

```ts
logicX = (pageX - viewportRect.x) / canvasScale
logicY = (pageY - viewportRect.y) / canvasScale
```

但真正要理解的是：

```text
viewportRect 解決「投影片現在放在哪裡」；
canvasScale 解決「畫面距離怎麼縮回邏輯距離」。
```

# 三、為什麼 `viewportRef.getBoundingClientRect()` 是入口

要理解這件事，先抓住一個核心：

```text
PPTist 元素資料的 left / top，
都是相對於「邏輯投影片左上角」的位置。
```

也就是說，元素資料不是相對於：

```text
整個瀏覽器頁面
整個 Canvas 容器
viewport-wrapper
螢幕左上角
```

而是相對於：

```text
viewport 這張邏輯投影片的左上角
```

所以當使用者在畫面上點一下時，PPTist 要問的不是：

```text
滑鼠在整個頁面哪裡？
```

而是：

```text
滑鼠點在這張投影片裡的哪裡？
```

要回答這個問題，第一步就必須先知道：

```text
這張投影片的左上角目前在畫面哪裡？
```

這就是：

```ts
viewportRef.value.getBoundingClientRect()
```

存在的原因。

---

## 1. 滑鼠事件給的是「全局位置」

例如使用者雙擊畫面，事件會給：

```ts
e.pageX
e.pageY
```

你可以先把它理解成：

```text
滑鼠在整個頁面中的位置。
```

假設：

```ts
e.pageX = 800
e.pageY = 420
```

這只能表示：

```text
滑鼠目前在頁面座標 (800, 420)
```

但這還不是投影片內部座標。

因為投影片本身可能放在畫面的某個位置，例如：

```text
投影片左上角在畫面上的位置是 (300, 120)
```

那滑鼠點對投影片來說，實際上是在：

```ts
x = 800 - 300 = 500
y = 420 - 120 = 300
```

也就是：

```text
滑鼠距離投影片左上角 500px、300px。
```

這一步就是 `getBoundingClientRect()` 要幫你解決的。

---

## 2. `getBoundingClientRect()` 取得的是 viewport 的畫面位置

在 PPTist 裡會看到：

```ts
const viewportRect = viewportRef.value.getBoundingClientRect()
```

這行的重點不是「取得一個矩形」而已。

它真正回答的是：

```text
viewport 這張投影片內容層，
目前在畫面上的左上角在哪裡？
```

所以：

```ts
viewportRect.x
viewportRect.y
```

或：

```ts
viewportRect.left
viewportRect.top
```

可以先理解成：

```text
投影片左上角目前在畫面上的位置。
```

因此：

```ts
e.pageX - viewportRect.x
e.pageY - viewportRect.y
```

代表：

```text
滑鼠相對於投影片左上角的畫面座標。
```

---

## 3. 為什麼不是直接用 `canvasRef`

因為 `canvasRef` 代表的是整個畫布工作區。

它可能包含：

```text
投影片本體
投影片外面的灰色區域
標尺
操作層
拖曳空間
其他畫布工具
```

但元素資料的 `left / top` 不是相對於整個工作區。

元素資料是相對於：

```text
投影片本身的左上角
```

所以如果用 `canvasRef` 當基準，算出來的座標會變成：

```text
滑鼠相對於整個編輯區的位置
```

這不是元素資料需要的座標。

PPTist 需要的是：

```text
滑鼠相對於投影片內容層的位置
```

所以要用 `viewportRef`。

---

## 4. 為什麼不是直接用 `viewportStyles.left / top`

理論上，`viewportStyles.left / top` 也描述投影片外框的位置。

但 source 裡使用：

```ts
viewportRef.value.getBoundingClientRect()
```

更穩，因為它直接從 DOM 取得目前實際顯示後的位置。

也就是說，它拿到的是：

```text
瀏覽器最後排版、定位、transform 後的實際畫面矩形。
```

這對互動換算很重要。

因為滑鼠事件拿到的是畫面上的座標，所以最好也用 DOM 實際畫面位置來相減。

這樣整條換算會比較一致：

```text
滑鼠畫面位置
-
viewport 實際畫面位置
=
滑鼠相對於投影片的畫面位置
```

---

## 5. 為什麼扣完還要除以 `canvasScale`

扣掉 `viewportRect.x / y` 之後，只是得到：

```text
滑鼠相對於投影片左上角的畫面座標
```

但 PPTist 元素資料需要的是：

```text
邏輯座標
```

假設目前：

```ts
canvasScale = 2
```

代表投影片被放大 2 倍顯示。

如果滑鼠距離投影片左上角的畫面距離是：

```ts
relativeX = 500
relativeY = 300
```

那換回邏輯畫布就是：

```ts
logicX = 500 / 2
logicY = 300 / 2
```

也就是：

```ts
logicX = 250
logicY = 150
```

所以完整公式是：

```ts
const viewportRect = viewportRef.value.getBoundingClientRect()

const logicX = (e.pageX - viewportRect.x) / canvasScale.value
const logicY = (e.pageY - viewportRect.y) / canvasScale.value
```

可以拆成兩步理解：

```text
第一步：
e.pageX - viewportRect.x
取得滑鼠相對於投影片左上角的畫面位置。

第二步：
/ canvasScale
把縮放後的畫面位置換回邏輯畫布位置。
```

---

## 6. `transform-origin: 0 0` 為什麼重要

在 `Canvas/index.vue` 裡，`viewport` 有：

```scss
.viewport {
  position: absolute;
  top: 0;
  left: 0;
  transform-origin: 0 0;
}
```

這代表：

```text
viewport 是從左上角開始縮放。
```

這一點很重要。

因為 PPTist 的元素座標也是從投影片左上角開始算：

```ts
element.left
element.top
```

所以顯示和互動可以共用同一個基準點：

```text
投影片左上角
```

也就是：

```text
顯示時：
邏輯座標從 viewport 左上角出發，乘上 canvasScale。

互動時：
滑鼠座標先扣 viewport 左上角，再除以 canvasScale。
```

如果 `transform-origin` 是中心點，縮放時內容會以中心點展開，左上角位置會受到額外影響。

那麼公式就不能簡單寫成：

```ts
logicX = (pageX - viewportRect.x) / canvasScale
logicY = (pageY - viewportRect.y) / canvasScale
```

因為你還要處理縮放中心造成的偏移。

所以 `transform-origin: 0 0` 的意義是：

```text
讓 viewport 的縮放基準和元素邏輯座標基準一致。
```

這樣互動座標反推才可以保持簡單。

---

## 7. 這章真正要記住的版本

```text
viewportRef.getBoundingClientRect()
不是單純取得 DOM 尺寸。

它是互動換算的起點，
因為它告訴系統：

「投影片左上角目前在畫面哪裡？」
```

完整換算可以記成：

```text
滑鼠在頁面上的位置
-
viewport 左上角在畫面上的位置
=
滑鼠相對於投影片左上角的畫面位置

滑鼠相對於投影片左上角的畫面位置
/
canvasScale
=
滑鼠在邏輯畫布中的位置
```

公式就是：

```ts
const viewportRect = viewportRef.value.getBoundingClientRect()

const logicX = (e.pageX - viewportRect.x) / canvasScale.value
const logicY = (e.pageY - viewportRect.y) / canvasScale.value
```

最核心的一句話是：

```text
viewportRect 解決「投影片左上角在哪裡」；
canvasScale 解決「畫面距離怎麼縮回邏輯距離」。
```

這樣會比原本清楚，因為它不只是說「要知道 viewport 在哪裡」，而是直接說清楚：

```text
PPTist 元素資料是以 viewport 左上角為原點，
所以互動換算一定要先找到 viewport 左上角。
```

---

# 四、互動換算的三種基本模型

PPTist 的互動換算不要只背公式。

比較好的理解方式是先問：

```text
這次滑鼠操作，系統到底想算什麼？
```

大致可以分成三種問題：

```text
1. 滑鼠點在投影片裡的哪裡？
2. 滑鼠從起點到終點，總共拖了多少？
3. 滑鼠框出來的矩形，在投影片裡是哪個範圍？
```

這三種問題，對應三種換算模型：

```text
點座標換算
位移量換算
矩形換算
```

---

## 1. 點座標換算：滑鼠點在投影片裡的哪裡？

點座標換算要回答的是：

```text
這個滑鼠點，落在邏輯投影片的哪個位置？
```

用在這些情境：

```text
雙擊新增文字：
  要知道新文字放在哪裡。

旋轉元素：
  要知道滑鼠點相對於元素中心在哪裡。

拖曳建立元素：
  要知道選區左上角落在投影片哪裡。
```

---

### 為什麼點座標要扣 `viewportRect`

滑鼠事件給你的通常是：

```ts
pageX
pageY
```

它代表的是：

```text
滑鼠在整個頁面上的位置。
```

但元素資料需要的是：

```text
滑鼠在投影片裡的位置。
```

所以第一步要先扣掉投影片左上角的位置：

```ts
relativeX = pageX - viewportRect.x
relativeY = pageY - viewportRect.y
```

這一步得到的是：

```text
滑鼠相對於投影片左上角的畫面位置。
```

但這還是畫面座標，還不是邏輯座標。

---

### 為什麼點座標還要除以 `canvasScale`

因為畫面上的投影片可能已經被縮放。

假設：

```ts
canvasScale = 2
```

滑鼠看起來距離投影片左上角：

```ts
relativeX = 500
relativeY = 300
```

換回邏輯畫布就是：

```ts
logicX = 500 / 2
logicY = 300 / 2
```

所以完整公式是：

```ts
const viewportRect = viewportRef.value.getBoundingClientRect()

const logicX = (pageX - viewportRect.x) / canvasScale.value
const logicY = (pageY - viewportRect.y) / canvasScale.value
```

可以把它拆成兩步：

```text
第一步：
pageX - viewportRect.x
= 滑鼠相對於投影片左上角的畫面位置

第二步：
/ canvasScale
= 把畫面位置換回邏輯畫布位置
```

一句話記：

```text
點座標 = 先找投影片內的位置，再縮回邏輯座標。
```

---

## 2. 位移量換算：滑鼠總共拖了多少？

位移量換算要回答的是：

```text
滑鼠從起點到現在，總共移動了多少距離？
```

用在這些情境：

```text
拖曳元素：
  要知道元素 left / top 要增加多少。

縮放元素：
  要知道寬高要變化多少。

調整 shape keypoint：
  要知道控制點移動多少。

拖曳線條端點或控制點：
  要知道 start / end / broken / curve / cubic 要變化多少。
```

---

### 為什麼位移量不用扣 `viewportRect`

位移量是兩個滑鼠點相減：

```ts
currentPageX - startPageX
currentPageY - startPageY
```

這本身已經是距離。

例如：

```ts
startPageX = 800
currentPageX = 860
```

那：

```ts
currentPageX - startPageX = 60
```

代表滑鼠在畫面上往右移動了 `60px`。

這個 `60px` 不關心投影片左上角在哪裡。

因為不管投影片放在畫面的 `(300, 120)`，還是 `(500, 200)`，滑鼠從 `800` 拖到 `860`，畫面位移都一樣是 `60px`。

所以位移量不需要扣：

```ts
viewportRect.x
viewportRect.y
```

---

### 為什麼位移量還是要除以 `canvasScale`

因為這個 `60px` 是畫面上的距離。

如果：

```ts
canvasScale = 2
```

代表畫面被放大 2 倍。

那畫面上拖了 `60px`，回到邏輯世界其實只有：

```ts
60 / 2 = 30
```

所以公式是：

```ts
const moveX = (currentPageX - startPageX) / canvasScale.value
const moveY = (currentPageY - startPageY) / canvasScale.value
```

一句話記：

```text
位移量 = 不用找投影片左上角，只要把畫面距離縮回邏輯距離。
```

---

## 3. 矩形換算：滑鼠框出來的範圍是哪裡？

矩形換算要回答的是：

```text
使用者在畫面上框出來的矩形，
對應到邏輯投影片裡的哪個矩形？
```

用在這些情境：

```text
拖曳建立元素：
  要把畫面選區變成新元素的 left / top / width / height。

框選元素：
  要把畫面框選範圍變成邏輯選區，再和元素範圍比較。
```

矩形比較特別，因為它同時包含兩種資訊：

```text
left / top：
  這個矩形從哪裡開始。

width / height：
  這個矩形有多大。
```

所以它其實是：

```text
點座標換算 + 位移量換算
```

---

### `left / top` 是點座標

`left / top` 問的是：

```text
矩形左上角在投影片裡的哪裡？
```

所以它要用點座標換算。

公式是：

```ts
const left = (minX - viewportRect.x) / canvasScale.value
const top = (minY - viewportRect.y) / canvasScale.value
```

因為它要做兩件事：

```text
1. 扣掉 viewport 左上角
2. 除以 canvasScale
```

---

### `width / height` 是距離量

`width / height` 問的是：

```text
這個矩形有多寬、多高？
```

它不是問矩形在投影片哪裡，而是問拖曳形成的距離。

所以它不用扣 `viewportRect`。

公式是：

```ts
const width = (maxX - minX) / canvasScale.value
const height = (maxY - minY) / canvasScale.value
```

因為：

```text
maxX - minX
maxY - minY
```

本身已經是畫面距離。

只要再除以 `canvasScale`，就會變成邏輯寬高。

---

## 矩形換算完整公式

所以完整矩形換算會長這樣：

```ts
const viewportRect = viewportRef.value.getBoundingClientRect()

const left = (minX - viewportRect.x) / canvasScale.value
const top = (minY - viewportRect.y) / canvasScale.value
const width = (maxX - minX) / canvasScale.value
const height = (maxY - minY) / canvasScale.value
```

這段不要硬背。

要這樣拆：

```text
left / top：
  問「矩形從哪裡開始」
  所以是點座標換算。

width / height：
  問「矩形有多大」
  所以是位移量換算。
```

一句話記：

```text
矩形換算 = 左上角用點座標，寬高用距離量。
```

---

## 三種模型的差異表

| 模型                     | 它在問什麼     | 要不要扣 `viewportRect` | 要不要除以 `canvasScale` |
| ---------------------- | --------- | ------------------: | ------------------: |
| 點座標換算                  | 滑鼠點在投影片哪裡 |                   要 |                   要 |
| 位移量換算                  | 滑鼠總共拖了多少  |                  不要 |                   要 |
| 矩形換算的 `left / top`     | 矩形從哪裡開始   |                   要 |                   要 |
| 矩形換算的 `width / height` | 矩形有多大     |                  不要 |                   要 |

---

## 這章真正要記住的版本

```text
點座標：
  要回答「滑鼠點在投影片裡哪裡？」
  所以要扣 viewportRect，再除以 canvasScale。

位移量：
  要回答「滑鼠拖了多少？」
  因為兩個滑鼠點相減已經是距離，
  所以不用扣 viewportRect，只要除以 canvasScale。

矩形：
  left / top 是起點，所以用點座標換算；
  width / height 是距離，所以用位移量換算。
```

最核心的一句話是：

```text
扣不扣 viewportRect，看你是不是在算「位置」；
除不除 canvasScale，看你是不是要從「畫面尺度」回到「邏輯尺度」。
```

# 五、為什麼互動座標必須反向映射

要理解互動座標為什麼要反向映射，先抓住一件事：

```text
PPTist 的元素資料不是存畫面座標，
而是存邏輯畫布座標。
```

例如一個元素資料是：

```ts
{
  left: 100,
  top: 50,
  width: 200,
  height: 80,
}
```

這些數字的意思不是：

```text
畫面上真的距離左邊 100px、上面 50px
```

而是：

```text
在 1000 × 562.5 這張邏輯投影片裡，
元素位於 left = 100、top = 50。
```

所以不管目前畫布被放大、縮小，元素資料本身都應該維持在邏輯座標裡。

---

## 1. 顯示方向：邏輯資料被放大成畫面結果

PPTist 顯示畫布時，流程是：

```text
邏輯畫布資料
-> 乘上 canvasScale
-> 顯示成畫面結果
```

也就是：

```text
邏輯座標 × canvasScale = 畫面座標
邏輯距離 × canvasScale = 畫面距離
```

例如：

```ts
canvasScale = 2
```

代表畫布被放大 2 倍顯示。

那麼邏輯世界裡的：

```ts
left = 100
width = 200
```

顯示到畫面上會變成：

```ts
畫面 left = 100 × 2 = 200
畫面 width = 200 × 2 = 400
```

所以使用者看到的是：

```text
已經被 canvasScale 放大後的結果。
```

---

## 2. 互動方向：畫面操作要縮回邏輯資料

問題來了。

使用者拖曳、點擊、框選時，滑鼠事件拿到的是：

```text
畫面上的位置或距離
```

但 PPTist 最後要更新的是：

```text
邏輯畫布裡的元素資料
```

所以互動時必須把顯示流程倒過來：

```text
畫面操作
-> 除以 canvasScale
-> 回到邏輯資料
```

也就是：

```text
畫面座標 ÷ canvasScale = 邏輯座標
畫面距離 ÷ canvasScale = 邏輯距離
```

這就是「反向映射」。

它不是額外技巧，而是因為：

```text
顯示時有乘上 canvasScale；
互動寫回資料時，就必須除回來。
```

---

## 3. 用拖曳元素來看最清楚

假設目前：

```ts
canvasScale = 2
```

畫面被放大 2 倍。

現在使用者把元素往右拖了：

```ts
40px
```

這個 `40px` 是畫面距離。

如果你直接寫回：

```ts
element.left = element.left + 40
```

就會錯。

因為在 `canvasScale = 2` 的情況下：

```text
邏輯世界移動 20px
畫面上才會看起來移動 40px
```

所以正確寫法應該是：

```ts
element.left = element.left + 40 / 2
```

也就是：

```ts
element.left = element.left + 20
```

這樣下一次畫面顯示時：

```ts
20 × 2 = 40
```

使用者才會看到元素真的移動了 `40px`。

---

## 4. 如果不反向映射會發生什麼事

假設你不除以 `canvasScale`，直接把畫面距離寫回元素資料。

在：

```ts
canvasScale = 2
```

時，使用者畫面上拖了：

```ts
40px
```

你直接寫入邏輯資料：

```ts
element.left += 40
```

下一次畫面顯示時，這個 `40` 又會被乘上 `canvasScale`：

```ts
40 × 2 = 80
```

結果使用者明明只拖了 `40px`，元素卻看起來跑了 `80px`。

這就是錯誤的根源：

```text
把已經放大後的畫面距離，
又當成邏輯距離存回資料，
下一次顯示時又被放大一次。
```

所以互動座標一定要反向映射。

---

## 5. 更精準的判斷標準

可以先記這句：

```text
如果資料來源是「畫面事件座標」或「畫面拖曳距離」，
而結果要寫回「邏輯資料」，
就要除以 canvasScale。
```

這句比下面這句更精準：

```text
寫回元素資料就一定要除以 canvasScale。
```

因為不一定。

例如使用者在屬性面板直接輸入：

```ts
left = 100
```

這個 `100` 本來就可以被設計成邏輯座標。

那它已經是資料世界的值，不是滑鼠畫面距離，所以不需要再除以 `canvasScale`。

真正要判斷的是：

```text
輸入來源是不是畫面座標？
輸出目標是不是邏輯資料？
```

---

## 6. 這章真正要記住的版本

```text
PPTist 顯示時：
邏輯資料 × canvasScale = 畫面結果

PPTist 互動時：
畫面操作 ÷ canvasScale = 邏輯資料
```

為什麼要反向？

```text
因為滑鼠事件拿到的是縮放後的畫面結果，
但元素資料要存回未縮放的邏輯畫布。
```

如果不反向映射，就會變成：

```text
把畫面距離直接存成邏輯距離，
下一次顯示時又被 canvasScale 放大一次，
導致元素移動、縮放、框選結果全部偏掉。
```

最核心一句話是：

```text
顯示時乘出去，互動時除回來。
```

---

# 六、從 `Canvas/index.vue` 看整個互動入口

這一章要先釐清一件事：

```text
Canvas/index.vue 不是所有互動座標公式的實作檔案，
而是整個畫布互動系統的「組裝入口」。
```

它做的事情主要有三個：

```text
1. 建立畫布 DOM 結構
2. 準備互動換算需要的共同資料
3. 把不同互動行為交給不同 hook
```

所以看 `Canvas/index.vue` 時，不要期待所有拖曳、縮放、旋轉細節都寫在這裡。

你真正要看的是：

```text
它把哪些東西組合在一起，
讓後面的互動 hook 可以完成座標換算。
```

---

## 1. 為什麼 `Canvas/index.vue` 是互動組裝入口

所有畫布互動最後都會需要三類資訊：

```text
1. 目前要操作哪些元素資料
2. 目前投影片在畫面上的位置
3. 目前畫布縮放倍率是多少
```

對回 source，就是：

```text
elementList：
  目前投影片上的元素資料。

viewportRef：
  取得 viewport 在畫面上的實際位置。

canvasScale：
  把畫面座標或畫面距離換回邏輯座標的倍率。
```

所以 `Canvas/index.vue` 的核心價值不是「自己完成所有互動計算」，而是：

```text
它同時拿得到 elementList、viewportRef、canvasScale，
所以它可以把這些共同上下文交給不同互動 hook。
```

---

## 2. 先看它建立的畫布 DOM 結構

`Canvas/index.vue` 裡最重要的結構可以先簡化成：

```vue
<div class="viewport-wrapper">
  <div class="operates">
    <!-- 操作框、對齊線、背景等 -->
  </div>

  <div class="viewport" ref="viewportRef">
    <!-- 真正的元素內容 -->
  </div>
</div>
```

這裡可以先這樣分：

```text
viewport-wrapper：
  縮放後的投影片外框。

viewport：
  真正承載元素內容的邏輯投影片層。

viewportRef：
  指向 viewport，用來取得投影片內容層目前在畫面上的位置。
```

這很重要，因為元素資料的 `left / top` 是相對於：

```text
viewport 這張投影片內容層的左上角
```

不是相對於整個 `canvas`，也不是相對於瀏覽器左上角。

所以後面互動換算才會需要：

```ts
viewportRef.value.getBoundingClientRect()
```

---

## 3. `viewport-wrapper` 和 `viewport` 都和 `canvasScale` 有關，但責任不同

`viewport-wrapper` 會使用：

```ts
width: viewportStyles.width * canvasScale + 'px'
height: viewportStyles.height * canvasScale + 'px'
left: viewportStyles.left + 'px'
top: viewportStyles.top + 'px'
```

它負責回答的是：

```text
縮放後的投影片外框，
現在在畫面上要多大、放在哪裡？
```

也就是：

```text
wrapper 管「畫面上的外框佔位」。
```

而 `viewport` 會使用：

```ts
transform: `scale(${canvasScale})`
```

它負責回答的是：

```text
固定邏輯畫布裡的內容，
現在要用幾倍顯示出來？
```

也就是：

```text
viewport 管「內容視覺縮放」。
```

所以這裡不是縮放兩次。

更精準地說：

```text
viewport-wrapper：
  用縮放後尺寸撐出畫面上的投影片外框。

viewport：
  把邏輯內容用 canvasScale 顯示在這個外框裡。
```

---

## 4. 為什麼這個 DOM 結構和互動換算有關

顯示時，PPTist 是這樣做的：

```text
邏輯元素資料
-> 放在 viewport 裡
-> viewport 用 canvasScale 顯示
-> wrapper 撐出縮放後的畫面外框
```

互動時剛好反過來：

```text
滑鼠畫面座標
-> 透過 viewportRef 找到 viewport 左上角
-> 扣掉 viewport 左上角
-> 除以 canvasScale
-> 回到 viewport 裡的邏輯座標
```

也就是：

```ts
const viewportRect = viewportRef.value.getBoundingClientRect()

const logicX = (pageX - viewportRect.x) / canvasScale.value
const logicY = (pageY - viewportRect.y) / canvasScale.value
```

所以 `Canvas/index.vue` 裡的 DOM 結構不是只影響畫面顯示。

它同時也決定了：

```text
互動座標要以哪一層 DOM 作為反推基準。
```

在 PPTist 裡，這個基準就是：

```text
viewport
```

---

## 5. Canvas 把不同互動交給不同 hook

`Canvas/index.vue` 會建立或呼叫這些 hook：

```ts
const viewportRef = useTemplateRef<HTMLElement>('viewportRef')

const { dragViewport, viewportStyles } = useViewportSize(canvasRef)

const {
  mouseSelection,
  mouseSelectionVisible,
  mouseSelectionQuadrant,
  updateMouseSelection,
} = useMouseSelection(elementList, viewportRef)

const { dragElement } = useDragElement(
  elementList,
  alignmentLines,
  canvasScale
)

const { dragLineElement } = useDragLineElement(elementList)

const {
  scaleElement,
  scaleMultiElement,
} = useScaleElement(elementList, alignmentLines, canvasScale)

const { rotateElement } = useRotateElement(
  elementList,
  viewportRef,
  canvasScale
)

const { rotateGroupElement } = useRotateGroupElement(
  elementList,
  viewportRef,
  canvasScale
)

const { moveShapeKeypoint } = useMoveShapeKeypoint(
  elementList,
  canvasScale
)
```

這段不要只是看成一堆 hook。

要看成一張分工表。

---

## 6. 這些 hook 各自需要什麼

### `useMouseSelection(elementList, viewportRef)`

它需要：

```text
elementList：
  用來判斷哪些元素被框選到。

viewportRef：
  用來取得 viewport 位置，把框選範圍換回邏輯座標。
```

框選時重點是：

```text
畫面框選範圍
-> 透過 viewportRef 換成邏輯框選範圍
-> 和 elementList 裡的元素範圍比較
```

---

### `useDragElement(elementList, alignmentLines, canvasScale)`

它需要：

```text
elementList：
  知道要移動哪些元素。

canvasScale：
  把滑鼠畫面位移換成邏輯位移。

alignmentLines：
  拖曳時處理吸附對齊線。
```

拖曳元素時重點是：

```text
畫面位移 / canvasScale
= 元素 left / top 的邏輯變化量
```

---

### `useScaleElement(elementList, alignmentLines, canvasScale)`

它需要：

```text
elementList：
  知道要縮放哪些元素。

canvasScale：
  把操作點畫面位移換成邏輯尺寸變化。

alignmentLines：
  縮放時處理吸附對齊線。
```

縮放元素時重點是：

```text
操作點拖曳距離
-> 必要時處理旋轉方向
-> 除以 canvasScale
-> 更新 left / top / width / height
```

---

### `useRotateElement(elementList, viewportRef, canvasScale)`

它需要：

```text
elementList：
  取得元素中心點。

viewportRef：
  取得 viewport 位置，把滑鼠點換成相對於投影片的位置。

canvasScale：
  把滑鼠畫面座標換成邏輯座標。
```

旋轉元素時重點是：

```text
滑鼠畫面點
-> 扣 viewportRect
-> 除以 canvasScale
-> 得到邏輯滑鼠點
-> 和元素中心點算角度
```

---

### `useMoveShapeKeypoint(elementList, canvasScale)`

它需要：

```text
elementList：
  取得 shape 元素資料與 keypoints。

canvasScale：
  把拖曳距離換成邏輯距離。
```

調整 shape keypoint 時重點是：

```text
畫面拖曳距離 / canvasScale
= keypoint 的邏輯變化量
```

---

### `useDragLineElement(elementList)`

這個比較特別。

它從外面只接收：

```text
elementList
```

但它內部會自己從 `mainStore` 取：

```text
canvasScale
```

所以不要誤會成它沒有使用縮放倍率。

線條端點拖曳時重點是：

```text
畫面拖曳距離 / canvasScale
= 線條端點或控制點的邏輯變化量
```

---

### `useViewportSize(canvasRef)`

它負責的是整張畫布外框的顯示位置與大小。

其中 `dragViewport()` 比較特別，因為它拖曳的是整張投影片在畫面中的位置。

所以它改的是：

```text
viewportLeft / viewportTop
```

不是元素邏輯資料。

因此拖曳整張畫布時：

```text
不需要除以 canvasScale
```

---

## 7. 這章真正要看懂的主線

`Canvas/index.vue` 的主線可以整理成：

```text
第一步：
建立 viewport-wrapper 和 viewport，
讓畫布有明確的顯示外框與內容層。

第二步：
用 viewportRef 指向 viewport，
讓互動 hook 可以取得投影片內容層在畫面上的位置。

第三步：
從 store 取得 canvasScale，
讓互動 hook 可以把畫面座標換回邏輯座標。

第四步：
取得 elementList，
讓互動 hook 可以讀取與更新元素資料。

第五步：
把不同互動行為分發給不同 hook：
框選、拖曳、縮放、旋轉、shape keypoint、線條端點、拖曳畫布。
```

也就是：

```text
Canvas/index.vue 本身像一個中控台。

它不負責寫完所有互動公式，
但它負責把「畫布 DOM」、「縮放倍率」、「元素資料」和「互動 hook」接起來。
```

---

## 8. 這章真正要記住的版本

```text
Canvas/index.vue 是畫布互動系統的組裝入口。

它準備三個最重要的共同上下文：

1. viewportRef
   用來取得投影片內容層目前在畫面上的位置。

2. canvasScale
   用來把畫面座標或畫面距離換回邏輯座標。

3. elementList
   用來讀取與更新投影片元素資料。
```

然後它把不同互動交給不同 hook：

```text
useMouseSelection：
  框選元素。

useDragElement：
  拖曳元素。

useScaleElement：
  縮放元素。

useRotateElement / useRotateGroupElement：
  旋轉元素或群組。

useMoveShapeKeypoint：
  調整 shape 控制點。

useDragLineElement：
  拖曳線條端點或控制點。

useViewportSize.dragViewport：
  拖曳整張畫布。
```

最核心的一句話是：

```text
Canvas/index.vue 把顯示結構和互動邏輯接起來：
viewportRef 解決「投影片在哪裡」；
canvasScale 解決「畫面尺度怎麼回到邏輯尺度」；
elementList 解決「最後要改哪些元素資料」。
```

# 七、案例一：雙擊空白處新增文字

## 這個案例在解什麼問題

雙擊新增文字是最單純的「點座標換算」。

使用者做的是：

```text
在畫面上的某一點雙擊
```

但 PPTist 要建立的是一個文字元素資料：

```ts
{
  left,
  top,
  width,
  height,
}
```

其中：

```ts
left
top
```

不是畫面座標，而是**邏輯投影片座標**。

所以這個案例真正要解的是：

```text
使用者雙擊的畫面位置，
要如何換成文字元素在邏輯投影片裡的 left / top？
```

---

## 為什麼不能直接用 `e.pageX / e.pageY`

滑鼠事件給的是：

```ts
e.pageX
e.pageY
```

它代表的是：

```text
滑鼠在整個頁面中的位置
```

但文字元素的 `left / top` 要的是：

```text
滑鼠在投影片裡的位置
```

所以第一步要先知道：

```text
投影片左上角目前在畫面哪裡？
```

這就是：

```ts
const viewportRect = viewportRef.value.getBoundingClientRect()
```

的作用。

---

## 換算流程

雙擊新增文字的換算流程可以拆成兩步。

第一步，先取得滑鼠相對於投影片左上角的畫面位置：

```ts
const relativeX = e.pageX - viewportRect.x
const relativeY = e.pageY - viewportRect.y
```

這時候的 `relativeX / relativeY` 還是畫面座標。

第二步，再除以 `canvasScale`，換回邏輯座標：

```ts
const left = relativeX / canvasScale.value
const top = relativeY / canvasScale.value
```

合起來就是：

```ts
const left = (e.pageX - viewportRect.x) / canvasScale.value
const top = (e.pageY - viewportRect.y) / canvasScale.value
```

---

## 對回 `Canvas/index.vue`

`Canvas/index.vue` 的 `handleDblClick()` 可以先理解成這條流程：

```text
1. 使用者在畫布上雙擊
2. 確認目前不是在操作既有元素
3. 取得 viewport 的畫面位置
4. 把雙擊點換成邏輯座標
5. 用這個邏輯座標建立文字元素
```

核心程式可以壓成：

```ts
const viewportRect = viewportRef.value.getBoundingClientRect()

const left = (e.pageX - viewportRect.x) / canvasScale.value
const top = (e.pageY - viewportRect.y) / canvasScale.value

createTextElement({
  left,
  top,
  width: 200 / canvasScale.value,
  height: 0,
})
```

---

## 為什麼 `width` 也要除以 `canvasScale`

這裡最容易被忽略的是：

```ts
width: 200 / canvasScale.value
```

它的意思是：

```text
希望新文字框在畫面上看起來大約是 200px 寬。
```

但元素資料裡的 `width` 是邏輯寬度，不是畫面寬度。

假設目前：

```ts
canvasScale = 2
```

如果你直接寫：

```ts
width: 200
```

那顯示到畫面上會變成：

```ts
200 * 2 = 400
```

文字框看起來就會變成 `400px` 寬。

所以 PPTist 寫成：

```ts
width: 200 / 2
```

也就是：

```ts
width: 100
```

下一次畫面顯示時：

```ts
100 * 2 = 200
```

畫面上才會接近預期的 `200px` 寬。

所以這裡可以整理成：

```text
left / top：
  把雙擊點的畫面位置換成邏輯位置。

width：
  把預設畫面寬度換成邏輯寬度。
```

---

## 這個案例真正要記住的版本

```text
雙擊新增文字時，
滑鼠點擊位置來自畫面，
但文字元素資料要存在邏輯畫布裡。

所以：

left = (滑鼠 X - viewport 左上角 X) / canvasScale
top  = (滑鼠 Y - viewport 左上角 Y) / canvasScale
```

而預設寬度也是同樣邏輯：

```text
希望畫面上看起來是 200px，
所以資料裡要存 200 / canvasScale。
```

最核心的一句話是：

```text
雙擊新增文字不只是「取得滑鼠位置」，
而是把「畫面上的雙擊點」轉成「邏輯投影片裡的新文字位置」。
```

---

# 八、案例二：拖曳建立元素

## 這個案例在解什麼問題

拖曳建立元素和雙擊新增文字不一樣。

雙擊新增文字只需要處理一個點：

```text
滑鼠雙擊點
-> 文字元素 left / top
```

但拖曳建立元素要處理的是一個矩形：

```text
滑鼠按下的位置
+
滑鼠放開的位置
=
使用者拖出來的畫面選區
```

PPTist 最後要建立的是元素資料：

```ts
{
  left,
  top,
  width,
  height,
}
```

而這四個值都必須是**邏輯畫布裡的資料**，不是畫面座標。

所以這個案例真正要解的是：

```text
使用者在畫面上拖出來的選區，
要如何換成邏輯畫布裡的新元素矩形？
```

---

## 拖曳建立元素其實分成兩段

這個流程不要一開始就看公式。

先把它拆成兩段：

```text
第一段：
ElementCreateSelection.vue
負責收集滑鼠拖曳位置，並在畫面上顯示選區。

第二段：
useInsertFromCreateSelection.ts
負責把畫面選區換成邏輯元素資料。
```

也就是：

```text
ElementCreateSelection.vue
= 畫面層

useInsertFromCreateSelection.ts
= 資料轉換層
```

這個分工很重要。

`ElementCreateSelection.vue` 本身不是在決定元素真正的 `left / top / width / height`。

它主要是在做：

```text
使用者拖到哪裡？
畫面上要顯示多大的建立選區？
```

真正把選區轉成元素資料的是後面的 `useInsertFromCreateSelection.ts`。

---

## 第一步：先取得拖曳的起點與終點

使用者按下滑鼠時，`ElementCreateSelection.vue` 會記錄起點：

```ts
startPageX
startPageY
```

滑鼠移動時，會更新目前位置：

```ts
currentPageX
currentPageY
```

最後 mouseup 時，丟出：

```ts
emit('created', { start, end })
```

這裡的 `start` 和 `end` 都還是**畫面座標**。

它們只是表示：

```text
使用者從哪個畫面位置開始拖
使用者在哪個畫面位置結束拖
```

它們還不是邏輯座標，也還不能直接變成元素資料。

---

## 第二步：先把 start / end 整理成標準畫面矩形

這一步很重要。

拖曳建立元素時，使用者不一定永遠從左上拖到右下。

可能有四種方向：

```text
左上 -> 右下
右下 -> 左上
左下 -> 右上
右上 -> 左下
```

所以不能直接把 `startPageX / startPageY` 當成 `left / top`。

要先整理出：

```ts
const minX = Math.min(start.x, end.x)
const maxX = Math.max(start.x, end.x)
const minY = Math.min(start.y, end.y)
const maxY = Math.max(start.y, end.y)
```

這四個值代表的是畫面選區的標準矩形：

```text
minX / minY：
  畫面矩形的左上角

maxX / maxY：
  畫面矩形的右下角
```

所以這一步是在做：

```text
不管使用者從哪個方向拖，
都先整理成一個標準的畫面矩形。
```

---

## 第三步：把畫面矩形換成邏輯元素矩形

整理完畫面矩形後，才開始做座標換算。

畫面矩形有四個資訊：

```text
minX
minY
maxX
maxY
```

但元素資料需要的是：

```text
left
top
width
height
```

所以要轉成：

```ts
const left = (minX - viewportRect.x) / canvasScale.value
const top = (minY - viewportRect.y) / canvasScale.value
const width = (maxX - minX) / canvasScale.value
const height = (maxY - minY) / canvasScale.value
```

這裡要分成兩種理解。

---

## `left / top` 是點座標換算

`left / top` 問的是：

```text
新元素的左上角，在邏輯投影片裡的哪裡？
```

所以它要先扣掉 viewport 左上角：

```ts
minX - viewportRect.x
minY - viewportRect.y
```

這一步得到的是：

```text
新元素左上角相對於投影片左上角的畫面位置
```

接著再除以 `canvasScale`：

```ts
left = (minX - viewportRect.x) / canvasScale.value
top = (minY - viewportRect.y) / canvasScale.value
```

這樣才會得到邏輯座標。

---

## `width / height` 是距離量換算

`width / height` 問的是：

```text
這個新元素有多寬、多高？
```

它不是在問「它在哪裡」，而是在問「它的尺寸是多少」。

所以不用扣 `viewportRect`。

因為：

```ts
maxX - minX
maxY - minY
```

本身已經是畫面上的寬高距離。

只要除以 `canvasScale`，就能換成邏輯尺寸：

```ts
width = (maxX - minX) / canvasScale.value
height = (maxY - minY) / canvasScale.value
```

一句話記：

```text
left / top：
  是位置，所以要扣 viewportRect，再除以 canvasScale。

width / height：
  是尺寸，所以只要除以 canvasScale。
```

---

## 用一個例子看懂

假設目前：

```ts
viewportRect.x = 300
viewportRect.y = 120
canvasScale = 2
```

使用者從畫面位置：

```ts
start = { x: 800, y: 420 }
```

拖到：

```ts
end = { x: 1000, y: 520 }
```

那畫面矩形是：

```ts
minX = 800
minY = 420
maxX = 1000
maxY = 520
```

先算元素左上角：

```ts
left = (800 - 300) / 2 = 250
top = (420 - 120) / 2 = 150
```

再算元素寬高：

```ts
width = (1000 - 800) / 2 = 100
height = (520 - 420) / 2 = 50
```

所以最後建立出的邏輯元素資料會是：

```ts
{
  left: 250,
  top: 150,
  width: 100,
  height: 50,
}
```

畫面顯示時再乘回 `canvasScale = 2`，它就會顯示成使用者剛剛拖出來的畫面選區大小。

---

## 對回 source 的責任分工

這一段可以這樣讀：

```text
ElementCreateSelection.vue
```

負責：

```text
1. 記錄滑鼠起點 startPageX / startPageY
2. 追蹤滑鼠目前位置 currentPageX / currentPageY
3. 在畫面上顯示正在拖曳的選區
4. mouseup 時 emit created，把 start / end 丟出去
```

而：

```text
useInsertFromCreateSelection.ts
```

負責：

```text
1. 接收 start / end
2. 用 min / max 整理成標準畫面矩形
3. 取得 viewportRect
4. 把畫面矩形轉成邏輯元素矩形
5. 根據建立類型產生文字、shape、image、line 等元素資料
```

所以可以這樣記：

```text
ElementCreateSelection.vue
= 先讓使用者在畫面上拖出一個選區

useInsertFromCreateSelection.ts
= 再把這個畫面選區轉成真正要存進 slides 的元素資料
```

---

## 這個案例真正要記住的版本

```text
拖曳建立元素不是只算一個點，
而是把使用者拖出的「畫面矩形」轉成「邏輯元素矩形」。
```

完整流程是：

```text
1. 記錄拖曳起點與終點
2. 用 minX / maxX / minY / maxY 整理成標準畫面矩形
3. left / top 用點座標換算
4. width / height 用距離量換算
5. 建立元素資料
```

最核心的一句話是：

```text
拖曳建立元素 = 畫面選區矩形 → 邏輯元素矩形。
```

其中：

```text
left / top 決定「新元素從哪裡開始」；
width / height 決定「新元素有多大」。
```

# 九、案例三：滑鼠框選元素

## 這個案例在解什麼問題

滑鼠框選看起來像只是：

```text
使用者在畫面上拉出一個框
```

但對 PPTist 來說，真正要解的問題不是「怎麼畫出框」，而是：

```text
使用者在畫面上拉出的框，
對應到邏輯投影片裡是哪個範圍？

這個邏輯範圍裡，
有哪些元素應該被選中？
```

所以滑鼠框選可以拆成兩層：

```text
第一層：座標換算
畫面框選範圍 -> 邏輯選區 mouseSelection

第二層：命中判斷
邏輯選區 mouseSelection -> selected element ids
```

一句話先記：

```text
框選不是拿畫面上的框直接選元素；
而是先把畫面框轉成邏輯框，再用邏輯框去命中元素。
```

---

## 為什麼不能直接用畫面上的框去選元素

PPTist 的元素資料是邏輯座標，例如：

```ts
element.left
element.top
element.width
element.height
```

這些值不是畫面上的實際 px，而是投影片邏輯世界裡的資料。

例如：

```ts
element.left = 100
element.top = 50
element.width = 200
element.height = 80
```

意思是：

```text
元素在邏輯投影片中：
x 從 100 到 300
y 從 50 到 130
```

但滑鼠事件拿到的是畫面座標，例如：

```ts
startPageX
startPageY
currentPageX
currentPageY
```

所以如果你直接拿畫面上的框去比對元素資料，就會變成：

```text
畫面座標的框
vs
邏輯座標的元素
```

這兩個不是同一個座標系，結果一定會偏。

所以框選前一定要先做這件事：

```text
畫面框選範圍
-> 換算成邏輯選區
-> 再和元素邏輯範圍比較
```

---

## 先把框選流程拆成 5 步

滑鼠框選可以先不要看 source 細節，先看流程：

```text
1. mousedown：
   記錄滑鼠按下的位置

2. mousemove：
   根據目前滑鼠位置，算出框選範圍

3. 座標換算：
   把畫面上的框選範圍換成邏輯選區

4. mouseup：
   用邏輯選區去比對每個元素的邏輯範圍

5. 更新 activeElementIdList：
   把命中的元素設為選中
```

對回 source，可以這樣理解：

```text
useMouseSelection.ts
= 處理「畫面框選 -> 邏輯選區 -> 選中元素 ID」的 hook
```

---

## 第一步：mousedown 時，先記錄框選起點

使用者按下滑鼠時，PPTist 會先記錄滑鼠起點。

這個起點一開始是畫面座標：

```ts
startPageX
startPageY
```

但框選最後要拿來和元素資料比較，所以起點需要轉成邏輯座標。

換算方式是點座標換算：

```ts
const startLogicX = (startPageX - viewportRect.x) / canvasScale.value
const startLogicY = (startPageY - viewportRect.y) / canvasScale.value
```

這裡分成兩步理解：

```text
startPageX - viewportRect.x
= 滑鼠起點相對於投影片左上角的畫面位置

/ canvasScale
= 把畫面位置縮回邏輯畫布位置
```

所以 mousedown 之後，可以先得到：

```ts
mouseSelection.left = startLogicX
mouseSelection.top = startLogicY
mouseSelection.width = 0
mouseSelection.height = 0
```

這時候框還沒拉開，所以寬高都是 0。

---

## 第二步：mousemove 時，計算滑鼠拖了多少

滑鼠移動時，會取得目前滑鼠位置：

```ts
currentPageX
currentPageY
```

接著會計算滑鼠從起點到現在拖了多少：

```ts
currentPageX - startPageX
currentPageY - startPageY
```

這是畫面距離。

所以要除以 `canvasScale`，變成邏輯距離：

```ts
const moveX = (currentPageX - startPageX) / canvasScale.value
const moveY = (currentPageY - startPageY) / canvasScale.value
```

這裡不要扣 `viewportRect`。

原因是：

```text
currentPageX - startPageX
本身已經是兩個滑鼠點之間的距離。

距離不需要知道投影片左上角在哪裡。
```

所以框選裡同時有兩種換算：

```text
起點 left / top：
  是位置，所以要扣 viewportRect，再除以 canvasScale。

拖曳距離 width / height：
  是距離，所以不用扣 viewportRect，只要除以 canvasScale。
```

---

## 第三步：為什麼要判斷拖曳方向

這是框選最容易卡住的地方。

因為使用者不一定都從左上角往右下角拖。

他可能有四種拖法：

```text
1. 往右下拖
2. 往左下拖
3. 往左上拖
4. 往右上拖
```

但元素命中判斷需要的是一個標準矩形：

```text
selectionLeft
selectionTop
selectionRight
selectionBottom
```

也就是：

```text
不管使用者怎麼拖，
最後都要整理成：

左邊界在哪裡？
上邊界在哪裡？
右邊界在哪裡？
下邊界在哪裡？
```

這就是為什麼 source 裡會有類似：

```ts
mouseSelectionQuadrant
```

它的目的不是單純記錄方向而已，而是幫助系統判斷：

```text
目前這個框選矩形的真正 left / top 應該在哪裡。
```

---

## 用數字看懂拖曳方向

假設 mousedown 的邏輯起點是：

```ts
startLogicX = 250
startLogicY = 150
```

也就是使用者從邏輯座標：

```text
(250, 150)
```

開始拖。

---

### 情況一：往右下拖

假設使用者往右下拖，換算後得到：

```ts
moveX = 100
moveY = 50
```

因為是往右下，所以框選範圍就是：

```ts
left = startLogicX
top = startLogicY
width = moveX
height = moveY
```

也就是：

```ts
left = 250
top = 150
width = 100
height = 50
```

這個選區範圍是：

```text
x: 250 -> 350
y: 150 -> 200
```

這很好理解。

---

### 情況二：往左上拖

但如果使用者往左上拖，換算後可能是：

```ts
moveX = -50
moveY = -50
```

這時候不能直接寫：

```ts
left = 250
top = 150
width = -50
height = -50
```

因為一般矩形資料不會希望寬高是負數。

真正的框選範圍應該是：

```text
x: 200 -> 250
y: 100 -> 150
```

所以要整理成：

```ts
left = startLogicX + moveX
top = startLogicY + moveY
width = Math.abs(moveX)
height = Math.abs(moveY)
```

代入數字：

```ts
left = 250 + (-50) = 200
top = 150 + (-50) = 100
width = 50
height = 50
```

所以最後選區是：

```ts
{
  left: 200,
  top: 100,
  width: 50,
  height: 50,
}
```

這就是為什麼框選不能只看：

```ts
width = currentPageX - startPageX
height = currentPageY - startPageY
```

還要看使用者是往哪個方向拖。

---

## 四種方向整理表

可以用這張表記：

| 拖曳方向 | `moveX` | `moveY` | 選區 left          | 選區 top           | 選區 width     | 選區 height    |
| ---- | ------: | ------: | ---------------- | ---------------- | ------------ | ------------ |
| 右下   |       正 |       正 | `startX`         | `startY`         | `abs(moveX)` | `abs(moveY)` |
| 左下   |       負 |       正 | `startX + moveX` | `startY`         | `abs(moveX)` | `abs(moveY)` |
| 左上   |       負 |       負 | `startX + moveX` | `startY + moveY` | `abs(moveX)` | `abs(moveY)` |
| 右上   |       正 |       負 | `startX`         | `startY + moveY` | `abs(moveX)` | `abs(moveY)` |

這張表的核心是：

```text
width / height 永遠用正數；
left / top 要根據拖曳方向重新調整。
```

所以 `mouseSelectionQuadrant` 可以理解成：

```text
它幫助 source 判斷：
目前使用者是從哪個方向拉框，
因此真正的 left / top 要怎麼修正。
```

---

## 第四步：mouseup 時，用邏輯選區比對元素

mousemove 只是更新框選範圍。

真正決定哪些元素被選中，通常是在 mouseup。

這時候 PPTist 會拿到一個邏輯選區：

```ts
mouseSelection.value.left
mouseSelection.value.top
mouseSelection.value.width
mouseSelection.value.height
```

可以先轉成四個邊界：

```ts
const selectionLeft = mouseSelection.left
const selectionTop = mouseSelection.top
const selectionRight = mouseSelection.left + mouseSelection.width
const selectionBottom = mouseSelection.top + mouseSelection.height
```

每個元素也可以轉成四個邊界：

```ts
const elementLeft = element.left
const elementTop = element.top
const elementRight = element.left + element.width
const elementBottom = element.top + element.height
```

接下來就可以比較：

```text
邏輯選區
vs
元素邏輯範圍
```

注意，這兩邊現在都是邏輯座標，所以可以正確比較。

---

## 第五步：判斷元素是否被框到

框選有兩種常見命中規則。

---

### 規則一：完整包含

完整包含的意思是：

```text
元素必須整個落在框選範圍裡，才算被選中。
```

條件可以理解成：

```ts
const fullyContained =
  elementLeft >= selectionLeft &&
  elementRight <= selectionRight &&
  elementTop >= selectionTop &&
  elementBottom <= selectionBottom
```

也就是：

```text
元素左邊界 >= 選區左邊界
元素右邊界 <= 選區右邊界
元素上邊界 >= 選區上邊界
元素下邊界 <= 選區下邊界
```

這代表元素完全被包在選區裡。

---

### 規則二：有交集就算

有交集的意思是：

```text
元素只要碰到框選範圍，就可能被選中。
```

條件可以理解成：

```ts
const intersected =
  elementRight >= selectionLeft &&
  elementLeft <= selectionRight &&
  elementBottom >= selectionTop &&
  elementTop <= selectionBottom
```

這代表：

```text
元素沒有完全在選區外面，
所以它和選區有重疊。
```

---

## Ctrl / Shift 的作用

`useMouseSelection.ts` 裡會判斷：

```ts
ctrlOrShiftKeyActive
```

這個不是座標換算。

它是命中規則的切換。

可以先這樣理解：

```text
沒有按 Ctrl / Shift：
  通常偏向完整包含。
  元素要完整落在選區內，才算被選中。

按住 Ctrl / Shift：
  通常偏向交集判斷。
  元素只要和選區有交集，也可能被選中。
```

所以框選有兩層邏輯，不能混在一起：

```text
第一層：座標換算
把畫面拖曳轉成邏輯選區。

第二層：命中規則
決定元素要完整包含才算選中，還是有交集就算選中。
```

你原本筆記裡也有提到 Ctrl / Shift 會影響完整包含或交集判斷，這裡建議把它放在「命中規則」下面講，會比較不容易混。

---

## 和拖曳建立元素的差異

滑鼠框選和拖曳建立元素很像。

因為它們前半段都在做：

```text
畫面矩形 -> 邏輯矩形
```

但兩者目的不同。

---

### 拖曳建立元素

拖曳建立元素是：

```text
畫面矩形
-> 邏輯元素矩形
-> 建立新元素
```

最後會產生：

```ts
{
  left,
  top,
  width,
  height,
}
```

這些值會寫進新元素資料。

---

### 滑鼠框選元素

滑鼠框選元素是：

```text
畫面矩形
-> 邏輯選區矩形
-> 比對既有元素
-> 產生 activeElementIdList
```

它不會建立新元素。

它最後要得到的是：

```ts
activeElementIdList
```

也就是：

```text
哪些元素被選中了。
```

所以可以這樣記：

```text
拖曳建立元素：
  畫面矩形 -> 邏輯矩形 -> 建立元素資料

滑鼠框選元素：
  畫面矩形 -> 邏輯矩形 -> 找出元素 ID
```

---

## 用完整例子走一次

假設目前：

```ts

viewportRect.y = 120
```

使用者在畫面座標按下滑鼠：

```ts
startPageX = 800
startPageY = 420
```

先換成邏輯起點：

```ts
startLogicX = (800 - 300) / 2 = 250
startLogicY = (420 - 120) / 2 = 150
```

所以框選起點是：

```text
邏輯座標 (250, 150)
```

---

### 使用者往右下拖

滑鼠移到：

```ts
currentPageX = 1000
currentPageY = 520
```

畫面位移是：

```ts
currentPageX - startPageX = 200
currentPageY - startPageY = 100
```

換成邏輯位移：

```ts
moveX = 200 / 2 = 100
moveY = 100 / 2 = 50
```

因為是往右下拖，所以選區是：

```ts
mouseSelection = {
  left: 250,

  width: 100,
  height: 50,
}
```

選區範圍是：

```text
x: 250 -> 350
y: 150 -> 200
```

---

### 使用者往左上拖

如果滑鼠改成移到：

```ts

currentPageY = 320
```

畫面位移是：

```ts
currentPageX - startPageX = -100
currentPageY - startPageY = -100
```

換成邏輯位移：

```ts
moveX = -100 / 2 = -50
moveY = -100 / 2 = -50
```

因為是往左上拖，所以選區不是從 `(250, 150)` 往右下長大。

而是要修正成：

```ts
mouseSelection = {
  left: 250 + (-50),
  top: 150 + (-50),
  width: 50,

```

也就是：

```ts
mouseSelection = {
  left: 200,
  top: 100,
  width: 50,
  height: 50,
}
```

選區範圍是：

```text
x: 200 -> 250
y: 100 -> 150
```

這就是框選裡 `mouseSelectionQuadrant` 存在的原因。

## 對回 source 的責任分工

讀 `useMouseSelection.ts` 時，可以按照下面順序看：

```text
1. 它從 mainStore 拿 canvasScale
   用來把畫面距離換成邏輯距離。

2. 它接收 viewportRef
   用來取得 viewportRect，知道投影片左上角在哪裡。

3. mousedown 時
   記錄 startPageX / startPageY，
   並把起點換成邏輯起點。

4. mousemove 時
   用 currentPageX - startPageX 算畫面位移，
   再除以 canvasScale 變成邏輯位移。

5. 根據拖曳方向更新 mouseSelectionQuadrant
   用來修正 left / top。

6. 更新 mouseSelection
   讓畫面上可以看到框選範圍。

7. mouseup 時
   用 mouseSelection 的邏輯範圍
   去比對 elementList 裡每個元素的邏輯範圍。

8. 排除 hiddenElementIdList 裡的元素
   隱藏元素不應該被框選。

9. 根據 Ctrl / Shift
   決定用完整包含還是交集判斷。

10. 最後更新 activeElementIdList
    讓命中的元素變成選中狀態。
```

原本筆記也有整理 `useMouseSelection.ts` 的這些責任，但建議像上面這樣用「事件流程」重排，會更容易讀 source。

另一句也要記：

```text
框選和拖曳建立元素很像，
都是畫面矩形轉邏輯矩形；

差別是：
拖曳建立元素會產生新元素，
滑鼠框選元素會產生選中的元素 ID。
```


| 階段        | 處理的資料                                  | 座標系       |
| --------- | -------------------------------------- | --------- |
| mousedown | `startPageX / startPageY`              | 畫面座標      |
| 起點換算      | `startLogicX / startLogicY`            | 邏輯座標      |
| mousemove | `currentPageX - startPageX`            | 畫面距離      |
| 位移換算      | `moveX / moveY`                        | 邏輯距離      |
| 整理方向      | `mouseSelection.left/top/width/height` | 邏輯選區      |
| mouseup   | `mouseSelection` vs `elementList`      | 邏輯座標比對    |
| 最後結果      | `activeElementIdList`                  | 被選中的元素 ID |

# 十、案例四：拖曳元素

## 這個案例在解什麼問題

拖曳元素時，使用者做的是：

```text
在畫面上按住元素，然後移動滑鼠
```

但 PPTist 要更新的是元素資料：

```ts
element.left
element.top
```

這兩個值是邏輯畫布座標，不是畫面座標。

所以這個案例真正要解的是：

```text
滑鼠在畫面上拖了多少距離，
要如何換成元素在邏輯畫布中的 left / top 變化量？
```

---

## 拖曳元素處理的是「位移量」，不是「點座標」

拖曳元素和雙擊新增文字不一樣。

雙擊新增文字問的是：

```text
滑鼠點在投影片裡哪裡？
```

所以它需要：

```text
扣 viewportRect
```

但拖曳元素問的是：

```text
滑鼠從按下到現在，總共移動了多少？
```

所以它處理的是位移量。

位移量來自：

```ts
currentPageX - startPageX
currentPageY - startPageY
```

這兩個值本身已經是畫面距離。

所以拖曳元素不需要扣：

```ts
viewportRect.x
viewportRect.y
```

它只需要把畫面距離換成邏輯距離：

```ts
moveX = (currentPageX - startPageX) / canvasScale.value
moveY = (currentPageY - startPageY) / canvasScale.value
```

---

## 為什麼要加回元素原始位置

拖曳元素不是直接把 `moveX / moveY` 當成元素位置。

`moveX / moveY` 只是：

```text
這次拖曳造成的位移量
```

真正的新位置應該是：

```text
元素原本的位置 + 這次拖曳的邏輯位移
```

所以公式是：

```ts
targetLeft = elOriginLeft + moveX
targetTop = elOriginTop + moveY
```

這裡的：

```ts
elOriginLeft
elOriginTop
```

代表滑鼠按下那一刻，元素原本的邏輯位置。

所以拖曳元素的完整邏輯是：

```text
記住元素原始位置
-> 計算滑鼠畫面位移
-> 畫面位移 / canvasScale
-> 得到邏輯位移
-> 原始位置 + 邏輯位移
-> 得到元素新位置
```

---

## 對回 `useDragElement.ts`

`useDragElement.ts` 在拖曳開始時，會記錄兩類資訊：

```text
1. 滑鼠起點
2. 元素原始位置
```

可以理解成：

```ts
const startPageX = e.pageX
const startPageY = e.pageY

const elOriginLeft = element.left
const elOriginTop = element.top
```

滑鼠移動時，取得目前滑鼠位置：

```ts
currentPageX
currentPageY
```

先算畫面位移：

```ts
currentPageX - startPageX
currentPageY - startPageY
```

再除以 `canvasScale.value`，換成邏輯位移：

```ts
let moveX = (currentPageX - startPageX) / canvasScale.value
let moveY = (currentPageY - startPageY) / canvasScale.value
```

最後加回元素原始位置：

```ts
let targetLeft = elOriginLeft + moveX
let targetTop = elOriginTop + moveY
```

這裡的主線可以記成：

```text
滑鼠拖多少，不是元素就直接加多少。
滑鼠拖多少，要先除以 canvasScale，才是元素資料該加多少。
```

---

## 為什麼不用 `viewportRef.getBoundingClientRect()`

這裡很容易和雙擊新增文字搞混。

雙擊新增文字要知道：

```text
滑鼠點在投影片裡的哪個位置
```

所以需要：

```ts
pageX - viewportRect.x
```

但拖曳元素只要知道：

```text
滑鼠從起點到現在移動了多少
```

所以只要：

```ts
currentPageX - startPageX
```

因為兩個滑鼠座標相減後，已經把「整個頁面上的絕對位置」消掉了。

例如：

```ts
startPageX = 800
currentPageX = 840
```

代表滑鼠往右移動：

```ts
840 - 800 = 40
```

這個 `40px` 跟 viewport 在畫面哪裡沒有關係。

所以拖曳元素不需要扣 viewport 位置，只需要除以 `canvasScale`。

---

## 用一個數字例子確認

假設目前：

```ts
canvasScale = 2
element.left = 100
element.top = 50
```

滑鼠按下時，元素原始位置是：

```ts
elOriginLeft = 100
elOriginTop = 50
```

使用者在畫面上往右拖了 `40px`，往下拖了 `20px`。

畫面位移是：

```ts
currentPageX - startPageX = 40
currentPageY - startPageY = 20
```

換成邏輯位移：

```ts
moveX = 40 / 2 = 20
moveY = 20 / 2 = 10
```

加回元素原始位置：

```ts
targetLeft = 100 + 20 = 120
targetTop = 50 + 10 = 60
```

所以元素資料最後應該變成：

```ts
element.left = 120
element.top = 60
```

畫面顯示時再乘回 `canvasScale = 2`，使用者看到的位移就是：

```ts
20 * 2 = 40
10 * 2 = 20
```

剛好符合使用者拖曳的畫面距離。

---

## 如果不除以 `canvasScale` 會怎樣

假設：

```ts
canvasScale = 2
```

使用者畫面上往右拖了：

```ts
40px
```

如果你直接寫：

```ts
element.left = element.left + 40
```

那資料層會真的移動 `40`。

但畫面顯示時又會乘上 `canvasScale`：

```ts
40 * 2 = 80
```

結果使用者只拖了 `40px`，元素看起來卻移動了 `80px`。

所以拖曳元素一定要做：

```ts
moveX = 畫面位移 / canvasScale
moveY = 畫面位移 / canvasScale
```

---

## 還有一個 source 細節：吸附對齊可能會修正結果

`useDragElement.ts` 不只是單純算：

```ts
targetLeft = elOriginLeft + moveX
targetTop = elOriginTop + moveY
```

它後面還可能會處理：

```text
對齊線
吸附
多元素拖曳
邊界或輔助線修正
```

但這些是拖曳結果的後續修正。

座標換算的主線仍然是：

```text
畫面位移
-> 除以 canvasScale
-> 邏輯位移
-> 加回元素原始位置
```

---

## 這個案例真正要記住的版本

```text
拖曳元素不是算「滑鼠現在在哪裡」，
而是算「滑鼠從起點到現在移動了多少」。
```

所以它使用的是位移量換算：

```ts
moveX = (currentPageX - startPageX) / canvasScale
moveY = (currentPageY - startPageY) / canvasScale
```

再加回元素原始位置：

```ts
targetLeft = elOriginLeft + moveX
targetTop = elOriginTop + moveY
```

最核心的一句話是：

```text
拖曳元素 = 畫面位移 → 邏輯位移 → 加回元素原始位置。
```

另一句也要記：

```text
雙擊新增文字要扣 viewportRect，因為它在算「點的位置」；
拖曳元素不用扣 viewportRect，因為它在算「兩個滑鼠點之間的距離」。
```

---

# 十一、案例五：縮放元素

## 這個案例在解什麼問題

縮放元素指的是：

```text
使用者拖曳元素外框上的操作點，
讓元素變大或變小。
```

例如拖右下角、左上角、右邊、下邊這些控制點。

這和「拖曳元素」不一樣。

拖曳元素主要改的是：

```ts
left
top
```

縮放元素可能同時改：

```ts
left
top
width
height
```

所以這個案例真正要解的是：

```text
使用者在畫面上拖動操作點的距離，
要如何換成元素在邏輯畫布中的位置與尺寸變化？
```

---

## 先抓住縮放元素的核心

縮放元素不是單純改 `width / height`。

更精準地說，它是在做：

```text
固定某一邊或某一個角，
拖動另一邊或另一個角，
重新算出元素的新 left / top / width / height。
```

例如：

```text
拖右下角：
  左上角大致固定
  右下角跟著滑鼠動
  所以主要改 width / height

拖左上角：
  右下角大致固定
  左上角跟著滑鼠動
  所以 left / top / width / height 都可能改
```

所以縮放元素要同時回答兩個問題：

```text
1. 元素變多寬、多高？
2. 元素左上角有沒有跟著改變？
```

---

## 縮放元素的基本流程

不管是拖哪個操作點，底層流程都可以先理解成：

```text
1. mousedown 時，記住元素原始資料
   originLeft / originTop / originWidth / originHeight

2. mousemove 時，算滑鼠畫面位移
   currentPageX - startPageX
   currentPageY - startPageY

3. 把畫面位移換成邏輯位移
   moveX = 畫面位移 X / canvasScale
   moveY = 畫面位移 Y / canvasScale

4. 根據拖的是哪個操作點，重新計算
   left / top / width / height
```

所以主線是：

```text
操作點畫面位移
-> 除以 canvasScale
-> 邏輯位移
-> 根據操作點更新元素矩形
```

---

## 一、未旋轉元素：畫面方向和元素方向一致

如果元素沒有旋轉，事情比較直覺。

因為元素的水平 / 垂直方向，和畫面的水平 / 垂直方向是一樣的。

所以滑鼠畫面位移可以直接換成邏輯位移：

```ts
const moveX = (currentPageX - startPageX) / canvasScale.value
const moveY = (currentPageY - startPageY) / canvasScale.value
```

接著根據操作點決定怎麼改元素資料。

---

## 拖右下角：主要改 `width / height`

假設拖的是右下角。

可以理解成：

```text
左上角固定，
右下角被滑鼠拉動。
```

所以元素的新尺寸大致是：

```ts
newWidth = originWidth + moveX
newHeight = originHeight + moveY
```

而 `left / top` 通常不需要改：

```ts
newLeft = originLeft
newTop = originTop
```

所以拖右下角的直覺是：

```text
往右拖，width 變大；
往下拖，height 變大。
```

---

## 拖左上角：`left / top / width / height` 都會改

如果拖的是左上角，情況就不一樣。

可以理解成：

```text
右下角固定，
左上角被滑鼠拉動。
```

所以如果你把左上角往右拖，元素左邊界會往右移，寬度會變小。

大致可以理解成：

```ts
newLeft = originLeft + moveX
newTop = originTop + moveY
newWidth = originWidth - moveX
newHeight = originHeight - moveY
```

所以拖左上角時，會同時影響：

```text
left
top
width
height
```

這就是為什麼縮放元素比拖曳元素複雜。

拖曳元素只是在移動整個矩形。

縮放元素是在改變矩形的邊界。

---

## 操作點決定哪些值會改

你可以先用這張表理解：

| 操作點 | 直覺       | 可能影響                             |
| --- | -------- | -------------------------------- |
| 右邊  | 拉右邊界     | `width`                          |
| 左邊  | 拉左邊界     | `left`, `width`                  |
| 下邊  | 拉下邊界     | `height`                         |
| 上邊  | 拉上邊界     | `top`, `height`                  |
| 右下角 | 拉右邊界與下邊界 | `width`, `height`                |
| 左上角 | 拉左邊界與上邊界 | `left`, `top`, `width`, `height` |
| 右上角 | 拉右邊界與上邊界 | `top`, `width`, `height`         |
| 左下角 | 拉左邊界與下邊界 | `left`, `width`, `height`        |

所以縮放元素的重點不是只有：

```text
位移 / canvasScale
```

而是：

```text
位移 / canvasScale 之後，
還要根據操作點決定怎麼改 left / top / width / height。
```

---

## 二、已旋轉元素：畫面方向和元素方向不一致

如果元素已經旋轉，事情會變複雜。

因為畫面上的水平拖曳，不一定等於元素自身的水平變化。

例如元素旋轉了 45 度。

這時你在畫面上往右拖，對元素自己來說，可能同時包含：

```text
元素自身的水平變化
元素自身的垂直變化
```

所以已旋轉元素不能直接做：

```ts
moveX = x / canvasScale.value
moveY = y / canvasScale.value
```

它需要先把畫面位移轉成「元素自身座標軸」上的位移。

---

## 已旋轉元素的流程

已旋轉元素可以這樣理解：

```text
1. 先取得滑鼠畫面位移
   x = currentPageX - startPageX
   y = currentPageY - startPageY

2. 根據元素 rotate，把畫面位移轉到元素自己的方向
   revisedX / revisedY

3. 再除以 canvasScale
   logicX = revisedX / canvasScale
   logicY = revisedY / canvasScale

4. 根據操作點更新 left / top / width / height
```

所以已旋轉元素不是不用除以 `canvasScale`。

而是多了一步：

```text
先修正旋轉方向，
再除以 canvasScale。
```

可以壓成：

```text
畫面拖曳距離
-> 根據 rotate 轉成元素自身方向的距離
-> 除以 canvasScale
-> 得到邏輯尺寸變化
-> 更新元素矩形
```

---

## 為什麼旋轉後不能直接用畫面 x / y

假設元素沒有旋轉。

你往右拖 `40px`，很自然就是：

```text
元素寬度增加某個量
```

但如果元素旋轉 90 度。

畫面上往右拖，對元素自己來說，可能更像是在改變它的高度方向。

所以 source 需要先把滑鼠位移根據元素角度做轉換。

這件事的本質是：

```text
畫面座標軸
和
元素自身座標軸
不再重合。
```

所以要先把畫面位移轉回元素自身的座標軸，才知道這次拖曳到底是在增加寬度，還是增加高度。

---

## 三、多選縮放：先縮放整體外框，再推回每個元素

多選縮放又再多一層。

因為使用者不是只縮放單一元素，而是縮放一組元素。

這時 PPTist 會先把多個元素看成一個整體外框。

可以理解成：

```text
多選元素
-> 形成一個群組外框
-> 使用者拖曳群組外框的操作點
-> 群組外框產生縮放比例
-> 每個子元素根據這個比例重新計算位置與尺寸
```

所以多選縮放不是簡單地把每個元素都加同一個 `moveX / moveY`。

而是要根據整體外框的縮放比例重新分配。

---

## 多選縮放的直覺公式

假設整體外框寬度從：

```ts
groupWidth = 400
```

變成：

```ts
newGroupWidth = 800
```

那水平縮放比例就是：

```ts
scaleX = newGroupWidth / groupWidth
```

也就是：

```ts
scaleX = 2
```

如果某個子元素原本距離群組左邊界 `50`，寬度是 `100`。

縮放後它會變成：

```ts
newRelativeLeft = 50 * 2 = 100
newWidth = 100 * 2 = 200
```

所以多選縮放的核心是：

```text
不是每個元素移動同樣距離，
而是每個元素按照群組外框的縮放比例重新定位與變形。
```

---

## 對回 `useScaleElement.ts`

讀 `useScaleElement.ts` 時，可以先不要一開始鑽進所有細節。

先用這條主線看：

```text
1. 使用者拖曳操作點

2. source 算出滑鼠畫面位移

3. 如果元素有旋轉，先把畫面位移轉成元素自身方向的位移

4. 除以 canvasScale，換成邏輯位移

5. 根據操作點，計算新的 left / top / width / height

6. 如果是多選縮放，先算整體外框縮放比例，再更新每個子元素
```

這樣讀 source 會比直接背 `moveX`、`moveY`、`revisedX`、`revisedY` 清楚很多。

---

## 這個案例真正要記住的版本

```text
縮放元素 = 拖曳操作點，改變元素矩形的邊界。
```

它和拖曳元素的差別是：

```text
拖曳元素：
  整個矩形一起移動，所以主要改 left / top。

縮放元素：
  某些邊界被拉動，所以可能同時改 left / top / width / height。
```

最核心流程是：

```text
操作點畫面位移
-> 必要時先依 rotate 修正成元素自身方向
-> 除以 canvasScale
-> 得到邏輯位移
-> 根據操作點更新 left / top / width / height
```

多選縮放則是：

```text
操作點畫面位移
-> 得到群組外框的新尺寸
-> 算出 scaleX / scaleY
-> 把每個子元素依比例重新定位與縮放
```

最重要的一句話是：

```text
縮放元素不是「把滑鼠位移加到元素上」；
而是「用操作點拖曳改變元素矩形的邊界」。
```

---

# 十二、案例六：旋轉元素

## 這個案例在解什麼問題

旋轉元素時，使用者做的是：

```text id="anpz57"
拖曳元素上方的旋轉控制點
```

但 PPTist 最後要更新的是元素資料裡的：

```ts id="mpdnqm"
rotate
```

所以這個案例真正要解的是：

```text id="ee2u47"
滑鼠目前在元素中心點的哪個方向？
這個方向對應到幾度？
```

也就是說，旋轉不是在算：

```text id="ik6wtg"
滑鼠拖了多少距離
```

而是在算：

```text id="zfmqzb"
滑鼠點和元素中心點形成的角度
```

---

## 旋轉元素的核心：中心點 + 滑鼠點

要算旋轉角度，至少需要兩個點：

```text id="i1plva"
1. 元素中心點
2. 滑鼠目前位置
```

元素中心點來自元素資料：

```ts id="utk5dk"
centerX = element.left + element.width / 2
centerY = element.top + element.height / 2
```

這裡的 `centerX / centerY` 是**邏輯座標**。

因為：

```ts id="5im8y7"
element.left
element.top
element.width
element.height
```

本來就都存在邏輯畫布裡。

但是滑鼠事件拿到的是：

```ts id="w9ef8g"
currentPageX
currentPageY
```

這是畫面座標。

所以問題來了：

```text id="k6brjj"
不能拿畫面座標的滑鼠點，
直接去減邏輯座標的元素中心點。
```

兩者必須先轉到同一套座標系。

---

## 第一步：先把滑鼠點換成邏輯座標

旋轉時，滑鼠位置要先經過點座標換算。

流程是：

```text id="x2hh5x"
滑鼠畫面位置
-> 扣掉 viewport 左上角
-> 除以 canvasScale
-> 得到滑鼠在邏輯畫布中的位置
```

對應公式是：

```ts id="12m93y"
const viewportRect = viewportRef.value.getBoundingClientRect()

const mouseX = (currentPageX - viewportRect.left) / canvasScale.value
const mouseY = (currentPageY - viewportRect.top) / canvasScale.value
```

這時候：

```ts id="3k2oao"
mouseX
mouseY
```

才可以和：

```ts id="91x266"
centerX
centerY
```

放在一起計算。

因為它們現在都在邏輯畫布座標系裡。

---

## 第二步：用滑鼠點和中心點形成向量

當滑鼠點和元素中心點都變成邏輯座標後，就可以算向量。

source 裡可以看到類似：

```ts id="taecli"
const x = mouseX - centerX
const y = centerY - mouseY
```

這個意思是：

```text id="gpudon"
x：
  滑鼠相對於中心點的水平距離

y：
  滑鼠相對於中心點的垂直距離
```

注意這裡的 `y` 是：

```ts id="d5muef"
centerY - mouseY
```

而不是：

```ts id="2w77id"
mouseY - centerY
```

原因是瀏覽器畫面座標通常是：

```text id="rnn2wd"
往下是正方向
```

但很多角度計算習慣用數學座標系：

```text id="z0yjn2"
往上是正方向
```

所以這裡把 y 方向反過來，方便後面用角度函數計算方向。

---

## 第三步：把向量轉成角度

有了：

```ts id="vegr51"
x
y
```

之後，就可以算角度：

```ts id="9x3j7t"
const angle = getAngleFromCoordinate(x, y)
```

這個 `angle` 的直覺意思是：

```text id="lm0tqg"
滑鼠目前在元素中心點的哪個方向
```

再把這個方向轉成元素的 `rotate`。

所以旋轉元素的主線是：

```text id="fcxdhm"
元素資料
-> 算出元素中心點

滑鼠事件
-> 換成邏輯滑鼠點

邏輯滑鼠點 - 元素中心點
-> 得到方向向量

方向向量
-> 算出角度

角度
-> 更新 rotate
```

---

## 為什麼旋轉不是位移量換算

這裡很容易和拖曳元素搞混。

拖曳元素問的是：

```text id="md5qgw"
滑鼠從起點到現在移動了多少？
```

所以它用的是：

```ts id="3o051c"
currentPageX - startPageX
currentPageY - startPageY
```

但旋轉元素問的是：

```text id="i29dwh"
滑鼠現在相對於元素中心點在哪個方向？
```

所以它需要的是：

```text id="twaur4"
滑鼠目前位置
+
元素中心點位置
```

因此旋轉元素不是單純的位移量換算，而是：

```text id="sg2bn0"
點座標換算 + 向量角度計算
```

這也是為什麼旋轉需要 `viewportRef`。

因為它必須先把滑鼠畫面點換成邏輯點，才能和元素中心點算角度。

---

## 對回 `useRotateElement.ts`

`useRotateElement.ts` 可以用這條主線讀：

```text id="mxj4vb"
1. 從元素資料算出中心點
2. 取得 viewportRect
3. mousemove 時取得滑鼠畫面位置
4. 把滑鼠畫面位置換成邏輯畫布位置
5. 用滑鼠邏輯點減元素中心點，得到方向向量
6. 用方向向量算角度
7. 更新元素 rotate
```

核心程式可以壓成：

```ts id="nofj9m"
const centerX = element.left + element.width / 2
const centerY = element.top + element.height / 2

const viewportRect = viewportRef.value.getBoundingClientRect()

const mouseX = (currentPageX - viewportRect.left) / canvasScale.value
const mouseY = (currentPageY - viewportRect.top) / canvasScale.value

const x = mouseX - centerX
const y = centerY - mouseY

const angle = getAngleFromCoordinate(x, y)
```

真正重要的是：

```text id="g9436h"
mouseX / mouseY 和 centerX / centerY 必須在同一套座標系裡。
```

所以滑鼠點必須先除以 `canvasScale`，不能直接拿畫面座標來算。

---

## 用一個簡單例子理解

假設元素中心點在邏輯座標：

```ts id="1w6p1f"
centerX = 500
centerY = 300
```

目前：

```ts id="15q5n3"
canvasScale = 2
viewportRect.left = 100
viewportRect.top = 50
```

滑鼠畫面位置是：

```ts id="taz9hg"
currentPageX = 1100
currentPageY = 650
```

先換成邏輯滑鼠點：

```ts id="v8txfu"
mouseX = (1100 - 100) / 2 = 500
mouseY = (650 - 50) / 2 = 300
```

這代表滑鼠剛好在元素中心點。

如果滑鼠移到畫面位置：

```ts id="q6ezlf"
currentPageX = 1300
currentPageY = 650
```

則：

```ts id="odsh1l"
mouseX = (1300 - 100) / 2 = 600
mouseY = (650 - 50) / 2 = 300
```

向量是：

```ts id="1box6q"
x = 600 - 500 = 100
y = 300 - 300 = 0
```

代表滑鼠在元素中心點的右方。

這時角度就會是「指向右方」的角度。

這就是旋轉計算的本質。

---

## 群組旋轉也是同一個邏輯

`useRotateGroupElement.ts` 和單一元素旋轉的核心一樣。

差別只在中心點來源不同。

單一元素旋轉時：

```text id="mgb53c"
中心點來自單一元素的 left / top / width / height
```

群組旋轉時：

```text id="7k23dw"
中心點來自整個群組外框
```

也就是先算出多個元素共同形成的範圍，再取得群組中心點。

但後面的流程一樣：

```text id="6qp0cc"
滑鼠畫面點
-> 扣 viewportRect
-> 除以 canvasScale
-> 得到邏輯滑鼠點
-> 和群組中心點形成向量
-> 算角度
-> 更新群組內元素旋轉
```

---

## 這個案例真正要記住的版本

```text id="ozpm2z"
旋轉元素不是算滑鼠拖了多少，
而是算滑鼠目前在元素中心點的哪個方向。
```

所以它需要：

```text id="s3frpy"
1. 元素中心點
2. 滑鼠邏輯點
3. 兩者形成的向量
4. 向量對應的角度
```

最核心流程是：

```text id="hoq1b7"
滑鼠畫面點
-> 扣 viewportRect
-> 除以 canvasScale
-> 得到滑鼠邏輯點
-> 減掉元素中心點
-> 得到方向向量
-> 算出 rotate
```

最重要的一句話是：

```text id="lniccr"
旋轉必須先把滑鼠點換成邏輯座標，
因為元素中心點本來就在邏輯座標裡。
```

# 十三、案例七：調整 shape keypoint

## 這個案例在解什麼問題

調整 shape keypoint 指的是：

```text
使用者拖曳 shape 上的控制點，
改變 shape 內部的形狀。
```

它和拖曳元素、縮放元素不一樣。

拖曳元素主要改：

```ts
left
top
```

縮放元素主要改：

```ts
left
top
width
height
```

但調整 shape keypoint 通常不是改元素外框，而是改：

```ts
keypoints
path
```

也就是說，它改的是：

```text
這個 shape 內部應該長什麼樣子。
```

所以這個案例真正要解的是：

```text
使用者在畫面上拖動 shape 控制點的距離，
要如何換成 shape 的 keypoint 變化，
並重新生成新的 path？
```

---

## 先理解 keypoint 是什麼

shape 元素本身會有外框：

```ts
left
top
width
height
```

這些描述的是：

```text
整個 shape 元素放在哪裡、有多大。
```

但 shape 的實際外觀不一定只是普通矩形。

例如：

```text
箭頭
流程圖圖形
對話框
圓角矩形
特殊形狀
```

這些形狀內部可能有一些可調整的位置。

例如：

```text
箭頭的箭頭寬度
對話框的小尖角位置
圓角矩形的圓角程度
某些形狀的凹陷或凸出位置
```

這些可調整的參數，就可以先理解成：

```text
keypoint
```

所以 keypoint 不是在說：

```text
整個元素在哪裡
```

而是在說：

```text
元素外框內部，某個形狀控制點的位置或比例。
```

---

## 為什麼調整 keypoint 不是直接改 DOM

使用者拖的是畫面上的控制點，看起來像是在移動一個小圓點。

但 PPTist 不會只是把那個 DOM 小圓點移動一下。

因為真正要改的是 shape 的資料。

也就是：

```text
keypoint 改了
-> shape 公式重新計算
-> 產生新的 path
-> shape 外觀改變
```

所以流程不是：

```text
拖曳控制點
-> 移動 DOM
```

而是：

```text
拖曳控制點
-> 更新 shape 控制參數
-> 重新生成 shape path
-> 重新渲染 shape
```

---

## 基本流程

調整 shape keypoint 的流程可以拆成：

```text
1. mousedown 時，記住滑鼠起點與 keypoint 原始值

2. mousemove 時，計算滑鼠畫面位移

3. 把畫面位移除以 canvasScale，換成邏輯位移

4. 根據 keypoint 的 relative 類型，把邏輯位移換成新的 keypoint

5. 用新的 keypoints 重新呼叫 shape formula

6. 生成新的 path
```

主線就是：

```text
畫面拖曳距離
-> / canvasScale
-> 邏輯拖曳距離
-> keypoint 變化
-> formula(width, height, keypoints)
-> 新 path
```

---

## 第一步：先把滑鼠位移換成邏輯位移

`useMoveShapeKeypoint.ts` 在 mousemove 時會取得目前滑鼠位置：

```ts
currentPageX
currentPageY
```

然後和起點相減：

```ts
currentPageX - startPageX
currentPageY - startPageY
```

這得到的是畫面位移。

但 keypoint 是要影響 shape 的邏輯資料，所以要除以 `canvasScale`：

```ts
const moveX = (currentPageX - startPageX) / canvasScale.value
const moveY = (currentPageY - startPageY) / canvasScale.value
```

這一步和拖曳元素很像。

差別是：

```text
拖曳元素：
  邏輯位移會加到 element.left / element.top

調整 keypoint：
  邏輯位移會轉成 keypoint 的變化
```

---

## 第二步：根據 `relative` 類型更新 keypoint

這裡是 shape keypoint 比較抽象的地方。

source 會根據 shape formula 裡定義的 `relative` 類型來處理。

例如：

```text
center
left
right
top
bottom
```

你可以先這樣理解：

```text
relative 決定這個 keypoint 是相對於 shape 的哪個基準在移動。
```

例如：

```text
center：
  控制點可能以 shape 中心或某個內部基準為參考。

left / right：
  控制點可能和 shape 的左右方向有關。

top / bottom：
  控制點可能和 shape 的上下方向有關。
```

所以同樣是滑鼠往右拖 `20px`，不一定每種 keypoint 都是單純：

```ts
keypoint += 20
```

而是要看它的公式定義。

更精準地說：

```text
moveX / moveY 只是滑鼠造成的邏輯位移；
真正的 keypoint 怎麼改，要交給 shape formula 的規則決定。
```

---

## 第三步：重新生成 path

keypoint 更新後，PPTist 會重新計算 shape 的 path。

可以理解成：

```ts
path = pathFormula.formula(width, height, keypoints)
```

這代表：

```text
同一個 shape 外框 width / height，
搭配不同 keypoints，
會生成不同的 path。
```

也就是說：

```text
width / height 決定 shape 的外框大小；
keypoints 決定 shape 外框內部的形狀細節；
formula 負責把它們轉成真正的 path。
```

所以調整 keypoint 的結果不是直接改外框，而是：

```text
外框可能不變，
但 shape 內部輪廓改變。
```

---

## 用一個簡單例子理解

假設有一個 shape：

```ts
width = 200
height = 100
```

它有一個 keypoint 控制某個內部位置。

使用者把控制點往右拖了 `40px`。

目前：

```ts
canvasScale = 2
```

那這個拖曳在邏輯世界裡其實是：

```ts
moveX = 40 / 2 = 20
```

所以 source 不會直接把 keypoint 改 `40`，而是會用邏輯位移 `20` 去計算新的 keypoint。

接著重新呼叫：

```ts
pathFormula.formula(width, height, keypoints)
```

產生新的 path。

所以畫面上看到的是：

```text
shape 的內部形狀被調整了。
```

而不是：

```text
整個 shape 被移動了 40px。
```

---

## 和拖曳元素、縮放元素的差異

可以用這張表理解：

| 操作                | 使用者拖的是什麼    | 主要改什麼資料                       | 結果           |
| ----------------- | ----------- | ----------------------------- | ------------ |
| 拖曳元素              | 整個元素        | `left / top`                  | 元素整體移動       |
| 縮放元素              | 外框操作點       | `left / top / width / height` | 元素外框變大或變小    |
| 調整 shape keypoint | shape 內部控制點 | `keypoints / path`            | shape 內部輪廓改變 |

所以 keypoint 調整的重點是：

```text
它不是移動元素，也不是縮放元素外框，
而是改變 shape 的內部幾何參數。
```

---

## 對回 `useMoveShapeKeypoint.ts`

讀 `useMoveShapeKeypoint.ts` 時，可以用這條主線：

```text
1. 使用者拖曳 shape 控制點

2. source 記住滑鼠起點與 keypoint 原始位置

3. mousemove 時取得 currentPageX / currentPageY

4. 計算畫面位移：
   currentPageX - startPageX
   currentPageY - startPageY

5. 除以 canvasScale，變成邏輯位移：
   moveX / moveY

6. 根據 relative 類型，把 moveX / moveY 轉成新的 keypoint

7. 把新的 keypoints 丟回 pathFormula.formula(width, height, keypoints)

8. 產生新的 path，更新 shape 外觀
```

核心可以壓成：

```ts
const moveX = (currentPageX - startPageX) / canvasScale.value
const moveY = (currentPageY - startPageY) / canvasScale.value

const keypoint = getNextKeypoint(originKeypoint, moveX, moveY, relative)

const path = pathFormula.formula(width, height, keypoints)
```

這段的重點不是函式名稱，而是資料流：

```text
畫面位移
-> 邏輯位移
-> keypoint
-> path
```

---

## 這個案例真正要記住的版本

```text
調整 shape keypoint 不是在移動整個元素，
也不是在改元素外框大小。

它是在改 shape 內部的控制參數。
```

完整流程是：

```text
滑鼠拖曳控制點
-> 算出畫面位移
-> 除以 canvasScale，變成邏輯位移
-> 根據 relative 規則更新 keypoint
-> 用 width / height / keypoints 重新生成 path
-> shape 外觀改變
```

最核心的一句話是：

```text
拖曳 shape keypoint = 畫面位移 → 邏輯位移 → 控制參數變化 → 重新生成 path。
```

另一句也要記：

```text
拖曳元素改位置；
縮放元素改外框；
調整 keypoint 改 shape 內部幾何。
```

# 十四、案例八：拖曳線條端點與控制點

## 這個案例在解什麼問題

拖曳線條端點或控制點，指的是使用者拖動：

```text id="acz6da"
線條起點
線條終點
折線控制點
曲線控制點
三次貝茲曲線控制點
```

它和拖曳一般元素不一樣。

一般元素拖曳時，主要是改：

```ts id="lvk39d"
element.left
element.top
```

但線條元素除了有外框位置，還有線條內部的點位資料：

```ts id="jfsdbx"
start
end
broken
broken2
curve
cubic
```

所以這個案例真正要解的是：

```text id="q382xb"
使用者在畫面上拖動線條某個端點或控制點，
要如何換成線條內部資料的變化，
並重新整理整條線的外框？
```

---

## 先理解線條有兩層資料

線條元素最容易混的地方是：它同時有兩層座標。

### 第一層：線條外框

例如：

```ts id="u7z0jq"
element.left
element.top
element.width
element.height
```

這一層描述的是：

```text id="o9py6j"
整條線條元素在邏輯畫布中的外框位置與大小。
```

也就是這條線的 bounding box。

---

### 第二層：線條內部點

例如：

```ts id="lqs3tg"
element.start
element.end
element.broken
element.curve
element.cubic
```

這一層描述的是：

```text id="ypqezd"
線條內部的起點、終點、折線點、曲線控制點。
```

這些點通常不是直接相對於整張投影片，而是相對於線條自己的外框。

你可以先這樣理解：

```text id="534ylb"
線條外框 left / top：
  決定這條線整體放在投影片哪裡。

線條內部 start / end / control points：
  決定這條線在自己的外框裡怎麼畫。
```

---

## 為什麼線條比一般元素麻煩

一般元素拖曳時，可以理解成：

```text id="370u13"
畫面位移
-> / canvasScale
-> 邏輯位移
-> element.left / top 改變
```

但線條端點拖曳時，不只是移動整條線。

例如你拖的是線條起點，終點可能不動。

這時候整條線的外框可能也會改變。

所以線條拖曳更像是：

```text id="fd1wo8"
畫面位移
-> / canvasScale
-> 邏輯位移
-> 更新某個線條點
-> 重新計算整條線的 bounding box
-> 重新計算 start / end / control points 的相對位置
```

這就是它比較難的地方。

---

## `useDragLineElement.ts` 仍然會用 `canvasScale`

`useDragLineElement.ts` 比較特別。

它不是像 `useDragElement()` 那樣從 `Canvas/index.vue` 參數拿到 `canvasScale`。

而是自己在 hook 裡從 `mainStore` 取：

```ts id="c83ps7"
const { canvasScale } = storeToRefs(useMainStore())
```

所以不要誤會成：

```text id="6k0a7p"
useDragLineElement 沒有接收 canvasScale，
所以它沒有做座標換算。
```

正確理解是：

```text id="hdemvb"
它仍然有做座標換算，只是 canvasScale 是在 hook 內部取得。
```

---

## 第一步：先把畫面位移換成邏輯位移

拖曳線條端點時，source 會先記錄滑鼠起點：

```ts id="8s3vb9"
startPageX
startPageY
```

mousemove 時取得目前滑鼠位置：

```ts id="wwjxdf"
currentPageX
currentPageY
```

兩者相減得到畫面位移：

```ts id="9jxe9k"
currentPageX - startPageX
currentPageY - startPageY
```

但線條資料是邏輯資料，所以要除以 `canvasScale`：

```ts id="hh23u3"
const moveX = (currentPageX - startPageX) / canvasScale.value
const moveY = (currentPageY - startPageY) / canvasScale.value
```

這一步和拖曳元素一樣。

差別在於後面更新的資料不同。

```text id="p5pb8p"
拖曳一般元素：
  邏輯位移加到 element.left / top。

拖曳線條端點：
  邏輯位移加到 start / end / broken / curve / cubic 對應的控制點。
```

---

## 第二步：先把線條點轉成「投影片上的絕對邏輯位置」

這裡是這章最重要的地方。

線條內部點通常是相對於線條外框的。

所以要拖曳某個點時，可以先把它轉成投影片上的絕對邏輯位置。

例如線條起點可以理解成：

```ts id="12wlde"
absoluteStartX = element.left + element.start[0]
absoluteStartY = element.top + element.start[1]
```

線條終點可以理解成：

```ts id="3jpzez"
absoluteEndX = element.left + element.end[0]
absoluteEndY = element.top + element.end[1]
```

這樣做的意思是：

```text id="ik0gl1"
先把線條自己的內部座標，
轉成整張邏輯投影片上的座標。
```

因為滑鼠拖曳換算出來的 `moveX / moveY` 也是邏輯畫布中的位移。

所以它們必須在同一層座標裡才能相加。

---

## 第三步：根據 command 決定要移動哪個點

`useDragLineElement.ts` 會根據不同 command 處理不同點。

可以先這樣理解：

```text id="m0bayp"
START：
  拖曳線條起點。

END：
  拖曳線條終點。

C：
  拖曳折線或曲線的中間控制點。

C1：
  拖曳三次貝茲曲線第一個控制點。

C2：
  拖曳三次貝茲曲線第二個控制點。
```

command 的作用是告訴系統：

```text id="bxuz9z"
這次滑鼠拖曳，應該改線條裡的哪一個點。
```

例如拖曳起點時：

```ts id="kcmu5p"
newAbsoluteStartX = absoluteStartX + moveX
newAbsoluteStartY = absoluteStartY + moveY
```

拖曳終點時：

```ts id="09yo2q"
newAbsoluteEndX = absoluteEndX + moveX
newAbsoluteEndY = absoluteEndY + moveY
```

如果拖的是曲線控制點，就是更新 `curve`、`cubic` 對應的控制點。

---

## 第四步：重新計算線條外框

更新某個線條點後，線條的整體範圍可能會變。

例如原本線條從：

```text id="52cwae"
x = 300 到 x = 600
```

如果你把起點拖到：

```text id="s41tfi"
x = 200
```

那整條線的外框左邊界就應該變成 `200`。

所以 source 需要重新找出所有相關點的最小值與最大值：

```ts id="lb9zwa"
minX = Math.min(allPointX)
minY = Math.min(allPointY)
maxX = Math.max(allPointX)
maxY = Math.max(allPointY)
```

然後新的線條外框就是：

```ts id="j0725x"
left = minX
top = minY
width = maxX - minX
height = maxY - minY
```

這一步很重要。

因為線條的外框必須包住目前所有起點、終點和控制點。

---

## 第五步：把所有線條點轉回「相對於新外框」的位置

重新計算外框後，線條內部點也要跟著重新整理。

因為 `start / end / broken / curve / cubic` 不是單純的投影片絕對座標，而是線條外框內部的相對座標。

所以要做：

```ts id="shs45s"
newStart = [
  newAbsoluteStartX - newLeft,
  newAbsoluteStartY - newTop,
]

newEnd = [
  newAbsoluteEndX - newLeft,
  newAbsoluteEndY - newTop,
]
```

控制點也是類似：

```text id="uqu2dl"
newControlPointX = absoluteControlPointX - newLeft
newControlPointY = absoluteControlPointY - newTop
```

這一步可以理解成：

```text id="0lzkbk"
先用投影片絕對座標算清楚整條線的新位置；
再把點位轉回線條自己的內部座標。
```

所以完整流程是：

```text id="a0knpg"
內部相對座標
-> 轉成投影片絕對邏輯座標
-> 套用拖曳位移
-> 重新計算 bounding box
-> 再轉回新外框內的相對座標
```

這是線條拖曳最核心的地方。

---

## 為什麼不能只改 `start / end`

你可能會想：

```text id="unetv7"
拖曳起點時，直接改 start 不就好了？
```

不完全行。

因為如果線條端點被拖到原本外框之外，線條的 `left / top / width / height` 也要跟著改。

例如：

```text id="jy0i74"
線條原本外框 left = 300
起點 startX = 0
終點 endX = 200
```

代表線條絕對範圍是：

```text id="t05e3p"
x = 300 到 x = 500
```

如果你把起點往左拖 `100`，新的絕對起點是：

```text id="ukyrym"
x = 200
```

那線條外框應該變成：

```text id="3t8tnl"
left = 200
width = 300
```

而不是只讓 `startX = -100`。

所以 source 會重新整理線條外框和內部點位，讓線條資料保持一致。

---

## 折線、曲線、貝茲曲線也是同一套

不管拖的是：

```text id="z9o4sc"
起點 START
終點 END
折線控制點 C
三次貝茲曲線控制點 C1 / C2
```

底層都可以用同一個模型理解：

```text id="98ouem"
1. 畫面位移 / canvasScale，得到邏輯位移
2. 找出這次要改的線條點
3. 把該點轉成投影片絕對邏輯座標
4. 套用邏輯位移
5. 重新計算整條線的 bounding box
6. 把所有點轉回相對於新 bounding box 的座標
7. 更新線條資料
```

差別只在於：

```text id="0b5u2k"
這次 command 指向哪一個點。
```

---

## 和一般拖曳元素的差異

可以用這張表理解：

| 操作      | 更新對象                          | 是否重新計算外框 | 重點            |
| ------- | ----------------------------- | -------: | ------------- |
| 拖曳一般元素  | `left / top`                  |     通常不用 | 整個元素一起移動      |
| 拖曳線條起點  | `start` + 外框                  |        要 | 一端動，整條線範圍可能改變 |
| 拖曳線條終點  | `end` + 外框                    |        要 | 一端動，整條線範圍可能改變 |
| 拖曳線條控制點 | `broken / curve / cubic` + 外框 |        要 | 控制點會改變線條形狀與範圍 |

所以拖曳線條不是單純：

```text id="1pss6k"
element.left += moveX
```

而是：

```text id="aq3dnd"
某個線條點改變
-> 線條整體範圍重新計算
-> 內部點位重新轉成相對座標
```

---

## 對回 `useDragLineElement.ts`

讀 `useDragLineElement.ts` 時，可以用這條主線：

```text id="k437st"
1. hook 內部取得 canvasScale

2. 使用者拖曳線條端點或控制點

3. 記錄 startPageX / startPageY

4. mousemove 時取得 currentPageX / currentPageY

5. 計算畫面位移

6. 除以 canvasScale，得到邏輯位移

7. 根據 command 判斷要更新 START / END / C / C1 / C2 哪個點

8. 把線條相關點轉成投影片上的絕對邏輯座標

9. 套用邏輯位移

10. 重新計算 minX / minY / maxX / maxY

11. 更新 line element 的 left / top / width / height

12. 把 start / end / broken / curve / cubic 轉回相對於新外框的座標

13. 寫回線條資料
```

這樣讀 source 會比只記：

```text id="q61uhe"
moveX = 畫面位移 / canvasScale
```

清楚很多。

因為 `/ canvasScale` 只是第一步，後面真正複雜的是線條資料的重新整理。

---

## 這個案例真正要記住的版本

```text id="fnk3sw"
拖曳線條端點或控制點，
不是在移動整個元素，
而是在改線條內部某個點。
```

完整流程是：

```text id="7bnqyh"
畫面位移
-> 除以 canvasScale，得到邏輯位移
-> 找到這次要拖的線條點
-> 將線條點轉成投影片絕對邏輯座標
-> 套用邏輯位移
-> 重新計算線條 bounding box
-> 把所有線條點轉回新外框內的相對座標
-> 更新線條資料
```

最核心的一句話是：

```text id="v5c1k1"
拖曳線條 = 更新某個線條點 + 重新整理整條線的外框與內部相對座標。
```

另一句也要記：

```text id="m18oie"
canvasScale 解決的是「畫面位移換成邏輯位移」；
而線條自己的難點是「絕對邏輯座標和線條內部相對座標之間的轉換」。
```

---

# 十五、例外案例：拖曳整張畫布不用除以 `canvasScale`

## 這個案例在解什麼問題

前面很多案例都在說：

```text
畫面位移
-> 除以 canvasScale
-> 邏輯位移
```

例如：

```text
拖曳元素
縮放元素
調整 shape keypoint
拖曳線條端點
```

它們都要除以 `canvasScale`，是因為最後改的是：

```text
元素在邏輯畫布裡的資料
```

但拖曳整張畫布不是這樣。

拖曳整張畫布時，使用者做的是：

```text
把目前畫面上的投影片整張往左、往右、往上、往下移動
```

也就是常見的 pan viewport。

它不是在改投影片裡面的元素位置。

它改的是：

```ts
viewportLeft
viewportTop
```

而這兩個值代表的是：

```text
投影片外框目前在編輯區畫面中的位置
```

所以這個案例真正要解的是：

```text
滑鼠在畫面上拖了多少，
投影片外框就跟著在畫面上移動多少。
```

這是畫面層操作，不是邏輯資料操作。

---

## 先分清楚：拖元素和拖畫布不是同一件事

拖曳元素時，改的是元素資料：

```ts
element.left
element.top
```

這些值是邏輯座標。

所以滑鼠畫面位移要先除以 `canvasScale`：

```ts
moveX = (currentPageX - startPageX) / canvasScale.value
moveY = (currentPageY - startPageY) / canvasScale.value

element.left = originLeft + moveX
element.top = originTop + moveY
```

但拖曳整張畫布時，改的是：

```ts
viewportLeft
viewportTop
```

這些值是畫面位置。

它們描述的是：

```text
投影片外框離編輯區左上角多少 px。
```

所以滑鼠畫面位移可以直接加上去：

```ts
viewportLeft = originLeft + (currentPageX - startPageX)
viewportTop = originTop + (currentPageY - startPageY)
```

這裡不需要除以 `canvasScale`。

---

## 為什麼拖畫布不用除以 `canvasScale`

因為 `canvasScale` 的用途是：

```text
把縮放後的畫面距離，換回未縮放的邏輯畫布距離。
```

但拖曳整張畫布時，系統不是要回到邏輯畫布。

系統只是要改：

```text
投影片外框在畫面上的位置
```

假設目前：

```ts
canvasScale = 2
```

使用者把整張畫布往右拖了：

```text
40px
```

這個 `40px` 是畫面位移。

如果你真的想讓投影片外框在畫面上跟著滑鼠移動 `40px`，就應該直接寫：

```ts
viewportLeft = originLeft + 40
```

如果你錯誤地除以 `canvasScale`：

```ts
viewportLeft = originLeft + 40 / 2
```

就會變成：

```ts
viewportLeft = originLeft + 20
```

結果使用者滑鼠拖了 `40px`，但畫布只跟著移動 `20px`。

這樣拖曳手感就會錯。

所以拖曳整張畫布不除以 `canvasScale`，是因為：

```text
滑鼠移動多少畫面距離，
投影片外框就應該移動多少畫面距離。
```

---

## 對回 `useViewportSize.ts`

`useViewportSize.ts` 裡的 `dragViewport()` 可以先這樣理解：

```text
1. mousedown 時記住滑鼠起點
2. 同時記住 viewport 原本的顯示位置
3. mousemove 時算滑鼠畫面位移
4. 把畫面位移直接加到 viewportLeft / viewportTop
```

核心程式可以壓成：

```ts
const startPageX = e.pageX
const startPageY = e.pageY

const originLeft = viewportLeft.value
const originTop = viewportTop.value

const moveX = currentPageX - startPageX
const moveY = currentPageY - startPageY

viewportLeft.value = originLeft + moveX
viewportTop.value = originTop + moveY
```

這裡沒有：

```ts
/ canvasScale.value
```

因為 `viewportLeft / viewportTop` 本來就是畫面位置。

---

## 用一個例子對比

假設目前：

```ts
canvasScale = 2
```

使用者畫面上往右拖了：

```text
40px
```

### 情況一：拖曳元素

拖曳元素時，改的是邏輯資料。

所以：

```ts
logicMoveX = 40 / 2 = 20
```

元素資料只加 `20`：

```ts
element.left = element.left + 20
```

畫面顯示時再乘上 `2`：

```ts
20 * 2 = 40
```

使用者看到元素剛好移動 `40px`。

---

### 情況二：拖曳整張畫布

拖曳畫布時，改的是外框顯示位置。

所以不需要回到邏輯座標。

直接：

```ts
viewportLeft = viewportLeft + 40
```

這樣投影片外框就會在畫面上移動 `40px`，剛好跟著滑鼠走。

---

## 為什麼這是「例外」

它是例外，不是因為它特殊到不用座標換算。

而是因為它的輸出目標不同。

前面的互動大多是：

```text
畫面操作
-> 邏輯資料
```

所以要除以 `canvasScale`。

但拖曳整張畫布是：

```text
畫面操作
-> 畫面位置
```

所以不除以 `canvasScale`。

判斷標準可以寫成：

```text
如果最後要改的是元素邏輯資料：
  要除以 canvasScale。

如果最後要改的是投影片外框的畫面位置：
  不要除以 canvasScale。
```

---

## 這個案例真正要記住的版本

```text
拖曳整張畫布，不是在移動投影片裡的元素，
而是在移動投影片外框在編輯區中的顯示位置。
```

所以：

```ts
viewportLeft = originLeft + (currentPageX - startPageX)
viewportTop = originTop + (currentPageY - startPageY)
```

不需要：

```ts
/ canvasScale
```

最核心的一句話是：

```text
拖元素 = 改邏輯資料，要除以 canvasScale。
拖畫布 = 改畫面位置，不除以 canvasScale。
```

另一句也要記：

```text
canvasScale 是用來把畫面距離換回邏輯距離；
但 viewportLeft / viewportTop 本來就是畫面距離。
```

---

# 十六、最重要的判斷準則

前面看了很多案例之後，最後要收斂成一個問題：

```text
看到一段互動 source 時，
我要怎麼判斷它到底要不要除以 canvasScale？
```

不要用下面這種方式硬背：

```text
拖曳元素要除
拖曳畫布不要除
旋轉要除
框選要除
```

這樣很容易記亂。

比較好的判斷方式是先問兩件事：

```text
1. 這個值的來源是哪一層？
   是滑鼠事件拿到的畫面座標？
   還是資料本來就已經是邏輯座標？

2. 這個值最後要去哪一層？
   要寫回元素邏輯資料？
   還是只是在改畫面顯示位置？
```

真正要看的不是「這個操作叫什麼」，而是：

```text
輸入來源是哪一層？
輸出目標是哪一層？
```

---

## 先把三種情況分清楚

### 情況一：畫面操作 → 邏輯資料

這種情況要除以 `canvasScale`。

例如：

```text
拖曳元素
縮放元素
拖曳建立元素
滑鼠框選
拖曳線條端點
調整 shape keypoint
```

因為滑鼠事件拿到的是畫面上的距離或位置，但元素資料存在邏輯畫布裡。

所以必須做：

```text
畫面尺度 / canvasScale = 邏輯尺度
```

例如拖曳元素：

```ts
moveX = (currentPageX - startPageX) / canvasScale.value
moveY = (currentPageY - startPageY) / canvasScale.value
```

因為最後要改的是：

```ts
element.left
element.top
```

這些是邏輯資料。

---

### 情況二：畫面操作 → 畫面位置

這種情況不要除以 `canvasScale`。

代表案例就是：

```text
拖曳整張畫布
```

拖曳整張畫布時，最後改的是：

```ts
viewportLeft
viewportTop
```

這兩個值描述的是：

```text
投影片外框在編輯區畫面中的位置
```

它們本來就是畫面位置。

所以滑鼠在畫面上拖了 `40px`，畫布外框就應該在畫面上移動 `40px`。

不需要除以 `canvasScale`。

```ts
viewportLeft.value = originLeft + (currentPageX - startPageX)
viewportTop.value = originTop + (currentPageY - startPageY)
```

---

### 情況三：邏輯資料 → 邏輯資料

這種情況也不要再除以 `canvasScale`。

例如使用者在屬性面板直接輸入：

```ts
left = 100
top = 50
```

如果這個表單本來就被設計成輸入邏輯座標，那它已經是資料世界的值。

這時如果再做：

```ts
left = 100 / canvasScale.value
```

反而會錯。

所以要記住：

```text
只有當來源是畫面座標或畫面距離時，
才需要考慮除以 canvasScale。

如果來源本來就是邏輯資料，
不要再除一次。
```

---

## 再補一個判斷：是「位置」還是「距離」

判斷要不要除以 `canvasScale` 之外，還要再判斷：

```text
這次算的是位置？
還是距離？
```

因為這會決定要不要扣 `viewportRect`。

---

### 1. 如果算的是「滑鼠點在投影片哪裡」

這是位置問題。

要先扣掉 viewport 左上角，再除以 `canvasScale`。

```ts
logicX = (pageX - viewportRect.x) / canvasScale.value
logicY = (pageY - viewportRect.y) / canvasScale.value
```

例如：

```text
雙擊新增文字
旋轉元素時計算滑鼠點
拖曳建立元素的 left / top
```

因為你要問的是：

```text
這個滑鼠點，相對於投影片左上角在哪裡？
```

所以要扣 `viewportRect`。

---

### 2. 如果算的是「滑鼠拖了多少」

這是距離問題。

不用扣 `viewportRect`，只要除以 `canvasScale`。

```ts
moveX = (currentPageX - startPageX) / canvasScale.value
moveY = (currentPageY - startPageY) / canvasScale.value
```

例如：

```text
拖曳元素
縮放元素
調整 shape keypoint
拖曳線條端點
```

因為：

```ts
currentPageX - startPageX
```

本身已經是兩個滑鼠點之間的畫面距離。

它不需要知道 viewport 在哪裡。

---

## 所以完整判斷流程是這樣

看到一段互動 source 時，可以照這個順序問：

```text
第一步：
這個值是從滑鼠事件來的嗎？

如果不是，而是已經存在資料裡的邏輯值：
  不要除以 canvasScale。

如果是：
  進入第二步。
```

```text
第二步：
這個滑鼠值最後要寫回邏輯資料嗎？

如果要寫回 element.left / top / width / height、keypoints、path、line points：
  要除以 canvasScale。

如果只是改 viewportLeft / viewportTop 這種畫面顯示位置：
  不要除以 canvasScale。
```

```text
第三步：
這次算的是位置還是距離？

如果是位置：
  先扣 viewportRect，再除以 canvasScale。

如果是距離：
  直接除以 canvasScale。
```

可以壓成一句話：

```text
畫面 → 邏輯：要除以 canvasScale。
畫面 → 畫面：不要除以 canvasScale。
邏輯 → 邏輯：不要再除以 canvasScale。

算位置：先扣 viewportRect。
算距離：不用扣 viewportRect。
```

---

## 判斷表

| 操作                       | 來源             | 目標                           | 類型     | 是否扣 `viewportRect` | 是否除以 `canvasScale` |
| ------------------------ | -------------- | ---------------------------- | ------ | -----------------: | -----------------: |
| 雙擊新增文字                   | 滑鼠畫面點          | 新文字 `left / top`             | 位置     |                  要 |                  要 |
| 雙擊新增文字的預設寬度              | 畫面預設寬度 `200px` | 新文字 `width`                  | 距離     |                 不要 |                  要 |
| 拖曳建立元素的 `left / top`     | 畫面選區左上角        | 新元素 `left / top`             | 位置     |                  要 |                  要 |
| 拖曳建立元素的 `width / height` | 畫面選區寬高         | 新元素 `width / height`         | 距離     |                 不要 |                  要 |
| 滑鼠框選起點                   | 滑鼠畫面點          | 邏輯選區起點                       | 位置     |                  要 |                  要 |
| 滑鼠框選寬高                   | 畫面拖曳距離         | 邏輯選區寬高                       | 距離     |                 不要 |                  要 |
| 拖曳元素                     | 畫面拖曳距離         | 元素 `left / top` 變化量          | 距離     |                 不要 |                  要 |
| 縮放元素                     | 操作點畫面位移        | 元素矩形變化                       | 距離     |                 不要 |                  要 |
| 旋轉元素                     | 滑鼠畫面點          | 滑鼠邏輯點，再算角度                   | 位置     |                  要 |                  要 |
| 群組旋轉                     | 滑鼠畫面點          | 滑鼠邏輯點，再算群組角度                 | 位置     |                  要 |                  要 |
| 調整 shape keypoint        | 畫面拖曳距離         | `keypoints / path`           | 距離     |                 不要 |                  要 |
| 拖曳線條端點                   | 畫面拖曳距離         | `start / end` 等線條資料          | 距離     |                 不要 |                  要 |
| 拖曳線條控制點                  | 畫面拖曳距離         | `broken / curve / cubic`     | 距離     |                 不要 |                  要 |
| 拖曳整張畫布                   | 畫面拖曳距離         | `viewportLeft / viewportTop` | 畫面位置變化 |                 不要 |                 不要 |
| 屬性面板直接輸入位置               | 邏輯數值           | 元素 `left / top`              | 已是邏輯資料 |                 不要 |                 不要 |

---

## 特別注意：旋轉不是把角度除以 `canvasScale`

旋轉元素這一列容易誤會。

旋轉時，最後更新的是：

```ts
element.rotate
```

但不是把 `rotate` 除以 `canvasScale`。

真正發生的是：

```text
滑鼠畫面點
-> 扣 viewportRect
-> 除以 canvasScale
-> 得到滑鼠邏輯點
-> 和元素中心點計算角度
-> 得到 rotate
```

所以旋轉需要除以 `canvasScale`，是因為：

```text
滑鼠點要先轉成邏輯座標。
```

不是因為角度本身要縮放。

---

## 這章真正要記住的版本

```text
不要問：
這個操作是不是拖曳？

要問：
這個值從哪一層來？
最後要寫到哪一層去？
```

最重要的判斷準則是：

```text
畫面事件座標 / 畫面拖曳距離
-> 要寫回邏輯資料
-> 除以 canvasScale

畫面事件座標 / 畫面拖曳距離
-> 只改畫面顯示位置
-> 不除以 canvasScale

資料本來就是邏輯座標
-> 不要再除以 canvasScale
```

再補一個位置 / 距離判斷：

```text
算「滑鼠點在哪裡」：
  扣 viewportRect，再除以 canvasScale。

算「滑鼠拖了多少」：
  不扣 viewportRect，只除以 canvasScale。
```

最核心一句話是：

```text
除不除 canvasScale，看你是不是要從畫面尺度回到邏輯尺度；
扣不扣 viewportRect，看你是不是在算相對於投影片左上角的位置。
```

# 十七、用資料流重新看整個互動層

前面已經分別看過新增、框選、拖曳、縮放、旋轉、shape keypoint、線條端點、拖曳畫布。

這一章要做的是把它們收斂成一張總圖。

先抓一句話：

```text
PPTist 的互動層，本質上是在把「畫面事件」轉成「資料更新」。
```

但不是所有資料更新都在同一層。

有些互動會更新邏輯資料：

```text
element.left / top
element.width / height
mouseSelection
keypoints
line start / end / curve
rotate
```

有些互動只更新畫面顯示位置：

```text
viewportLeft / viewportTop
```

所以整個互動層可以分成兩條主線：

```text
畫面操作 -> 邏輯資料
畫面操作 -> 畫面位置
```

前者通常要除以 `canvasScale`。
後者不需要除以 `canvasScale`。

---

## 一、先看共同入口：`Canvas/index.vue`

`Canvas/index.vue` 不是每一個互動公式的實作檔案。

它比較像畫布互動系統的中控台。

它準備三個最重要的共同上下文：

```text
viewportRef
canvasScale
elementList
```

這三個東西分別解決不同問題：

```text
viewportRef：
  告訴互動 hook，投影片內容層目前在畫面哪裡。

canvasScale：
  把畫面距離或畫面位置換回邏輯尺度。

elementList：
  提供目前投影片上的元素資料，讓 hook 可以讀取或更新。
```

所以 `Canvas/index.vue` 的角色可以整理成：

```text
Canvas/index.vue
  -> 建立 viewport / viewport-wrapper DOM 結構
  -> 取得 viewportRef
  -> 取得 canvasScale
  -> 取得 elementList
  -> 把不同互動交給不同 hook
```

也就是：

```text
Canvas/index.vue 負責組裝上下文；
各個 hook 負責具體互動邏輯。
```

---

## 二、第一類資料流：畫面操作 → 邏輯資料

這是大多數互動的共同模式。

流程是：

```text
滑鼠事件
-> 取得畫面座標或畫面位移
-> 換算成邏輯座標或邏輯位移
-> 更新元素或選區資料
```

其中最關鍵的是：

```text
只要最後要進入邏輯資料世界，
就要把畫面尺度換回邏輯尺度。
```

也就是：

```text
畫面尺度 / canvasScale = 邏輯尺度
```

---

## 1. `useMouseSelection`：畫面框選 → 邏輯選區 → 選中元素

`useMouseSelection` 的資料流是：

```text
滑鼠按下與拖曳
-> 取得畫面起點與目前位置
-> 起點扣 viewportRect，再除以 canvasScale
-> 拖曳距離除以 canvasScale
-> 得到邏輯選區 mouseSelection
-> 用 mouseSelection 比對 elementList
-> 得到 activeElementIdList
```

更精準地拆：

```text
起點 left / top：
  是位置
  -> 扣 viewportRect
  -> 除以 canvasScale

寬高 width / height：
  是拖曳距離
  -> 不扣 viewportRect
  -> 除以 canvasScale
```

所以它的本質是：

```text
畫面框選矩形
-> 邏輯選區矩形
-> 命中元素
```

---

## 2. `useDragElement`：畫面拖曳距離 → 元素邏輯位移

`useDragElement` 的資料流是：

```text
滑鼠拖曳元素
-> currentPageX - startPageX
-> currentPageY - startPageY
-> 得到畫面位移
-> 除以 canvasScale
-> 得到邏輯位移
-> 加回元素原始 left / top
-> 更新元素位置
```

可以壓成：

```text
畫面位移
-> / canvasScale
-> 邏輯位移
-> element.left / element.top
```

它不需要 `viewportRect`，因為它算的是距離，不是滑鼠點在投影片裡的位置。

---

## 3. `useScaleElement`：操作點位移 → 元素矩形變化

`useScaleElement` 的資料流是：

```text
拖曳縮放操作點
-> 取得操作點的畫面位移
-> 如果元素有旋轉，先把位移轉成元素自身方向
-> 除以 canvasScale
-> 得到邏輯位移
-> 根據操作點更新 left / top / width / height
```

可以壓成：

```text
操作點畫面位移
-> 必要時依 rotate 修正方向
-> / canvasScale
-> 邏輯尺寸變化
-> element.left / top / width / height
```

它比拖曳元素複雜，是因為縮放不是整個矩形平移，而是在改變矩形邊界。

---

## 4. `useRotateElement / useRotateGroupElement`：滑鼠點 → 邏輯滑鼠點 → 角度

旋轉的資料流是：

```text
滑鼠畫面點
-> 扣 viewportRect
-> 除以 canvasScale
-> 得到滑鼠邏輯點
-> 和元素中心點或群組中心點形成向量
-> 計算角度
-> 更新 rotate
```

可以壓成：

```text
畫面滑鼠點
-> 邏輯滑鼠點
-> 與中心點算角度
-> rotate
```

這裡要注意：

```text
不是把 rotate 除以 canvasScale。
```

而是先把滑鼠點轉成邏輯座標，才能和元素中心點計算角度。

---

## 5. `useMoveShapeKeypoint`：畫面位移 → keypoint → path

`useMoveShapeKeypoint` 的資料流是：

```text
拖曳 shape 控制點
-> 取得畫面位移
-> 除以 canvasScale
-> 得到邏輯位移
-> 根據 relative 規則更新 keypoint
-> 用 width / height / keypoints 重新計算 path
-> 更新 shape 外觀
```

可以壓成：

```text
畫面位移
-> / canvasScale
-> 邏輯位移
-> keypoints
-> path
```

它的重點是：

```text
拖曳 keypoint 不是移動整個元素，
而是改 shape 內部幾何參數。
```

---

## 6. `useDragLineElement`：畫面位移 → 線條點 → 重新整理線條資料

`useDragLineElement` 的資料流是：

```text
拖曳線條端點或控制點
-> 取得畫面位移
-> 除以 canvasScale
-> 得到邏輯位移
-> 找到要更新的線條點
-> 將線條點轉成投影片絕對邏輯座標
-> 套用邏輯位移
-> 重新計算線條 bounding box
-> 把 start / end / broken / curve / cubic 轉回新外框內的相對座標
-> 更新線條資料
```

可以壓成：

```text
畫面位移
-> / canvasScale
-> 邏輯位移
-> 線條點改變
-> 重新整理 bounding box
-> 重新整理內部相對座標
```

這裡要記住：

```text
canvasScale 只解決「畫面位移變邏輯位移」；
線條自己的難點是「絕對邏輯座標」和「線條內部相對座標」之間的轉換。
```

---

## 三、第二類資料流：畫面操作 → 畫面位置

這一類目前最典型的案例是：

```text
useViewportSize.dragViewport
```

它的資料流是：

```text
拖曳整張畫布
-> 取得畫面位移
-> 直接加到 viewportLeft / viewportTop
-> 改變投影片外框在畫面上的位置
```

可以壓成：

```text
畫面位移
-> viewportLeft / viewportTop
-> 不除以 canvasScale
```

為什麼不除？

因為 `viewportLeft / viewportTop` 本來就是畫面位置。

它不是元素邏輯座標。

所以滑鼠在畫面上拖了 `40px`，畫布外框就應該在畫面上移動 `40px`。

---

## 四、用總表看整個互動層

| Hook                           | 輸入       | 中間換算                                  | 輸出                                     |
| ------------------------------ | -------- | ------------------------------------- | -------------------------------------- |
| `useMouseSelection`            | 畫面框選點與距離 | 位置扣 `viewportRect`，距離除以 `canvasScale` | 邏輯選區、選中元素 ID                           |
| `useDragElement`               | 畫面拖曳距離   | `/ canvasScale`                       | 元素 `left / top`                        |
| `useScaleElement`              | 操作點畫面位移  | 旋轉修正後 `/ canvasScale`                 | 元素 `left / top / width / height`       |
| `useRotateElement`             | 滑鼠畫面點    | 扣 `viewportRect`，再 `/ canvasScale`    | 邏輯滑鼠點，進一步算 `rotate`                    |
| `useRotateGroupElement`        | 滑鼠畫面點    | 扣 `viewportRect`，再 `/ canvasScale`    | 邏輯滑鼠點，進一步算群組旋轉                         |
| `useMoveShapeKeypoint`         | 畫面拖曳距離   | `/ canvasScale`                       | `keypoints / path`                     |
| `useDragLineElement`           | 畫面拖曳距離   | `/ canvasScale`，再整理線條座標               | `start / end / broken / curve / cubic` |

## 五、這章真正要記住的版本

整個互動層可以用兩種資料流理解。

第一種：

```text
畫面操作
-> 邏輯資料
```

代表案例：

```text
框選元素
拖曳元素
縮放元素
旋轉元素
調整 shape keypoint
拖曳線條端點
```

這一類通常要做：

```text
畫面位置或距離
-> 換算成邏輯位置或距離
-> 更新元素或選區資料
```

第二種：

```text
畫面操作
-> 畫面位置
```

代表案例：

```text
拖曳整張畫布
```

這一類不需要除以 `canvasScale`。

最核心一句話是：

```text
Canvas/index.vue 負責把 viewportRef、canvasScale、elementList 交給各個 hook；
各個 hook 再把畫面事件轉成對應的資料更新。
```

另一句也要記：

```text
凡是進入邏輯資料世界，就要從畫面尺度換回邏輯尺度；
凡是仍然停在畫面顯示世界，就維持畫面尺度。
```

# 十八、常見誤解整理

這一章不是單純列出錯誤觀念，而是幫你在讀 source 時快速修正判斷。

PPTist 畫布互動最容易混的地方有三類：

```text
1. 把滑鼠座標誤認成畫布座標
2. 把畫面尺度誤認成邏輯尺度
3. 把所有拖曳都誤認成同一種資料更新
```

所以後面每個誤解都用同一個格式整理：

```text
誤解是什麼？
為什麼容易誤會？
正確理解是什麼？
一句話怎麼記？
```

---

## 誤解一：`pageX / pageY` 就是畫布座標

### 錯誤想法

```text
滑鼠事件給了 pageX / pageY，
那這不就是滑鼠在畫布上的座標嗎？
```

不是。

`pageX / pageY` 只是滑鼠在整份 document 裡的位置。

它回答的是：

```text
滑鼠現在在整個頁面哪裡？
```

但 PPTist 元素資料需要的是：

```text
滑鼠在投影片邏輯畫布裡哪裡？
```

這兩件事不同。

---

### 正確理解

要把 `pageX / pageY` 變成邏輯畫布座標，至少要經過兩步：

```text
第一步：
扣掉 viewport 左上角位置

第二步：
除以 canvasScale
```

公式是：

```ts
logicX = (pageX - viewportRect.x) / canvasScale
logicY = (pageY - viewportRect.y) / canvasScale
```

其中：

```text
pageX / pageY：
  滑鼠在頁面上的位置。

viewportRect.x / y：
  投影片左上角在畫面上的位置。

canvasScale：
  畫面尺度回到邏輯尺度的倍率。
```

---

### 一句話記

```text
pageX / pageY 只是滑鼠在頁面上的位置；
扣 viewportRect、除 canvasScale 之後，才是畫布邏輯座標。
```

---

## 誤解二：`viewportRect.x / y` 是元素位置

### 錯誤想法

```text
viewportRect.x / y 看起來也是 x、y，
那它是不是某個元素的位置？
```

不是。

`viewportRect.x / y` 描述的是：

```text
viewport 這個 DOM 元素目前在畫面上的位置。
```

它不是投影片裡某個元素的 `left / top`。

---

### 正確理解

`viewportRect.x / y` 的角色是：

```text
告訴系統：投影片左上角目前在畫面哪裡。
```

它是互動換算的基準點。

例如：

```ts
viewportRect.x = 300
viewportRect.y = 120
```

代表：

```text
投影片左上角目前在畫面座標 (300, 120)。
```

如果滑鼠點是：

```ts
pageX = 800
pageY = 420
```

那麼滑鼠相對於投影片左上角的畫面位置就是：

```ts
relativeX = 800 - 300
relativeY = 420 - 120
```

---

### 一句話記

```text
viewportRect 不是元素位置；
它是「投影片左上角在畫面上的位置」。
```

---

## 誤解三：`viewportRect.x / y` 和 `viewportRect.left / top` 是兩套座標

### 錯誤想法

```text
有些 source 用 viewportRect.x / y，
有些 source 用 viewportRect.left / top，
是不是代表兩套不同座標？
```

通常不是。

在 `getBoundingClientRect()` 回傳的 `DOMRect` 裡，可以先這樣理解：

```text
x    ≈ left
y    ≈ top
```

PPTist source 裡兩種寫法都有出現。

---

### 正確理解

讀 source 時不要被名字卡住。

你要看它們扮演的角色：

```text
viewportRect.x / left：
  viewport 左上角的 X 位置。

viewportRect.y / top：
  viewport 左上角的 Y 位置。
```

所以看到：

```ts
e.pageX - viewportRect.x
```

或：

```ts
currentPageX - viewportRect.left
```

都可以先理解成：

```text
滑鼠 X 位置 - 投影片左上角 X 位置
```

---

### 一句話記

```text
x / y 和 left / top 在這裡都是 viewport 左上角位置；
讀 source 時重點看角色，不要被屬性名稱干擾。
```

---

## 誤解四：`canvasScale` 只是 CSS transform 的值

### 錯誤想法

```text
canvasScale 不就是 transform: scale(...) 用的樣式值嗎？
```

不只如此。

`canvasScale` 同時連接顯示層和互動層。

---

### 正確理解

`canvasScale` 至少有三個角色：

```text
1. 決定 viewport-wrapper 的縮放後外框尺寸

2. 決定 viewport 內容要用幾倍顯示

3. 互動時把畫面座標或畫面距離換回邏輯座標
```

顯示時：

```text
邏輯資料 × canvasScale = 畫面結果
```

互動時：

```text
畫面操作 ÷ canvasScale = 邏輯資料
```

所以 `canvasScale` 不是純 CSS 狀態，而是畫布顯示與互動之間的共同倍率。

---

### 一句話記

```text
canvasScale 顯示時負責放大縮小；
互動時負責把畫面尺度除回邏輯尺度。
```

---

## 誤解五：所有拖曳都要除以 `canvasScale`

### 錯誤想法

```text
前面拖曳元素、縮放元素、拖線條都要除以 canvasScale，
所以是不是所有拖曳都要除？
```

不是。

要不要除以 `canvasScale`，不是看「是不是拖曳」，而是看：

```text
最後要改的是邏輯資料？
還是畫面位置？
```

---

### 正確理解

如果拖曳結果要寫回邏輯資料，就要除。

例如：

```text
拖曳元素：
  改 element.left / top

縮放元素：
  改 element.left / top / width / height

調整 shape keypoint：
  改 keypoints / path

拖曳線條端點：
  改 start / end / curve / cubic
```

這些都進入邏輯資料世界，所以要除以 `canvasScale`。

但拖曳整張畫布時，改的是：

```ts
viewportLeft
viewportTop
```

這是投影片外框在畫面上的位置，本來就是畫面座標，所以不要除。

---

### 一句話記

```text
拖元素、拖線條、拖 keypoint：改資料世界，要除。
拖整張畫布：改顯示世界，不除。
```

---

## 誤解六：框選只是畫面效果

### 錯誤想法

```text
框選不就是畫面上拉一個框嗎？
```

不只如此。

框選的視覺效果在畫面上，但真正用來判斷元素是否被選中的範圍，必須轉成邏輯座標。

---

### 正確理解

框選流程是：

```text
使用者在畫面上拖出框
-> 轉成邏輯選區 mouseSelection
-> 用邏輯選區比對元素邏輯範圍
-> 決定 activeElementIdList
```

元素資料是邏輯座標：

```ts
element.left
element.top
element.width
element.height
```

所以框選範圍也必須是邏輯座標。

否則就會變成：

```text
用畫面座標的框
去比對邏輯座標的元素
```

這樣一定會偏。

---

### 一句話記

```text
框選不是只畫一個框；
它是把畫面框轉成邏輯框，再用邏輯框命中元素。
```

---

## 誤解七：旋轉只是在算角度，不需要座標換算

### 錯誤想法

```text
旋轉只是算角度，應該不用 canvasScale 吧？
```

不是。

旋轉確實最後是在算角度，但算角度之前，需要先把滑鼠點轉成邏輯座標。

---

### 正確理解

旋轉需要兩個點：

```text
1. 元素中心點
2. 滑鼠目前位置
```

元素中心點來自元素資料：

```ts
centerX = element.left + element.width / 2
centerY = element.top + element.height / 2
```

這是邏輯座標。

滑鼠事件拿到的是畫面座標。

所以滑鼠點必須先轉成邏輯座標：

```ts
mouseX = (currentPageX - viewportRect.left) / canvasScale.value
mouseY = (currentPageY - viewportRect.top) / canvasScale.value
```

然後才能和元素中心點相減，算出角度。

---

### 一句話記

```text
旋轉不是把角度除以 canvasScale；
而是先把滑鼠點除回邏輯座標，再和中心點算角度。
```

---

## 誤解八：`width / height` 和 `left / top` 的換算方式一樣

### 錯誤想法

```text
left / top / width / height 都是元素資料，
所以換算方式應該一樣。
```

不完全一樣。

它們都是邏輯資料，但來源類型不同。

---

### 正確理解

`left / top` 通常是位置問題：

```text
這個點在投影片裡哪裡？
```

所以要：

```text
扣 viewportRect
再除以 canvasScale
```

例如：

```ts
left = (minX - viewportRect.x) / canvasScale.value
top = (minY - viewportRect.y) / canvasScale.value
```

`width / height` 通常是距離問題：

```text
這個矩形有多寬、多高？
```

所以不用扣 `viewportRect`，只要：

```text
畫面距離 / canvasScale
```

例如：

```ts
width = (maxX - minX) / canvasScale.value
height = (maxY - minY) / canvasScale.value
```

---

### 一句話記

```text
left / top 是位置，要扣 viewportRect；
width / height 是距離，不用扣 viewportRect。
兩者都要除以 canvasScale，因為都要回到邏輯尺度。
```

---

## 誤解九：只要看到 `/ canvasScale`，就代表在改元素位置

### 錯誤想法

```text
看到 / canvasScale，
應該就是在改 element.left / top。
```

不是。

`/ canvasScale` 只代表：

```text
把畫面尺度換回邏輯尺度。
```

它不一定是在改元素位置。

---

### 正確理解

不同互動中，`/ canvasScale` 可能用來產生不同資料：

```text
雙擊新增文字：
  產生新文字的 left / top / width

滑鼠框選：
  產生邏輯選區 mouseSelection

拖曳元素：
  產生元素 left / top 的變化量

縮放元素：
  產生 width / height 或 left / top 的變化量

旋轉元素：
  產生滑鼠邏輯點，再拿去算角度

調整 shape keypoint：
  產生 keypoint 的邏輯位移

拖曳線條：
  產生線條端點或控制點的邏輯位移
```

所以看到 `/ canvasScale` 時，要問的是：

```text
這個畫面值要被轉成哪一種邏輯資料？
```

---

### 一句話記

```text
/ canvasScale 只代表「畫面尺度 → 邏輯尺度」；
不代表一定是在改 left / top。
```

---

## 誤解十：線條元素和一般矩形元素一樣處理

### 錯誤想法

```text
線條也是元素，所以拖曳線條端點應該和拖曳元素差不多。
```

只對一半。

線條是元素，但它不是單純的矩形元素。

---

### 正確理解

一般元素主要看：

```ts
left
top
width
height
```

線條還有內部點：

```ts
start
end
broken
curve
cubic
```

拖曳線條端點或控制點時，不只是改 `left / top`。

它通常要做：

```text
畫面位移
-> / canvasScale
-> 邏輯位移
-> 更新某個線條點
-> 重新計算 bounding box
-> 把線條點轉回新外框內的相對座標
```

所以線條案例的難點不是只有 `/ canvasScale`。

更大的難點是：

```text
線條點的絕對邏輯座標
和
線條外框內部相對座標
之間的轉換。
```

---

### 一句話記

```text
拖曳線條不是移動整個矩形；
而是更新線條點，重新整理外框與內部相對座標。
```

# 十九、30 秒速查版

這一章不是重新講原理，而是讓你讀 source 時可以快速判斷：

```text
看到 pageX / pageY、viewportRect、canvasScale 時，
我現在到底該怎麼換算？
```

先記三個角色：

```text
pageX / pageY：
  滑鼠在畫面或 document 裡的位置。

viewportRect：
  投影片 viewport 左上角目前在畫面哪裡。

canvasScale：
  畫面尺度和邏輯尺度之間的換算倍率。
```

---

## 1. 先問：這次是在算「位置」還是「距離」？

這是第一個判斷。

### 算位置

意思是：

```text
滑鼠點在投影片裡哪裡？
```

例如：

```text
雙擊新增文字
旋轉元素時計算滑鼠點
拖曳建立元素的 left / top
框選起點
```

公式：

```ts
logicX = (pageX - viewportRect.x) / canvasScale
logicY = (pageY - viewportRect.y) / canvasScale
```

記法：

```text
位置 = 先扣 viewportRect，再除以 canvasScale
```

因為你要先知道：

```text
滑鼠相對於投影片左上角在哪裡
```

---

### 算距離

意思是：

```text
滑鼠從起點到現在拖了多少？
```

例如：

```text
拖曳元素
縮放元素
調整 shape keypoint
拖曳線條端點
拖曳線條控制點
```

公式：

```ts
moveX = (currentPageX - startPageX) / canvasScale
moveY = (currentPageY - startPageY) / canvasScale
```

記法：

```text
距離 = 不扣 viewportRect，只除以 canvasScale
```

因為：

```ts
currentPageX - startPageX
```

本身已經是畫面距離，不需要再知道投影片左上角在哪裡。

---

## 2. 再問：結果要寫到哪一層？

這是第二個判斷。

### 畫面操作 → 邏輯資料

要除以 `canvasScale`。

例如：

```text
元素 left / top
元素 width / height
mouseSelection
keypoints
path
line start / end / curve / cubic
```

記法：

```text
畫面尺度要進入邏輯資料世界，就要除以 canvasScale。
```

---

### 畫面操作 → 畫面位置

不要除以 `canvasScale`。

代表案例：

```text
拖曳整張畫布
```

因為最後改的是：

```ts
viewportLeft
viewportTop
```

它們本來就是畫面位置。

公式：

```ts
viewportLeft = originLeft + (currentPageX - startPageX)
viewportTop = originTop + (currentPageY - startPageY)
```

記法：

```text
拖畫布是改顯示位置，不是改元素資料，所以不除以 canvasScale。
```

---

### 邏輯資料 → 邏輯資料

也不要再除以 `canvasScale`。

例如屬性面板直接輸入：

```ts
left = 100
top = 50
```

如果這個 `100 / 50` 本來就是邏輯座標，就不能再除一次。

記法：

```text
已經是邏輯資料，就不要再除以 canvasScale。
```

---

## 3. 三個最常用公式

### 點座標公式

用在「滑鼠點在投影片哪裡」。

```ts
const logicX = (pageX - viewportRect.x) / canvasScale
const logicY = (pageY - viewportRect.y) / canvasScale
```

一句話：

```text
點座標 = 滑鼠位置 - 投影片左上角，再除以 canvasScale。
```

---

### 位移量公式

用在「滑鼠拖了多少」。

```ts
const moveX = (currentPageX - startPageX) / canvasScale
const moveY = (currentPageY - startPageY) / canvasScale
```

一句話：

```text
位移量 = 滑鼠前後差值，直接除以 canvasScale。
```

---

### 矩形公式

用在「畫面矩形變成邏輯矩形」。

```ts
const left = (minX - viewportRect.x) / canvasScale
const top = (minY - viewportRect.y) / canvasScale
const width = (maxX - minX) / canvasScale
const height = (maxY - minY) / canvasScale
```

一句話：

```text
left / top 是位置，要扣 viewportRect；
width / height 是距離，不用扣 viewportRect；
四個值都要回到邏輯尺度，所以都要除以 canvasScale。
```

---

## 4. 各案例一眼判斷

| 操作             | 本質                   | 換算方式                                                            |
| -------------- | -------------------- | --------------------------------------------------------------- |
| 雙擊新增文字         | 畫面點 → 文字邏輯位置         | 扣 `viewportRect`，再除以 `canvasScale`                              |
| 拖曳建立元素         | 畫面矩形 → 元素邏輯矩形        | `left/top` 扣 `viewportRect`；`width/height` 不扣；都除以 `canvasScale` |
| 滑鼠框選           | 畫面矩形 → 邏輯選區          | 同矩形公式                                                           |
| 拖曳元素           | 畫面位移 → 元素邏輯位移        | 不扣 `viewportRect`，除以 `canvasScale`                              |
| 縮放元素           | 操作點位移 → 元素矩形變化       | 必要時先修正旋轉方向，再除以 `canvasScale`                                    |
| 旋轉元素           | 畫面滑鼠點 → 邏輯滑鼠點 → 角度   | 先扣 `viewportRect`，再除以 `canvasScale`，再算角度                        |
| shape keypoint | 畫面位移 → keypoint/path | 不扣 `viewportRect`，除以 `canvasScale`                              |
| 線條端點 / 控制點     | 畫面位移 → 線條內部點         | 不扣 `viewportRect`，除以 `canvasScale`，再整理線條座標                      |
| 拖曳整張畫布         | 畫面位移 → 畫面位置          | 不扣 `viewportRect`，不除以 `canvasScale`                             |

---

## 5. 最容易混的三句話

```text
算滑鼠「在哪裡」：
  扣 viewportRect，再除以 canvasScale。

算滑鼠「拖了多少」：
  不扣 viewportRect，只除以 canvasScale。

拖曳整張畫布：
  改的是 viewportLeft / viewportTop，
  它們本來就是畫面位置，所以不除以 canvasScale。
```

---

## 6. 最終記憶版

```text
viewportRect 解決：
  投影片左上角在畫面哪裡。

canvasScale 解決：
  畫面尺度怎麼回到邏輯尺度。

扣不扣 viewportRect：
  看你是不是在算「位置」。

除不除 canvasScale：
  看你是不是要從「畫面尺度」回到「邏輯尺度」。
```

最核心一句話：

```text
位置要先扣 viewportRect；
畫面尺度進邏輯資料要除以 canvasScale；
拖整張畫布改的是畫面位置，所以不除。
```

這樣的速查版會比原本清楚，因為它把判斷順序拆成：

```text
1. 先判斷位置 / 距離
2. 再判斷輸出是邏輯資料 / 畫面位置
3. 最後套公式
```

# 二十一、這篇最後要帶走的結論

這篇最終不是要你背一堆公式，而是要建立一個心智模型：

```text
PPTist 的互動層，本質上是在處理「畫面世界」和「邏輯畫布世界」之間的轉換。
```

使用者操作時，滑鼠事件拿到的是：

```text
畫面世界的位置或距離
```

但 PPTist 真正要維護的是：

```text
邏輯畫布裡的元素資料
```

例如：

```ts
element.left
element.top
element.width
element.height
element.rotate
keypoints
line.start
line.end
```

這些都不是單純的畫面座標，而是建立在固定邏輯畫布上的資料。

所以整篇的主線可以收斂成一句話：

```text
把使用者在畫面上的操作，
轉成 PPTist 邏輯畫布裡可以使用的資料。
```

---

## 一、顯示方向和互動方向剛好相反

PPTist 顯示畫布時，是從資料世界走向畫面世界：

```text
邏輯資料
-> 乘上 canvasScale
-> 畫面顯示結果
```

也就是：

```text
邏輯座標 × canvasScale = 畫面座標
邏輯距離 × canvasScale = 畫面距離
```

但使用者互動時，方向剛好反過來：

```text
畫面操作
-> 除以 canvasScale
-> 邏輯資料
```

也就是：

```text
畫面座標 ÷ canvasScale = 邏輯座標
畫面距離 ÷ canvasScale = 邏輯距離
```

所以最基本的直覺是：

```text
顯示時乘出去；
互動時除回來。
```

---

## 二、`viewportRect` 和 `canvasScale` 各自解決不同問題

整篇裡最重要的兩個角色是：

```text
viewportRect
canvasScale
```

但它們不是同一種東西。

---

### `viewportRect` 解決的是：投影片左上角在哪裡？

滑鼠事件給你的 `pageX / pageY` 是整個頁面裡的位置。

但 PPTist 元素資料是相對於：

```text
投影片左上角
```

所以如果要算滑鼠點在投影片裡的哪裡，就必須先扣掉：

```text
viewport 左上角的位置
```

這就是 `viewportRef.getBoundingClientRect()` 的作用。

```text
viewportRect 解決「滑鼠相對於投影片左上角在哪裡」。
```

---

### `canvasScale` 解決的是：畫面尺度怎麼回到邏輯尺度？

扣掉 `viewportRect` 之後，你得到的仍然是畫面上的距離。

但資料要存在邏輯畫布裡。

所以還要除以 `canvasScale`。

```text
canvasScale 解決「畫面距離怎麼縮回邏輯距離」。
```

---

## 三、看到互動 source 時，先問兩個問題

不要先問：

```text
這個操作是不是拖曳？
這個 hook 有沒有 canvasScale？
```

比較好的判斷方式是問：

```text
1. 輸入來源是哪一層？
2. 輸出目標是哪一層？
```

---

### 情況一：畫面操作 → 邏輯資料

這種情況要除以 `canvasScale`。

例如：

```text
雙擊新增文字
拖曳建立元素
滑鼠框選
拖曳元素
縮放元素
旋轉元素
調整 shape keypoint
拖曳線條端點或控制點
```

因為它們的共同點是：

```text
滑鼠事件來自畫面世界，
但最後要更新邏輯資料。
```

所以要做：

```text
畫面尺度 / canvasScale = 邏輯尺度
```

---

### 情況二：畫面操作 → 畫面位置

這種情況不要除以 `canvasScale`。

代表案例是：

```text
拖曳整張畫布
```

因為拖曳整張畫布改的是：

```ts
viewportLeft
viewportTop
```

這兩個值描述的是：

```text
投影片外框在編輯區畫面中的位置
```

它們本來就是畫面位置，所以不需要回到邏輯畫布。

---

### 情況三：邏輯資料 → 邏輯資料

這種情況也不要再除以 `canvasScale`。

例如屬性面板直接輸入：

```ts
left = 100
top = 50
```

如果這個 `100 / 50` 本來就代表邏輯座標，那它已經在資料世界裡了，不需要再除一次。

---

## 四、再問一次：這次是在算位置，還是在算距離？

除了判斷要不要除以 `canvasScale`，還要判斷要不要扣 `viewportRect`。

---

### 算位置：要扣 `viewportRect`

如果你問的是：

```text
滑鼠點在投影片裡哪裡？
```

那就要：

```ts
logicX = (pageX - viewportRect.x) / canvasScale
logicY = (pageY - viewportRect.y) / canvasScale
```

例如：

```text
雙擊新增文字
旋轉元素時計算滑鼠點
拖曳建立元素的 left / top
框選起點
```

因為這些都在問：

```text
某個點相對於投影片左上角在哪裡？
```

---

### 算距離：不用扣 `viewportRect`

如果你問的是：

```text
滑鼠從起點到現在拖了多少？
```

那就只要：

```ts
moveX = (currentPageX - startPageX) / canvasScale
moveY = (currentPageY - startPageY) / canvasScale
```

例如：

```text
拖曳元素
縮放元素
調整 shape keypoint
拖曳線條端點
```

因為兩個滑鼠點相減後，已經是距離，不需要再知道投影片左上角在哪裡。

---

## 五、整篇所有案例其實都能歸到同一套模型

可以這樣整理：

```text
雙擊新增文字：
  畫面點 -> 邏輯文字位置

拖曳建立元素：
  畫面矩形 -> 邏輯元素矩形

滑鼠框選：
  畫面矩形 -> 邏輯選區 -> 命中元素

拖曳元素：
  畫面位移 -> 邏輯位移 -> element.left / top

縮放元素：
  操作點畫面位移 -> 邏輯位移 -> 元素矩形變化

旋轉元素：
  畫面滑鼠點 -> 邏輯滑鼠點 -> 與中心點算角度

調整 shape keypoint：
  畫面位移 -> 邏輯位移 -> keypoints -> path

拖曳線條：
  畫面位移 -> 邏輯位移 -> 線條點 -> 重整 bounding box

拖曳整張畫布：
  畫面位移 -> viewportLeft / viewportTop
```

除了拖曳整張畫布以外，大多數互動都會進入邏輯資料世界，所以都需要把畫面尺度換回邏輯尺度。

---

## 六、最終判斷規則

最後可以把整篇濃縮成這幾句：

```text
1. 來源是畫面座標或畫面距離，目標是邏輯資料：
   要除以 canvasScale。

2. 來源是畫面座標或畫面距離，目標仍然是畫面位置：
   不要除以 canvasScale。

3. 來源本來就是邏輯資料：
   不要再除以 canvasScale。

4. 算滑鼠點在投影片哪裡：
   要扣 viewportRect。

5. 算滑鼠拖了多少距離：
   不用扣 viewportRect。
```

## 七、這篇真正要帶走的一句話

如果只能記一句話，就是：

```text
viewportRect 解決「投影片左上角在哪裡」；
canvasScale 解決「畫面尺度怎麼回到邏輯尺度」；
輸出目標決定「到底要不要除以 canvasScale」。
```

再壓得更短一點：

```text
算位置，看 viewportRect；
回邏輯，看 canvasScale；
拖畫布，不回邏輯，所以不除。
```

這樣收尾會比原本更清楚，因為它不是只說「反向映射」，而是把整篇真正要帶走的三個判斷全部收斂起來：

```text
1. 我現在在算位置還是距離？
2. 我現在要回到邏輯資料還是留在畫面位置？
3. viewportRect 和 canvasScale 在這裡各自負責什麼？
```
