# 為什麼要這樣設計 `.frame` / `.viewport`

核心原因其實不是「瀏覽器流程很玄」，而是這樣設計可以同時滿足兩個目標，而這兩個目標本來就互相衝突：

1. 元素資料想永遠用同一套固定座標系
2. 畫面又必須能隨容器大小縮放顯示

如果只看結果，這套設計其實是在做一句話：

`用固定尺寸的內部世界寫資料，用可變尺寸的外部盒子負責顯示。`

拿你的例子說：

- `.viewport = 1000 x 562.5`
  這是「資料世界」
- `.frame = 600 x 337.5`
  這是「顯示世界」

為什麼要分成兩個世界？因為如果不分，麻煩會很多。

## 如果沒有這種設計，會怎樣

只有兩種常見做法：

### 1. 直接讓 `.viewport` 本身就是 `600 x 337.5`

這樣元素資料就得跟著變。原本在邏輯世界裡是：

```ts
left: 100
top: 50
width: 200
height: 80
```

到了 600 寬畫面，可能就得變成：

```ts
left: 60
top: 30
width: 120
height: 48
```

問題是：

- 每次縮放都要重算所有元素
- 滑鼠拖曳、框選、旋轉都要一直在不同尺寸間換算
- 縮圖、主畫布、播放頁會很難共用同一份資料

### 2. 只對 `.viewport` 做 `transform: scale(0.6)`，但沒有 `.frame`

這樣雖然畫面看起來縮小了，但外部排版還是容易把它當成 `1000 x 562.5` 那張大投影片。

問題是：

- 外界不知道它其實只想顯示成 `600 x 337.5`
- 容器佔位、裁切、置中、對齊會變得不直觀
- 你會一直遇到「看起來是小的，為什麼排版上還像大的」這種混亂

## 所以 `.frame + .viewport` 的設計，本質上是在拆責任

- `.viewport` 負責：資料與座標系穩定
- `.frame` 負責：畫面上實際佔多大

## 這樣的好處

你可以把它理解成「先保留原稿，再決定展示尺寸」。

- `.viewport` 像原稿，永遠是 `1000 x 562.5`
- `.frame` 像展示框，今天可以是 `600 x 337.5`，明天也可以是 `800 x 450`
- `transform: scale(...)` 只是把原稿縮放到展示框裡

所以這樣設計不是為了配合瀏覽器流程本身，而是為了工程上的穩定性：

- 資料模型固定
- 元素不用各自處理縮放
- 互動座標換算有一致規則
- 縮圖、主畫布、播放頁可以共用同一套邏輯

## 三種做法比較表

| 做法 | `.viewport` 尺寸 | `transform: scale()` | 元素資料座標 | 優點 | 缺點 | 適合情境 |
|---|---|---:|---|---|---|---|
| 1. 固定邏輯畫布 + 外層承接顯示尺寸 | 固定 `1000 x 562.5` | 有，例如 `scale(0.6)` | 固定邏輯座標，例如 `left: 100` | 資料模型穩定；元素不用各自縮放；滑鼠換算規則一致；縮圖、主畫布、播放頁可共用同一套資料 | 需要理解 `.frame / .viewport` 兩層分工 | 畫布編輯器、投影片系統、白板、縮圖與主畫布共用資料 |
| 2. 直接把 `.viewport` 改成顯示尺寸，不用 `scale()` | 直接是 `600 x 337.5` | 無 | 也要跟著變成顯示尺寸座標，例如 `left: 60` | 結構直觀；不用理解 transform 對 layout 的影響 | 每次縮放都要重算元素；互動換算容易亂；不同畫面尺寸難共用資料 | 很簡單、固定尺寸的小型展示頁 |
| 3. 直接把 `.viewport` 改成顯示尺寸，同時又 `scale(0.6)` | `600 x 337.5` | 有，例如 `scale(0.6)` | 不一致，容易混亂 | 幾乎沒有實際好處 | 會 double scale；最後變成 `360 x 202.5`；邏輯與顯示都混在一起 | 不建議 |

一句話判斷：

- 做投影片編輯器這類系統，通常應選 `1`
- 只做單純固定版面展示，`2` 有時可以接受
- `3` 基本上是錯的
